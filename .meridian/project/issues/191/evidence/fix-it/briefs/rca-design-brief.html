Issue Overview

prepare-implementation / implement-epic path disconnect
Recipe pipeline handoff failure — wrong artifact base paths
High Severity

implement-epic hardcodes artifact paths to the product epics directory (.meridian/product/roadmap/epics/{epic_id}/) while prepare-implementation writes all design artifacts to the issue STM context directory ({stm_base}/{issue}/context/design/). These are entirely separate directory trees with no bridging step, so implement-epic always starts from the wrong base and can never find the artifacts that prepare-implementation produced.

🔍

Root Cause Analysis

Primary Cause
core/components/recipes/implement-epic/SKILL.md lines 94–107
The pre-flight block hardcodes artifact_base = "{product_base}/roadmap/epics/{epic_id}" and derives all four artifact paths (plan_yaml_path, features_yaml_path, scenarios_yaml_path, tech_yaml_path) from this base. This makes implement-epic always look in the product epics directory tree for its input artifacts.
Why this happened
prepare-implementation was redesigned (TDD-first, issue #183) to write artifacts into {stm_base}/{issue}/context/design/ — the issue STM — but implement-epic's path resolution was never updated to match. The two recipes evolved independently and now point to different base directories.
Secondary Cause
core/components/recipes/prepare-implementation/SKILL.md lines 42–65 (pre-flight)
prepare-implementation parses issue from branch name and resolves stm_base but never creates a GitHub issue, feature branch, or STM directory. It assumes start-feature already ran. There is no issue/branch/STM setup in prepare-implementation's own workflow.
Why this happened
The recipe was designed as a pure artifact-production step (planning phase) that assumed upstream setup was complete. This made it dependent on start-feature or start-feature-planning being invoked separately — but the issue expected it to be self-contained, covering the full "ultra god mode" of planning context setup.
Tertiary Cause
core/components/recipes/prepare-implementation/SKILL.md lines 1207–1217 (Step 20 — Lock All Artifacts)
The lock step iterates artifacts at path ${stm_base}/${issue}/${artifact} (flat), but the actual DRAFT artifacts are written to {stm_base}/{issue}/context/design/${artifact}. This means the lock step silently finds nothing (files absent at flat path) and exits without locking anything or raising an error.
Why this happened
The path template in the lock step was not updated when the prepare-implementation TDD redesign (issue #183) introduced the context/design/ subdirectory structure. This compounds the primary disconnect — even if implement-epic's paths were corrected, the artifacts may never actually reach LOCKED status due to this secondary bug.

Fix Strategy

Chosen Approach

Two coordinated intent.yaml changes trigger two rebakes. (1) prepare-implementation/reference/intent.yaml gains new constraints covering issue resolution, branch creation, and STM initialization (Phase 0 — Setup), plus a corrected artifact lock path constraint. A rebake produces an updated SKILL.md. (2) implement-epic/reference/intent.yaml has its artifact path constraints updated to reflect {stm_base}/{issue}/context/design/ as the canonical source. A rebake produces an updated SKILL.md. The result: prepare-implementation fully sets up the working context (issue, branch, STM, design artifacts) and implement-epic reads all of it from the issue STM — a single source of truth.

1
Update prepare-implementation/reference/intent.yaml — add setup phase constraints and fix artifact lock path
modify core/components/recipes/prepare-implementation/reference/intent.yaml
Add constraints mirroring start-feature: C_SETUP_1 (resolve/create GitHub issue), C_SETUP_2 (create feature branch), C_SETUP_3 (initialize STM directory), C_SETUP_4 (lock step uses context/design/ path). Add failure conditions F_SETUP_1 through F_SETUP_3. Add scenario S_SETUP: developer invokes prepare-implementation with no prior start-feature run — recipe creates issue, branch, STM, and completes all planning producing fully locked artifacts in {stm_base}/{issue}/context/design/.
Expected Outcome
intent.yaml reflects the full lifecycle of prepare-implementation including setup phase. Constraint schema is complete with no ambiguity about artifact locations.

Alternatives Considered

Alt 1 — Fix only implement-epic: update its path resolution to read from {stm_base}/{issue}/context/design/ where prepare-implementation already writes. Leave prepare-implementation unchanged.
✕
Solves the path mismatch but leaves the missing setup problem intact. Users still must run start-feature separately before prepare-implementation. The lock step bug in prepare-implementation also remains, meaning artifacts may never reach LOCKED status regardless of the path fix. Two bugs remain for one fix applied.
Alt 2 — Make prepare-implementation copy or symlink its output artifacts into the product epics directory tree where implement-epic currently expects them. Preserve implement-epic unchanged.
✕
Creates artifact duplication across two directory trees with no single source of truth. Symlinks are fragile across checkouts, CI, and worktrees. The product epics path is a product-planning concern, not an implementation concern — filling it with issue-specific design artifacts (blast radius, baseline tests) pollutes the product directory semantics. This is a workaround that avoids fixing the actual path incoherence.
Alt 3 — Add a new bridging recipe/step between prepare-implementation and implement-epic that reads from context/design/ and writes to product epics path (or vice versa).
✕
Overengineered. The problem is a path mismatch in two existing recipes — adding a third recipe to bridge them introduces an extra invocation step and a new failure surface without removing the underlying incoherence. Users would need to know to run the bridge recipe or it would be silently skipped.
Alt 4 — Change prepare-implementation's output to write artifacts directly to the product epics path ({product_base}/roadmap/epics/{epic_id}/) to match what implement-epic expects, leaving implement-epic unchanged.
✕
Regresses the TDD redesign (#183). The issue STM (context/design/) was specifically chosen to co-locate issue-level working context with implementation state. Writing design artifacts to the product epics tree severs that co-location, removes blast radius and baseline test artifacts from issue context, and makes the STM incomplete.

Risk Assessment

# Risk Mitigation Severity
1 Rebaking implement-epic changes behavior for the entire epic workflow, not just the path resolution. A poorly authored rebake could drop constraints or alter the TDD loop structure. Diff the rebaked SKILL.md against the original before accepting. Verify that only the pre-flight path block and Step 1 contract changed — all other steps (eval-generator, test-writer, code-builder, judge, quality-auditor) must be structurally identical. The intent.yaml change is scoped to path constraints only. High
2 Existing runs of prepare-implementation that wrote artifacts to context/design/ may have plan.yaml in DRAFT state (due to the lock step bug). implement-epic will hard-halt on C1 (status != LOCKED) even after the path fix. The fix must include correcting the lock step path in prepare-implementation. Any existing DRAFT artifacts at context/design/ must be manually locked (set status: LOCKED) or prepare-implementation must be re-run for those issues after the fix is deployed. High
3 Adding Phase 0 setup to prepare-implementation may conflict with users who already ran start-feature before invoking prepare-implementation. Double branch creation or duplicate issue creation could corrupt the working context. The setup phase must be idempotent. Branch creation should use the same existence check as repo-orchestrator in start-feature (switch if exists, create if absent). Issue resolution must check for an existing open issue before creating a new one. STM directory initialization is always safe with mkdir -p. Medium
4 architecture.yaml and quality-standards.yaml paths in implement-epic are product-level and should NOT be changed to the issue STM path. If the rebake incorrectly moves these paths, the quality gate will break for all features. The intent.yaml change must explicitly distinguish the four design artifacts (issue-STM-sourced) from the two architecture artifacts (product-base-sourced). Review the rebaked SKILL.md to confirm architecture_yaml_path and quality_standards_path still reference {product_base}/architecture/. Medium
5 The fix changes where implement-epic reads artifacts, so any existing documentation, ADRs, or operator runbooks referencing the old product epics path for implementation artifacts will become stale. Search docs/ and core/components/memory/ for references to {product_base}/roadmap/epics/ in the context of implement-epic inputs and update any found references as part of the same PR. Low
