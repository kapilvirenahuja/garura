Issue #195 RCA + Design Checkpoint
fix: capture-learning extracts zero knowledge — only scans resolution-trace.yaml
github.com/kapilvirenahuja/meridian-os/issues/195  ·  Generated: 2026-04-12
Observed Symptom

Running capture-learning against 8 closed Phoenix issues produces zero knowledge candidates every time. The extraction phase exits cleanly with "zero candidates" despite those issues containing rich evidence (judge reports, remediation files, quality reports, RCA artifacts, implementation plans).

1

Root Cause

File
core/components/recipes/capture-learning/SKILL.md
Location
Step 3 — Locate Resolution Traces (lines 136–139)
Logic
find {stm_archive}/{YYYY-MM}/{issue}/evidence/ -name "resolution-trace.yaml" -type f
Why Wrong
The find pattern is an over-specific literal filename match. The intent (C12) says resolution traces are the "primary" promotion signal — not the only signal. The recipe implemented "primary" as "exclusive", ignoring all other evidence artifacts (judge reports, RCA files, design artifacts, remediation records) that carry substantial learnable knowledge. The scan produces an empty list for any issue that did not run prepare-implementation, which is the majority of real issues.
Intent Misalignment — C12 in capture-learning/reference/intent.yaml
"Resolution traces with resolved_from='llm' are the primary promotion signal — decisions where the system lacked stored knowledge."
The word "primary" implies other signals may exist, but neither the recipe nor the agent implements any secondary extraction path. The phrase "reusable patterns from resolution traces" was interpreted as "only resolution-trace.yaml files" rather than "traces of resolution activity" — evidence showing how decisions were made.
2

Contributing Cause

File
core/components/agents/knowledge-extractor.md
Location
EXTRACT Mode, Step 2 (line 61)
Logic
Extract all entries where resolved_from == "llm"
Why Wrong
The extraction signal is hardcoded to a single YAML field that only exists in resolution-trace.yaml files. Even if Step 3 were broadened to collect other evidence files, the agent would still produce zero candidates — judge reports, remediation files, and RCA artifacts do not contain the resolved_from field. Broadening the file scan without updating the extraction logic would leave the core gap intact.

Proposed Fix Strategy

Broaden scan + extend extraction to multiple evidence types
Fix at both layers simultaneously: (1) update intent.yaml C12 to name additional evidence types as extraction sources, (2) rebake recipe so Step 3 scans for all recognized evidence artifact types rather than only resolution-trace.yaml, (3) extend knowledge-extractor EXTRACT mode to understand diverse artifact formats and extract knowledge signals from each. The synthesis and deduplication logic is preserved unchanged — only the input acquisition and signal extraction change.
Why chosen: Addresses both the primary bottleneck (recipe scan) and the contributing cause (extraction logic) in one coherent change. Starting at intent.yaml ensures the fix goes through the correct compilation pipeline (rebake) rather than violating the "do not edit SKILL.md manually" constraint. The synthesis and LTM write pipeline is sound and reused unchanged.
6

Alternatives Considered

Alt 1: Broaden Step 3 scan only, keep resolved_from: llm signal
Rejected
Update SKILL.md Step 3 to scan for more file types, but leave the knowledge-extractor extraction logic unchanged (still only extracts resolved_from: 'llm' entries).
Rejection Reason
Does not solve the problem. The other evidence types (judge reports, remediation files, RCA artifacts) do not contain resolved_from: 'llm' fields. Broadening the scan would pass more files to knowledge-extractor, but the extractor would still find zero qualifying entries and still produce zero candidates. Fixes the symptom of file discovery while leaving the actual extraction gap intact.
Alt 2: Enforce resolution-trace.yaml production from all recipes
Rejected
Require every recipe (start-feature, implement-epic, fix-it, commit-code) to produce a resolution-trace.yaml with resolved_from entries. Adapter logic transforms existing evidence into resolution-trace format before capture-learning runs.
Rejection Reason
High blast radius requiring changes to 5+ independent recipes, each with their own intent.yaml and rebake requirements. Creates artificial data transformation between recipes. Produces a synthetic artifact that doesn't naturally emerge from the work, reducing its quality as a knowledge signal. The chosen approach is lower risk and more natural — it teaches capture-learning to understand evidence as it actually exists, rather than forcing all evidence into one format.
Alt 3: Post-process archived issues via LLM summarization only
Rejected
Replace structured extraction entirely with a single LLM prompt reading all evidence files and free-form extracting knowledge candidates without per-format heuristics.
Rejection Reason
Less deterministic and less auditable. Structured extraction per artifact type produces traceable provenance (which file, which section, which signal) that feeds into deduplication and classification. Free-form LLM summarization loses the source_trace_path traceability and makes SE-7, SE-10, SE-11 evals harder to verify. Also conflicts with C9 (holistic organization) which requires deliberate synthesis, not arbitrary extraction.

Risk Assessment

Risk Severity Mitigation
SKILL.md is a compiled artifact — direct edits violate the stated constraint. If the fix edits SKILL.md directly instead of going through rebake, the intent_hash in compilation metadata will become stale, creating drift between intent.yaml and the compiled recipe. High Fix MUST follow the intent.yaml → /create-recipe --rebake path. The execution plan explicitly gates on this. Reviewer must verify intent_hash is updated in the rebaked SKILL.md.
knowledge-extractor EXTRACT mode changes could alter synthesis behavior for resolution-trace.yaml (the existing working case). Regression risk on issues that do produce resolution traces. Medium Preserve the existing resolved_from: 'llm' extraction branch unchanged as a first-class path. New artifact type parsers are additive code paths, not replacements. Test the agent against a known resolution-trace.yaml to verify existing behavior is unaffected.
Different evidence artifact schemas (judge reports, RCA files, remediation records) are not formally specified — their field names and structure may vary across recipes and evolve over time. Hardcoding field names in the extractor creates brittleness. Medium Use pattern-matching heuristics that degrade gracefully (if expected field absent, skip that artifact rather than fail). Document the extraction heuristics per artifact type. Log unexpected schemas in candidates.yaml summary rather than erroring.
Broader scan surface may produce too many candidates per issue, violating C13's 3-5 synthesized patterns target and F11's 7-candidate ceiling. Rich evidence directories could generate 20+ raw signals before synthesis. Low The synthesis step already groups and consolidates raw signals. Preserve the 3-5 target guidance and the 7-candidate ceiling from C13/F11. The synthesis step is the natural throttle — already designed to compress many raw signals into few patterns.
C15 zero-candidate exit currently triggers when no resolved_from: 'llm' entries exist. After the fix, if C15 is not updated in intent.yaml, the recipe may incorrectly exit clean when only the narrow trace check fails but other evidence exists. Low Update C15 in intent.yaml explicitly to state "when no extractable evidence signals exist across all recognized artifact types." Include this in the rebake so the compiled recipe's Step 4 zero-candidate logic reflects the updated condition.
