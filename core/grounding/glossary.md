# Garura Glossary

Canonical definitions of every Garura concept. Referenced by all agents during self-development.

---

## Three-Layer Hierarchy

The foundation of Garura's execution model: Plays (human-invocable workflows) orchestrate Agents (domain experts) which invoke Skills (bounded capabilities). Each layer has a distinct invocation rule and responsibility.

| Concept | Definition | Examples | Relationships | Anti-definition |
|---------|------------|----------|---------------|-----------------|
| **Play** | A human-invocable ordered workflow that orchestrates agents to accomplish a named goal. Plays define the sequence of steps, the agents involved, and the checkpoints required. Encoded as a compiled SKILL.md derived from `reference/intent.yaml`. | `/ship`, `/enhance`, `/start-feature` | Invokes Agents via Task tool; compiled from `intent.yaml` by `/play-creator`; produces STM artifacts; may be Atomic or High-Order | NOT: a skill. Skills are model-invocable only; plays are user-invocable. NOT: an agent. Agents perform domain work; plays sequence it. |
| **Sub-play** | A play that chains other plays in sequence to accomplish a higher-order goal. Delegates entire workflow steps to atomic plays rather than invoking agents directly for domain work. | `ship` chains `commit-code → create-pr → merge-pr` | Is a High-Order Play; composed of Atomic Plays; each atomic play runs independently with its own JSON contract | NOT: an Atomic Play. Atomic plays never chain others. NOT: a play that performs domain work directly. |
| **Atomic Play** | A standalone play that produces exactly one artifact or performs one bounded operation. Never chains other plays. The primitive building block of higher-order workflows. | `commit-code`, `create-pr`, `merge-pr`, `start-feature` | Building block of Sub-plays; invokes ≤2 agents; produces a single bounded output | NOT: a High-Order Play. High-order plays chain multiple atomics. NOT: a multi-artifact producer in one invocation. |
| **High-Order Play** | A play that chains ≤5 atomic plays in sequence. Human-only invocable. Coordinates a pipeline of atomic outcomes into a compound workflow. | `ship` (commit → PR → merge), `enhance` (scope → design → implement → validate) | Composes Atomic Plays; human-only invocable; the top-level entry point for complex pipelines | NOT: an Atomic Play. NOT: a play that invokes domain agents directly — it delegates to atomic plays which invoke agents. |
| **Agent** | An autonomous decision-maker with a single domain of expertise. Receives a JSON contract from a play, performs domain reasoning, loads relevant LTM via Context Crafting, and invokes skills to produce STM artifacts. Named `{domain}-{role}`. | `tech-designer`, `code-builder`, `quality-validator`, `repo-orchestrator` | Invoked by Plays via Task tool; invokes Skills; follows R1–R4 to assemble context; returns enriched JSON contract | NOT: a skill. Skills are model-invocable bounded capabilities; agents are domain experts that invoke skills. NOT: a play. |
| **Domain Agent** | An agent performing primary domain work — design, implementation, validation, analysis. Counts toward the play's agent-call budget. The primary decision-maker in any play step. | `tech-designer`, `code-builder`, `quality-validator`, `knowledge-extractor` | Sub-type of Agent; contributes to play's agent budget; distinct from Utility Agents which are exempt | NOT: a Utility Agent. Utility agents perform infrastructure work and are exempt from agent budgets. |
| **Utility Agent** | An agent performing infrastructure operations — commits, file writes to `.garura/`, status tracking. Exempt from the play's agent-call budget. Runs without consuming domain reasoning capacity. | `repo-orchestrator`, `scriber` | Sub-type of Agent; infrastructure-only; exempt from agent budget; typically runs in background | NOT: a Domain Agent. Utility agents do not make domain decisions; they execute pre-decided actions. |
| **Skill** | A model-invocable bounded capability with defined input/output contracts. Invoked only by agents, never directly by humans or plays. Each skill has a `SKILL.md` specifying what it does, what it requires, and what it produces. | `write-evidence`, `analyze-pr`, `create-commit`, `draft-lld` | Invoked by Agents only; produces STM artifacts; authored in `core/components/skills/`; deployed to `~/.claude/skills/` | NOT: a play. Plays are human-invocable workflows; skills are model-invocable capabilities. NOT: an agent. Skills have no autonomy; they execute a single bounded task. |
| **Play-scoped sub-role** | A context-isolated sub-agent role defined within a specific play for a narrowly bounded task. Not a standalone agent in `core/components/agents/`. Exists only within the play's execution context and cannot be invoked from outside. | `test-writer` inside `implement`, `judge` inside `enhance` | Exists within a play step only; NOT a registered standalone agent; enforces Context isolation | NOT: a standalone Agent. Cannot be invoked outside its parent play. NOT: a Skill. Sub-roles are context-specific decision-makers, not bounded capabilities. |

## Memory Architecture

Garura uses two memory tiers — Long-Term Memory (LTM) that persists across sessions and Short-Term Memory (STM) that is per-issue. The Knowledge Base (KB) is the capability catalog sub-folder within Core LTM. "Knowledge," "KB," and "Core LTM" are distinct and must not be conflated.

> **Transitional (#434).** The KB / Knowledge / `domain-taxonomy` / Capability(KB) / Intent-Epic concepts in the table below describe the PRE-ProductOS memory model. The `knowledge/` folder (the old KB) has been **deleted**. These rows are kept only because the legacy plays (`specify` / `design` / `arch`) still reference them until #434 Phase E retires those plays. The new persistent data model is the **ProductOS Model (v1)** — see the section directly after this one.

| Concept | Definition | Examples | Relationships | Anti-definition |
|---------|------------|----------|---------------|-----------------|
| **LTM (Long-Term Memory)** | Persistent knowledge that survives across sessions. Contains standards, formats, domain knowledge, and project-specific decisions. Split into Core LTM (framework-wide) and Product LTM (project-specific). Deployed at setup time; never created during play execution. | `~/.garura/core/memory/` (Core), `{product_base}/` (Product) | Contains Core LTM and Product LTM; consulted by agents during Context Crafting via R1–R4 protocol | NOT: STM. STM is per-issue and ephemeral in scope; LTM persists across all issues and sessions. NOT: the context window. |
| **Core LTM** | Framework-wide organizational knowledge authored in `core/components/memory/` and deployed to `~/.garura/core/memory/` (global) or `.garura/core/memory/` (project). Contains standards, rules, formats, and KB. Used by all plays regardless of the product being built. | `~/.garura/core/memory/standards/rules/`, `~/.garura/core/memory/knowledge/` | Sub-type of LTM; consulted at R3 in the resolution protocol; contains Standards, Formats, Knowledge, and KB | NOT: the KB alone. KB is the `knowledge/domain-taxonomy/` sub-folder of Core LTM. NOT: Knowledge (the `knowledge/` sub-folder). Core LTM is the entire deployed memory tree, not one sub-folder. |
| **Product LTM** | Project-specific product knowledge produced and updated by `capture-learning`. Lives at the path resolved from `product_base` in `.garura/core/config.yaml`. Organized by SDLC stage (research/, specification/, etc.). Updated incrementally as epics are completed. | `{product_base}/specification/intent-epics/`, `{product_base}/research/` | Sub-type of LTM; updated by `knowledge-extractor` in ENRICH mode; consulted at R2 in the resolution protocol | NOT: Core LTM. Product LTM is project-specific; Core LTM is framework-wide. NOT: STM. Product LTM persists across issues; STM is per-issue. |
| **STM (Short-Term Memory)** | Per-issue artifacts produced during play execution. Lives at `.garura/project/issues/{N}/`. Persists indefinitely as an audit trail but is scoped to one issue. Contains evidence, checkpoints, status files, and context baseline. | `.garura/project/issues/243/evidence/`, `.garura/project/issues/243/checkpoint/` | Written by Scriber via write-evidence skill; read by agents during Context Crafting; per-issue scope; audit trail | NOT: the context window. The context window is transient per session; STM persists on disk. NOT: Context — Bundle. NOT: LTM. |
| **KB (Knowledge Base)** | The capability catalog in `core/components/memory/knowledge/domain-taxonomy/`. Organized as Domain → Capability → Intent Epic. Used exclusively by the product planning pipeline (`specify` / `design` / `arch`). | `core/components/memory/knowledge/domain-taxonomy/user-management.md` | Sub-folder of Core LTM; provides the capability catalog for product planning; organized by Domain | NOT: Core LTM. Core LTM is the entire `~/.garura/core/memory/` tree; KB is only the `knowledge/domain-taxonomy/` sub-folder. NOT: the Knowledge sub-folder (which covers arch/, domain/, product/, quality/ — NOT domain-taxonomy/). |
| **Standards** | Rules, conventions, and quality criteria governing how agents work. Lives in `core/components/memory/standards/rules/`. Deployed to `~/.garura/core/memory/standards/`. Enforced during play execution and reconciliation. | `resolution.md`, `commits.md`, `_index.md` | Sub-folder of Core LTM; loaded by agents via Context Crafting at R3; distinct from Formats | NOT: Formats. Formats are templates and output shapes; Standards are rules that must be followed. NOT: guidelines. Standards are enforced, not advisory. |
| **Formats** | Templates and output shape definitions for artifacts. Lives in `core/components/memory/standards/templates/` and `formats/`. Defines the expected structure of every deliverable. | YAML schemas for `intent-epic.yaml`, `checkpoint.json`, `resolution-trace.yaml` | Sub-folder of Core LTM; read by skills during Spec Crafting; defines artifact structure | NOT: Standards. Standards are rules; Formats are shapes. NOT: STM artifacts themselves — Formats are templates; STM artifacts are instances. |
| **Knowledge** | Searchable reference material organized by sub-domain: arch/, domain/, product/, quality/. Lives in `core/components/memory/knowledge/` (excluding `domain-taxonomy/`). Provides background knowledge for agent reasoning at R3. | `core/components/memory/knowledge/arch/`, `core/components/memory/knowledge/quality/` | Sub-folder of Core LTM; distinct from KB (KB = `domain-taxonomy/` only); read by agents at R3 | NOT: KB. KB is specifically the `domain-taxonomy/` capability catalog. NOT: Core LTM (the full tree). NOT: Product LTM. |
| **Context — Baseline** | A snapshot of what `prepare` knew at planning time. Stored in `{stm_base}/{issue}/context/`. Captured when the epic plan was produced, before implementation began. Serves as the "before" state in reconciliation. | `.garura/project/issues/42/context/understanding.md`, `.garura/project/issues/42/context/approach.yaml` | Part of STM; input to ANALYZE mode; compared against implementation outcomes during Reconciliation; see: Context — Bundle, Context — Crafting, Context — ltm_context | NOT: the context window. NOT: Context — Bundle. Context baseline is a persisted planning snapshot; context bundle is a runtime assembly for a skill invocation. |
| **Evidence** | Proof-of-work artifacts written to `.garura/{issue}/evidence/{play-name}/` at the end of each play step. Documents what was done, what was decided, and how decisions were reached. The audit trail of a play execution. | `build-report.yaml`, `reconciliation-proposals.yaml`, `resolution-trace.yaml` | Written to STM by Scriber; organized under `{issue}/evidence/{play}/`; committed via evidence self-commit | NOT: STM itself. STM is the container; evidence is content within STM. NOT: a checkpoint. Checkpoints request approval for upcoming action; evidence documents completed work. |
| **Resolution trace** | A per-decision trace written by agents to `{stm_base}/{issue}/evidence/{play}/resolution-trace.yaml` when following the R1–R4 protocol. Documents which domains were consulted, which artifacts were read, and whether the answer came from LTM or LLM fallback. | `.garura/project/issues/42/evidence/enhance/resolution-trace.yaml` with `resolved_from: "llm"` | Written to STM via Scriber; produced during Context Crafting; demonstrates the LTM resolution path taken | NOT: the LTM artifacts themselves. Resolution trace documents the path taken; LTM artifacts contain the knowledge read. |
| **Capability** | A product feature unit in the KB capability catalog. Named `{PREFIX}-F001` (e.g., `UM-F001`). Defines a bounded feature with inclusion rules, success criteria, and failure scenarios. Groups under a Domain; expands into Intent Epics. | `UM-F001` (user registration), `PM-F003` (payment processing) | Part of KB; grouped under Domain (KB); expands into Intent Epics via `specify` | NOT: an Intent Epic. A capability is a KB definition; an Intent Epic is a structured work unit generated from a capability for a specific project. |
| **Domain (KB)** | A knowledge taxonomy domain in the KB capability catalog. One Markdown file per domain in `domain-taxonomy/`. Each domain contains multiple Capabilities. Provides the organizing structure for product planning. | user-management, commerce, payments, auth | Contained within KB; groups Capabilities; one file per domain in `domain-taxonomy/` | NOT: a bounded context in the architecture sense. KB domains are knowledge taxonomy categories, not runtime service boundaries. |

## ProductOS Model (v1)

The persistent product data model introduced under #434 (Realign Garura to the ProductOS Command Model). It keeps only three things forever — structure, intent, decisions — and generates everything else on demand. Defined by the v1 schemas at `core/components/memory/standards/schemas/product-os/`.

| Concept | Definition | Schema | Lifecycle |
|---------|------------|--------|-----------|
| **ProductOS** | The persistent product model: a 3-level tree Domain → Capability → Functionality, plus personas and journeys attached to nodes. The "structure" ProductOS keeps forever. | `product-os.yaml` | permanent |
| **Capability (ProductOS)** | A node at the middle level of the tree, the unit of INTENT — it carries ICE. NOT the legacy KB `{PREFIX}-F001` catalog entry. | `product-os.yaml` | permanent |
| **Functionality** | A leaf node under a capability. Its functionality-level ICE is the build input. | `product-os.yaml` | permanent |
| **ICE (ProductOS)** | Intent (goals, constraints, failures), Context (persona, systems, scope), Expectations (outcomes), attached to a capability or functionality node. The build unit at functionality level. | `ice.yaml` | permanent |
| **Decision / ADR (ProductOS)** | A decision record = an ADR at any level (product, capability, functionality, framework). Accepted decisions are never edited — superseded by a new record. | `decision.yaml` | permanent |
| **Lens** | The 5 realization lenses /realize produces for a capability — ux, architecture, run, quality, agentic. One canonical copy per capability so deliveries don't drift. | `lens/<type>.yaml` | permanent |
| **Epic (ProductOS)** | A vertical slice of a functionality — the unit of delivery and the grain a GitHub issue is cut at. Written by the shaping pipeline, deleted when its issue merges. | `epic.yaml` | temporary |

Storage tiers: permanent items (structure, ICE, decisions, lenses) and temporary epics both live in the product-os folder; stories, tests, and build detail stay in STM and are regenerated on demand.

**Status:** the v1 schemas are locked. The plays that fill them (`/vision`, `/understand`, `/shape`, `/realize`, `/grill`, `/implement`, …) are not yet built — the legacy plays (`specify`/`design`/`arch`/`enhance`/`ship`) remain the operating pipeline until #434 Phase E retires them.

## Execution Model

The execution model is governed by the Four Crafts Architecture: **Intent Crafting** (framework author declares goal/constraints in `intent.yaml`), **Prompt Crafting** (play passes only the JSON contract to agents), **Context Crafting** (agent assembles LTM + STM for skill), and **Spec Crafting** (skill reads templates, fills them, writes to STM). The term "Context" has four distinct meanings — each is a separate row below.

| Concept | Definition | Examples | Relationships | Anti-definition |
|---------|------------|----------|---------------|-----------------|
| **JSON Contract** | The primary communication mechanism in task-driven plays. A single JSON object enriched by each agent invocation. Passed from play → agent and returned enriched. Core fields: `intent_path`, `stm_base`, `stm.input`, `stm.output`, `task_id`, `notes`, `step_failure`. | `{"intent_path": "reference/intent.yaml", "stm_base": ".garura/project/issues/42"}` | Passed by Play to Agent; returned enriched; governs what an agent sees; may include Context — ltm_context field | NOT: a full intent document. The contract references intent via `intent_path` but does not inline it. NOT: a task description or instruction. |
| **Intent** | The clean triple — goal, constraints, and failure conditions — declared in `reference/intent.yaml` of a play or skill. The "why" behind a workflow, and the constrained end state to reach. Authored once; hash-locked into the compiled play; never modified at runtime. Constraints are labeled C-N; failure conditions F-N. Success scenarios and recovery are NOT in Intent — they live in the **Expectation** layer (see ICE). | `core/components/plays/enhance/reference/intent.yaml` with `goal`, `constraints`, `failure_conditions` | Produced by Intent Crafting; compiled into SKILL.md; referenced at runtime via `intent_path` in JSON contract | NOT: the JSON contract. The contract is a runtime message; intent is a design-time declaration. NOT: a task description. NOT: the home of scenarios/success/recovery. |
| **ICE** | The three artifacts every piece of work is built from: **Intent** (goal, constraints, failure conditions), **Context** (the tech, design, standards, and build pathway — in Garura, the plays/skills/agents the work runs in), and **Expectation** (success scenarios + recovery). Intent is human-authored and stable; Context is assembled from memory; Expectation is generated from Intent+Context and vetted at a checkpoint. Defined in `docs/philosophy/intent-driven-development.md`. | Intent triple + Context surround + Expectation spec | Intent → IDD Element 1; Context → Elements 5+7; Expectation → generated spec verified at Element 8 | NOT: a new field set inside intent.yaml. ICE splits across layers; Intent stays the clean triple. |
| **Expectation** | The generated spec layer of ICE. Two parts: **success scenarios** (acceptance and the checkable definition of done, merged — the stop/go target and the source of evals) and **recovery** (how to continue toward the intent when blocked). Generated from Intent + Context, vetted at a human checkpoint before it governs anything. | `success_scenarios` (persona/given/then + measure) and `recovery` (per failure: direction + autonomous\|human handoff) | Generated from Intent+Context; vetted at checkpoint; success scenarios feed the encrypted evals; recovery feeds the recovery handoff plan | NOT: part of Intent. Scenarios/success/recovery are never hand-authored into intent.yaml. NOT: two buckets for acceptance vs success — those are one. |
| **Intent Crafting** | The framework author activity of producing `reference/intent.yaml` — specifying goal, constraints (C-N), and failure conditions (F-N) for a play or skill (success scenarios and recovery are generated into Expectation, not authored here). One of the Four Crafts. The foundation all subsequent crafts depend on. | Writing `reference/intent.yaml` for the `enhance` play before running `/play-creator --build` | One of the Four Crafts; produces `intent.yaml`; prerequisite for `/play-creator --build`; defines what success means | NOT: writing SKILL.md directly. SKILL.md is compiled from intent.yaml by `/play-creator`. Direct SKILL.md edits bypass intent. |
| **Prompt Crafting** | The discipline of passing ONLY the JSON contract to agents — no extra inline instructions. The play's responsibility is to wire the contract correctly, not add runtime instructions that bypass intent constraints. Keeps agent behavior deterministic. | Play calls `Task tool → agent` passing only the JSON contract, nothing else | One of the Four Crafts; enforces clean contract boundaries; prevents context contamination of agents | NOT: writing prompts. The craft is about what NOT to include (extra instructions) as much as what to include. |
| **Context — Crafting** | The agent activity of discovering relevant LTM paths, reading required STM artifacts, and assembling the ≤12K token context bundle a skill needs to execute. The agent's primary responsibility per the Four Crafts model. Follows the R1–R4 resolution protocol. | Agent reads `resolution.md` at R3, loads `context-baseline` from STM, selects relevant `intent-epic.yaml` | One of the Four Crafts; follows R1–R4; produces Context — Bundle; reads Context — Baseline from STM; triggered by Context — ltm_context in JSON contract; generates Resolution trace | NOT: the context window itself. Context Crafting is the assembly activity; the context window is the model environment. see: Context — Bundle, Context — Baseline, Context — ltm_context |
| **Spec Crafting** | The skill activity of reading format templates from LTM, filling them with substantive content, and writing the resulting artifact to STM. The skill's primary responsibility per the Four Crafts model. The production phase of a skill invocation. | Skill reads `intent-epic.yaml` schema from Formats, fills all fields, writes to `{stm_base}/{issue}/evidence/` | One of the Four Crafts; performed by Skills; consumes Formats from LTM; produces STM artifacts | NOT: Context Crafting. Spec Crafting happens after context is assembled. NOT: writing documentation. |
| **Context — Bundle** | The assembled ≤12K token context that an agent passes to a skill invocation. Composed of selected LTM excerpts and STM artifacts relevant to the skill's task. The concrete output of Context Crafting. | Subset of: `resolution.md` excerpt + `context-baseline` + relevant `intent-epic.yaml` + recent `evidence` | Output of Context Crafting; input to Skill; bounded by ≤12K token limit; see: Context — Crafting, Context — Baseline, Context — ltm_context | NOT: the context window (unlimited model context). Context bundle is a curated subset assembled by the agent, not the full session context. NOT: Context — Baseline. |
| **Checkpoint** | An approval gate written by Scriber to `.garura/{issue}/checkpoint/{play}/`. Status lifecycle: `PENDING_APPROVAL` → `APPROVED` / `REJECTED`. Human response required unless auto-approval criteria are met. Governs high-impact actions (commits, PRs, destructive operations). | `.garura/project/issues/42/checkpoint/enhance/` with `status: PENDING_APPROVAL` | Written to STM by Scriber; requires Tether/Vanish response; may be auto-approved if criteria met; distinct from Evidence | NOT: evidence. Evidence documents completed work; a checkpoint requests approval for upcoming action. NOT: a status file. |
| **Tether / Vanish** | The canonical user responses at checkpoint gates. `Tether` (or `T`) = proceed with the proposed action. `Vanish` (or `V`) = cancel the action. Parsed case-insensitively. Required at all high-impact checkpoints. | User types "Tether" at a checkpoint gate to approve a PR; types "Vanish" to abort | Part of the Checkpoint approval lifecycle; defined in CLAUDE.md and AGENTS.md; applies to commits, PRs, destructive actions | NOT: optional conventions. All plays requiring human approval MUST use Tether/Vanish semantics — no inline `AskUser` calls. |
| **Auto-approval** | The play behavior of skipping a human checkpoint when all low-risk criteria defined in `reference/intent.yaml` are met. Prevents unnecessary friction for safe read-only or non-destructive operations. | Read-only analysis steps; steps that only write evidence with no side effects | Conditional path at Checkpoint; defined in intent.yaml constraints; skips Tether/Vanish prompt when criteria pass | NOT: the default behavior. Auto-approval is explicitly gated on criteria defined in intent. Manual approval is the default. |
| **Signals** | User CLI invocations that enter the Garura system (e.g., `/ship`, `/enhance`, `/start-feature`). All signals enter exclusively via plays. IDD Element 2. The only sanctioned entry point to Garura workflows. | `/ship`, `/enhance`, `/start-feature`, `/play-creator`, `/sud:install` | Entry point to the play system; maps to IDD "Signals" element; triggers Play execution | NOT: tool calls. Signals are user-typed play invocations, not model tool calls. NOT: agent actions. |
| **NWWI** | "No Work Without an Issue." All work that produces checkpoints must be linked to a GitHub issue. Enforced at `commit-code`. Implemented via the two-phase write pattern: `start-feature` writes first to `_pending/` then moves to `{issue}/` once the issue number is known. | `start-feature` creates issue, then writes STM to `_pending/` until issue number resolved; `commit-code` rejects commits without issue linkage | Enforced at `commit-code`; defined in ADR 008; implemented by two-phase write pattern | NOT: a suggestion. NWWI is a hard architectural invariant — a play that produces checkpoints without an issue is non-compliant. |
| **Evidence self-commit** | The pattern where every play commits its own STM artifacts at the end (Stage 7) via `repo-orchestrator`. Non-blocking — does not gate play completion. Ensures every play leaves an auditable trail in the repository. | `ship` play commits its evidence YAML files after merge completes | Defined in ADR 012; executed by repo-orchestrator as utility agent; writes Evidence to STM repository | NOT: the feature commit. Evidence self-commit is a separate non-blocking STM write; the feature commit contains code changes. |
| **Agent-first pattern** | The principle that plays NEVER execute domain work directly using tools. All domain work is delegated to agents via the Task tool. Plays are pure orchestrators; agents are domain experts. Unconditional — no exceptions for "simple" tasks. | Play uses `Task tool → agent` instead of running `git commit` directly | Enforced in CLAUDE.md and AGENTS.md; prevents plays from accumulating domain knowledge; maintains separation of concerns | NOT: permission to bypass agents for trivial tasks. The pattern is unconditional — domain work always goes through agents. |
| **Pause / Resume** | The ability to interrupt and resume a play mid-execution. Tracked via a status file at `{stm_base}/{issue}/status/{play}.json`. Each completed step is recorded so the play can restart from the last incomplete step without re-executing completed work. | User interrupts `enhance` mid-execution; on resume, play reads status file and skips already-completed steps | Requires Status file; defined per-play in SKILL.md; allows recovery from mid-play failures | NOT: a checkpoint. Pause/Resume tracks execution state; a checkpoint requests human approval for an upcoming action. |
| **Status file** | JSON file at `{stm_base}/{issue}/status/{play}.json` tracking play execution state. Written by Scriber after each step completes. Enables Pause/Resume by recording which steps have been completed in a given play run. | `{"play": "enhance", "current_step": 5, "completed_steps": [1,2,3,4]}` | Written to STM via Scriber; read at play start for resume detection; enables Pause/Resume pattern | NOT: a Checkpoint. Status files track execution progress; checkpoints request human approval. NOT: evidence. |
| **Structured failure protocol** | Standardized failure return format for agents when they cannot complete a task. Fields: `what_failed`, `why`, `domain_assessment` (with `responsible_domain`), `suggested_fix`. Enables plays to route failures to the correct recovery action. | Agent returns `failure.domain_assessment.responsible_domain: "design"` when a design gap prevents implementation | Defined in `docs/framework/structured-failure-protocol.md`; consumed by plays for recovery routing; triggers Recovery | NOT: an error log. Structured failure is a typed contract returned to the play, not a free-text log entry. |
| **Recovery** | One concept: how to continue, directionally, toward the intent's end state when something blocks the way. Lives in the **Expectation** layer (generated, vetted); at runtime the validator builds a recovery handoff plan and routes it to an autonomous fix or human review. The autonomous branch is executed via the structured-failure mechanism — route to the responsible agent, retry max 2, else escalate. Defined in `docs/framework/intent-driven-recovery.md`. | Validator: "unit tests at 50%, here are the failing ones, raise to green" → autonomous fix loop (max 2) or escalate to human | Part of Expectation (ICE); triggered by Structured failure protocol / failed evals; autonomous branch bounded at 2 retries; feeds intent-resolver at Level 4 | NOT: three separate recovery notions — one concept at all levels; the core is how an agent recovers to meet the intent. NOT: infinite retries. |
| **Context isolation** | The requirement that certain agents (judge, evals-engineer, play-scoped evaluators) MUST NOT see implementation code or prior outputs. Enforced via strict JSON contract separation — only pass what the agent should see. Prevents anchoring bias in evaluation. | Judge agent receives only intent + scenarios + outcome summary; NOT the full code diff or prior agent outputs | Enforced via JSON Contract field selection; critical for Reconciliation judges; see: Play-scoped sub-role | NOT: a security boundary. Context isolation is for inference quality — it prevents bias, not unauthorized access. |
| **Scriber** | A utility agent running on a lightweight model (`haiku`) that writes evidence, checkpoints, and status artifacts to the `.garura/` whitelist. Runs in background (`run_in_background: true`). Exempt from agent budget. | Scriber writes `build-report.yaml` to `.garura/project/issues/42/evidence/enhance/` | Sub-type of Utility Agent; invokes write-evidence skill; enforces Whitelist at write boundary; exempt from agent budget | NOT: a domain agent. Scriber performs no domain reasoning; it only writes pre-formatted artifacts supplied by other agents. |
| **Write-evidence / Whitelist** | `write-evidence` is the single skill that enforces the `.garura/` folder whitelist at every write boundary. The whitelist (ADR 017) is the strict list of allowed folders under `.garura/` — specifically `evidence/`, `checkpoint/`, `status/`, `context/`. All STM writes route through write-evidence. | Only paths matching whitelist patterns are written: `{issue}/evidence/`, `{issue}/checkpoint/`, `{issue}/status/` | Invoked by Scriber; enforces Whitelist (ADR 017); prevents unauthorized writes to `.garura/` | NOT: an OS file permission system. The whitelist is a design-time contract enforced in the skill, not filesystem permissions. |
| **Blast radius** | The scope of impact of a proposed change, measured by files affected and domains crossed. Documented in `{issue}/context/blast-radius/`. Used by the Scope gate to determine if the work is right-sized for the `enhance` play. | Change touching 3 files in 1 domain has smaller blast radius than change spanning 8 files across 3 domains | Assessed during Scope gate; documented in STM `context/`; also read by knowledge-extractor §Step 1 | NOT: a risk level. Blast radius measures scope/spread, not probability or severity of failure. |
| **Scope gate** | The `enhance` play's initial assessment of whether an enhancement is right-sized. Thresholds: files ≤15, domains ≤2, effort ≤1 day. ## Reconciliation

Reconciliation is the process of comparing the context baseline (what `prepare` knew) against implementation outcomes (what actually happened) and updating Product LTM. The `knowledge-extractor` agent operates in three modes — ANALYZE (produces proposals), ENRICH (writes approved proposals), and FAST (lightweight post-merge) — to execute this process. "Tier" in this section refers to the reconciliation classification (Tiers 1/2/3); see also Tier — Context Audience for the distinct second usage.

|---------|------------|----------|---------------|-----------------|

## Protocols

Protocols are the enforced decision rules and communication contracts that govern how agents resolve domain questions (R1–R4 resolution), return failures (structured failure protocol), and handle artifact authority (LOCKED vs. DRAFT). The `ltm_context` contract field is the trigger that activates the R1–R4 protocol.

| Concept | Definition | Examples | Relationships | Anti-definition |
|---------|------------|----------|---------------|-----------------|
| **LTM Resolution Protocol (R1–R4)** | A four-step hierarchy governing how agents make domain decisions. **R1:** Identify domains relevant to the question. **R2:** Check project LTM — LOCKED artifacts stop descent; DRAFT artifacts require R3 confirmation. **R3:** Check Core LTM for standards and background knowledge. **R4:** LLM fallback when no LTM answer exists, flagged as `resolved_from: "llm"`. | Agent choosing a test strategy: R1 identifies "testing" domain, R2 finds no locked artifact, R3 reads `quality` standards, R4 falls back to LLM | Governs Context Crafting; produces Resolution trace; defined in `resolution.md` and ADR 015; triggered by Context — ltm_context | NOT: strictly sequential in all cases. A LOCKED artifact at R2 stops resolution immediately — R3 and R4 are skipped. |
| **LOCKED artifact** | A project LTM artifact listed in `ltm_context.locked_artifacts` that is authoritative for its domain. Encountering a LOCKED artifact at R2 stops the resolution — no descent to R3 or R4. The single source of truth for its domain question. | Locked `physical-architecture.yaml` means all technology stack decisions come from this file exclusively | Part of R2 resolution step; stops resolution descent; produced by DRAFT → VALIDATE → LOCKED lifecycle | NOT: DRAFT. DRAFT artifacts are advisory only and require R3 confirmation. NOT: all project LTM artifacts. Only explicitly locked artifacts stop descent. |
| **DRAFT artifact** | A project LTM artifact that is advisory only. Does not stop resolution at R2 — the agent must descend to R3 (Core LTM) for confirmation before relying on the content. A provisional state in the artifact lifecycle. | DRAFT `quality-profile.yaml` means its quality targets are provisional pending Core LTM confirmation | Part of R2 resolution step; advisory only; requires R3 descent; part of DRAFT → VALIDATE → LOCKED lifecycle | NOT: LOCKED. LOCKED stops resolution; DRAFT continues it. NOT: invalid or wrong — DRAFT is a lifecycle state, not a quality indicator. |
| **LLM fallback** | R4 in the resolution protocol. Invoked when no LTM artifact answers the domain question. The decision is flagged as `resolved_from: "llm"` in the resolution trace. Flags a knowledge gap that `capture-learning` should fill. | Agent deciding on error handling pattern when no relevant standards or project LTM artifact exists | R4 in the LTM Resolution Protocol; flagged in Resolution trace; primary candidate for `capture-learning` to persist | NOT: the preferred resolution path. R1–R3 must be exhausted before R4. An LLM fallback indicates a knowledge gap to be closed, not a valid long-term answer. |
| **Context — ltm_context** | An optional field in the JSON contract. When present, signals to the agent that it must follow the full R1–R4 LTM resolution protocol. Contains: `project_base`, `core_base`, `query_domains`, and `locked_artifacts`. | `{"ltm_context": {"project_base": "/path/to/product", "query_domains": ["testing"], "locked_artifacts": ["physical-architecture.yaml"]}}` | Optional field in JSON Contract; triggers R1–R4 resolution; see: LTM Resolution Protocol, Context — Bundle, Context — Crafting, Context — Baseline | NOT: the Context Crafting activity. `ltm_context` is a data field in the JSON contract; Context Crafting is the agent's assembly process that the field triggers. |

## Play Compilation

Plays are produced by the `intent.yaml → /play-creator --build → SKILL.md` pipeline. The compiled SKILL.md is hash-locked to its intent; direct edits are prohibited. The product planning pipeline (`specify → design → arch`) produces Intent Epics from KB capabilities. All generated artifacts follow the DRAFT → VALIDATE → LOCKED authority lifecycle.

| Concept | Definition | Examples | Relationships | Anti-definition |
|---------|------------|----------|---------------|-----------------|
| **Play maturity levels (L1–L5)** | Historical five-level classification of play sophistication from L1 ("structured non-deterministic") to L5 ("dark factory"). Appears in ADRs 001–003 for historical context only. | L1: human-run, non-deterministic. L2: compiled, intent-driven (current). L3: eval-driven. L4: self-improving. L5: autonomous. | Historical only; superseded by ADR 017; see: Level 2 (Compiled, Intent-Driven) for current state | NOT: the current design criterion. | **Level 2 (Compiled, Intent-Driven)** | The current operating level of all Garura plays. SKILL.md is compiled from `intent.yaml` by `/play-creator`. ## Self-Development

Self-development is the Garura practice of using Garura to build and improve Garura itself. The `core/grounding/` directory is the self-development layer — distinct from `core/components/` (what gets deployed) and `docs/` (what external readers see).
