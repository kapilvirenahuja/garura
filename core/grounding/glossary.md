# Garura Glossary

Canonical definitions of every Garura concept. Referenced by all agents during self-development.

---

## Three-Layer Hierarchy

The foundation of Garura's execution model: Plays (human-invocable workflows) orchestrate Agents (domain experts) which invoke Skills (bounded capabilities). Each layer has a distinct invocation rule and responsibility.

| Concept | Definition | Examples | Relationships | Anti-definition |
|---------|------------|----------|---------------|-----------------|
| **Play** | A human-invocable ordered workflow that orchestrates agents to accomplish a named goal. Plays define the sequence of steps, the agents involved, and the checkpoints required. Encoded as a compiled SKILL.md derived from its ICE source (`reference/ice.md`). | `/vision`, `/implement`, `/fix-bug` | Invokes Agents via Task tool; compiled from `reference/ice.md` by `/play-creator` (changed via `/play-editor`); produces STM artifacts; may be Atomic or High-Order | NOT: a skill. Skills are model-invocable only; plays are user-invocable. NOT: an agent. Agents perform domain work; plays sequence it. |
| **Sub-play** | A play that chains other plays in sequence to accomplish a higher-order goal. Delegates entire workflow steps to atomic plays rather than invoking agents directly for domain work. | `measure` opens with `start-change` and closes by chaining `commit-change → propose-change → review-change → merge-change` | Is a High-Order Play; composed of Atomic Plays; each atomic play runs independently with its own JSON contract | NOT: an Atomic Play. Atomic plays never chain others. NOT: a play that performs domain work directly. |
| **Atomic Play** | A standalone play that produces exactly one artifact or performs one bounded operation. Never chains other plays. The primitive building block of higher-order workflows. | `commit-change`, `propose-change`, `merge-change`, `start-change` | Building block of Sub-plays; invokes ≤2 agents; produces a single bounded output | NOT: a High-Order Play. High-order plays chain multiple atomics. NOT: a multi-artifact producer in one invocation. |
| **High-Order Play** | A play that chains ≤5 atomic plays in sequence. Human-only invocable. Coordinates a pipeline of atomic outcomes into a compound workflow. | `measure` (opens its branch via `start-change`, closes via `commit-change → propose-change → review-change → merge-change`) | Composes Atomic Plays; human-only invocable; the top-level entry point for complex pipelines | NOT: an Atomic Play. NOT: a play that invokes domain agents directly — it delegates to atomic plays which invoke agents. |
| **Agent** | An autonomous decision-maker with a single domain of expertise. Receives a JSON contract from a play, performs domain reasoning, loads relevant LTM via Context Crafting, and invokes skills to produce STM artifacts. Named `{domain}-{role}`. | `tech-designer`, `code-builder`, `quality-auditor`, `repo-orchestrator` | Invoked by Plays via Task tool; invokes Skills; follows R1–R4 to assemble context; returns enriched JSON contract | NOT: a skill. Skills are model-invocable bounded capabilities; agents are domain experts that invoke skills. NOT: a play. |
| **Domain Agent** | An agent performing primary domain work — design, implementation, validation, analysis. Counts toward the play's agent-call budget. The primary decision-maker in any play step. | `tech-designer`, `code-builder`, `quality-auditor`, `market-analyst` | Sub-type of Agent; contributes to play's agent budget; distinct from Utility Agents which are exempt | NOT: a Utility Agent. Utility agents perform infrastructure work and are exempt from agent budgets. |
| **Utility Agent** | An agent performing infrastructure operations — commits, file writes to `.garura/`, status tracking. Exempt from the play's agent-call budget. Runs without consuming domain reasoning capacity. | `repo-orchestrator`, `scriber` | Sub-type of Agent; infrastructure-only; exempt from agent budget; typically runs in background | NOT: a Domain Agent. Utility agents do not make domain decisions; they execute pre-decided actions. |
| **Skill** | A model-invocable bounded capability with defined input/output contracts. Invoked only by agents, never directly by humans or plays. Each skill has a `SKILL.md` specifying what it does, what it requires, and what it produces. | `write-evidence`, `analyze-pr`, `create-commit`, `draft-lld` | Invoked by Agents only; produces STM artifacts; authored in `core/components/skills/`; deployed to `~/.claude/skills/` | NOT: a play. Plays are human-invocable workflows; skills are model-invocable capabilities. NOT: an agent. Skills have no autonomy; they execute a single bounded task. |
| **Play-scoped sub-role** | A context-isolated sub-agent role defined within a specific play for a narrowly bounded task. Not a standalone agent in `core/components/agents/`. Exists only within the play's execution context and cannot be invoked from outside. | the isolated, clean-context content-quality judge inside `/vision` and the other grounding plays | Exists within a play step only; NOT a registered standalone agent; enforces Context isolation | NOT: a standalone Agent. Cannot be invoked outside its parent play. NOT: a Skill. Sub-roles are context-specific decision-makers, not bounded capabilities. |

## Memory Architecture

Garura uses two memory tiers — Long-Term Memory (LTM) that persists across sessions and Short-Term Memory (STM) that is per-issue. The Knowledge Base (KB) is the `knowledge/` shelf tree within Core LTM. "KB" and "Core LTM" are distinct and must not be conflated.

> **Retired (#434).** The pre-ProductOS memory model — the `domain-taxonomy/` capability catalog, `{PREFIX}-F001` capabilities, and Intent Epics — was deleted together with the legacy plays (`specify` / `design` / the old planning `arch`). The persistent product data model is now the **ProductOS Model (v1)** — see the section directly after this one. The KB lives on as the knowledge shelves (see the KB row below).

| Concept | Definition | Examples | Relationships | Anti-definition |
|---------|------------|----------|---------------|-----------------|
| **LTM (Long-Term Memory)** | Persistent knowledge that survives across sessions. Contains standards, formats, domain knowledge, and project-specific decisions. Split into Core LTM (framework-wide) and Product LTM (project-specific). Deployed at setup time; never created during play execution. | `~/.garura/core/memory/` (Core), `{product_base}/` (Product) | Contains Core LTM and Product LTM; consulted by agents during Context Crafting via R1–R4 protocol | NOT: STM. STM is per-issue and ephemeral in scope; LTM persists across all issues and sessions. NOT: the context window. |
| **Core LTM** | Framework-wide organizational knowledge authored in `core/components/memory/` and deployed to `~/.garura/core/memory/` (machine-global). Contains standards (rules, schemas, templates) and the knowledge shelves. Used by all plays regardless of the product being built. | `~/.garura/core/memory/standards/rules/`, `~/.garura/core/memory/knowledge/` | Sub-type of LTM; consulted at R3 in the resolution protocol; contains Standards, Formats, and the KB shelves | NOT: the KB alone. KB is the `knowledge/` shelf tree within Core LTM; Core LTM is the entire deployed memory tree, not one sub-folder. |
| **Product LTM** | Project-specific product knowledge — in the ProductOS command model this is the persistent product model: the spine, the grounding docs, and the decision records, at the product base resolved from `.garura/core/config.yaml`. Written by the strategy and realize plays; updated after delivery by `/learn`. | the spine, `domain.md` / `capability.md` / `functionality.md` grounding docs, decision records | Sub-type of LTM; consulted at R2 in the resolution protocol; updated by `/learn` | NOT: Core LTM. Product LTM is project-specific; Core LTM is framework-wide. NOT: STM. Product LTM persists across issues; STM is per-issue. |
| **STM (Short-Term Memory)** | Per-issue artifacts produced during play execution. Lives at `.garura/project/issues/{N}/`. Persists indefinitely as an audit trail but is scoped to one issue. Contains specs, evidence, checkpoints, status files, context, and review artifacts. | `.garura/project/issues/243/evidence/`, `.garura/project/issues/243/checkpoint/` | Written by Scriber via write-evidence skill; read by agents during Context Crafting; per-issue scope; audit trail | NOT: the context window. The context window is transient per session; STM persists on disk. NOT: Context — Bundle. NOT: LTM. |
| **KB (Knowledge Base)** | The knowledge shelf tree at `core/components/memory/knowledge/` (deployed machine-global to `~/.garura/core/memory/knowledge/`). Shelves: `architecture/`, `domains/`, `review/`, `technology/`, `work-intelligence/`. The domain shelves ground `/vision`; the work-intelligence shelf feeds `/next`. | `knowledge/domains/ecommerce.md`, `knowledge/work-intelligence/` | Sub-folder of Core LTM; consulted by agents at R3; extended via `propose-kb-node` / `validate-kb-extension` | NOT: the retired `domain-taxonomy/` capability catalog — that model was deleted under #434. NOT: Core LTM (the full tree). |
| **Standards** | Rules, conventions, and quality criteria governing how agents work. Lives in `core/components/memory/standards/rules/`. Deployed to `~/.garura/core/memory/standards/`. Enforced during play execution. | `resolution.md`, `commits.md`, `_index.md` | Sub-folder of Core LTM; loaded by agents via Context Crafting at R3; distinct from Formats | NOT: Formats. Formats are templates and output shapes; Standards are rules that must be followed. NOT: guidelines. Standards are enforced, not advisory. |
| **Formats** | Templates and output shape definitions for artifacts. Lives in `core/components/memory/standards/templates/` and `standards/schemas/`. Defines the expected structure of every deliverable. | the ProductOS v1 schemas (`standards/schemas/product-os/`), `standards/templates/checkpoint.md`, `standards/templates/approval-prompt.md` | Sub-folder of Core LTM; read by skills during Spec Crafting; defines artifact structure | NOT: Standards. Standards are rules; Formats are shapes. NOT: STM artifacts themselves — Formats are templates; STM artifacts are instances. |
| **Evidence** | Proof-of-work artifacts written to `{stm_base}/{issue}/evidence/{play}/` at the end of each play step. Documents what was done, what was decided, and how decisions were reached. The audit trail of a play execution. | `build-report.yaml`, `findings.yaml`, `resolution-trace.yaml` | Written to STM by Scriber; organized under `{issue}/evidence/{play}/`; committed via evidence self-commit | NOT: STM itself. STM is the container; evidence is content within STM. NOT: a checkpoint. Checkpoints request approval for upcoming action; evidence documents completed work. |
| **Resolution trace** | A per-decision trace written by agents to `{stm_base}/{issue}/evidence/{play}/resolution-trace.yaml` when following the R1–R4 protocol. Documents which domains were consulted, which artifacts were read, and whether the answer came from LTM or LLM fallback. | `.garura/project/issues/42/evidence/fix-bug/resolution-trace.yaml` with `resolved_from: "llm"` | Written to STM via Scriber; produced during Context Crafting; demonstrates the LTM resolution path taken | NOT: the LTM artifacts themselves. Resolution trace documents the path taken; LTM artifacts contain the knowledge read. |
| **Domain (KB)** | A domain shelf in the KB — one Markdown file per domain under `knowledge/domains/`, capturing the domain from the user's point of view. Grounds `/vision`'s domain and capability proposals so they are grounded in the shelves, never invented. | `knowledge/domains/ecommerce.md`, `knowledge/domains/payments.md` | Contained within KB; consulted by `/vision`; genuinely new domains enter via KB-node proposals | NOT: a bounded context in the architecture sense. KB domain shelves are reference knowledge, not runtime service boundaries. NOT: a spine domain entry — the spine records the product's own domains. |

## ProductOS Model (v1)

The persistent product data model introduced under #434 (Realign Garura to the ProductOS Command Model). It keeps only three things forever — structure, intent, decisions — and generates everything else on demand. Defined by the v1 schemas at `core/components/memory/standards/schemas/product-os/`.

| Concept | Definition | Schema | Lifecycle |
|---------|------------|--------|-----------|
| **ProductOS** | The persistent product model: a 3-level tree Domain → Capability → Functionality, plus personas and journeys attached to nodes. The "structure" ProductOS keeps forever. | `product-os.yaml` | permanent |
| **Capability (ProductOS)** | A node at the middle level of the tree, the unit of INTENT — it carries ICE. NOT the legacy KB `{PREFIX}-F001` catalog entry. | `product-os.yaml` | permanent |
| **Functionality** | A leaf node under a capability. Its functionality-level ICE is the build input. | `product-os.yaml` | permanent |
| **ICE (ProductOS)** | Intent (goals, constraints, failures), Context (persona, systems, scope), Expectations (outcomes), attached to a capability or functionality node. The build unit at functionality level. | `ice.yaml` | permanent |
| **Decision / ADR (ProductOS)** | A decision record = an ADR at any level (product, capability, functionality, framework). Accepted decisions are never edited — superseded by a new record. | `decision.yaml` | permanent |
| **Slice** | A vertical product increment — a bundle of functionalities, possibly across several capabilities of one domain, that together deliver something usable. Cut by `/shape`, planned by `/roadmap`, realized lens by lens, stamped realized by `/measure`. | `slice.yaml` | permanent |
| **Lens** | The realization lens docs the realize plays produce for a SLICE — quality, ux, agentic, architecture, measure, run, plus the marketing lens doc from the functional pipe. Stored on the slice; `/measure` stamps the slice realized only when all seven lens docs line up. | `lens/<type>.yaml` | permanent |
| **Epic (ProductOS)** | One functionality sliced end-to-end through its components — the unit of delivery and the grain a GitHub issue is cut at. Cut by `/grill`; kept and stamped delivered when its issue merges (ADR 019). | `epic.yaml` | permanent (kept when delivered) |

Storage tiers: the structure, ICE, decisions, lenses, slices, and epics (kept when delivered — ADR 019) live in the product-os folder; stories, tests, and build detail stay in STM and are regenerated on demand.

**Status:** the v1 schemas are locked and the plays that fill them are built and operating — `/vision`, `/understand`, `/shape`, `/roadmap`, the realize lens plays, `/grill`, `/implement`, and the rest of the command model. The legacy pipeline (`specify`/`design`/the old planning `arch`, and `enhance`/`ship`) was retired under #434; today's `/arch` is the non-functional lens play, unrelated to the old planning `arch`.

## Execution Model

The execution model is governed by the Four Crafts Architecture: **Intent Crafting** (framework author declares goal/constraints in the play's ICE source, `reference/ice.md`), **Prompt Crafting** (play passes only the JSON contract to agents), **Context Crafting** (agent assembles LTM + STM for skill), and **Spec Crafting** (skill reads templates, fills them, writes to STM). The term "Context" has several distinct meanings — each is a separate row below.

| Concept | Definition | Examples | Relationships | Anti-definition |
|---------|------------|----------|---------------|-----------------|
| **JSON Contract** | The primary communication mechanism in task-driven plays. A single JSON object enriched by each agent invocation. Passed from play → agent and returned enriched. Core fields: `intent_path`, `stm_base`, `stm.input`, `stm.output`, `task_id`, `notes`, `step_failure`. | `{"intent_path": "reference/ice.md", "stm_base": ".garura/project/issues/42"}` | Passed by Play to Agent; returned enriched; governs what an agent sees; may include Context — ltm_context field | NOT: a full intent document. The contract references intent via `intent_path` but does not inline it. NOT: a task description or instruction. |
| **Intent** | The clean triple — goal, constraints, and failure conditions — declared in the `## Intent` section of a play's ICE source (`reference/ice.md`). The "why" behind a workflow, and the constrained end state to reach. Authored once; compiled into the play; never modified at runtime. Constraints are labeled C-N; failure conditions F-N. Success scenarios and recovery are NOT in Intent — they live in the **Expectation** section (see ICE). | `core/components/plays/vision/reference/ice.md` — `## Intent` with constraints and failure conditions | Produced by Intent Crafting; compiled into SKILL.md; referenced at runtime via `intent_path` in JSON contract | NOT: the JSON contract. The contract is a runtime message; intent is a design-time declaration. NOT: a task description. NOT: the home of scenarios/success/recovery. |
| **ICE** | The three artifacts every piece of work is built from: **Intent** (goal, constraints, failure conditions), **Context** (the tech, design, standards, and build pathway — in Garura, the plays/skills/agents the work runs in), and **Expectation** (success scenarios + recovery). Intent is human-authored and stable; Context is assembled from memory; Expectation is generated from Intent+Context and vetted at a checkpoint. Defined in `docs/philosophy/intent-driven-development.md`. | Intent triple + Context surround + Expectation spec | Intent → IDD Element 1; Context → Elements 5+7; Expectation → generated spec verified at Element 8 | NOT: a bloated Intent. ICE splits across layers; Intent stays the clean triple. |
| **Expectation** | The generated spec layer of ICE. Two parts: **success scenarios** (acceptance and the checkable definition of done, merged — the stop/go target and the source of evals) and **recovery** (how to continue toward the intent when blocked). Generated from Intent + Context, vetted at a human checkpoint before it governs anything. | `success_scenarios` (persona/given/then + measure) and `recovery` (per failure: direction + autonomous\|human handoff) | Generated from Intent+Context; vetted at checkpoint; success scenarios feed the encrypted evals; recovery feeds the recovery handoff plan | NOT: part of Intent. Scenarios/success/recovery are never hand-authored into the Intent section. NOT: two buckets for acceptance vs success — those are one. |
| **Intent Crafting** | The framework author activity of producing the Intent section of `reference/ice.md` — specifying goal, constraints (C-N), and failure conditions (F-N) for a play (success scenarios and recovery are generated into Expectation, not hand-authored). One of the Four Crafts. The foundation all subsequent crafts depend on. | Writing the Intent for a new play before `/play-creator` compiles it | One of the Four Crafts; produces the ICE source; prerequisite for `/play-creator`; defines what success means | NOT: writing SKILL.md directly. SKILL.md is compiled from the ICE source; intent changes never go in as direct SKILL.md edits (only non-intent surface edits may, with a deviation note). |
| **Prompt Crafting** | The discipline of passing ONLY the JSON contract to agents — no extra inline instructions. The play's responsibility is to wire the contract correctly, not add runtime instructions that bypass intent constraints. Keeps agent behavior deterministic. | Play calls `Task tool → agent` passing only the JSON contract, nothing else | One of the Four Crafts; enforces clean contract boundaries; prevents context contamination of agents | NOT: writing prompts. The craft is about what NOT to include (extra instructions) as much as what to include. |
| **Context — Crafting** | The agent activity of discovering relevant LTM paths, reading required STM artifacts, and assembling the ≤12K token context bundle a skill needs to execute. The agent's primary responsibility per the Four Crafts model. Follows the R1–R4 resolution protocol. | Agent reads `resolution.md` at R3, loads the relevant STM evidence, selects the relevant grounding docs | One of the Four Crafts; follows R1–R4; produces Context — Bundle; reads STM artifacts; triggered by Context — ltm_context in JSON contract; generates Resolution trace | NOT: the context window itself. Context Crafting is the assembly activity; the context window is the model environment. see: Context — Bundle, Context — ltm_context |
| **Spec Crafting** | The skill activity of reading format templates from LTM, filling them with substantive content, and writing the resulting artifact to STM. The skill's primary responsibility per the Four Crafts model. The production phase of a skill invocation. | Skill reads a lens template from `standards/schemas/product-os/`, fills all fields, writes the draft to STM | One of the Four Crafts; performed by Skills; consumes Formats from LTM; produces STM artifacts | NOT: Context Crafting. Spec Crafting happens after context is assembled. NOT: writing documentation. |
| **Context — Bundle** | The assembled ≤12K token context that an agent passes to a skill invocation. Composed of selected LTM excerpts and STM artifacts relevant to the skill's task. The concrete output of Context Crafting. | Subset of: `resolution.md` excerpt + the relevant grounding docs + recent `evidence` | Output of Context Crafting; input to Skill; bounded by ≤12K token limit; see: Context — Crafting, Context — ltm_context | NOT: the context window (unlimited model context). Context bundle is a curated subset assembled by the agent, not the full session context. |
| **Checkpoint** | An approval gate written by Scriber to `{stm_base}/{issue}/checkpoint/{play}/`. Status lifecycle: `PENDING_APPROVAL` → `APPROVED` / `REJECTED`. Human response required unless auto-approval criteria are met. Governs high-impact actions (commits, PRs, destructive operations). | `.garura/project/issues/42/checkpoint/implement/` with `status: PENDING_APPROVAL` | Written to STM by Scriber; requires Tether/Orbit/Vanish response; may be auto-approved if criteria met; distinct from Evidence | NOT: evidence. Evidence documents completed work; a checkpoint requests approval for upcoming action. NOT: a status file. |
| **Tether / Orbit / Vanish** | The canonical user responses at checkpoint gates. `Tether` = proceed with the proposed action. `Orbit` (with feedback) = re-run the gated stage incorporating the feedback. `Vanish` = cancel the action. Parsed case-insensitively. Required at all high-impact checkpoints. | User types "Tether" at a checkpoint gate to approve; "Orbit" plus feedback to redo the stage; "Vanish" to abort | Part of the Checkpoint approval lifecycle; canonical format in `standards/templates/approval-prompt.md`; applies to commits, PRs, destructive actions | NOT: optional conventions. All plays requiring human approval MUST use this protocol — no inline `AskUser` calls. |
| **Auto-approval** | The play behavior of skipping a human checkpoint when all low-risk criteria defined in the play's ICE constraints are met. Prevents unnecessary friction for safe read-only or non-destructive operations. | Read-only analysis steps; steps that only write evidence with no side effects | Conditional path at Checkpoint; defined in the ICE constraints; skips the approval prompt when criteria pass | NOT: the default behavior. Auto-approval is explicitly gated on criteria defined in intent. Manual approval is the default. |
| **Signals** | User CLI invocations that enter the Garura system (e.g., `/vision`, `/implement`, `/fix-bug`). All signals enter exclusively via plays. IDD Element 2. The only sanctioned entry point to Garura workflows. | `/vision`, `/implement`, `/fix-bug`, `/next`, `/play-creator` | Entry point to the play system; maps to IDD "Signals" element; triggers Play execution | NOT: tool calls. Signals are user-typed play invocations, not model tool calls. NOT: agent actions. |
| **NWWI** | "No Work Without an Issue." All work that produces checkpoints must be linked to a tracked issue. `start-change` resolves or creates the issue at the head of every unit of work; `commit-change` writes commits that reference it. STM written before the issue number is known goes to the configured pending path, then moves to `{issue}/`. | `start-change` resolves/creates the issue and initializes STM; `commit-change` messages reference the issue | Enforced by `start-change` / `commit-change`; defined in ADR 008; implemented by the two-phase (pending → issue) write pattern | NOT: a suggestion. NWWI is a hard architectural invariant — a play that produces checkpoints without an issue is non-compliant. |
| **Evidence self-commit** | The pattern where a play commits its own STM artifacts at its close via `repo-orchestrator`. Non-blocking — does not gate play completion. Ensures every play leaves an auditable trail in the repository. | the Standard Play Close records the self-commit SHA(s) in its Commit Reference line (`play-close.md`) | Defined in ADR 012; executed by repo-orchestrator as utility agent; writes Evidence to STM repository | NOT: the feature commit. Evidence self-commit is a separate non-blocking STM write; the feature commit contains code changes. |
| **Agent-first pattern** | The principle that plays NEVER execute domain work directly using tools. All domain work is delegated to agents via the Task tool. Plays are pure orchestrators; agents are domain experts. Unconditional — no exceptions for "simple" tasks. | Play uses `Task tool → agent` instead of running `git commit` directly | Enforced in CLAUDE.md and AGENTS.md; prevents plays from accumulating domain knowledge; maintains separation of concerns | NOT: permission to bypass agents for trivial tasks. The pattern is unconditional — domain work always goes through agents. |
| **Pause / Resume** | The ability to interrupt and resume a play mid-execution. Tracked via a status file at `{stm_base}/{issue}/status/{play}.json`. Each completed step is recorded so the play can restart from the last incomplete step without re-executing completed work. | User interrupts `implement` mid-execution; on resume, play reads status file and skips already-completed steps | Requires Status file; defined per-play in SKILL.md; allows recovery from mid-play failures | NOT: a checkpoint. Pause/Resume tracks execution state; a checkpoint requests human approval for an upcoming action. |
| **Status file** | JSON file at `{stm_base}/{issue}/status/{play}.json` tracking play execution state. Written by Scriber after each step completes. Enables Pause/Resume by recording which steps have been completed in a given play run. | `{"play": "implement", "current_step": 5, "completed_steps": [1,2,3,4]}` | Written to STM via Scriber; read at play start for resume detection; enables Pause/Resume pattern | NOT: a Checkpoint. Status files track execution progress; checkpoints request human approval. NOT: evidence. |
| **Structured failure protocol** | Standardized failure return format for agents when they cannot complete a task. Fields: `what_failed`, `why`, `domain_assessment` (with `responsible_domain`), `suggested_fix`. Enables plays to route failures to the correct recovery action. | Agent returns `failure.domain_assessment.responsible_domain: "design"` when a design gap prevents implementation | Defined in `docs/framework/structured-failure-protocol.md`; consumed by plays for recovery routing; triggers Recovery | NOT: an error log. Structured failure is a typed contract returned to the play, not a free-text log entry. |
| **Recovery** | One concept: how to continue, directionally, toward the intent's end state when something blocks the way. Lives in the **Expectation** layer (generated, vetted); at runtime the validator builds a recovery handoff plan and routes it to an autonomous fix or human review. The autonomous branch is executed via the structured-failure mechanism — route to the responsible agent, retry max 2, else escalate. Defined in `docs/framework/intent-driven-recovery.md`. | Validator: "unit tests at 50%, here are the failing ones, raise to green" → autonomous fix loop (max 2) or escalate to human | Part of Expectation (ICE); triggered by Structured failure protocol / failed evals; autonomous branch bounded at 2 retries; feeds intent-resolver at Level 4 | NOT: three separate recovery notions — one concept at all levels; the core is how an agent recovers to meet the intent. NOT: infinite retries. |
| **Context isolation** | The requirement that certain agents (judge, evals-engineer, play-scoped evaluators) MUST NOT see implementation code or prior outputs. Enforced via strict JSON contract separation — only pass what the agent should see. Prevents anchoring bias in evaluation. | Judge agent receives only intent + scenarios + outcome summary; NOT the full code diff or prior agent outputs | Enforced via JSON Contract field selection; critical for verification judges (`quality-auditor` in `/implement`, the content-quality judge in the grounding plays); see: Play-scoped sub-role | NOT: a security boundary. Context isolation is for inference quality — it prevents bias, not unauthorized access. |
| **Scriber** | A utility agent running on a lightweight model (`haiku`) that writes evidence, checkpoints, and status artifacts to the `.garura/` whitelist. Runs in background (`run_in_background: true`). Exempt from agent budget. | Scriber writes `build-report.yaml` to `.garura/project/issues/42/evidence/implement/` | Sub-type of Utility Agent; invokes write-evidence skill; enforces Whitelist at write boundary; exempt from agent budget | NOT: a domain agent. Scriber performs no domain reasoning; it only writes pre-formatted artifacts supplied by other agents. |
| **Write-evidence / Whitelist** | `write-evidence` is the single skill that enforces the `.garura/` folder whitelist at every write boundary. The whitelist (ADR 017) is the strict list of allowed paths under `.garura/` — `core/` (excluding `memory/`), `product/{product,ux,arch}/`, and `project/issues/{n}/{specs,evidence,checkpoint,context,review}/`. All STM writes route through write-evidence. | Only paths matching whitelist patterns are written: `{issue}/evidence/`, `{issue}/checkpoint/`, `{issue}/specs/` | Invoked by Scriber; enforces Whitelist (ADR 017); prevents unauthorized writes to `.garura/` | NOT: an OS file permission system. The whitelist is a design-time contract enforced in the skill, not filesystem permissions. |
| **Blast radius** | The scope of impact of a proposed change, measured from the change surface and the dependency graph. Computed by the `compute-blast-radius` skill, which emits `blast-radius.yaml`. | Change touching 3 files in 1 domain has smaller blast radius than change spanning 8 files across 3 domains | Computed by `compute-blast-radius` from `change_surface` + `dependency_graph`; consumed by `test-engineer` (Phase 2B) to bound test scope | NOT: a risk level. Blast radius measures scope/spread, not probability or severity of failure. |

## Learning Loop

Closing the loop is `/learn`'s job: after a unit of work ships, read what actually happened and update the living product model to match reality. It replaces the retired capture/codify/distill/enrich/reap pipeline and the `knowledge-extractor` agent's ANALYZE/ENRICH/FAST modes — those concepts are gone.

| Concept | Definition | Examples | Relationships | Anti-definition |
|---------|------------|----------|---------------|-----------------|
| **Learning loop (`/learn`)** | The play that closes the loop after a unit of work ships. Reads the outcome signals the pipeline produced — the measure lens (baseline/target/realized), the validate verdicts and fix reports, the run lens, and the delivered epic/slice status — and refreshes the living product model's MEANING: one-liners, NFR levels, status promotions, the grounding-doc sections learning changed. Each material learning lands as an append-only decision record; the tree skeleton is never restructured. One delivered unit per run; one human checkpoint before anything persists. | `/learn` after an epic ships | Reads the model from the spine; writes spine + grounding docs + decisions through the change pipeline; successor of the retired capture/codify/distill/enrich/reap loop | NOT: code review. NOT: a rebuild of the model — it refreshes meaning, never the structure. |
| **Tier — Context Audience** | The audience-separation classification applied to context bundles (documented in `docs/philosophy/idsd.md`). Describes what a context bundle is assembled for and which audience tier it targets, controlling what each agent type is permitted to see. | Context bundle assembled for a judge agent (tier: evaluation only) vs. code-builder agent (tier: implementation) | Appears in idsd.md context bundle architecture; see: Context — Bundle, Context isolation | NOT: a priority ranking. Audience tiers separate what each agent type sees in its context bundle. |
| **ADR (Architecture Decision Record)** | A formal record of an architectural decision about Garura itself, stored in `docs/adr/` and numbered sequentially (ADR 001, ADR 002, etc.). Product-level decisions in a target product live in the ProductOS decision records (`decision.yaml`), not here. | `docs/adr/017-folder-whitelist.md`, `docs/adr/016-agent-json-contract.md` | Stored in `docs/adr/`; framework counterpart of the ProductOS Decision record | NOT: a changelog. ADRs capture architectural decisions with rationale and impact, not feature change logs. NOT: the ProductOS decision record. |
| **Trinity (Slice Trinity Model)** | The 3-3-1 spine of the slice plays (#434, decision 23): the **lens trinity** (quality, ux, agentic — what the slice should be) + the **foundation trinity** (arch, measure, run — what makes it real; the load-bearing one) + **grill** (the closer, a prime of one). Lens plays read the slice hub only; foundation plays read the hub plus all three lens files; grill reads everything. | A slice runs lens (quality → ux → agentic) ‖ main ‖ foundation (arch → measure → run) ‖ main ‖ grill | Defined in the lens schema index (`standards/schemas/product-os/lens/_index.md`, decision 23); stability principle: lens churn is normal, foundation churn is an event | NOT: the Three-Layer Hierarchy (play/agent/skill). NOT: the old implement+validate "trinity execution" cycle from the retired `enhance` play — that meaning is retired; grill → implement → validate → launch → deploy is now the **execute** pipeline. |

## Protocols

Protocols are the enforced decision rules and communication contracts that govern how agents resolve domain questions (R1–R4 resolution), return failures (structured failure protocol), and handle artifact authority (LOCKED vs. DRAFT). The `ltm_context` contract field is the trigger that activates the R1–R4 protocol.

| Concept | Definition | Examples | Relationships | Anti-definition |
|---------|------------|----------|---------------|-----------------|
| **LTM Resolution Protocol (R1–R4)** | A four-step hierarchy governing how agents make domain decisions. **R1:** Identify domains relevant to the question. **R2:** Check project LTM — LOCKED artifacts stop descent; DRAFT artifacts require R3 confirmation. **R3:** Check Core LTM for standards and background knowledge. **R4:** LLM fallback when no LTM answer exists, flagged as `resolved_from: "llm"`. | Agent choosing a test strategy: R1 identifies "testing" domain, R2 finds no locked artifact, R3 reads `quality` standards, R4 falls back to LLM | Governs Context Crafting; produces Resolution trace; defined in `resolution.md` and ADR 015; triggered by Context — ltm_context | NOT: strictly sequential in all cases. A LOCKED artifact at R2 stops resolution immediately — R3 and R4 are skipped. |
| **LOCKED artifact** | A project LTM artifact listed in `ltm_context.locked_artifacts` that is authoritative for its domain. Encountering a LOCKED artifact at R2 stops the resolution — no descent to R3 or R4. The single source of truth for its domain question. | Locked `physical-architecture.yaml` means all technology stack decisions come from this file exclusively | Part of R2 resolution step; stops resolution descent; produced by DRAFT → VALIDATE → LOCKED lifecycle | NOT: DRAFT. DRAFT artifacts are advisory only and require R3 confirmation. NOT: all project LTM artifacts. Only explicitly locked artifacts stop descent. |
| **DRAFT artifact** | A project LTM artifact that is advisory only. Does not stop resolution at R2 — the agent must descend to R3 (Core LTM) for confirmation before relying on the content. A provisional state in the artifact lifecycle. | DRAFT `quality-profile.yaml` means its quality targets are provisional pending Core LTM confirmation | Part of R2 resolution step; advisory only; requires R3 descent; part of DRAFT → VALIDATE → LOCKED lifecycle | NOT: LOCKED. LOCKED stops resolution; DRAFT continues it. NOT: invalid or wrong — DRAFT is a lifecycle state, not a quality indicator. |
| **LLM fallback** | R4 in the resolution protocol. Invoked when no LTM artifact answers the domain question. The decision is flagged as `resolved_from: "llm"` in the resolution trace. Flags a knowledge gap that `/learn` should fill. | Agent deciding on error handling pattern when no relevant standards or project LTM artifact exists | R4 in the LTM Resolution Protocol; flagged in Resolution trace; primary candidate for `/learn` to persist | NOT: the preferred resolution path. R1–R3 must be exhausted before R4. An LLM fallback indicates a knowledge gap to be closed, not a valid long-term answer. |
| **Context — ltm_context** | An optional field in the JSON contract. When present, signals to the agent that it must follow the full R1–R4 LTM resolution protocol. Contains: `project_base`, `core_base`, `query_domains`, and `locked_artifacts`. | `{"ltm_context": {"project_base": "/path/to/product", "query_domains": ["testing"], "locked_artifacts": ["physical-architecture.yaml"]}}` | Optional field in JSON Contract; triggers R1–R4 resolution; see: LTM Resolution Protocol, Context — Bundle, Context — Crafting | NOT: the Context Crafting activity. `ltm_context` is a data field in the JSON contract; Context Crafting is the agent's assembly process that the field triggers. |

## Play Compilation

Plays are produced by the `reference/ice.md → /play-creator → SKILL.md` pipeline. Intent changes go through the ICE source and recompile (`/play-editor` for existing plays); non-intent surface edits are made directly, recorded with a deviation note. Project LTM artifacts follow the DRAFT → VALIDATE → LOCKED authority lifecycle.

| Concept | Definition | Examples | Relationships | Anti-definition |
|---------|------------|----------|---------------|-----------------|
| **Play maturity levels (L1–L5)** | Five-level classification of framework maturity from L1 ("structured non-deterministic") to L5 ("dark factory"). Defined in ADR 013; Level 3 redefined by ADR 025. Numeric levels are never assigned to individual plays (ADR 017) — they name the framework's operating model. | L1: human-run, non-deterministic. L2: compiled, intent-driven (current). L3: deterministic skeleton, goal-loop interior (target — ADR 025). L4: runtime resolution (north star). L5: autonomous. | Framework-level only; see: Level 2 for current state, Level 3 for the target | NOT: a per-play badge. Play coherence is enforced by the ICE source, the compiler's gates, and evals — not by a level number. |
| **Level 2 (Compiled, Intent-Driven)** | The current operating level of all Garura plays. SKILL.md is compiled from the ICE source by `/play-creator`. The same intent input always produces the same execution path — including the interior steps of every box. Intent changes require recompilation; only non-intent surface edits are made directly, with a deviation note. | All plays in `core/components/plays/` are Level 2 — they have `reference/ice.md` and a compiled `SKILL.md` (`play-creator` itself excepted: it is the compiler bootstrap and has no ICE source) | Produced by `/play-creator`; references the ICE source; the current operational state | NOT: a target to be reached. Level 2 is where all current plays already are. NOT: a permanent ceiling — Level 3 (ADR 025) is the active target. |
| **Level 3 (Skeleton + Goal-Loop)** | The active target operating model (ADR 025). The compiled skeleton — command sequence, gates, evidence contract — stays deterministic and guaranteed; inside each box the agent runs a free loop until the intent is met, fenced by four walls: a machine-checkable stop condition, a checker executing the lens's deterministic gates, a turn/token budget, and an evidence schema. Governing rule: determinism lives at the skeleton, never inside the boxes. | A ship is always commit-change → propose-change → review-change → merge-change (skeleton); how commit-change groups and commits internally is the loop's call, closed by its stop condition | Defined in ADR 025; reached via epic #460; keeps the ICE → `/play-creator` pipeline; the loop self-checks only deterministic quality — design/UX/security judgment stays human or fresh-context reviewer | NOT: runtime intent resolution — that is Level 4, a north star deliberately not being built (ADR 013's rollback stands). NOT: self-judging subjective quality. |
| **ICE source (`reference/ice.md`)** | The per-play source of truth: the clean Intent triple (goal, constraints C-N, failure conditions F-N) plus the generated-and-vetted Expectation (success scenarios, recovery). Authored by the framework author; compiled by `/play-creator`; changed via `/play-editor`; never modified at runtime. | `core/components/plays/vision/reference/ice.md` — defines `/vision`'s goal, constraints, and expectation | Input to `/play-creator`; produces Compiled play; referenced at runtime via `intent_path` in JSON contract | NOT: a task description or runtime prompt. The ICE source is a design-time contract defining what success means, compiled before the play ever runs. |
| **Compiled play** | The `SKILL.md` produced by `/play-creator` from `reference/ice.md`. The artifact that agents execute at runtime. Intent changes require editing the ICE source and recompiling (via `/play-editor`); non-intent surface edits are allowed directly, recorded with a `Direct-edit deviation note`. | `core/components/plays/vision/SKILL.md` | Produced by `/play-creator` from the ICE source; deployed via `/install-garura` | NOT: the ICE source. The ICE source is the design-time contract; SKILL.md is the compiled artifact that agents execute. NOT: the place to make intent changes. |
| **/play-creator** | The play compiler. Interviews for the intent triple, generates the Expectation, identifies the skills, scripts, and agents the play needs, selects a workflow structure, generates evals, and emits the compiled `SKILL.md` plus bundled scripts for the play's mechanical work. It is the compiler bootstrap — it has no ICE source of its own, so changes to it are direct SKILL.md edits by definition. | `/play-creator` run to compile a new play; `/play-editor` to change an existing one | Compiles the ICE source into SKILL.md; companion `/play-editor` edits existing plays by editing their ICE source and recompiling | NOT: `/play-editor`, which modifies existing plays. NOT: a hand-editing pass — the compiled play is generated. |
| **Workflow structure (A/B/C)** | Classification of compiled play structure, catalogued in play-creator's `references/workflow-structures.md`. **Structure A:** full checkpoint flow. **Structure B:** fast execution flow. **Structure C:** chained plays (higher-order). A readiness-only shape covers analysis-only plays. | A play with human gates at each stage runs Structure A; a play that chains other plays end-to-end runs Structure C | Selected by `/play-creator` at compile time; determines how a play orchestrates agents and sub-plays | NOT: a quality ranking. Structure A is not superior to Structure C — they serve different orchestration needs. |
| **Eval types (SE/SCE)** | Two types of evaluation criteria in compiled plays. **Step Eval (SE-N):** a per-step pass criterion that must be satisfied before advancing. **Scenario Eval (SCE-N):** an end-to-end scenario evaluation validating the full play outcome against a named use case. | SE-1: "JSON contract contains `stm_base`." SCE-2: "The slice is stamped realized only when all seven lens docs line up." | Defined in Compiled plays (SKILL.md); SE-N guards individual steps; SCE-N validates end-to-end play quality | NOT: executable unit tests. Evals are natural language criteria evaluated by agents, not code-based test assertions. NOT: the same as each other — SE-N is per-step; SCE-N is end-to-end. |
| **DRAFT → VALIDATE → LOCKED lifecycle** | The three-state artifact authority lifecycle applied to project LTM artifacts consulted during resolution. `DRAFT` = initial generated state (advisory, does not stop R2 resolution). `VALIDATE` = under human review. `LOCKED` = authoritative (stops LTM resolution at R2; no further descent). | an architecture artifact starts DRAFT, moves to LOCKED after human review; agents treat it as authoritative thereafter | Governs artifact authority in Product LTM; intersects with R1–R4 at R2 | NOT: a code review lifecycle. This is artifact authority state, not pull request status. NOT: a sequential requirement — not all artifacts need to pass through all three states. |

## Self-Development

Self-development is the Garura practice of using Garura to build and improve Garura itself. The `core/grounding/` directory is the self-development layer — distinct from `core/components/` (what gets deployed) and `docs/` (what external readers see). Global and project deployment modes control where compiled components land.

| Concept | Definition | Examples | Relationships | Anti-definition |
|---------|------------|----------|---------------|-----------------|
| **Global mode** | The default Garura deployment: components land in `~/.claude/` (skills + agents) and shared memory goes machine-global to `~/.garura/core/memory/`. Deployed via `/install-garura`. Persists across all projects on the machine. | `~/.claude/skills/`, `~/.garura/core/memory/` | Deployed by /install-garura; contrasts with Project mode; the default for all Garura development work | NOT: Project mode. Global mode is permanent user-level deployment; project mode is ephemeral and gitignored. |
| **Project mode** | Repo-local Garura deployment for a target project. Components go to the target's `.claude/` (Claude Code) or `.agents/skills/` (Codex CLI), both gitignored; shared memory still goes machine-global to `~/.garura/core/memory/`; an install manifest records what was placed. Deployed via `/install-garura {target} --tool {claude\|codex}`. | `.claude/skills/` (gitignored), `.agents/skills/` (Codex) within the target repo | Deployed by /install-garura per target; manifest consumed by /uninstall-garura; contrasts with Global mode | NOT: Global mode. Project mode is repo-local and gitignored; global mode is permanent and user-level. |
| **/install-garura** | The garura-native meta-play that installs Garura into a target project or repository. Runs a per-tool adapter: **claude** (default) lays down `.claude/` skills + agents with model tiers resolved to Claude models; **codex** lays down `.agents/skills/` Agent Skills plus `AGENTS.md` and `~/.codex` profiles. Always writes the `.garura/` tooling tree, copies shared memory to machine-global `~/.garura`, and records an install manifest. The reverse is `/uninstall-garura`. | `/install-garura {target} --tool codex` | Deploys `core/components/`; adapter per host tool; manifest consumed by `/uninstall-garura`; run after source edits to refresh a deployment | NOT: `/sud:install` — the sudarshan harness port is a separate tool; garura self-hosts its own install (#434). NOT: single-host — it covers Claude Code and the OpenAI Codex CLI. |
| **core/grounding/** | A directory in the Garura repository for self-development documents — artifacts used by agents working on Garura itself, not deployed with the runtime. Distinct from `core/components/` (deployable components) and `docs/` (external reader documentation). First populated by this glossary. | `core/grounding/glossary.md` (this file) | Distinct from `core/components/` (deployed by `/install-garura`); not gitignored; self-development layer | NOT: `core/components/`. Files in `core/grounding/` are NOT deployed by `/install-garura`. NOT: `docs/`. docs/ is for external readers; `core/grounding/` is for Garura's internal self-development. |
| **IDD (Intent-Driven Development)** | The foundational paradigm underlying Garura. "Intent in → Quality out." Defines 5 elements: Intent model, Signals, Context bundles, Verification loops, Memory. Tool-agnostic — not specific to Claude Code or the Codex CLI. The "what" that IDSD operationalizes. | "IDD is to IDSD as Agile is to Scrum." The 5 IDD elements map directly to Garura mechanisms. | The foundational paradigm; operationalized by IDSD in Garura; defined in `docs/philosophy/intent-driven-development.md` | NOT: IDSD. IDD is the paradigm (the "what"); IDSD is its Garura-specific methodology (the "how"). NOT: limited to Garura — IDD is tool-agnostic. |
| **IDSD (Intent-Driven Software Development)** | The Garura-specific methodology operationalizing IDD. Maps IDD's 5 elements to Garura's concrete mechanisms: Intent → the plays' ICE sources, Signals → play invocations, Context bundles → ≤12K token assemblies, Verification → eval loops, Memory → LTM/STM architecture. | The entire Garura framework IS the IDSD implementation of IDD. | Operationalizes IDD; documented in `docs/philosophy/idsd.md`; instantiated by every Garura component | NOT: IDD. IDSD is the Garura-specific implementation; IDD is the tool-agnostic paradigm. NOT: a synonym for the Garura framework — IDSD is the methodology; Garura is the implementation. |
| **Intent primacy** | The architectural principle that intent is primary; plays are scaffolding. Plays exist for determinism now, but the long-term direction is for intent to replace plays as the direct execution unit — same determinism, without the scaffolding overhead. | "Intent is primary. Plays are scaffolding." The goal: give the intent to the model and get the same outcome as today's compiled play. | Principle in `docs/philosophy/architecture.md §Intent Primacy and Play Evolution`; drives play design toward pure intent-centricity | NOT: "plays are unimportant or to be eliminated now." Intent primacy describes the evolutionary direction, not a current deprecation of plays. |
