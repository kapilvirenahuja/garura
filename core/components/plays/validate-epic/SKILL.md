# validate-epic

Given committed code from implement-epic on a feature branch and locked verification scenarios from prepare-epic, deploy to a preview/staging environment and execute system-level verification. E2E tests run all 3 scenario tiers in order: baseline, new, regression. A judge exercises the running product independently via the deployed environment — this is QA acceptance, not developer self-check. When judge failures occur, remediation routes back to implement-epic for code changes — validate-epic does not invoke code-builder directly. After implement-epic fixes and re-commits, validate-epic re-deploys and re-runs. Max 3 fix iterations. A feature-steward generates manual test scenarios for anything e2e automation cannot cover. The final output is a QA verdict: ACCEPT or REJECT.

## Recovery

When an agent returns `status: "failed"`:

1. **Deployment environment down (F2):** Retry health check — max 3 attempts with exponential backoff. If all fail, hard halt.
2. **E2E test infrastructure failure:** Retry test execution — max 2 attempts. If persistent, halt with infrastructure error report.
3. **Eval generation failure:** Retry once with the same specs. After 2 failures, halt.
4. **Judge failure (agent error, not eval failure):** Verify encrypted file is readable. If corrupted, regenerate evals and retry.
5. **Judge eval failures (behavioral):** Enter fix loop — route to implement-epic. Max 3 iterations (C9).
6. **Baseline tier failure (F10):** Do NOT proceed to Tier 2/3. Enter fix loop immediately.
7. **Feature-steward failure:** Retry once. If persistent, generate placeholder checklist and flag for manual follow-up.
8. **implement-epic fix timeout:** If implement-epic takes longer than expected, wait. Do not bypass or invoke code-builder directly (C8, F4).

**Pre-flight failures (C1, C2, C13) are not recoverable** — hard halt with clear error message.
