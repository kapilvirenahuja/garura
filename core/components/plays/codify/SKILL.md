# codify

Given an existing codebase (single repo or multi-repo, any size from 10k to millions of LOC), reverse-engineer the Garura product LTM that the code already embodies. codify is the brownfield counterpart to the /specify + /design + /arch trinity — it produces the same artifact shape via code inference instead of user interview. Every proposal is staged under `{stm_base}/{issue}/evidence/codify/proposals/` with `source_type: inferred_from_code`. No writes to product LTM occur during this play — /garura:enrich is the sole promotion path.

## Role

You are the orchestrator. You own the workflow. You delegate every inference task to agents via JSON contracts — never execute inference work directly. You DO perform the deterministic scan via the `scan-codebase` skill and the final aggregation via `aggregate-codify-proposals`.

**Forbidden:** Direct source code reading as a primary input; direct architecture/domain reasoning; any write to `{product_base}` or `~/.garura/core/memory/`.

Utility agents: 1 (repo-orchestrator). Exempt from budget.

## Arguments

```
/codify [--issue <issue-number>] [--path <dir> ...] [--incremental] [--target <phase>] [--size-limit-mb N] [--time-limit-s N]
```

- `--issue` — optional. GitHub issue to anchor STM output. If absent, resolved from branch name `codify/{issue}-*` or `feat/{issue}-codify-*`; else a new issue is created by `project-orchestrator`.
- `--path` — repeatable. Repository root(s) to scan. Defaults to current directory.
- `--incremental` — opt-in skip of LOCKED product LTM artifacts.
- `--target` — optional filter: `foundation | scope | epics | quality | architecture | experience | all` (default `all`).
- `--size-limit-mb` — scan-index.json soft cap. Default 10.
- `--time-limit-s` — scan wall-clock soft cap. Default 600.
