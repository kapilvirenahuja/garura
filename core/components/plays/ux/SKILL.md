# ux

Take one shaped **slice** — a vertical product increment from /shape, the thing you actually
deliver — and write its **ux lens**: just enough to anchor the intended experience and let
the build figure the rest. The slice is the unit of realization: you pick a slice and run
quality → ux → agentic → arch → run on it, then ship it. A slice has no ICE of its own — its
**hub** is the union of its functionalities' ICE (which may span several capabilities) plus
the profile. The lens is three things and only three: the **screens** the slice needs, each
with a low-fidelity layout; the **states** each screen can hold; and the product's **visual
core** — color and typography. The screens make every functionality of the slice visible, so
a human can look at them and confirm the shape and the ICE under it are right. Accessibility
is **not** restated — it lives in the profile, and the build connects it. Flows are not
specified — the build derives them from the screens and the journeys. It writes only the ux
lens, plus the visual-core decision.

**Pipeline position: none.** /ux is a realization, model-building play. It opens no delivery
issue and cuts no branch, so the D2 pipeline-position rule injects neither a `start-change`
head nor a close sequence. It writes the persistent product model directly. It runs after
/shape, and by convention second in the realize sequence — but takes **no** dependency on the
quality lens. It reads the hub (the slice's functionalities' ICE + the profile box) only;
never another lens.

## Role

You are the orchestrator. You own the workflow and the step order. You delegate the one piece
of judgment — turning the slice's functionalities and personas/journeys into screens (with
layouts) and their states, plus the visual core — to the `product-os-keeper` agent via a JSON
contract over files on disk, and you run every mechanical part (the readiness + hub
resolution, the draft validation, the snapshot, the allowlisted persist, and the post-apply
checks) through bundled scripts. You never write the model YAML yourself, you never write the
slice record, the profile, or another lens, and you never persist before the human approves
the single checkpoint (C11).

**Forbidden:** hand-writing lens/decision YAML; writing the slice record, the functionalities'
ICE, the profile, or another lens by any route; reading another realize lens to derive the
screens (ux reads the hub); restating accessibility in the lens (it lives in the profile);
persisting by any route other than `scripts/apply_ux.py`; persisting before Step 3 approval.

**Agent boundaries:**

| Agent | Domain | Skills it invokes | Phases |
|-------|--------|-------------------|--------|
| `product-os-keeper` | Turn the slice's functionalities' ICE and personas/journeys into low-fidelity screens (with layouts), the states each holds, and the product's visual core (color + typography) recorded as a decision; **search the KB's technology/architecture shelves via kb-search** to ground the visual core, navigation pattern, and responsive strategy in best-fit learnings (gaps → a recorded proposal); cover every functionality the slice bundles | `kb-search`, `author-ux-lens` | Draft |

`product-os-keeper` is the single **domain agent** this play uses (1 of the ≤5 budget).
No utility agents are needed — git/issue machinery is absent (position none).

## Pre-flight

| Check | Constraint | Action on Failure |
|-------|-----------|-------------------|
| Resolve config + `product_base` (`.garura/core/config.yaml`) | — | Hard halt |
| Profile firmed (`set`) AND the slice exists with every functionality ICE resolved + rich | C1 | Hard halt (REC1) |

Resolve config mechanically, then resolve the slice + hub. /ux has no branch or issue
(position none):

```
python3 scripts/preflight.py --play ux --config .garura/core/config.yaml
python3 scripts/check_ready_slice.py --product-base <product_base> --slice <slice>
```

`preflight.py` returns config facts (`product_base`, `stm_base`, `evidence_record`).
`check_ready_slice.py` is the readiness gate **and hub resolver** (C1): it halts unless the
profile is `set` and the slice — the play argument (e.g. `/ux slice-token-data-spine` or
`/ux <domain>/<slice-id>`) — exists with every `functionalities[].ice_ref` resolving to a rich
ICE. An ice_ref that does not resolve **fails loud** (a broken hub, never a silent pass). On
success it emits the slice context the draft step uses: `slice_file`, `domain`, `lens_dir`,
and the resolved `functionality_ices`. If the profile is `directional`, the slice is missing,
or a functionality ICE is unresolved/thin, hard halt and route to /understand or /shape
(REC1).

**Resume check:** if `{stm_base}_realize/ux/status/<slice-id>.json` exists, resume — skip
completed steps, reset any in-progress step to pending, continue from the first incomplete.

## Workflow

### Phase: Draft

**Step 1 — Draft ux lens** · Owner: `product-os-keeper` · Depends on: pre-flight
The agent reads the slice's hub (its functionalities' ICE, resolved by the readiness gate) and
its personas/journeys, then **searches the KB's technology/architecture shelves through
`kb-search`** for the patterns that fit this product's conditions + surfaces and bases the
visual core, the navigation pattern, and the responsive strategy on them. It invokes
`author-ux-lens` to draft the screens (with low-fidelity layouts), the states each holds, and
the product's visual core — reusing the product's visual-core decision if one exists, else
drafting it — to STM, grounding every screen and covering every functionality the slice bundles,
plus a `choices` block recording the KB-grounded pattern choices (gaps → a recorded proposal):

    {
      "task":    "search the KB's technology/architecture shelves for the ux patterns that fit this product (conditions + surfaces); from this slice's functionalities + personas/journeys, draft the ux lens — screens (with low-fidelity layouts) that make every functionality visible, the states each holds, and the product's visual core (color + typography) recorded as a decision; ground the visual core, navigation pattern, and responsive strategy in matched KB learnings (or recorded proposals) in the manifest `choices` block; cover every functionality the slice bundles so a human can validate the shape",
      "inputs":  { "slice_ref": "<domain>/<slice-id>",
                   "slice_file": "<product_base>/<slice_file>",
                   "functionality_ices": [ "<product_base>/<ice_ref>", "..." ],
                   "profile_path": "<product_base>/product-os/profile.yaml",
                   "kb_search": "<skills>/kb-search/scripts/kb_search.py",
                   "kb_root": "<knowledge_dir>",
                   "product_base": "<product_base>",
                   "lens_rel": "<lens_dir>/ux.yaml" },
      "outputs": { "draft_dir":  "<working>/draft/",
                   "lens":       "<working>/draft/<lens_dir>/ux.yaml",
                   "manifest":   "<working>/draft/ux-manifest.yaml",
                   "proposals":  "<working>/draft/proposals/" }
    }

`slice_file`, `lens_dir`, and `functionality_ices` come from `check_ready_slice.py`; `<skills>`
is the active skills base and `<knowledge_dir>` is the KB root (the `knowledge/` dir from
config's `ltm` section). The skill reads the hub + the KB **read-only** and never another lens.
It returns the contract with the output paths on disk — never inline content.
**SE-1 (F1/C1):** the readiness + hub gate passed at pre-flight — the profile is `set` and the
slice exists with every functionality ICE resolved + rich; otherwise the run halted (REC1).

### Phase: Validate

**Step 2 — Validate the draft** · Owner: play · Depends on: Step 1
Run the ux validator over the draft before the checkpoint. Coverage is read straight from the
slice record — every functionality the slice bundles must be visualized:

```
python3 scripts/validate_ux.py --draft <working>/draft \
        --manifest <working>/draft/ux-manifest.yaml \
        --slice-file <product_base>/<slice_file>
```

**SE-3 (F3/C3):** the lens is the three blocks only — `content` has exactly
screens/states/design_system, each non-empty; no flows, no accessibility, no
gates/components/environments.
**SE-4 (F4/C4):** every screen grounds to a real source (a functionality's ICE or a
persona/journey) and the visual core grounds on a decision that resolves; no element
ungrounded.
**SE-5 (F5/C5):** just enough — every screen has a purpose and a low-fidelity layout, states
are enumerated per declared screen, and design_system carries a palette and typography.
**SE-6 (F6/C6):** coverage holds — every functionality the slice bundles (read from the slice
record) is visualized by at least one screen.
**SE-7 (F7/C7):** no grounding source is another lens — ux read the hub only.
**SE-8 (F8/C8):** the visual core names a decision that resolves to a drafted record.
**SE-10 (F10/C10):** the lens and any decision carry their required v1 fields and valid enums
(`type: ux`, a `slice_ref`).
On any GAP, apply REC3/REC4/REC5/REC6/REC7/REC8/REC10 and re-run before the checkpoint.

**Step 2b — Grounding check (KB)** · Owner: play · Depends on: Step 1
Assert the UX pattern choices are grounded in the KB — a matched learning on the
technology/architecture shelf, or a recorded gap proposal:

```
python3 scripts/check_kb_grounding.py --manifest <working>/draft/ux-manifest.yaml \
        --kb-root <knowledge_dir> --proposals-dir <working>/draft/proposals
```

**SE-12 (F12/C12):** `check_kb_grounding.py` exits 0 — the manifest's `choices` (the visual
core, the navigation pattern, the responsive strategy) each ground in a `technology/*` or
`architecture/*` learning that resolves on a shelf, or a recorded proposal that exists; none
invented from the model's taste.
On any GAP, apply REC12 (search the KB and ground the choice, or raise a proposal) and re-run.

### Phase: Checkpoint (mandatory — never skipped, C11)

**Step 3 — Human review** · Owner: play · Depends on: Step 2
Present the proposed ux lens **inline** as the validation surface for the shape — the screens
with their purpose and low-fidelity layout, the states each holds, and the visual core
(palette + typography), each screen with the functionality or persona/journey it came from,
and the visual-core decision — for the human to confirm the shape and ICE under it hold. This
checkpoint is always presented and never skippable. Approve → continue to persist; cancel →
halt with nothing written to the model.
**SE-11 (F11/C11):** the lens is persisted only after this approval — Step 4 is the sole
writer and depends on this step; no product-model file is written before Step 4.

### Phase: Apply

**Step 4 — Persist** · Owner: play · Depends on: Step 3
First **snapshot** the slice's lens folder, the slice record, and the profile (so Step 5 can
prove only the ux lens changed). The snapshot is taken here, at the gated apply step — never
at pre-flight — so a resume can never compare post-apply against post-apply. `<slice_dir>` is
`<product_base>/product-os/<domain>/slices/<slice-id>` (the folder beside the record):

```
mkdir -p <working>/slice-before
cp -R <slice_dir>/. <working>/slice-before/ 2>/dev/null || true   # lens/ + decisions/ (may not exist yet)
cp <product_base>/<slice_file> <working>/slice-record-before.yaml
cp <product_base>/product-os/profile.yaml <working>/profile-before.yaml
```

Then persist on the fixed allowlist. `apply_ux.py` writes the ux lens (overwrite — the
re-derive) and copies decisions skip-if-exists; it is handed only the draft, which holds only
the lens + decisions, so it cannot write the slice record, the ICE, the profile, or another
lens:

```
python3 scripts/apply_ux.py --draft <working>/draft --product-base <product_base> \
        --out-manifest <working>/apply-manifest.json
```

**SE-2 (F2/C2):** the apply manifest's `written` set holds only the slice's `lens/ux.yaml` and
`decisions/*.yaml`; its `refused` set is empty (the draft held nothing out of scope).

**Step 5 — Verify persisted** · Owner: play · Depends on: Step 4
Confirm the slice record is byte-identical (realize never writes it), then check the lens
folder against the snapshot:

```
cmp <working>/slice-record-before.yaml <product_base>/<slice_file>   # must match (F2/F9)
python3 scripts/check_ux.py --cap-before <working>/slice-before \
        --cap-dir <slice_dir> \
        --profile-before <working>/profile-before.yaml \
        --profile-after <product_base>/product-os/profile.yaml
```

**SE-9 (F9/C9):** the slice record and profile are byte-identical; within the slice folder
every file is byte-identical to its snapshot except `lens/ux.yaml`; no accepted decision was
edited in place; nothing was removed.
On any GAP, apply REC2/REC9 and re-run.

### Phase: Scenario Validation

**Step 6 — Scenario evals** · Owner: play · Depends on: Step 5
- **SCE-1 (S1 — product designer, first run):** the slice's `ux.yaml` exists with a
  `slice_ref` and non-empty screens/states/design_system; every other product-model file is
  byte-identical; the lens validates v1.
- **SCE-2 (S2 — product owner, validates the shape):** every functionality the slice bundles
  maps to at least one screen in the manifest; none unvisualized.
- **SCE-3 (S3 — ux researcher, grounded):** every screen names a functionality's ICE or
  persona/journey and the visual core names a decision that resolves; none ungrounded.
- **SCE-4 (S4 — design lead, just enough):** every screen has a purpose and a layout, states
  are enumerated, the visual core carries palette + typography, and there is no pixel-level or
  exhaustive component detail.
- **SCE-5 (S5 — architect, hub-only):** no quality/agentic/architecture/run lens of the slice
  was touched and no screen grounds on another lens.
- **SCE-6 (S6 — product owner, re-run):** a second run changes only the slice's `ux.yaml` (and
  possibly a new decision); the slice record, other lenses, ICE, and profile are
  byte-identical; no accepted decision edited in place.
- **SCE-7 (S7 — reviewer, the checkpoint):** the checkpoint showed the lens (screens, states,
  visual core) as the validation surface inline before any write.
- **SCE-8 (S8 — ux researcher, KB-grounded):** the manifest's `choices` block lists the visual
  core, navigation pattern, and responsive strategy, each grounded in a `technology/*` or
  `architecture/*` learning that resolves on a shelf, or a proposal file that exists;
  `check_kb_grounding.py` is clean.

## Recovery

| For | Trigger | Direction | Handoff |
|-----|---------|-----------|---------|
| F1 | the slice is absent, a functionality ICE is unresolved/thin, or the profile is not firmed | halt and route to /shape (to shape the slice) or /understand (to enrich + firm) before /ux | human |
| F2 | a write touched something beyond this slice's ux lens or a decision | revert the out-of-scope write; /ux writes only the slice's ux lens (and the decision) | autonomous |
| F3 | the lens carries content beyond the three blocks or the wrong shape | strip it back to the screens/states/design_system shape the schema requires (a11y belongs to the profile, flows to the build) | autonomous |
| F4 | an invented/ungrounded element | drop it, or re-tie the screen to one of the slice's functionalities' ICE or a persona/journey and the visual core to a recorded decision; never keep an invented element | autonomous |
| F5 | the lens over- or under-specifies (missing purpose/layout, unenumerated states, a design_system missing palette or typography) | add the missing piece and strip any over-specification | autonomous |
| F6 | a functionality of the slice is covered by no screen | add the screen(s) that visualize the missing functionality so the whole shape is validatable | autonomous |
| F7 | /ux read or depended on another lens | remove the dependency; /ux derives only from the slice's hub | autonomous |
| F8 | the visual core was set with no decision recorded | record the slice-level visual-core decision before persisting; reuse the existing product decision if one exists | autonomous |
| F9 | a non-lens/non-decision file changed, or an accepted decision was edited in place | restore it and re-apply only the ux lens (and the new decision), after a human confirms the restore | human |
| F10 | the lens or a decision fails v1 schema validation | re-emit the failing artifact to conform before the play completes | autonomous |
| F11 | the lens was persisted before the checkpoint was approved | revert the premature write and re-present the checkpoint; persist only after the human approves | human |
| F12 | a UX pattern choice (visual core, navigation, or responsive strategy) rests on neither a matched KB learning nor a recorded proposal | search the KB's technology/architecture shelves via kb-search and ground the choice in the best-fit learning, or raise a KB-learning-gap proposal; never keep a taste-only choice | autonomous |

## Pause and Resume

Steps run top to bottom. On entry, resolve config, run the readiness + hub gate, resolve the
target slice from the play argument or the in-progress draft, check the status marker, skip
completed steps, reset any in-progress step to pending, and continue. The pre-apply snapshot
is captured at Step 4 (the gated apply step) and preserved on resume, so the non-destructive
comparison always diffs against true pre-apply state. A fresh start with no marker runs
everything and creates the marker at Step 1.
