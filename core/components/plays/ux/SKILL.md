# ux

Take one shaped capability and write its **ux lens** — just enough to anchor the intended
experience and let the build figure the rest. The lens is three things and only three: the
**screens** the capability needs, each with a low-fidelity layout; the **states** each
screen can hold; and the product's **visual core** — color and typography. The screens make
every shaped slice of the capability visible, so a human can look at them and confirm the
shape and the ICE under it are right. /shape composes the capability's functionalities into
vertical slices; /ux makes those slices visible as screens a person can validate.
Accessibility is **not** restated in the lens — it lives in the product profile, and the
build connects it. Flows are not specified — the build derives them from the screens and the
journeys. Every screen is grounded in a shaped slice (and its functionality ICE) or a
persona/journey; the visual core is a deliberate choice recorded as a decision the whole
product references. It writes only the ux lens, plus the visual-core decision.

**Pipeline position: none.** /ux is a realization, model-building play. It opens no delivery
issue and cuts no branch, so the D2 pipeline-position rule injects neither a `start-change`
head nor a close sequence. It writes the persistent product model directly. It runs after
/shape (a capability must be shaped before its experience can be drawn), and by convention
after /quality in the realize sequence — but takes **no** dependency on the quality lens. It
reads the hub (the capability's ICE + the profile box) and the shape (its slices, personas,
journeys); never another lens.

## Role

You are the orchestrator. You own the workflow and the step order. You delegate the one
piece of judgment — turning the shaped slices, personas/journeys, and the product's visual
core into screens (with layouts) and their states — to the `product-os-keeper` agent via a
JSON contract over files on disk, and you run every mechanical part (the readiness gate, the
draft validation, the snapshot, the allowlisted persist, and the post-apply checks) through
bundled scripts. You never write the model YAML yourself, you never write the profile, the
slices, or another lens, and you never persist before the human approves the single
checkpoint (C11).

**Forbidden:** hand-writing lens/decision YAML; writing the ICE, the profile, the slices, or
another lens by any route; reading another realize lens to derive the screens (ux reads the
hub + the shape); restating accessibility in the lens (it lives in the profile); persisting
by any route other than `scripts/apply_ux.py`; persisting before Step 3 approval.

**Agent boundaries:**

| Agent | Domain | Skill it invokes | Phases |
|-------|--------|------------------|--------|
| `product-os-keeper` | Turn the capability's shaped slices (and their functionality ICE) and personas/journeys into low-fidelity screens (with layouts), the states each holds, and the product's visual core (color + typography) recorded as a decision; cover every slice-bound functionality | `author-ux-lens` | Draft |

`product-os-keeper` is the single **domain agent** this play uses (1 of the ≤5 budget).
No utility agents are needed — git/issue machinery is absent (position none).

## Workflow

### Phase: Draft

**Step 1 — Draft ux lens** · Owner: `product-os-keeper` · Depends on: pre-flight
The agent invokes `author-ux-lens` to read the capability's rich ICE, its shaped slices (and
the functionality ICE they reference), and its personas/journeys, then draft the screens
(with low-fidelity layouts), the states each holds, and the product's visual core — reusing
the product's visual-core decision if one exists, else drafting it — to STM, grounding every
screen and declaring the to-cover set in the manifest:

    {
      "task":    "from this capability's shaped slices + personas/journeys, draft the ux lens — screens (with low-fidelity layouts) that make every slice visible, the states each holds, and the product's visual core (color + typography) recorded as a decision; cover every slice-bound functionality so a human can validate the shape",
      "inputs":  { "capability_path": "<product_base>/product-os/<capability>",
                   "slices_dir":      "<product_base>/product-os/<domain>/slices",
                   "product_base":    "<product_base>" },
      "outputs": { "draft_dir":  "<working>/draft/",
                   "lens":       "<working>/draft/.../lens/ux.yaml",
                   "manifest":   "<working>/draft/ux-manifest.yaml" }
    }

The skill reads the ICE + profile + slices + personas/journeys **read-only** and never
another lens. It returns the contract with the output paths on disk — never inline content.
**SE-1 (F1/C1):** the readiness gate passed at pre-flight — the profile is `set` and the
capability is `active` with rich ICE; otherwise the run halted (REC1).

### Phase: Validate

**Step 2 — Validate the draft** · Owner: play · Depends on: Step 1
Run the ux validator over the draft before the checkpoint. It recomputes the to-cover set
independently from the live slices + capability dirs, so coverage cannot be faked (and if
the slices name functionalities none of which resolve under the capability, it fails loud
rather than passing on an empty set):

```
python3 scripts/validate_ux.py --draft <working>/draft \
        --manifest <working>/draft/ux-manifest.yaml \
        --capability-dir <product_base>/product-os/<capability> \
        --slices-dir <product_base>/product-os/<domain>/slices
```

**SE-3 (F3/C3):** the lens is the three blocks only — `content` has exactly
screens/states/design_system, each non-empty; no flows, no accessibility, no
gates/components/environments.
**SE-4 (F4/C4):** every screen grounds to a real source (slice/ICE or persona/journey) and
the visual core grounds on a decision that resolves; no element ungrounded.
**SE-5 (F5/C5):** just enough — every screen has a purpose and a low-fidelity layout, states
are enumerated per declared screen, and design_system carries a palette and typography.
**SE-6 (F6/C6):** coverage holds — the independently recomputed to-cover set equals the
manifest's `covers`, every slice-bound functionality is visualized by at least one screen,
and the tree was read correctly (no loud-guard trip).
**SE-7 (F7/C7):** no grounding source is another lens — ux read the hub + the shape only.
**SE-8 (F8/C8):** the visual core names a decision that resolves to a drafted record.
**SE-10 (F10/C10):** the lens and any decision carry their required v1 fields and valid
enums (`type: ux`).
On any GAP, apply REC3/REC4/REC5/REC6/REC7/REC8/REC10 and re-run before the checkpoint.

### Phase: Checkpoint (mandatory — never skipped, C11)

**Step 3 — Human review** · Owner: play · Depends on: Step 2
Present the proposed ux lens **inline** as the validation surface for the shape — the screens
with their purpose and low-fidelity layout, the states each holds, and the visual core
(palette + typography), each screen with the slice/ICE or persona/journey it came from, and
the visual-core decision — for the human to confirm the shape and ICE under it hold. This
checkpoint is always presented and never skippable. Approve → continue to persist; cancel →
halt with nothing written to the model.
**SE-11 (F11/C11):** the lens is persisted only after this approval — Step 4 is the sole
writer and depends on this step; no product-model file is written before Step 4.

### Phase: Apply

**Step 4 — Persist** · Owner: play · Depends on: Step 3
First **snapshot** the capability folder and the profile (so Step 5 can prove only the ux
lens changed and the profile was untouched). The snapshot is taken here, at the gated apply
step — never at pre-flight — so a resume can never compare post-apply against post-apply:

```
mkdir -p <working>/cap-before
cp -R <product_base>/product-os/<capability>/. <working>/cap-before/
cp <product_base>/product-os/profile.yaml <working>/profile-before.yaml
```

Then persist on the fixed allowlist. `apply_ux.py` writes the ux lens (overwrite — the
re-derive) and copies decisions skip-if-exists; it is handed only the draft, which holds only
the lens + decisions, so it cannot write the ICE, the profile, the slices, or another lens:

```
python3 scripts/apply_ux.py --draft <working>/draft --product-base <product_base> \
        --out-manifest <working>/apply-manifest.json
```

**SE-2 (F2/C2):** the apply manifest's `written` set holds only the capability's `lens/ux.yaml`
and `decisions/*.yaml`; its `refused` set is empty (the draft held nothing out of scope).

**Step 5 — Verify persisted** · Owner: play · Depends on: Step 4
Check what landed against the snapshots:

```
python3 scripts/check_ux.py --cap-before <working>/cap-before \
        --cap-dir <product_base>/product-os/<capability> \
        --profile-before <working>/profile-before.yaml \
        --profile-after <product_base>/product-os/profile.yaml
```

**SE-9 (F9/C9):** every file in the capability folder is byte-identical to its snapshot
except `lens/ux.yaml`; the profile is byte-identical; no accepted decision was edited in
place; nothing was removed.
On any GAP, apply REC2/REC9 and re-run.

### Phase: Scenario Validation

**Step 6 — Scenario evals** · Owner: play · Depends on: Step 5
- **SCE-1 (S1 — product designer, first run):** `ux.yaml` exists with non-empty
  screens/states/design_system; every other product-model file is byte-identical; the lens
  validates v1.
- **SCE-2 (S2 — product owner, validates the shape):** every slice-bound functionality of the
  capability maps to at least one screen in the manifest; none unvisualized.
- **SCE-3 (S3 — ux researcher, grounded):** every screen names a slice/ICE or persona/journey
  and the visual core names a decision that resolves; none ungrounded.
- **SCE-4 (S4 — design lead, just enough):** every screen has a purpose and a layout, states
  are enumerated, the visual core carries palette + typography, and there is no pixel-level or
  exhaustive component detail.
- **SCE-5 (S5 — architect, hub-only):** no quality/agentic/architecture/run lens was touched
  and no screen grounds on another lens.
- **SCE-6 (S6 — product owner, re-run):** a second run changes only `ux.yaml` (and possibly a
  new decision); other lenses, ICE, profile, and slices are byte-identical; no accepted
  decision edited in place.
- **SCE-7 (S7 — reviewer, the checkpoint):** the checkpoint showed the lens (screens, states,
  visual core) as the validation surface inline before any write.

## Recovery

| For | Trigger | Direction | Handoff |
|-----|---------|-----------|---------|
| F1 | the capability is absent, not `active`, lacks rich ICE, or the profile is not firmed | halt and route to /shape (to shape it) or /understand (to enrich + firm) before /ux | human |
| F2 | a write touched something beyond this capability's ux lens or the visual-core decision | revert the out-of-scope write; /ux writes only the ux lens (and the decision) | autonomous |
| F3 | the lens carries content beyond the three blocks or the wrong shape | strip it back to the screens/states/design_system shape the schema requires (a11y belongs to the profile, flows to the build) | autonomous |
| F4 | an invented/ungrounded element | drop it, or re-tie the screen to a shaped slice/ICE or a persona/journey and the visual core to a recorded decision; never keep an invented element | autonomous |
| F5 | the lens over- or under-specifies (missing purpose/layout, unenumerated states, a design_system missing palette or typography) | add the missing piece and strip any over-specification | autonomous |
| F6 | a shaped slice's functionality is covered by no screen | add the screen(s) that visualize the missing functionality so the whole shape is validatable | autonomous |
| F7 | /ux read or depended on another lens | remove the dependency; /ux derives only from the ICE + profile + shape | autonomous |
| F8 | the visual core was set with no decision recorded | record the capability-level visual-core decision before persisting; reuse the existing product decision if one exists | autonomous |
| F9 | a non-lens/non-decision file changed, or an accepted decision was edited in place | restore it and re-apply only the ux lens (and the new decision), after a human confirms the restore | human |
| F10 | the lens or a decision fails v1 schema validation | re-emit the failing artifact to conform before the play completes | autonomous |
| F11 | the lens was persisted before the checkpoint was approved | revert the premature write and re-present the checkpoint; persist only after the human approves | human |

## Pause and Resume

Steps run top to bottom. On entry, resolve config, run the readiness gate, resolve the target
capability from the play argument or the in-progress draft, check the status marker, skip
completed steps, reset any in-progress step to pending, and continue. The pre-apply snapshot
is captured at Step 4 (the gated apply step) and preserved on resume, so the non-destructive
comparison always diffs against true pre-apply state. A fresh start with no marker runs
everything and creates the marker at Step 1.
