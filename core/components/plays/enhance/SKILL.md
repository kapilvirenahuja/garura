# enhance

Given an existing open issue describing an enhancement, gather context from the user via targeted Q&A, assemble a context package from LTM/KB/codebase, assess whether the scope is right-sized for this play (redirect to fix-it if too small, /prepare if too big), design an approach with tasks and self-evaluation criteria, implement autonomously, have a judge independently rate solution confidence, and ship via PR with full review. Two approval gates: a configurable mid-checkpoint after approach design (default OFF), and a mandatory PR checkpoint after review.

## Role

You are the orchestrator. You own the workflow. You delegate domain tasks to agents via JSON contracts — never execute domain work directly.

**Forbidden:** Direct `git`, `gh`, or codebase analysis commands. All domain work goes through agents (C11).

**Agent boundaries:**

| Agent | Type | Domain | Phases |
|-------|------|--------|--------|
| `project-orchestrator` | domain | Issues: validate issue exists and is open | Pre-flight |
| `tech-designer` | domain | Context assembly, approach design | Preparation |
| `code-builder` | domain | Implement from approach + context | Execution |
| `judge` | domain | Independent solution confidence rating | Execution |
| `quality-auditor` | domain | Quality gates (build/lint/typecheck/tests) | Execution |
| `repo-orchestrator` | utility | Branch setup, commit, PR, merge, evidence | Pre-flight, Finalize, Evidence |

Domain agents: 5 (project-orchestrator, tech-designer, code-builder, judge, quality-auditor). Within the ≤5 budget.
Utility agents: 1 (repo-orchestrator). Exempt from budget.

## Arguments

```
/enhance --issue <issue-number> [--approve-plan]
```

- `--issue` — required. The GitHub issue number describing the enhancement.
- `--approve-plan` — optional flag. When set, enables the mid-checkpoint (C9) after approach design. Default: OFF.

## Pre-flight

Execute these checks before any domain work:

| Check | Constraint | Action on Failure |
|-------|-----------|-------------------|
| Resolve `stm_base` from `.meridian/core/config.yaml` | — | Hard halt — config is required |
| Resolve `product_base` from `.meridian/core/config.yaml` | C5 | Soft — set `has_product_ltm=false`, continue without product LTM |
| Issue number resolvable | C1 | Hard halt — "No issue provided. Provide an issue number or run from an enhance/ branch." |
| Issue exists and is open | C1 | Hard halt — "Issue #{n} does not exist or is not open." |
| Current branch guard | C2 | Create enhance/{issue}-{slug} branch via repo-orchestrator |

```bash
# Resolve STM base path
stm_base=$(grep '^\s*base-path:' .meridian/core/config.yaml | head -1 | awk '{print $2}')

# C5: Resolve product base path for LTM grounding
product_base=$(grep -A1 '^product:' .meridian/core/config.yaml | grep 'base-path' | awk '{print $2}')
has_product_ltm=false
if [ -d "${product_base}architecture/" ]; then
  has_product_ltm=true
fi

# C1: Extract issue number from branch name or arguments
branch=$(git branch --show-current)
issue=$(echo "$branch" | grep -oE '/[0-9]+' | tr -d '/')
# If no issue from branch, expect it from user input
# If no issue at all → hard halt

# C2: Branch guard — determine if on main/master/default
default_branch=$(git remote show origin | grep 'HEAD branch' | awk '{print $NF}')
# If on default branch → need to create enhance/ branch
```

**Resume check:** If `{stm_base}/{issue}/status/enhance.json` exists, resume — skip completed steps, reset any `in_progress` to pending, continue from first incomplete.

### Phase: Preparation — Scope Gate

**Step 5 — Scope Gate**
Owner: play (direct — assessment logic)
Depends on: Step 4

Read `context/understanding.md`. Assess scope using C6 thresholds:
- Count estimated files to touch
- Count domains crossed
- Count new abstractions
- Count tests impacted
- Estimate effort

Present assessment:

```markdown
## Scope Assessment — Issue #{issue}

| Signal | Value | Range |
|--------|-------|-------|
| Files touched | {n} | ≤3 → fix-it / 4-15 → enhance / >15 → /prepare |
| Domains crossed | {n} | 1 → fix-it eligible / 1-2 → enhance / 3+ → /prepare |
| New abstractions | {n} | 0 → fix-it eligible / 0-2 → enhance / new bounded context → /prepare |
| Tests impacted | ~{n} | ≤5 → fix-it eligible / 6-30 → enhance / >30 → /prepare |
| Estimated effort | {estimate} | <1hr → fix-it / 1hr-1day → enhance / >1day → /prepare |

**Recommendation:** This enhancement is {too small for enhance — use fix-it / right-sized / too large — use /prepare}.
```

If redirecting → halt with recommendation. If right-sized → continue.

**Step 5 Eval:**
- **SE-6 (C6/F4):** Scope assessment is recorded in STM with all five signals evaluated: files touched, domains crossed, new abstractions, tests impacted, and estimated effort. When any signal places the enhancement out-of-range (files > 15, domains >= 3, effort > 1 day, or new bounded context), the scope assessment table is presented to the user with a recommendation to use /prepare or fix-it, and the play halts — no approach.yaml is created and no implementation is attempted.


- **SE-11 (C12/F3):** When self-eval finds failures, the fix loop runs at most 3 iterations. self-eval.yaml records the iteration count. - Read `domain_assessment.responsible_domain` to identify which agent can fix it
- Invoke the responsible agent with fix context + the original intent

## Pause and Resume

Steps are in execution order — run top to bottom.

**Issue detection:** Extract issue number from branch name during pre-flight (e.g., `enhance/42-add-scope-gate` → `42`). If not on an enhance/ branch, issue comes from user input.

**Status file:** `{stm_base}/{issue}/status/enhance.json`

```json
{
  "play": "enhance",
  "issue": "{issue}",
  "started_at": "{ISO timestamp}",
  "tasks": {
    "validate-issue": { "status": "completed", "completed_at": "..." },
    "create-branch": { "status": "completed", "completed_at": "..." },
    "qa-discovery": { "status": "in_progress", "started_at": "..." },

    "approach-design": { "status": "pending" },
    "mid-checkpoint": { "status": "pending" },

    "self-eval": { "status": "pending" },
    "judge-rating": { "status": "pending" },
    "judge-gate": { "status": "pending" },
    "quality-check": { "status": "pending" },
    "commit-pr": { "status": "pending" },
    "pr-review": { "status": "pending" },
    "pr-checkpoint": { "status": "pending" },
    "ship": { "status": "pending" },
    "evidence": { "status": "pending" }
  }
}
```

**Executor loop:**
```
resolve issue number from branch name or user input
check for status file at {stm_base}/{issue}/status/enhance.json

for each step in compiled order:
  if status file shows step "completed" -> skip
  if status file shows step "in_progress" -> reset to pending
  mark step "in_progress" in status file
  execute step
  mark step "completed" in status file
```

**Fresh start:** No status file → execute all steps, create status file on first step.
**Resume:** Status file exists → skip completed, continue from first incomplete.
**Pause:** User terminates session. Status file reflects last completed step.
