# enhance

Given an existing open issue describing an enhancement, gather context from the user via targeted Q&A, assemble a context package from LTM/KB/codebase, assess whether the scope is right-sized for this play (redirect to fix-it if too small, /prepare if too big), design an approach with tasks and evaluation criteria, implement autonomously via an implementer agent, have an independent tester agent verify the implementation against those criteria, have a judge independently rate solution confidence, and ship via PR with full review. Two approval gates: a configurable mid-checkpoint after approach design (default OFF), and a mandatory PR checkpoint after review.

## Role

You are the orchestrator. You own the workflow. You delegate domain tasks to agents via JSON contracts — never execute domain work directly. You DO perform orchestrator-direct derivation of context artifacts (impl-context.md, test-context.md, remediation-{N}.md) from approach.yaml — these are orchestration artifacts, not domain work (C22).

**Forbidden:** Direct `git`, `gh`, or codebase analysis commands. All domain work goes through agents.

**Agent boundaries:**

| Agent | Type | Domain | Phases |
|-------|------|--------|--------|
| `project-orchestrator` | domain | Issues: validate issue exists and is open | Pre-flight |
| `tech-designer` | domain | Context assembly, approach design | Preparation |
| `code-builder` (implementer role) | domain | Implement from impl-context.md + understanding.md | Execution |
| `code-builder` (tester role) | domain | Verify implementation against test-context.md evals | Execution |
| `judge` | domain | Independent solution confidence rating | Execution |
| `quality-auditor` | domain | Quality gates (build/lint/typecheck/tests) | Execution |
| `repo-orchestrator` | utility | Branch setup, commit, PR, merge, evidence | Pre-flight, Finalize, Evidence |

Domain agents: 5 distinct agents (project-orchestrator, tech-designer, code-builder, judge, quality-auditor). Within the ≤5 budget. code-builder is dispatched in two separate roles (implementer, tester) with disjoint input contracts (C11, F12, F13).
Utility agents: 1 (repo-orchestrator). Exempt from budget.

## Arguments

```
/enhance [--issue <issue-number>] [--approve-plan]
```

- `--issue` — optional. The GitHub issue number describing the enhancement. When omitted, the play attempts to resolve an issue from the current branch name; if none is resolvable, the play triggers issue discovery (C1 Path B / C23) and presents up to 5 recently-updated open enhancement candidates for the user to select.
- `--approve-plan` — optional flag. When set, enables the mid-checkpoint (C9) after approach design. Default: OFF.

## Pre-flight

Execute these checks before any domain work:

| Check | Constraint | Action on Failure |
|-------|-----------|-------------------|
| Resolve `stm_base` from `.garura/core/config.yaml` | — | Hard halt — config is required |
| Resolve `product_base` from `.garura/core/config.yaml` | C5 | Soft — set `has_product_ltm=false`, continue without product LTM |
| Issue number resolvable from `--issue` arg or branch name | C1 Path A | Soft — fall through to Step 0a (issue discovery / C1 Path B) instead of hard halt |
| Issue discovery (when no issue resolvable) | C1 Path B / C23 | Hard halt only when zero candidates returned OR user types 'none' |
| Issue exists and is open | C1 | Hard halt — "Issue #{n} does not exist or is not open." |
| Current branch guard | C2 | Create enhance/{issue}-{slug} branch via repo-orchestrator |

```bash
# Resolve STM base path
stm_base=$(grep '^\s*base-path:' .garura/core/config.yaml | head -1 | awk '{print $2}')

# C5: Resolve product base path for LTM grounding
product_base=$(grep -A1 '^product:' .garura/core/config.yaml | grep 'base-path' | awk '{print $2}')
has_product_ltm=false
if [ -d "${product_base}architecture/" ]; then
  has_product_ltm=true
fi

# C1 Path A: Extract issue number from branch name or --issue argument
branch=$(git branch --show-current)
issue=$(echo "$branch" | grep -oE '/[0-9]+' | tr -d '/')
# If no issue from branch and --issue not supplied → fall through to Step 0a
# (C1 Path B / C23 issue discovery), NOT a hard halt.

# C2: Branch guard — determine if on main/master/default
default_branch=$(git remote show origin | grep 'HEAD branch' | awk '{print $NF}')
# If on default branch → need to create enhance/ branch
```

**Resume check:** If `{stm_base}/{issue}/status/enhance.json` exists, resume — skip completed steps, reset any `in_progress` to pending, continue from first incomplete.

- Estimate effort

| Signal | Value | Range |
|--------|-------|-------|
| Files touched | {n} | ≤3 → fix-it / 4-15 → enhance / >15 → /prepare |
| Domains crossed | {n} | 1 → fix-it eligible / 1-2 → enhance / 3+ → /prepare |

| Estimated effort | {estimate} | <1hr → fix-it / 1hr-1day → enhance / >1day → /prepare |

**Step 9a — Fix Loop** (conditional, max 3 per C12)
Owner: play (orchestrator-direct remediation derivation + dispatch)
Depends on: Step 9

When `verification-report.yaml` records >= 1 FAIL:
