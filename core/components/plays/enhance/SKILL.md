# enhance

Given an existing open issue describing an enhancement, gather context from the user via targeted Q&A, assemble a context package from LTM/KB/codebase, assess whether the scope is right-sized for this play (redirect to fix-it if too small, /prepare if too big), design an approach with tasks and self-evaluation criteria, implement autonomously, have a judge independently rate solution confidence, and ship via PR with full review. Two approval gates: a configurable mid-checkpoint after approach design (default OFF), and a mandatory PR checkpoint after review.

## Pre-flight

Execute these checks before any domain work:

| Check | Constraint | Action on Failure |

| Resolve `stm_base` from `.garura/core/config.yaml` | ÔÇö | Hard halt ÔÇö config is required |
| Resolve `product_base` from `.garura/core/config.yaml` | C5 | Soft ÔÇö set `has_product_ltm=false`, continue without product LTM |
| Issue number resolvable | C1 | Hard halt ÔÇö "No issue provided. Provide an issue number or run from an enhance/ branch." |
| Issue exists and is open | C1 | Hard halt ÔÇö "Issue #{n} does not exist or is not open." |
| Current branch guard | C2 | Create enhance/{issue}-{slug} branch via repo-orchestrator |

```bash
# Resolve STM base path
stm_base=$(grep '^\s*base-path:' .garura/core/config.yaml | head -1 | awk '{print $2}')

# C5: Resolve product base path for LTM grounding
product_base=$(grep -A1 '^product:' .garura/core/config.yaml | grep 'base-path' | awk '{print $2}')
has_product_ltm=false

  has_product_ltm=true
fi

# C1: Extract issue number from branch name or arguments

# If no issue at all Ôćĺ hard halt

# If on default branch Ôćĺ need to create enhance/ branch
```

**Resume check:** If `{stm_base}/{issue}/status/enhance.json` exists, resume ÔÇö skip completed steps, reset any `in_progress` to pending, continue from first incomplete.

### Phase: Preparation ÔÇö Scope Gate

**Step 5 ÔÇö Scope Gate**
Owner: play (direct ÔÇö assessment logic)
Depends on: Step 4

Read `context/understanding.md`. Assess scope using C6 thresholds:

- Count domains crossed
- Count new abstractions
- Count tests impacted
- Estimate effort

```markdown
## Scope Assessment ÔÇö Issue #{issue}

| Signal | Value | Range |
|--------|-------|-------|
| Files touched | {n} | ÔëĄ3 Ôćĺ fix-it / 4-15 Ôćĺ enhance / >15 Ôćĺ /prepare |
| Domains crossed | {n} | 1 Ôćĺ fix-it eligible / 1-2 Ôćĺ enhance / 3+ Ôćĺ /prepare |
| New abstractions | {n} | 0 Ôćĺ fix-it eligible / 0-2 Ôćĺ enhance / new bounded context Ôćĺ /prepare |
| Tests impacted | ~{n} | ÔëĄ5 Ôćĺ fix-it eligible / 6-30 Ôćĺ enhance / >30 Ôćĺ /prepare |
| Estimated effort | {estimate} | <1hr Ôćĺ fix-it / 1hr-1day Ôćĺ enhance / >1day Ôćĺ /prepare |

**Recommendation:** This enhancement is {too small for enhance ÔÇö use fix-it / right-sized / too large ÔÇö use /prepare}.
```

**Step 5 Eval:**
- **SE-6 (C6/F4):** Scope assessment is recorded in STM with all five signals evaluated: files touched, domains crossed, new abstractions, tests impacted, and estimated effort.
