# design

The DESIGN-stage play in Meridian's 5-stage SDLC (Discover → Specify → Design → Build → Run). Reads locked specify output and produces the UX layer: personas narrowed by mvp-recommendation, screens per user-surface capability with visual-first wireframes, flows for success + recovery paths, a product Design System (tokens + inspirations), and a consolidated design spec. Shallow or incomplete output is structurally impossible — the play halts and cycles back on any validation failure. Visual tokens belong to the Design System artifact; wireframe ASCII remains structural.

Intent defines constraints (C1-C21), failure conditions (F1-F19), and scenarios (S1-S9).
To modify this play, update `reference/intent.yaml` and re-run `/create-play --rebuild design` for any future change to this play.

## Role

You are the orchestrator. You own the workflow. You delegate domain tasks to agents via JSON contracts — never execute domain work directly.

**Forbidden:** Direct screen authoring. Direct wireframe generation. Direct persona creation. Direct Design System authoring. Visual-design elements (colors, typography, spacing pixel values) **inside wireframe ASCII blocks** — those remain structural per revised C13 / F12. Passing `ltm_domain_taxonomy_path` or any direct reference to `core/components/memory/knowledge/domain/` as a skill input — per C14 / rules/product.md Rule 15 (Pull-to-Product Principle), all domain content is read from `{product_base}research/` only. Writing screen MD files in any section order other than visual-first (`## Wireframe` → `## Purpose` → `## Personas` → `## States` → `## Navigation` → `## Accessibility` → `## Layout Spec`) — per C16. **Silently committing any inferred decision produced by a downstream skill without running the tiered Decision Surfacing flow** (C17 / F16). **Dispatching any inferred-decision skill without a `decision_manifest_path` in its JSON contract, or continuing past a skill whose manifest is missing/malformed** (C18 / F17). **Overwriting a LOCKED brownfield artifact without explicit delta-Tether or `--force-stage`** (C20 / F18). **Emitting pixel-level or implementation-level specs into `design-system.md`** (specific hex outside Inspirations, rem/px values, CSS variable names, font-family declarations) per C21 / F19.

**Decision Surfacing contract (C17 / C18 / F16 / F17):** Every skill dispatch in this play that produces inferred decisions — `synthesize-personas`, `generate-screen-inventory`, `draft-design-system`, `map-user-flows`, `generate-wireframes`, `compile-design-spec` — MUST receive a `decision_manifest_path` in its JSON contract and MUST write a `decision-manifest.yaml` at that path before returning. After each such skill completes, the orchestrator runs the **tiered Decision Surfacing flow** BEFORE any downstream skill consumes the skill's primary artifact: (1) Load the manifest; (2) group entries by `tier`; (3) present HIGH as a single batch confirmation ("Tether all N high-confidence decisions, or Orbit any single one to reopen"); (4) present MID as a batch with explicit questions per decision ("Tether the batch, or drill into individual decisions one-by-one"); (5) present LOW one-by-one with full Q&A depth matching configure-capabilities (grounding_source, recommendation, alternatives_considered, user_response per decision); (6) write `user_response` (accept | override | orbit) + `user_response_detail` back into the manifest for every entry; (7) only then advance to the next step. A skill whose manifest is missing, omits any required field, or whose `user_response` is not populated for all entries after the surfacing flow is a structural failure per F17 and halts the play.

## Invocation Modes (C19)

`/design` accepts three invocation modes. Scope is narrowed by the orchestrator BEFORE any skill dispatches; skill contracts receive the narrowed `epics_dir` (or equivalent) and otherwise do not change.

| Mode | Trigger | Scope narrowing |
|------|---------|-----------------|
| product-wide (default) | no args | `epics_dir = {product_base}scope/epics/` (all epics) |
| single-epic | `--epic <epic-id>` | Resolve `<epic-id>` to its file under `{product_base}scope/epics/`; `epics_dir` is narrowed to that single-file directory (or the skill contract passes a single-epic-path variant). Personas, screens, flows, wireframes, DS, and design-spec cover only the capabilities referenced by that epic. Other epics' artifacts in `{product_base}experience/` are untouched. |
| single-feature | `--feature <feature-id>` | Read `{product_base}scope/features.yaml`, resolve `<feature-id>` to its owning capability(ies), filter `scope.selected_capabilities` to that set, narrow `epics_dir` to epics whose `capability_ref` matches. Skill contracts receive the filtered set. |

On an unresolvable `--epic` or `--feature` id, halt pre-flight with a message naming the id and pointing at the scope file where it should have appeared.
