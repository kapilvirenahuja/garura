The Bet

Meridian OS bets that intent-driven development will replace ad-hoc AI prompting. By giving AI assistants persistent project memory (SG1) and structured, auditable workflows (SG2), engineering teams will stop babysitting AI and start delegating to it. If we build the operating layer that makes every AI session start informed rather than blank, we capture the workflow layer that Copilot, Cursor, and general-purpose frameworks have left open. The five epics — from foundational memory and play execution to artifact generation, product discovery, and zero-friction onboarding — form the progressive stack that proves this thesis.

The Story

Today, software teams using AI assistants waste enormous time on context switching, prompt engineering, and re-explaining project state to every new session. Engineers spend 10-15 minutes per session re-establishing context that should have persisted. Development operations like feature planning, code review, and shipping are ad-hoc — each engineer prompts differently, producing inconsistent results with no audit trail.

Meridian OS addresses this by building a three-layer stack. First, a persistent memory layer (F1) ensures AI sessions start with full project intent, constraints, and recent decisions already loaded. Second, a play execution engine (F2) turns common development operations into structured, repeatable workflows that delegate to specialized agents — producing auditable evidence of what was done and why. Third, an artifact generation pipeline (F3) produces audience-separated documents from a single source of truth, ending the manual synchronization tax across product, engineering, and stakeholder docs.

On top of this foundation, the product discovery flow (F4) gives teams a structured path from idea to validated product vision within a single session, while zero-friction onboarding (F5) ensures adoption requires nothing more than a git clone and a single sync command. The result is a system where AI assistants have durable context, workflows are deterministic, and artifacts trace back to shared intent.

The strategic bet is that teams willing to adopt structured workflows — if the friction is low enough — will produce measurably better AI output than those relying on ad-hoc prompting. Meridian OS is built to prove that thesis.

Assumptions

  • Claude Code will remain the primary execution environment for at least 12 months
  • Teams are willing to adopt structured workflows if the friction is low enough
  • Intent-driven development produces measurably better AI output than ad-hoc prompting
  • The play/agent/skill hierarchy is sufficient to model most development workflows

Exclusions

  • Multi-provider support — Focus on Claude Code first; other LLM providers are future scope
  • Visual IDE plugins — CLI-first approach; IDE integration is downstream
  • Team collaboration features — Single-developer workflow first; team features after core loop is proven
