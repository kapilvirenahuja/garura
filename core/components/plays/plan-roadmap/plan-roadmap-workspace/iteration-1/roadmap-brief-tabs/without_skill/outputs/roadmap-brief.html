Meridian OS LOCKED

Roadmap Brief — Intent-Driven AI Development Operating Layer
Meridian OS is the operating layer for intent-driven AI development. It gives AI assistants persistent project memory, structured workflows (plays), and a shared understanding of intent — so every AI session starts informed, not blank. Teams stop babysitting AI and start delegating to it.

Problem

Software teams using AI assistants waste enormous time on context switching, prompt engineering, and re-explaining project state to every new AI session. There is no durable, structured way for an AI to understand a project's intent, constraints, and current state.

Near-Term
F1 Persistent Project Memory P1 Foundation
Enable AI assistants to retain full project intent, constraints, and decisions across sessions through structured short-term and long-term memory systems.
Today (Problem)
Every AI session starts from zero. Engineers waste 10-15 minutes per session re-explaining project context, architecture decisions, and current state. There is no persistent memory layer between sessions.
After (Outcome)
AI assistants will automatically load project intent, active constraints, and recent decisions at session start. Engineers begin working immediately without context re-entry.
Why Now
This directly enables SG1 (Persistent Context) and is foundational — every other epic depends on the memory layer existing.
Dependencies: None

Success Scenarios

Given: A project with prior work tracked in STM
When: Engineer starts a new Claude Code session
Then: AI loads project context including active issue, recent decisions, and constraints within 5 seconds
Given: An engineer completes a play that produces artifacts
When: A new session starts the next day
Then: AI can reference yesterday's artifacts by path and summarize their content without re-reading

Failure Conditions

AI session starts with no project context loaded despite STM directory existing
Memory loading takes more than 30 seconds, causing engineer to skip it
Memory from one project leaks into another project's session
F2 Play Execution Engine P1 Foundation
Build the core play execution engine that runs plays with agent delegation, checkpoint handling, and evidence collection.
Today (Problem)
Development operations like feature planning, code review, and shipping are currently ad-hoc. Each engineer prompts AI differently, producing inconsistent results with no audit trail.
After (Outcome)
Common development operations execute as structured plays — repeatable, auditable, and delegating to specialized agents. Every play produces evidence of what was done and why.
Why Now
This enables SG2 (Structured Workflows) and provides the execution backbone for all downstream features.
Dependencies: F1

Success Scenarios

Given: A play with 5 steps and 2 agent calls
When: Engineer invokes the play
Then: All steps execute in order, agents produce artifacts, evidence is written, and the play completes with a summary
Given: A play is interrupted mid-execution
When: Engineer re-invokes the play
Then: Execution resumes from the last incomplete step without re-running completed steps

Failure Conditions

Play silently skips a step without logging why
Agent call fails but play continues as if it succeeded
Evidence file is missing or contains only a stub after play completion
F3 Artifact Generation Pipeline P2
Implement the skill-based artifact generation system where specialized skills produce YAML artifacts and HTML briefs from structured inputs.
Today (Problem)
Product, engineering, and stakeholder artifacts are maintained manually in parallel. Changes in one doc aren't reflected in others. Teams waste hours keeping docs in sync.
After (Outcome)
Artifacts are generated from a single source of truth by specialized skills. Product spec, architecture, scenarios, and plans each have a structured YAML + HTML brief pair, all derived from the same product intent.
Why Now
This enables SG3 (Audience-Separated Artifacts) — each audience gets their view without manual translation.
Dependencies: F1 F2

Success Scenarios

Given: A locked product.yaml exists
When: Engineer runs /prepare-implementation
Then: 5 YAML artifacts and 5 HTML briefs are generated, each conforming to their schema
Given: A brief is generated
When: Reviewer opens it in a browser
Then: Tabbed layout renders, inline comments work, export produces valid JSON

Failure Conditions

Generated YAML artifact is missing required schema fields
HTML brief has external CDN dependencies or broken JavaScript
Artifact references a field that doesn't exist in the source YAML
Mid-Term
F4 Product Discovery Flow P2
Implement the /discover-product play flow from problem statement to locked product.yaml with market context, vision, and HTML brief review.
Today (Problem)
Starting a new product requires extensive manual research, document creation, and stakeholder alignment. There is no structured path from idea to validated product vision.
After (Outcome)
Engineers run /discover-product with a problem statement and get a structured product.yaml with market context, vision, strategic goals, and an HTML brief for stakeholder review — all within a single session.
Why Now
This enables SG4 (Progressive Enrichment) — the product artifact starts with discovery and enriches through downstream plays.
Dependencies: F2 F3

Success Scenarios

Given: Engineer has a product idea described in 1-2 sentences
When: They run /discover-product --phase draft
Then: product.yaml is written with market context, vision, 3+ strategic goals, and a product-brief.html is generated for review
Given: product.yaml exists in DRAFT status
When: Engineer runs /discover-product --phase lock
Then: Status changes to LOCKED and downstream plays can consume it

Failure Conditions

product.yaml has fewer than 3 strategic goals
product-brief.html has no feedback mechanism (missing inline comments)
LOCK phase silently overwrites content instead of only changing status
Long-Term
F5 Zero-Friction Onboarding P3
Ensure new projects can adopt Meridian OS with minimal setup — sync components, configure paths, and run first play within 5 minutes.
Today (Problem)
Adopting new development tools typically requires infrastructure changes, team training, and workflow migration. Engineers resist tools that add setup overhead.
After (Outcome)
A new project runs /sync-claude and immediately has access to all plays, skills, and agents. First play execution happens within 5 minutes of setup.
Why Now
This enables SG5 (Zero-Friction Adoption) — the barrier to entry is near zero.
Dependencies: F1 F2

Success Scenarios

Given: A fresh git repository with no Meridian configuration
When: Engineer clones meridian-os and runs /sync-claude
Then: All skills and agents are deployed to ~/.claude/, and running /discover-product works immediately
Given: Engineer has Meridian configured globally
When: They open a new project
Then: Plays are available without any project-specific setup

Failure Conditions

Setup requires more than 3 manual steps beyond git clone
/sync-claude overwrites user's existing ~/.claude/ customizations
First play fails because a config path was not resolved correctly

Risks

Claude Code API changes could break play execution model
Adoption requires teams to commit to structured workflows over ad-hoc prompting
Competing AI providers may build similar workflow layers natively

Assumptions

Claude Code will remain the primary execution environment for at least 12 months
Teams are willing to adopt structured workflows if the friction is low enough
Intent-driven development produces measurably better AI output than ad-hoc prompting
The play/agent/skill hierarchy is sufficient to model most development workflows

Out of Scope

Multi-provider support
Focus on Claude Code first; other LLM providers are future scope
Visual IDE plugins
CLI-first approach; IDE integration is downstream
Team collaboration features
Single-developer workflow first; team features after core loop is proven
