# garura:refactor

Given a target scope (files, modules, or functions), a stated smell or restructuring
goal, explicit constraints, an out-of-scope list, and a time-box, produce a series of
atomic, behaviour-preserving refactor commits. The play establishes a green baseline,
pins existing behaviour with characterization tests where coverage is thin, maps blast
radius, presents a numbered plan for human approval, then executes each refactor move
one commit at a time with a green gate after every commit. The public API surface,
dependencies, features, and bug behaviour are preserved exactly unless the user has
explicitly authorised changes. A final report captures baseline vs. final metrics,
structural changes, what did not change, and deferred follow-ups.

## Pre-flight

Execute these checks before any domain work:

| Check | Constraint | Action on Failure |
|-------|-----------|-------------------|
| Resolve `stm_base` from `.garura/core/config.yaml` | — | Hard halt — config required |
| Working tree is clean | C3 | Hard halt — instruct user to commit or stash first |
| Target scope provided | C1 | Hard halt — scope is required input |
| Extract issue from branch name | — | Hard halt if issue not resolvable |

```bash
# Resolve STM base path
stm_base=$(grep 'base-path' .garura/core/config.yaml | head -1 | awk '{print $2}')

# C3 — clean working tree
git status --porcelain
# Halt with message if output is non-empty:
# "Working tree is not clean. Commit or stash uncommitted changes before refactoring."

# Extract issue number from branch name (e.g., refactor/341-garura-refactor → 341)
branch=$(git branch --show-current)
issue=$(echo "$branch" | grep -oE '/[0-9]+' | tr -d '/')
# Halt if issue is empty

# Capture pre-play working-tree snapshot (for SE-11 eval)
git status --porcelain > {stm_base}/{issue}/evidence/refactor/pre-flight/working-tree-snapshot.txt
```

**Resume check:** If `{stm_base}/{issue}/status/garura-refactor.json` exists, resume —
skip completed steps, reset any `in_progress` to pending, continue from first incomplete.

## Recovery

Sourced from `reference/expectation.yaml` `recovery` — one entry per failure condition. When the validator detects a tripped failure, it builds a recovery handoff plan from the matching entry and routes it per `handoff`: `autonomous` loops the fix back to the builder; `human` escalates for a manual call.

| ID | For | Symptom (trigger) | Direction | Handoff |
|----|-----|-------------------|-----------|---------|
| REC1 | F1 | a commit leaves the suite red | revert the offending move; land it green before the history is presented | autonomous |
| REC2 | F2 | public API differs from baseline without authorisation | needs a user decision to authorise or revert — the builder cannot grant it | human |
| REC3 | F3 | a baseline-green test is missing/skipped/weakened | restore the test to baseline strength; revert the move that depended on weakening it | autonomous |
| REC4 | F4 | a commit mixes a behaviour change with refactor | split so the refactor stands alone; queue the behaviour part as a separate `fix:` | autonomous |
| REC5 | F5 | final test count below baseline | restore dropped tests so the count is at or above baseline | autonomous |
| REC6 | F6 | commits not atomic / wrong prefix | re-split and re-message so each commit is one move, correctly prefixed | autonomous |
| REC7 | F7 | thin coverage, no characterization tests before first refactor | author + commit `test:` characterization commits ahead of the first `refactor:` | autonomous |
| REC8 | F8 | new dependency/framework/pattern without authorisation | needs a user decision to authorise or remove — the builder cannot grant it | human |
| REC9 | F9 | out-of-scope edits in the diff | remove them from the diff; record them in FOLLOWUPS.md | autonomous |
| REC10 | F10 | execution began without the pre-edit report | produce + present the baseline result, blast-radius map, and plan before continuing | autonomous |
| REC11 | F11 | started on a red baseline or unclean tree | needs a user go/no-go to clean the tree or accept the state — the builder cannot decide | human |
| REC12 | F12 | final report missing a required section | fill the missing section from the session record before delivery | autonomous |

At Level 4, `intent-resolver` executes the autonomous entries (9 of 12) without a human; the human entries (REC2, REC8, REC11) always escalate.

```json
    "scenario-evals": { "status": "pending" },

  }

```

**Executor loop:**
```
resolve issue number from branch name
check for status file at {stm_base}/{issue}/status/garura-refactor.json

for each step in compiled order:
  if status file shows step "completed" → skip
  if status file shows step "skipped" → skip (conditional steps)
  if status file shows step "in_progress" → reset to pending (may not have finished)
  mark step "in_progress" in status file
  execute step
  mark step "completed" in status file
```

**Fresh start:** No status file → execute all steps, create status file on first step.
**Resume:** Status file exists → skip completed, continue from first incomplete.
**Execution loop resume:** `current_plan_step` in status file tracks which plan step was
last completed. On resume, restart from `current_plan_step + 1`. If the last step was
`in_progress`, revert its commit if it exists (commits are atomic — a partial step may
have produced a commit) and re-execute.
