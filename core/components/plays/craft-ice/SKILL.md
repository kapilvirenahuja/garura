# craft-ice

Given an epic or feature describing a substantial body of work, craft-ice interviews
the user and assembles the surrounding context to produce an **ICE-compliant
specification** — a clean **Intent** triple (goal, constraints, failure conditions), a
**Context** bundle (relevant memory, knowledge, and codebase grounding), and a
generated, approved **Expectation** (success scenarios + recovery). The specification
is approved at a checkpoint whose human-in-the-loop requirement is set by
configuration, after which an agent can select and run the right downstream play
(prepare or enhance) directly against the spec, without re-deriving the source. It is
the doorway from "we have an epic/feature" to "a downstream play has a complete,
checkable spec." It is for substantial work — quick, fix-it-scale changes are out of
scope and redirected.

## Role

You are the orchestrator. You own the workflow. You conduct the user-facing discovery
Q&A directly (a sub-agent cannot interview interactively), then delegate every domain
task to agents via JSON contracts — never execute their domain work directly.

**Agent boundaries (domain agents):**

| Agent | Domain | Phase |
|-------|--------|-------|
| `intent-crafter` | Intent authoring — applies the quality gate and writes the feature's clean triple via `author-intent-yaml`. Runs in **authoring mode**: the play pre-gathers the discovery Q&A; the agent does NOT interview. | Preparation |
| `tech-designer` | Context assembly — gathers LTM/KB/codebase grounding into the Context bundle (as in enhance). | Preparation |
| `expectation-crafter` | Expectation generation (**feature case**) — invokes `generate-feature-expectation` from the intent + Context bundle. | Preparation |

Domain agents: 3 (within the ≤5 budget). Utility agents: none — evidence/close is handled inline by the play.

## Pre-flight

Execute these checks before any domain work:

| Check | Constraint | Action on failure |
|-------|-----------|-------------------|
| PF1 — resolve `stm_base`, `product_base` from `.garura/core/config.yaml` | — | Hard halt — config required |
| PF2 — git repository present | — | Hard halt — not a git repo |
| PF3 — tracking issue resolved and STM anchored at `{stm_base}/{issue}/` | C4 | Hard block — no issue, no spec (traceability anchor is mandatory) |
| PF4 — scope guard: the work is substantial, not fix-it-scale | C5 | Graceful redirect — do NOT produce a spec; point the user to fix-it for quick fixes |
| PF5 — read `craft-ice.hitl` from `.garura/core/config.yaml` | C6 | Default `true` (human-in-the-loop ON) when absent/unreadable — fail safe toward human approval |

```bash
# Resolve paths
stm_base=$(yq '.stm.base-path' .garura/core/config.yaml)
product_base=$(yq '.product.base-path' .garura/core/config.yaml)

# Git repo check
git rev-parse --is-inside-work-tree

# Issue from branch (start-feature convention: {type}/{issue}-{slug})
branch=$(git branch --show-current)
issue=$(echo "$branch" | grep -oE '/[0-9]+' | tr -d '/')

# HITL config (C6) — default ON when absent/unreadable
hitl=$(yq '.["craft-ice"].hitl' .garura/core/config.yaml)
[ "$hitl" = "null" ] && hitl=true
```

**Scope guard (PF4 / C5).** Assess the work against the same signals the enhance play
uses (files touched, domains crossed, new abstractions, tests impacted, estimated
effort). If the work is fix-it-scale (e.g. a single-file, single-domain, sub-hour fix),
**do not produce a spec** — record the assessment to
`{stm_base}/{issue}/evidence/craft-ice/scope-assessment.md` and redirect the user to
`/fix-it`. craft-ice produces a full ICE spec only for substantial, sweeping work.

**Pre-flight eval:**

**SE-5 (F5):** No full specification exists for fix-it-scale work. Check
`{stm_base}/{issue}/specs/` against `{stm_base}/{issue}/evidence/craft-ice/scope-assessment.md`.
PASS: either the work was assessed substantial (not fix-it-scale), or no full
specification was produced. FAIL: a full specification exists for work assessed
fix-it-scale.

**Resume check:** If `{stm_base}/{issue}/status/craft-ice.json` exists, resume — skip
completed steps, reset any `in_progress` to pending, continue from first incomplete.

agents MUST NOT call TaskUpdate on them; they MAY TaskCreate discovered sub-work with
`addBlockedBy` pointing to the current step.

```

TaskCreate [T2] Author Intent triple            blockedBy: [T1]
TaskCreate [T3] Assemble Context bundle          blockedBy: [T1]
TaskCreate [T4] Generate Expectation             blockedBy: [T2, T3]
TaskCreate [T5] Approval checkpoint (HITL config-gated)  blockedBy: [T4]
TaskCreate [T6] Evidence & Close                 blockedBy: [T5]
```

On resume: `TaskList` → skip completed, reset `in_progress` → pending.

## Workflow

### Phase: Preparation

**Step 1 — Discovery Q&A**
Owner: play · Depends on: pre-flight · `Task: TaskUpdate [T1] → in_progress`

Read the source epic/feature (issue body + any referenced epic file). Then ask the
user targeted questions to fill gaps — goal and expected outcome, boundaries, what
would make the result unacceptable, who the feature's actors are. Write the digest to
`{stm_base}/{issue}/context/discovery.md`. This is play-led (the orchestrator owns user
interaction); no agent interviews.

`Task: TaskUpdate [T1] → completed`

---

**Step 2 — Author Intent triple**
Owner: `intent-crafter` · Depends on: Step 1 · `Task: TaskUpdate [T2] → in_progress`

```json
{
  "intent_path": "core/components/plays/craft-ice/reference/intent.yaml",
  "stm_base": "{stm_base}",
  "stm": {
    "input": { "discovery": "{stm_base}/{issue}/context/discovery.md" },
    "output": { "intent": "{stm_base}/{issue}/specs/intent.yaml" }
  },
  "task_id": "author-feature-intent"
}
```

Authoring mode: the agent reads the discovery digest, applies its quality gate, and
invokes `author-intent-yaml` (passing `output_base = {stm_base}/{issue}/specs/`) to
write the feature's clean triple. It does NOT interview. Every artifact must carry a
resolvable link back to the source epic/feature (C4/F4).

**SE-2 (F2/C2):** The delivered Intent layer is implementation-agnostic — it does not
name agents, skills, tools, technologies, file paths, or step-by-step instructions.
Check `{stm_base}/{issue}/specs/intent.yaml`. PASS: none present. FAIL: any
implementation detail present.

`Task: TaskUpdate [T2] → completed`

---

**Step 3 — Assemble Context bundle**
Owner: `tech-designer` · Depends on: Step 1 · `Task: TaskUpdate [T3] → in_progress`

```json
{
  "intent_path": "core/components/plays/craft-ice/reference/intent.yaml",
  "stm_base": "{stm_base}",
  "stm": {
    "input": { "discovery": "{stm_base}/{issue}/context/discovery.md" },
    "output": { "context_bundle": "{stm_base}/{issue}/context/context-bundle.md" }
  },
  "task_id": "assemble-context"
}
```

The agent follows its Context Loading protocol: searches LTM at
`~/.garura/core/memory/`, evaluates sufficiency, and falls back to
`research-domain-context` (web) when LTM is thin; it explores the codebase for the
integration points the feature touches and the actors it serves. Output is the Context
bundle (the "C" in ICE) written to `{stm_base}/{issue}/context/`, with each artifact
carrying a link back to the source epic/feature (C4/F4).

`Task: TaskUpdate [T3] → completed`

---

**Step 4 — Generate Expectation**
Owner: `expectation-crafter` (feature case) · Depends on: Steps 2, 3 · `Task: TaskUpdate [T4] → in_progress`

```json
{
  "intent_path": "{stm_base}/{issue}/specs/intent.yaml",
  "stm_base": "{stm_base}",
  "stm": {
    "input": {
      "context": "{stm_base}/{issue}/context/context-bundle.md",
      "rules_path": "core/components/memory/standards/rules/feature-expectation-generation.md"
    },
    "output": { "expectation": "{stm_base}/{issue}/specs/expectation.yaml" }
  },
  "task_id": "generate-feature-expectation"
}
```

The presence of `stm.input.context` selects the feature case — the agent invokes
`generate-feature-expectation` (passing `intent_path`, `context_path`, `output_path`,
and `rules_path`). The skill derives success scenarios grounded in the Context's actors
and exactly one recovery entry per failure condition, and writes the Expectation with
`vetted.status: pending` and `generated_from.intent` + `generated_from.context`
provenance.

**SE-1 (F1/C1):** The produced specification contains all three ICE layers — Intent
(`specs/intent.yaml`), Context (`context/`), Expectation (`specs/expectation.yaml`).
PASS: all three present and non-empty. FAIL: any layer missing or empty.

**SE-3 (F3/C3):** The delivered Expectation carries an approval marker and a provenance
link to the Intent. Check `{stm_base}/{issue}/specs/expectation.yaml`. PASS: a vetted
marker AND a `generated_from.intent` link that resolves to the feature's Intent. FAIL:
marker missing or provenance missing.

**SE-4 (F4):** Every artifact the specification contains carries a resolvable link back
to the source epic/feature — not merely a tracking-issue number. Check `specs/` and
`context/`. PASS: each produced artifact resolves back to the source epic/feature.
FAIL: any artifact lacks the link.

`Task: TaskUpdate [T4] → completed`

### Phase: Checkpoint (skippable per HITL config)

**Step 5 — Approval checkpoint**
Owner: play · Depends on: Step 4 · `Task: TaskUpdate [T5] → in_progress`

Read `craft-ice.hitl` (resolved at PF5).

- **HITL ON** — present the complete ICE spec (Intent + Context + Expectation file
  paths) to the user for review. On **Tether**, set `vetted.status: approved` (record
  `approved_by` + `approved_at`) in `specs/expectation.yaml`. On **Vanish**, return to
  the relevant Preparation step to revise. Nothing downstream consumes a `pending`
  expectation.
- **HITL OFF** — the checkpoint is skipped; auto-approve per policy and record that the
  approval was automatic (no human approver).

**SE-6 (F6):** The specification's approval is consistent with the human-in-the-loop
configuration. Check `specs/expectation.yaml` against `craft-ice.hitl`. PASS: the
approval state matches the configured setting — when HITL is required, a human approver
is recorded. FAIL: the approval contradicts the configuration (e.g. auto-approved when
HITL is required).

`Task: TaskUpdate [T5] → completed`

## Scenario Validation

Validate E2E outcomes against the success scenarios in `reference/expectation.yaml` —
one SCE per scenario, text from each scenario's `measure`:

**SCE-1 (S1 — the agent that selects the next play):** The specification contains a
non-empty Intent layer (goal + constraints + failure conditions), a non-empty Context
layer (memory/knowledge/codebase grounding), and a non-empty Expectation layer (success
scenarios + recovery) — all three present — and the Expectation is marked approved, so
the agent can select and run prepare or enhance directly against the spec.

**SCE-2 (S2 — the reviewer at the spec approval checkpoint):** The Expectation carries a
`generated_from.intent` provenance link and a `vetted.status` marker that, under a
human-in-the-loop configuration, a reviewer flips to approved at the checkpoint before
the agent in S1 can run anything.

**SCE-3 (S3 — auditor tracing the specification back to its origin):** The specification
references a tracking issue and every contained artifact carries a resolvable link back
to the source epic/feature.

## Recovery

Sourced from `reference/expectation.yaml` `recovery` — one entry per failure condition.
When the validator detects a tripped failure, it builds a recovery handoff plan from the
matching entry and routes it per `handoff`: `autonomous` loops the fix back to the
builder; `human` escalates for a manual call.

| ID | For | Symptom (trigger) | Direction | Handoff |
|----|-----|-------------------|-----------|---------|
| REC1 | F1 | the delivered specification is missing one or more of the three ICE layers | fill the absent layer by deriving it from material already gathered before the spec is considered complete | autonomous |
| REC2 | F2 | the delivered Intent layer carries implementation details (agents, skills, tools, technologies, paths, steps) | strip every implementation token so the Intent describes only goal, constraints, and failure conditions | autonomous |
| REC3 | F3 | the delivered Expectation lacks an approval marker or a provenance link to the Intent | restore the provenance link and route the Expectation to a human for the approval the builder cannot grant | human |
| REC4 | F4 | the delivered specification has no traceable link back to the source epic/feature or tracking issue | anchor the spec to the tracking issue and its source epic/feature — a human maps the work to the correct external tracking record | human |
| REC5 | F5 | a full specification exists for fix-it-scale work | a human judges the scope and redirects quick-fix-sized work away from the spec pipeline | human |
| REC6 | F6 | the spec's approval contradicts the human-in-the-loop configuration | re-run the review honoring the configured setting so a human grants approval whenever the configuration requires it | human |

At Level 4, `intent-resolver` executes the autonomous entries (REC1, REC2) without a
human; the human entries (REC3, REC4, REC5, REC6) always escalate.
