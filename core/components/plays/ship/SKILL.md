# ship

Deliver all uncommitted work on the current feature branch to the main branch in a single automated pipeline: commit all changes, create and submit a pull request, optionally run the `review-pr` quality gate, merge the PR, switch to main, pull latest, and delete the feature branch. Auto-proceeds end-to-end unless `review-pr.bypass=false` in `.garura/core/config.yaml` and `review-pr` returns a `block` or `escalate` verdict. End state: local checkout on main, main up to date with remote, feature branch gone, all work reachable from main's history.

## Role

You are the orchestrator of a Structure C play. You chain compiled plays in sequence — you do NOT invoke agents directly. Each sub-play manages its own agents, skills, and evals internally.

**Forbidden:** Direct `git` commands. Direct `gh`/`glab`/`bb` commands. Direct agent invocations for domain work. All domain work is delegated to sub-plays.

**Sub-play chain:**

| Step | Play | What It Does | Conditional |
|------|--------|--------------|---|
| 1 | `commit-code` | Commit all changes grouped by concern, mapped to issues, push branch | always |
| 1b | `distill` | Learning extraction — dispatched in background after Step 1, parallel with Step 2 | always (non-blocking) |
| 2 | `create-pr` | Create PR with quality checklist and evidence | always |
| 2.5 | `review-pr` | Diff-scoped quality review; emits findings + confidence + reviewers | only when `review-pr.bypass == false` (C7) |
| 3 | `merge-pr` | Merge PR, switch to main, pull latest, delete feature branch | always (gated by 2.5 verdict) |

## Pre-flight

Execute these checks before any sub-play invocation:

| Check | Constraint | Action on Failure |
|-------|-----------|-------------------|
| Resolve `stm_base` from `.garura/core/config.yaml` | — | Hard halt — config is required |
| Current branch is not main/master/default | C1 | Hard halt with message |
| Changed files exist (staged, unstaged, untracked) | implicit | Graceful exit — nothing to ship |
| Platform config exists | implicit | Hard halt — no platform configured |
| Resolve `review-pr.bypass` from `.garura/core/config.yaml` | C7 | Hard halt if key missing — "review-pr.bypass not set in config" |
| When `review-pr.bypass == false`: `core/components/plays/review-pr/SKILL.md` exists | C7 | Hard halt — "review-pr play missing; cannot enforce gate" |
| When `review-pr.bypass == false`: `core/components/memory/standards/rules/pr.md` exists at config-resolved path | C7 | Hard halt — "PR severity taxonomy missing" |

```bash
stm_base=$(grep 'base-path' .garura/core/config.yaml | head -1 | awk '{print $2}')
branch=$(git branch --show-current)
issue=$(echo "$branch" | grep -oE '/[0-9]+' | tr -d '/')

default_branch=$(git remote show origin | grep 'HEAD branch' | awk '{print $NF}')
# Halt if branch == default_branch or branch is "main" or "master"

git status --porcelain  # Graceful exit if empty

grep '^platform:' .garura/core/config.yaml  # Halt if unset

# C7 — resolve bypass
bypass=$(yq '.review-pr.bypass' .garura/core/config.yaml)
if [ "$bypass" = "false" ]; then
  test -f core/components/plays/review-pr/SKILL.md || { echo "review-pr missing"; exit 1; }
  taxonomy=$(yq '.standards.pr-severity-taxonomy' .garura/core/config.yaml)
  test -f "$taxonomy" || { echo "Taxonomy missing at $taxonomy"; exit 1; }
fi

# Resolve evidence.record flag (default true when absent)
evidence_record=$(grep -A1 '^evidence:' .garura/core/config.yaml | grep 'record:' | awk '{print $2}')
evidence_record=${evidence_record:-true}
```

Ship's pre-flight is lighter than individual sub-play pre-flights because each sub-play runs its own. Ship validates only cross-cutting concerns plus the gate-conditional preconditions.

**Resume check:** If `{stm_base}/{issue}/status/ship.json` exists, resume — skip completed steps, reset any `in_progress` to pending, continue from first incomplete.

## Workflow

### Phase: Commit

**Step 1 — Commit Changes**
Owner: `commit-code` (sub-play)
Depends on: pre-flight

```
Skill: commit-code
Context: approval_override: "auto-proceed"
```

**Approval override (C2):** Pass `approval_override: "auto-proceed"` to suppress all human checkpoints. Even low-confidence groupings and issue mappings auto-proceed.

**On failure:** Ship halts immediately with commit-code's failure details. Do not continue.

**After Step 1 — dispatch distill concurrently:** Immediately after commit-code completes, dispatch `distill` in the background (fire-and-forget). Distill uses `git diff` and does not need a PR to exist. Ship does NOT wait for distill before proceeding to Step 2.

```
Skill: distill (background)
Context:
  stm_base: {stm_base}
  issue: {issue}
  product_base: {product_base}
```

**Step 1 pipeline checks:**
- commit-code ran its own step evals (SE-1 through SE-8) — if any failed, it halted, and ship halts.
- F4 coverage: commit-code's evals verify issue references.
- F6 coverage: after commit-code, all changes are committed — verified by commit-code's SE-5.

---

### Phase: Pull Request

**Step 2 — Create Pull Request**
Owner: `create-pr` (sub-play)
Depends on: Step 1

```
Skill: create-pr
Context: approval_override: "auto-proceed"
```

**Approval override (C2):** Even low-confidence target branch and checklist auto-proceed.

create-pr reads commit records from STM (written by commit-code) to inform PR content.

**On failure:** Ship halts immediately with create-pr's failure details.

**Step 2 pipeline checks:**
- create-pr ran its own step evals — if any failed, it halted, and ship halts.
- F5 coverage: create-pr's evals verify PR has linked issue and correct target branch.

---

### Phase: Quality Gate (conditional — runs only when `review-pr.bypass == false`)

**Step 2.5 — Review PR**
Owner: `review-pr` (sub-play)
Depends on: Step 2
Skip when: `review-pr.bypass == true`

```
Skill: review-pr
Context:
  pr_number: {from create-pr STM output}
  approval_override: "auto-proceed"
```

review-pr reads `pr_number` from STM (written by create-pr in Step 2). It runs diff-scoped quality evaluation, computes confidence, selects reviewers, and posts a structured comment to the PR. It writes `confidence.yaml` containing the routing decision.

After review-pr completes, ship reads `{stm_base}/{issue}/evidence/review-pr/confidence.yaml` and inspects `routing`:

| routing | Ship action |
|---|---|
| `pass` | Continue to Step 3 (merge) |
| `block` | HALT before merge-pr. Surface findings, confidence, reviewers in the failure report. **F7 enforcement.** |
| `escalate` | HALT before merge-pr. Surface low-confidence findings + reviewers. **F7 enforcement.** |

**Step 2.5 pipeline checks (C7, F7):**
- review-pr ran its own step evals (SE-1 through SE-9) — if any failed, it halted, and ship halts.
- C7 coverage: When bypass=false, this step is non-skippable; halting on `block`/`escalate` is enforced by ship before invoking merge-pr.
- F7 coverage: Ship MUST NOT proceed to Step 3 when `routing` ∈ `{block, escalate}`.

**On failure of review-pr itself:** Ship halts with review-pr's failure details. Do not continue to merge-pr.

---

### Phase: Merge & Cleanup

**Step 3 — Merge, Switch, Pull, Delete**
Owner: `merge-pr` (sub-play)
Depends on: Step 2 (always) + Step 2.5 (when bypass=false; routing must be `pass`)

```
Skill: merge-pr
```

merge-pr reads PR number from STM (written by create-pr). It merges using default strategy (C3), switches to main, pulls latest (C4), and deletes the feature branch locally and remotely (C5). Conflicts are a hard stop (C6).

**On failure:** Ship halts immediately with merge-pr's failure details.

**Step 3 pipeline checks:**
- merge-pr ran its own step evals (SE-1 through SE-5) — if any failed, it halted, and ship halts.
- F1 coverage: merge-pr's SE-2 verifies branch deleted.
- F2 coverage: merge-pr's SE-1 verifies no conflicts.
- F3 coverage: merge-pr's SE-3 verifies on main and up to date.

---

### Phase: Post-merge Learning Capture

**Step 4 — Read Distill Outcome (non-blocking)**
Owner: play
Depends on: Step 3

Distill was dispatched in the background after Step 1. By the time Step 4 is reached it will usually have completed. Read its outcome from STM:

```bash
proposals_path="{stm_base}/{issue}/evidence/distill/proposals.yaml"
```

| Outcome | Ship action |
|---------|-------------|
| `proposals.yaml` exists | Log proposals path, continue |
| No file, distill still running | Log "distill pending", continue |
| No file, distill skipped | Log warning with reason, continue |

**Always non-blocking (C8).** Do NOT halt regardless of distill outcome.

### Phase: Scenario Validation

**Step 5 — Scenario Evals**
Owner: play
Depends on: Step 4

Sourced from `reference/expectation.yaml` `success_scenarios` — one eval per scenario, text from each `measure`.

- **SCE-1 (S1 — Developer):** After completion, local HEAD is on main, the local main branch is level with its remote (0 commits behind origin), the feature branch is absent from both `git branch` and `git branch -r`, and the feature commits are reachable from main's tip.
- **SCE-2 (S2 — Code Reviewer):** The platform reports the PR state as merged, the PR body links the originating issue and carries the quality checklist, and the merge commit for that PR is reachable from main's tip.
- **SCE-3 (S3 — Code Reviewer):** When the run executed with `review-pr.bypass == false`, the PR carries a `review-pr` structured comment containing P1/P2/P3 finding sections, a numeric confidence score, and the selected reviewers. Not applicable when bypass was true.
- **SCE-4 (S4 — Developer):** The ship evidence file records a distill outcome (completed with proposals_path, completed with no_learnings, or skipped with reason); when the PR was non-trivial with STM evidence available, proposals_path is non-null; and no file under `{product_base}` was created or modified during the ship run.

If any scenario eval fails: log failure in evidence, present to user.

**Step 6b — C1 Evidence file (gated by `evidence.record`).**

**Evidence lands on main.** Because ship ends on main after merge, the evidence self-commit goes directly to main. This is expected — the delivery record belongs alongside the merge commit.

**Sub-play evidence sweep (C9):** Before invoking repo-orchestrator, build a consolidated sweep list of all sub-play evidence files that are untracked in the working tree:

```
1. Read {stm_base}/{issue}/status/ship.json to determine which sub-plays completed.
   If ship.json is absent, probe all known paths directly.

2. For each completed sub-play in [commit-code, create-pr, merge-pr, distill]:
   probe {stm_base}/{issue}/evidence/{sub-play}/

3. Collect untracked files from each probed path:
   git status --porcelain {stm_base}/{issue}/evidence/{sub-play}/
   Include any file listed with "??" (untracked) status.

4. If any untracked sub-play evidence files found:
   Include them alongside ship's own evidence file in the self-commit file list.
   Never use "git add ." — only the explicitly enumerated paths.

5. If no untracked sub-play files found:
   Self-commit only ship's evidence file (normal behavior).
```

This sweep is non-blocking — any failure in the sweep enumeration is logged and ship continues. If no untracked files are found across all paths, no sweep commit is made beyond ship's own evidence file.

Invoke `repo-orchestrator` to self-commit the consolidated file list (ship evidence + any swept sub-play files) (ADR 012). Non-blocking on failure. Commit message: `chore(stm): record ship evidence for #{issue} (#{issue})`.

If `evidence.record` is `false`:
- Skip ship evidence file — do not write
- Skip sub-play evidence sweep (C9) — no sub-play evidence files will have been written
- Skip self-commit — nothing to commit
- C2 delivery report (Step 6a) is still emitted — it is never gated

## Recovery

Sourced from `reference/expectation.yaml` `recovery` — one entry per failure condition. When the validator detects a tripped failure, it builds a recovery handoff plan from the matching entry and routes it per `handoff`: `autonomous` loops the fix back to the builder; `human` escalates for a manual call.

| ID | For | Symptom (trigger) | Direction | Handoff |
|----|-----|-------------------|-----------|---------|
| REC1 | F1 | feature branch still present locally or on remote after the pipeline | delete the lingering branch, treating an already-deleted remote as success | autonomous |
| REC2 | F2 | merge conflicts exist between the feature branch and main | never auto-resolve — the conflicted state needs a user decision to resolve or decline before anything lands on main | human |
| REC3 | F3 | not on main, or local main behind remote | checkout main and fast-forward it to match the remote | autonomous |
| REC4 | F4 | a commit references the wrong issue or has no issue reference | reconcile the commit to its true issue — the correct issue mapping is external knowledge the builder lacks | human |
| REC5 | F5 | PR has no linked issue, or targets the wrong base branch | reconcile the issue linkage and base target — both require knowing the intended issue and base | human |
| REC6 | F6 | uncommitted work remains at the point the merge is attempted | the leftover uncommitted state needs a user go/no-go on how to reconcile it before any merge proceeds | human |
| REC7 | F7 | merged while review-pr.bypass was false and review-pr returned block/escalate | a merge past a blocking gate is an outward-facing override that needs a human go-ahead | human |
| REC8 | F8 | sub-play evidence files remain untracked after ship completes | sweep the untracked evidence paths and commit them alongside ship's own evidence | autonomous |

At Level 4, `intent-resolver` executes the autonomous entries (REC1, REC3, REC8) without a human; the human entries (REC2, REC4, REC5, REC6, REC7) always escalate.

## Pause and Resume

Steps are in execution order — run top to bottom.

**Issue detection:** Extract issue number from branch name during pre-flight (e.g., `feature/95-slug` → `95`).

**Status file:** `{stm_base}/{issue}/status/ship.json`

```json
{
  "play": "ship",
  "issue": "{issue}",
  "started_at": "{ISO timestamp}",
  "feature_branch": "{branch name before merge}",
  "review_pr_bypass": false,
  "tasks": {
    "commit-code": { "status": "completed" },
    "create-pr": { "status": "completed" },
    "review-pr": { "status": "in_progress", "bypass": false, "routing": null },
    "merge-pr": { "status": "pending" },
    "distill": { "status": "pending" },
    "scenario-evals": { "status": "pending" },
    "write-evidence": { "status": "pending" }
  }
}
```

**Executor loop:**
```
resolve issue number from branch name
resolve review-pr.bypass from config
check for status file at {stm_base}/{issue}/status/ship.json

for each step in compiled order:
  if step is review-pr AND bypass == true → skip
  if status file shows step "completed" → skip
  if status file shows step "skipped" → skip (distill uses "skipped" on failure)
  if status file shows step "in_progress" → reset to pending
  mark step "in_progress" in status file
  execute step
  mark step "completed" or "skipped" in status file (distill uses "skipped" on error)
```

**Fresh start:** No status file → execute all steps, create status file on first step.
**Resume:** Status file exists → skip completed sub-plays, continue from first incomplete.
**Important:** `feature_branch` is stored at start because after merge-pr the branch no longer exists — needed for scenario evals.
