# ship

Deliver all uncommitted work on the current feature branch to the main branch in a single automated pipeline: commit all changes, create and submit a pull request, optionally run the `review-pr` quality gate, merge the PR, switch to main, pull latest, and delete the feature branch. Auto-proceeds end-to-end unless `review-pr.bypass=false` in `.meridian/core/config.yaml` and `review-pr` returns a `block` or `escalate` verdict. End state: local checkout on main, main up to date with remote, feature branch gone, all work reachable from main's history.

## Role

You are the orchestrator of a Structure C play. You chain compiled plays in sequence — you do NOT invoke agents directly. Each sub-play manages its own agents, skills, and evals internally.

**Forbidden:** Direct `git` commands. Direct `gh`/`glab`/`bb` commands. Direct agent invocations for domain work. All domain work is delegated to sub-plays.

**Sub-play chain:**

| Step | Play | What It Does | Conditional |
|------|--------|--------------|---|
| 1 | `commit-code` | Commit all changes grouped by concern, mapped to issues, push branch | always |
| 2 | `create-pr` | Create PR with quality checklist and evidence | always |
| 2.5 | `review-pr` | Diff-scoped quality review; emits findings + confidence + reviewers | only when `review-pr.bypass == false` (C7) |
| 3 | `merge-pr` | Merge PR, switch to main, pull latest, delete feature branch | always (gated by 2.5 verdict) |

## Pre-flight

Execute these checks before any sub-play invocation:

| Check | Constraint | Action on Failure |
|-------|-----------|-------------------|
| Resolve `stm_base` from `.meridian/core/config.yaml` | — | Hard halt — config is required |
| Current branch is not main/master/default | C1 | Hard halt with message |
| Changed files exist (staged, unstaged, untracked) | implicit | Graceful exit — nothing to ship |
| Platform config exists | implicit | Hard halt — no platform configured |
| Resolve `review-pr.bypass` from `.meridian/core/config.yaml` | C7 | Hard halt if key missing — "review-pr.bypass not set in config" |
| When `review-pr.bypass == false`: `core/components/plays/review-pr/SKILL.md` exists | C7 | Hard halt — "review-pr play missing; cannot enforce gate" |
| When `review-pr.bypass == false`: `core/components/memory/standards/rules/pr.md` exists at config-resolved path | C7 | Hard halt — "PR severity taxonomy missing" |

```bash
stm_base=$(grep 'base-path' .meridian/core/config.yaml | head -1 | awk '{print $2}')
branch=$(git branch --show-current)
issue=$(echo "$branch" | grep -oE '/[0-9]+' | tr -d '/')

default_branch=$(git remote show origin | grep 'HEAD branch' | awk '{print $NF}')
# Halt if branch == default_branch or branch is "main" or "master"

git status --porcelain  # Graceful exit if empty

grep '^platform:' .meridian/core/config.yaml  # Halt if unset

# C7 — resolve bypass
bypass=$(yq '.review-pr.bypass' .meridian/core/config.yaml)
if [ "$bypass" = "false" ]; then
  test -f core/components/plays/review-pr/SKILL.md || { echo "review-pr missing"; exit 1; }
  taxonomy=$(yq '.standards.pr-severity-taxonomy' .meridian/core/config.yaml)
  test -f "$taxonomy" || { echo "Taxonomy missing at $taxonomy"; exit 1; }
fi
```

Ship's pre-flight is lighter than individual sub-play pre-flights because each sub-play runs its own. Ship validates only cross-cutting concerns plus the gate-conditional preconditions.

**Resume check:** If `{stm_base}/{issue}/status/ship.json` exists, resume — skip completed steps, reset any `in_progress` to pending, continue from first incomplete.

## Workflow

**Step 1 — Commit Changes**

**Approval override (C2):** Pass `approval_override: "auto-proceed"` to suppress all human checkpoints. Even low-confidence groupings and issue mappings auto-proceed.

**On failure:** Ship halts immediately with commit-code's failure details. Do not continue.

**Step 1 pipeline checks:**
- commit-code ran its own step evals (SE-1 through SE-8) — if any failed, it halted, and ship halts.
- F4 coverage: commit-code's evals verify issue references.
- F6 coverage: after commit-code, all changes are committed — verified by commit-code's SE-5.

### Phase: Pull Request

**Step 2 — Create Pull Request**
Owner: `create-pr` (sub-play)
Depends on: Step 1

```
Skill: create-pr
Context: approval_override: "auto-proceed"
```

**Approval override (C2):** Even low-confidence target branch and checklist auto-proceed.

create-pr reads commit records from STM (written by commit-code) to inform PR content.

**On failure:** Ship halts immediately with create-pr's failure details.

**Step 2 pipeline checks:**
- create-pr ran its own step evals — if any failed, it halted, and ship halts.
- F5 coverage: create-pr's evals verify PR has linked issue and correct target branch.

---

### Phase: Quality Gate (conditional — runs only when `review-pr.bypass == false`)

**Step 2.5 — Review PR**
Owner: `review-pr` (sub-play)
Depends on: Step 2
Skip when: `review-pr.bypass == true`

```
Skill: review-pr
Context:
  pr_number: {from create-pr STM output}
  approval_override: "auto-proceed"
```

review-pr reads `pr_number` from STM (written by create-pr in Step 2). It runs diff-scoped quality evaluation, computes confidence, selects reviewers, and posts a structured comment to the PR. It writes `confidence.yaml` containing the routing decision.

After review-pr completes, ship reads `{stm_base}/{issue}/evidence/review-pr/confidence.yaml` and inspects `routing`:

| routing | Ship action |
|---|---|
| `pass` | Continue to Step 3 (merge) |
| `block` | HALT before merge-pr. Surface findings, confidence, reviewers in the failure report. **F7 enforcement.** |
| `escalate` | HALT before merge-pr. Surface low-confidence findings + reviewers. **F7 enforcement.** |

**Step 2.5 pipeline checks (C7, F7):**
- review-pr ran its own step evals (SE-1 through SE-9) — if any failed, it halted, and ship halts.
- C7 coverage: When bypass=false, this step is non-skippable; halting on `block`/`escalate` is enforced by ship before invoking merge-pr.
- F7 coverage: Ship MUST NOT proceed to Step 3 when `routing` ∈ `{block, escalate}`.

**On failure of review-pr itself:** Ship halts with review-pr's failure details. Do not continue to merge-pr.

---

### Phase: Merge & Cleanup

**Step 3 — Merge, Switch, Pull, Delete**
Owner: `merge-pr` (sub-play)
Depends on: Step 2 (always) + Step 2.5 (when bypass=false; routing must be `pass`)

```
Skill: merge-pr
```

merge-pr reads PR number from STM (written by create-pr). It merges using default strategy (C3), switches to main, pulls latest (C4), and deletes the feature branch locally and remotely (C5). Conflicts are a hard stop (C6).

**On failure:** Ship halts immediately with merge-pr's failure details.

**Step 3 pipeline checks:**
- merge-pr ran its own step evals (SE-1 through SE-5) — if any failed, it halted, and ship halts.
- F1 coverage: merge-pr's SE-2 verifies branch deleted.
- F2 coverage: merge-pr's SE-1 verifies no conflicts.
- F3 coverage: merge-pr's SE-3 verifies on main and up to date.

---

### Phase: Post-merge Learning Capture

**Step 4 — Capture Learning (non-blocking)**
Owner: `capture-learning-fast` (sub-play)
Depends on: Step 3

```
Skill: capture-learning-fast
Context:
  stm_base: {stm_base}
  issue: {issue}
  product_base: {product_base}
```

**Non-blocking (C8):** This step is fire-and-forget. Ship invokes
capture-learning-fast and reads the outcome, then **always continues to Step 5
regardless of outcome**.

| Outcome | Ship action |
|---------|-------------|
| `{ status: "completed", no_learnings: true }` | Log "no learnings detected", continue |
| `{ status: "completed", proposals_path: "..." }` | Log proposals path, continue |
| `{ status: "skipped", reason: "..." }` | Log warning with reason, continue |

**On any failure of capture-learning-fast:** Log `"Step 4: capture-learning-fast
skipped — {reason}"` in evidence. Do NOT halt. Continue to Step 5 (Scenario Validation).

---

### Phase: Scenario Validation

**Step 5 — Scenario Evals**
Owner: play
Depends on: Step 4

- **SCE-1 (S1 — Developer):** Local checkout is on main/default branch. Main is up to date with remote. Feature branch is absent locally and on remote. Feature commits are visible in main's history. Developer can start next feature immediately.
- **SCE-2 (S2 — Code Reviewer):** PR state on platform is MERGED. PR is linked to the originating issue. Quality checklist is present in PR body. Merge commit is on main. All commits are traceable to the originating issue.
- **SCE-3 (S3 — Code Reviewer):** When ship ran with `review-pr.bypass == false`, the PR carries a `review-pr` structured comment with P1/P2/P3 sections, a confidence score, and the selected reviewers. Skipped when bypass was true.
- **SCE-4 (S4 — Developer):** capture-learning-fast outcome is recorded in the evidence file. If the PR was non-trivial and STM evidence was available, proposals_path is non-null. Product LTM contains no new or modified files from this ship run.

If any scenario eval fails: log failure in evidence, present to user.

---

### Phase: Evidence & Close

**Step 6 — Write Evidence**
Owner: play
Depends on: Step 5

Write evidence to `{stm_base}/{issue}/evidence/ship/{YYYYMMDD-HHMMSS}.md` containing:
- Pipeline steps completed (commit, PR, review-pr if run, merge, capture-learning-fast)
- `review-pr.bypass` value at run time
- review-pr routing decision (when run)
- PR number and merge SHA
- Branch deletion status
- capture-learning-fast outcome: status (completed/skipped), proposals_path (if any), no_learnings flag, reason (if skipped)
- Scenario eval results (SCE-1, SCE-2, SCE-3, SCE-4)
- Any failures encountered

Present final delivery report to user.

**Evidence lands on main.** Because ship ends on main after merge, the evidence self-commit goes directly to main. This is expected — the delivery record belongs alongside the merge commit.

Invoke `repo-orchestrator` to self-commit evidence files (ADR 012). Non-blocking on failure.

## Pause and Resume

Steps are in execution order — run top to bottom.

**Issue detection:** Extract issue number from branch name during pre-flight (e.g., `feature/95-slug` → `95`).

**Status file:** `{stm_base}/{issue}/status/ship.json`

```json
{
  "play": "ship",
  "issue": "{issue}",
  "started_at": "{ISO timestamp}",
  "feature_branch": "{branch name before merge}",
  "review_pr_bypass": false,
  "tasks": {
    "commit-code": { "status": "completed" },
    "create-pr": { "status": "completed" },
    "review-pr": { "status": "in_progress", "bypass": false, "routing": null },
    "merge-pr": { "status": "pending" },
    "capture-learning-fast": { "status": "pending" },
    "scenario-evals": { "status": "pending" },
    "write-evidence": { "status": "pending" }
  }
}
```

**Executor loop:**
```
resolve issue number from branch name
resolve review-pr.bypass from config
check for status file at {stm_base}/{issue}/status/ship.json

for each step in compiled order:
  if step is review-pr AND bypass == true → skip
  if status file shows step "completed" → skip
  if status file shows step "skipped" → skip (capture-learning-fast uses "skipped" on failure)
  if status file shows step "in_progress" → reset to pending
  mark step "in_progress" in status file
  execute step
  mark step "completed" or "skipped" in status file (capture-learning-fast uses "skipped" on error)
```

**Fresh start:** No status file → execute all steps, create status file on first step.
**Resume:** Status file exists → skip completed sub-plays, continue from first incomplete.
**Important:** `feature_branch` is stored at start because after merge-pr the branch no longer exists — needed for scenario evals.
