# reap

Post-epic learning extraction. After validate completes, read the build trinity and produce tiered, taxonomy-classified learning proposals staged in STM. Each proposal carries a two-level learning taxonomy (learning_category + sub_category) aligned to the core/components/memory/ KB structure. A human reviews proposals and Tethers to commit evidence or Vanishes to halt. No product LTM is written during this play.

```

- Every proposal in `proposals.yaml` carries a non-empty `learning_category` field.

- For proposals with `learning_category` in `{arch, quality, standards}`: `sub_category` is non-null and non-empty.
- For proposals with `learning_category` in `{domain, product}`: `sub_category` is null or empty.

**Step 2 Eval — SE-3 (C7a, F3):**

**Step 2 Eval — SE-4 (C4, F4):**
- `proposals.yaml` output path is under `{stm_base}/{issue}/evidence/reap/`. No path under `{product_base}` or `~/.garura/core/memory/` appears in the proposals output.

**Step 2 Eval — SE-5 (C6, F5):**
- Every proposal with `tier: 1` carries an `adr_draft_path` and an `impact` block with at least `risk` and `downstream_artifacts`.

**Step 2 Eval — SE-6 (C8, F6):**

**Step 2 Eval — SE-7 (C11):**

## Recovery

Sourced from `reference/expectation.yaml` `recovery` — one entry per failure condition. When the validator detects a tripped failure, it builds a recovery handoff plan from the matching entry and routes it per `handoff`: `autonomous` loops the fix back to the builder; `human` escalates for a manual call.

| ID | For | Symptom (trigger) | Direction | Handoff |
|----|-----|-------------------|-----------|---------|
| REC1 | F1 | extraction completed and user Tethered, but no proposals.yaml at the evidence path | write the missing proposals.yaml from the extraction output on hand — even a zero-proposal run writes the file with total_proposals: 0 | autonomous |
| REC2 | F2 | a proposal has no learning_category, or it is empty | classify the offending proposal from the finding already produced and fill the missing learning_category | autonomous |
| REC3 | F3 | a proposed-new taxonomy value lacks a complete taxonomy_justification block | decide whether to invent a new category or remap to canonical, and author the why-no-canonical-fits reasoning — a reviewer's call the builder cannot make | human |
| REC4 | F4 | a proposal targets, or the run writes, a path under product or global core LTM | re-target any proposal pointing at product/global core LTM back into the STM evidence path | autonomous |
| REC5 | F5 | a tier 1 proposal has no adr_draft_path or no impact block | generate the missing ADR draft and impact block from the Tier 1 finding the run already holds | autonomous |
| REC6 | F6 | a check-drift manifest was present, yet its findings appear re-derived rather than consumed | consume the manifest findings as-is, mark source: check-drift, and drop the re-derived duplicates | autonomous |
| REC7 | F7 | a flat-parent proposal (domain or product) carries a non-null sub_category | set sub_category to null on the offending proposal | autonomous |
| REC8 | F8 | a hierarchical-parent proposal (arch, quality, standards) has a null/empty sub_category and is not flagged proposed | assign the correct canonical child from the taxonomy tree to the offending proposal | autonomous |
| REC9 | F9 | a prior-play STM artifact was modified/renamed/deleted, or a write landed outside evidence/reap/ and status/reap.json | restore the altered prior-play artifacts to their original state — the unaltered source of truth is outside what the builder controls | human |
| REC10 | F10 | evidence was committed without proposals.yaml first being staged and presented for Tether/Vanish | pause and obtain the mandatory Tether/Vanish decision before any evidence commit — an authority the builder does not hold | human |

At Level 4, `intent-resolver` executes the autonomous entries (REC1, REC2, REC4, REC5, REC6, REC7, REC8) without a human; the human entries (REC3, REC9, REC10) always escalate.

```

**Executor loop:**
```
check for status file at {stm_base}/{issue}/status/reap.json

for each step in compiled order:
  if status file shows step "completed" → skip
  if status file shows step "in_progress" → reset to pending (may not have finished)
  mark step "in_progress" in status file
  execute step
  mark step "completed" in status file
```

**Fresh start:** No status file → execute all steps, create status file on first step.
**Resume:** Status file exists → skip completed, continue from first incomplete.
**Pause:** User terminates session. Status file reflects last completed step. Next invocation resumes.
