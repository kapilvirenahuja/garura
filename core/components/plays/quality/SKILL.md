# quality

Take one shaped **slice** — a vertical product increment from /shape, the thing you actually
deliver — and write its quality lens: the flat list of gates it must pass — "p99 < 150ms",
"no user enumeration", "WCAG 2.1 AA". The slice is the unit of realization: you pick a slice
and run quality → ux → agentic → arch → run on it, then ship it. A slice has no ICE of its
own — its **hub** is the union of its functionalities' ICE (which may span several
capabilities) plus the profile. Every gate comes from a profile target that applies to this
slice or from one of the slice's functionalities' ICE rules made checkable; none is invented.
/quality is the **first** of the five realize plays, so it reads only the slice's hub, never
another lens, and it sets the quality bar the later lenses (ux, agentic, arch, run) size
their depth against. It writes only the quality lens, plus a decision for any material choice.

**Pipeline position: start.** /quality OPENS the slice pipeline (quality → ux → agentic → arch → run → grill) and selects the slice: the D2 rule prepends `start-change` — resolve or create the slice-realization issue, cut the branch off fresh main, optional worktree, init STM — so the later lenses and /grill run on this already-started branch. No close sequence here; the slice change closes at /grill. It runs after /shape, since a slice must be shaped before it is realized. It writes the persistent product model directly, on the started branch. (#437)

## Role

You are the orchestrator. You own the workflow and the step order. You delegate the one
piece of judgment — turning the targets and ICE rules into gates — to the `product-os-keeper`
agent via a JSON contract over files on disk, and you run every mechanical part (the
readiness + hub resolution, the draft validation, the snapshot, the allowlisted persist, and
the post-apply checks) through bundled scripts. You never write the model YAML yourself, you
never write the slice record, the profile, or another lens, and you never persist before the
human approves the single checkpoint (C10).

**Forbidden:** hand-writing lens/decision YAML; writing the slice record, the functionalities'
ICE, the profile, or another lens by any route; reading another lens to derive the gates
(quality is first); persisting by any route other than `scripts/apply_quality.py`; persisting
before Step 3 approval.

**Agent boundaries:**

| Agent | Domain | Skill it invokes | Phases |
|-------|--------|------------------|--------|
| `product-os-keeper` | Turn the applicable profile targets + the slice's functionalities' ICE constraints/failures into a grounded, checkable list of gates; record a decision for any material choice | `author-quality-lens` | Draft |

`product-os-keeper` is the single **domain agent** this play uses (1 of the ≤5 budget).
No utility agents are needed — git/issue machinery is absent (position none). /quality does
**not** ground in the KB router: the gates come from the slice's own hub + profile, not from
a placement question.

## Pre-flight

| Check | Constraint | Action on Failure |
|-------|-----------|-------------------|
| Resolve config + `product_base` (`.garura/core/config.yaml`) | — | Hard halt |
| Profile firmed (`set`) AND the slice exists with every functionality ICE resolved + rich | C1 | Hard halt (REC1) |

Resolve config mechanically, then resolve the slice + hub. /quality has no branch or issue
(position none):

```
python3 scripts/preflight.py --play quality --config .garura/core/config.yaml
python3 scripts/check_ready_slice.py --product-base <product_base> --slice <slice>
```

`preflight.py` returns config facts (`product_base`, `stm_base`, `evidence_record`).
`check_ready_slice.py` is the readiness gate **and hub resolver** (C1): it halts unless the
profile is `set` and the slice — the play argument (e.g. `/quality slice-token-data-spine` or
`/quality <domain>/<slice-id>`) — exists with every `functionalities[].ice_ref` resolving to a
rich ICE. An ice_ref that does not resolve **fails loud** (a broken hub, never a silent pass).
On success it emits the slice context the draft step uses: `slice_file`, `domain`, `lens_dir`,
and the resolved `functionality_ices`. If the profile is `directional`, the slice is missing,
or a functionality ICE is unresolved/thin, hard halt and route to /understand or /shape
(REC1).

**Resume check:** if `{stm_base}_realize/quality/status/<slice-id>.json` exists, resume —
skip completed steps, reset any in-progress step to pending, continue from the first
incomplete.

## Workflow

### Phase: Start (injected — D2 position: start)

**Step 0 — start-change** · Owner: `start-change` (sub-play) · Depends on: pre-flight
Run the start-of-pipeline member as a sub-play, dispatched with `parent_run_id` so it
emits only its own C1 evidence and this play's close absorbs it. It resolves or creates
the slice-realization issue, cuts the branch off fresh main, sets up a worktree iff config calls for
it, and initializes the STM workspace. The later lenses (/ux, /agentic, /arch, /run) and /grill run on this already-started branch; /grill closes it.

    {
      "play":          "start-change",
      "parent_run_id": "<this run id>",
      "inputs":  { "title": "realize slice <slice-id>" },
      "outputs": { "result": "{stm_base}_quality/start/start-change.json" }
    }

start-change owns its own evals (issue anchored, branch off latest main, worktree per
config, STM initialized); they are not re-checked here.

### Phase: Draft

**Step 1 — Draft quality lens** · Owner: `product-os-keeper` · Depends on: pre-flight
The agent invokes `author-quality-lens` to read the slice's hub (its functionalities' ICE,
resolved by the readiness gate) and the profile box, pull the applicable targets, make the
ICE constraints/failures checkable, and draft the gates — plus a decision for any material
choice — to STM, grounding every gate in the manifest:

    {
      "task":    "from this slice's profile targets + its functionalities' ICE rules, draft the quality lens (gates only), grounding each gate; record a decision for any material choice",
      "inputs":  { "slice_ref": "<domain>/<slice-id>",
                   "slice_file": "<product_base>/<slice_file>",
                   "functionality_ices": [ "<product_base>/<ice_ref>", "..." ],
                   "profile_path": "<product_base>/product-os/profile.yaml",
                   "lens_rel": "<lens_dir>/quality.yaml" },
      "outputs": { "draft_dir":  "<working>/draft/",
                   "lens":       "<working>/draft/<lens_dir>/quality.yaml",
                   "manifest":   "<working>/draft/quality-manifest.yaml" }
    }

`slice_file`, `lens_dir`, and `functionality_ices` come from `check_ready_slice.py`. The skill
reads the hub **read-only** and never another lens (quality is first). It returns the contract
with the output paths on disk — never inline content.
**SE-1 (F1/C1):** the readiness + hub gate passed at pre-flight — the profile is `set` and the
slice exists with every functionality ICE resolved + rich; otherwise the run halted (REC1).

### Phase: Validate

**Step 2 — Validate the draft** · Owner: play · Depends on: Step 1
Run the lens validator over the draft before the checkpoint:

```
python3 scripts/validate_quality.py --draft <working>/draft \
        --manifest <working>/draft/quality-manifest.yaml
```

**SE-3 (F3/C3):** the lens is gates-only — `content` has exactly the `gates` key, a non-empty
list, no how-to-test/coverage/environments.
**SE-4 (F4/C4):** every gate in the manifest grounds to a profile target or a functionality's
ICE constraint/failure — none invented.
**SE-5 (F5/C5):** no gate is a bare word; each reads as a checkable condition.
**SE-6 (F6/C6):** no gate's grounding source is another lens — quality read none.
**SE-7 (F7/C7):** every gate flagged `material` carries a decision that resolves to a drafted
decision record.
**SE-9 (F9/C9):** the lens and any decision carry their required v1 fields and valid enums
(`type: quality`, a `slice_ref`).
On any GAP, apply REC3/REC4/REC5/REC6/REC7/REC9 and re-run before the checkpoint.

### Phase: Checkpoint (mandatory — never skipped, C10)

**Step 3 — Human review** · Owner: play · Depends on: Step 2
Present the proposed quality lens **inline** — the list of gates, each with the target or ICE
rule it came from, and any decisions — for the human. This checkpoint is always presented and
never skippable. Approve → continue to persist; cancel → halt with nothing written to the
model.
**SE-10 (F10/C10):** the lens is persisted only after this approval — Step 4 is the sole
writer and depends on this step; no product-model file is written before Step 4.

### Phase: Apply

**Step 4 — Persist** · Owner: play · Depends on: Step 3
First **snapshot** the slice's lens folder, the slice record, and the profile (so Step 5 can
prove only the quality lens changed). The snapshot is taken here, at the gated apply step —
never at pre-flight — so a resume can never compare post-apply against post-apply. `<slice_dir>`
is `<product_base>/product-os/<domain>/slices/<slice-id>` (the folder beside the record):

```
mkdir -p <working>/slice-before
cp -R <slice_dir>/. <working>/slice-before/ 2>/dev/null || true   # lens/ + decisions/ (may not exist yet)
cp <product_base>/<slice_file> <working>/slice-record-before.yaml
cp <product_base>/product-os/profile.yaml <working>/profile-before.yaml
```

Then persist on the fixed allowlist. `apply_quality.py` writes the quality lens (overwrite —
the re-derive) and copies decisions skip-if-exists; it is handed only the draft, which holds
only the lens + decisions, so it cannot write the slice record, the ICE, the profile, or
another lens:

```
python3 scripts/apply_quality.py --draft <working>/draft --product-base <product_base> \
        --out-manifest <working>/apply-manifest.json
```

**SE-2 (F2/C2):** the apply manifest's `written` set holds only the slice's `lens/quality.yaml`
and `decisions/*.yaml`; its `refused` set is empty (the draft held nothing out of scope).

**Step 5 — Verify persisted** · Owner: play · Depends on: Step 4
Confirm the slice record is byte-identical (realize never writes it), then check the lens
folder against the snapshot:

```
cmp <working>/slice-record-before.yaml <product_base>/<slice_file>   # must match (F2/F8)
python3 scripts/check_quality.py --cap-before <working>/slice-before \
        --cap-dir <slice_dir> \
        --profile-before <working>/profile-before.yaml \
        --profile-after <product_base>/product-os/profile.yaml
```

**SE-8 (F8/C8):** the slice record and profile are byte-identical; within the slice folder
every file is byte-identical to its snapshot except `lens/quality.yaml`; no accepted decision
was edited in place; nothing was removed.
On any GAP, apply REC2/REC8 and re-run.

### Phase: Scenario Validation

**Step 6 — Scenario evals** · Owner: play · Depends on: Step 5
- **SCE-1 (S1 — quality lead, first run):** the slice's `quality.yaml` exists with a
  `slice_ref` and a non-empty gates list; every other product-model file is byte-identical;
  the lens validates v1.
- **SCE-2 (S2 — security engineer, grounded):** every gate names a profile target or a
  functionality's ICE rule in the manifest; none ungrounded.
- **SCE-3 (S3 — QA engineer, checkable):** each gate is a clear pass/fail with a value where
  the target has one; no bare adjective.
- **SCE-4 (S4 — architect, it's first):** no ux/agentic/architecture/run lens of the slice was
  touched and no gate grounds on another lens.
- **SCE-5 (S5 — product owner, re-run):** a second run changes only the slice's `quality.yaml`
  (and possibly a new decision); the slice record, other lenses, ICE, and profile are
  byte-identical; no accepted decision edited in place.
- **SCE-6 (S6 — reviewer, the checkpoint):** the checkpoint showed the gates and decisions
  inline before any write.

## Recovery

| For | Trigger | Direction | Handoff |
|-----|---------|-----------|---------|
| F1 | the slice is absent, a functionality ICE is unresolved/thin, or the profile is not firmed | halt and route to /shape (to shape the slice) or /understand (to enrich + firm) before /quality | human |
| F2 | a write touched something beyond this slice's quality lens or a decision | revert the out-of-scope write; /quality writes only the slice's quality lens (and decisions) | autonomous |
| F3 | the lens carries non-gate content or the wrong shape | strip it back to the gates-only shape the schema requires | autonomous |
| F4 | an invented/ungrounded gate | drop it, or re-tie it to a profile target or a functionality's ICE constraint/failure; never keep an invented gate | autonomous |
| F5 | a vague or uncheckable gate | rewrite it as a concrete pass/fail with a value where the dimension needs one | autonomous |
| F6 | /quality read or depended on another lens | remove the dependency; /quality derives only from the slice's hub (it is first) | autonomous |
| F7 | a material gate choice with no decision recorded | record the slice-level decision for the choice before persisting | autonomous |
| F8 | a non-lens/non-decision file changed, or an accepted decision was edited in place | restore it and re-apply only the quality lens (and the new decision), after a human confirms the restore | human |
| F9 | the lens or a decision fails v1 schema validation | re-emit the failing artifact to conform before the play completes | autonomous |
| F10 | gates were persisted before the checkpoint was approved | revert the premature write and re-present the checkpoint; persist only after the human approves | human |

## Pause and Resume

Steps run top to bottom. On entry, resolve config, run the readiness + hub gate, resolve the
target slice from the play argument or the in-progress draft, check the status marker, skip
completed steps, reset any in-progress step to pending, and continue. The pre-apply snapshot
is captured at Step 4 (the gated apply step) and preserved on resume, so the non-destructive
comparison always diffs against true pre-apply state. A fresh start with no marker runs
everything and creates the marker at Step 1.
