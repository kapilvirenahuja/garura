# quality

Take one shaped capability and write its quality lens: the flat list of gates it must
pass — "p99 < 150ms", "no user enumeration", "WCAG 2.1 AA". Every gate comes from a
profile target that applies to this capability or from one of the capability's own ICE
rules made checkable; none is invented. /quality is the **first** of the five realize
plays, so it reads only the capability's ICE and the profile, never another lens, and
it sets the quality bar the later lenses (ux, agentic, arch, run) size their depth
against. It writes only the quality lens, plus a decision for any material choice.

**Pipeline position: none.** /quality is a realization, model-building play. It opens no
delivery issue and cuts no branch, so the D2 pipeline-position rule injects neither a
`start-change` head nor a close sequence. It writes the persistent product model
directly. It runs after /shape, since a capability must be shaped before it is realized.

## Role

You are the orchestrator. You own the workflow and the step order. You delegate the one
piece of judgment — turning the targets and ICE rules into gates — to the
`product-os-keeper` agent via a JSON contract over files on disk, and you run every
mechanical part (the readiness gate, the draft validation, the snapshot, the allowlisted
persist, and the post-apply checks) through bundled scripts. You never write the model
YAML yourself, you never write the profile or another lens, and you never persist before
the human approves the single checkpoint (C10).

**Forbidden:** hand-writing lens/decision YAML; writing the ICE, the profile, or another
lens by any route; reading another lens to derive the gates (quality is first);
persisting by any route other than `scripts/apply_quality.py`; persisting before Step 3
approval.

**Agent boundaries:**

| Agent | Domain | Skill it invokes | Phases |
|-------|--------|------------------|--------|
| `product-os-keeper` | Turn the applicable profile targets + the capability's ICE constraints/failures into a grounded, checkable list of gates; record a decision for any material choice | `author-quality-lens` | Draft |

`product-os-keeper` is the single **domain agent** this play uses (1 of the ≤5 budget).
No utility agents are needed — git/issue machinery is absent (position none). /quality
does **not** ground in the KB router: the gates come from the capability's own profile +
ICE, not from a placement question.

## Pre-flight

| Check | Constraint | Action on Failure |
|-------|-----------|-------------------|
| Resolve config + `product_base` (`.garura/core/config.yaml`) | — | Hard halt |
| Profile firmed (`set`) AND the target capability is `active` with rich ICE | C1 | Hard halt (REC1) |

Resolve config mechanically with the bundled resolver; /quality has no branch or issue
(position none):

```
python3 scripts/preflight.py --play quality --config .garura/core/config.yaml
python3 scripts/check_ready.py --product-base <product_base> --capability <capability>
```

`preflight.py` returns config facts (`product_base`, `stm_base`, `evidence_record`).
`check_ready.py` is the readiness gate (C1): it halts unless the profile is `set` and
the target capability is `active` with rich ICE. The **target capability** is the play
argument (e.g. `/quality order-management/checkout`). If the profile is `directional`,
the capability is not `active`, or its ICE is only a seed, hard halt and route to
/understand or /shape (REC1).

**Resume check:** if `{stm_base}_realize/quality/status/<capability>.json` exists,
resume — skip completed steps, reset any in-progress step to pending, continue from the
first incomplete.

## Workflow

### Phase: Draft

**Step 1 — Draft quality lens** · Owner: `product-os-keeper` · Depends on: pre-flight
The agent invokes `author-quality-lens` to read the capability's rich ICE and the
profile box, pull the applicable targets, make the ICE constraints/failures checkable,
and draft the gates — plus a decision for any material choice — to STM, grounding every
gate in the manifest:

    {
      "task":    "from this capability's profile targets + ICE rules, draft the quality lens (gates only), grounding each gate; record a decision for any material choice",
      "inputs":  { "capability_path": "<product_base>/product-os/<capability>",
                   "profile_path":    "<product_base>/product-os/profile.yaml" },
      "outputs": { "draft_dir":  "<working>/draft/",
                   "lens":       "<working>/draft/.../lens/quality.yaml",
                   "manifest":   "<working>/draft/quality-manifest.yaml" }
    }

The skill reads the ICE + profile **read-only** and never another lens (quality is
first). It returns the contract with the output paths on disk — never inline content.
**SE-1 (F1/C1):** the readiness gate passed at pre-flight — the profile is `set` and the
capability is `active` with rich ICE; otherwise the run halted (REC1).

### Phase: Validate

**Step 2 — Validate the draft** · Owner: play · Depends on: Step 1
Run the lens validator over the draft before the checkpoint:

```
python3 scripts/validate_quality.py --draft <working>/draft \
        --manifest <working>/draft/quality-manifest.yaml
```

**SE-3 (F3/C3):** the lens is gates-only — `content` has exactly the `gates` key, a
non-empty list, no how-to-test/coverage/environments.
**SE-4 (F4/C4):** every gate in the manifest grounds to a profile target or an ICE
constraint/failure — none invented.
**SE-5 (F5/C5):** no gate is a bare word; each reads as a checkable condition.
**SE-6 (F6/C6):** no gate's grounding source is another lens — quality read none.
**SE-7 (F7/C7):** every gate flagged `material` carries a decision that resolves to a
drafted decision record.
**SE-9 (F9/C9):** the lens and any decision carry their required v1 fields and valid
enums.
On any GAP, apply REC3/REC4/REC5/REC6/REC7/REC9 and re-run before the checkpoint.

### Phase: Checkpoint (mandatory — never skipped, C10)

**Step 3 — Human review** · Owner: play · Depends on: Step 2
Present the proposed quality lens **inline** — the list of gates, each with the target or
ICE rule it came from, and any decisions — for the human. This checkpoint is always
presented and never skippable. Approve → continue to persist; cancel → halt with nothing
written to the model.
**SE-10 (F10/C10):** the lens is persisted only after this approval — Step 4 is the sole
writer and depends on this step; no product-model file is written before Step 4.

### Phase: Apply

**Step 4 — Persist** · Owner: play · Depends on: Step 3
First **snapshot** the capability folder and the profile (so Step 5 can prove only the
quality lens changed and the profile was untouched). The snapshot is taken here, at the
gated apply step — never at pre-flight — so a resume can never compare post-apply against
post-apply:

```
mkdir -p <working>/cap-before
cp -R <product_base>/product-os/<capability>/. <working>/cap-before/
cp <product_base>/product-os/profile.yaml <working>/profile-before.yaml
```

Then persist on the fixed allowlist. `apply_quality.py` writes the quality lens
(overwrite — the re-derive) and copies decisions skip-if-exists; it is handed only the
draft, which holds only the lens + decisions, so it cannot write the ICE, the profile, or
another lens:

```
python3 scripts/apply_quality.py --draft <working>/draft --product-base <product_base> \
        --out-manifest <working>/apply-manifest.json
```

**SE-2 (F2/C2):** the apply manifest's `written` set holds only the capability's
`lens/quality.yaml` and `decisions/*.yaml`; its `refused` set is empty (the draft held
nothing out of scope).

**Step 5 — Verify persisted** · Owner: play · Depends on: Step 4
Check what landed against the snapshots:

```
python3 scripts/check_quality.py --cap-before <working>/cap-before \
        --cap-dir <product_base>/product-os/<capability> \
        --profile-before <working>/profile-before.yaml \
        --profile-after <product_base>/product-os/profile.yaml
```

**SE-8 (F8/C8):** every file in the capability folder is byte-identical to its snapshot
except `lens/quality.yaml`; the profile is byte-identical; no accepted decision was
edited in place; nothing was removed.
On any GAP, apply REC2/REC8 and re-run.

### Phase: Scenario Validation

**Step 6 — Scenario evals** · Owner: play · Depends on: Step 5
- **SCE-1 (S1 — quality lead, first run):** `quality.yaml` exists with a non-empty gates
  list; every other product-model file is byte-identical; the lens validates v1.
- **SCE-2 (S2 — security engineer, grounded):** every gate names a profile target or an
  ICE rule in the manifest; none ungrounded.
- **SCE-3 (S3 — QA engineer, checkable):** each gate is a clear pass/fail with a value
  where the target has one; no bare adjective.
- **SCE-4 (S4 — architect, it's first):** no ux/architecture/run/agentic lens was
  touched and no gate grounds on another lens.
- **SCE-5 (S5 — product owner, re-run):** a second run changes only `quality.yaml` (and
  possibly a new decision); other lenses, ICE, and profile are byte-identical; no
  accepted decision edited in place.
- **SCE-6 (S6 — reviewer, the checkpoint):** the checkpoint showed the gates and
  decisions inline before any write.

## Recovery

| For | Trigger | Direction | Handoff |
|-----|---------|-----------|---------|
| F1 | the capability is absent, not `active`, lacks rich ICE, or the profile is not firmed | halt and route to /shape (to shape it) or /understand (to enrich + firm) before /quality | human |
| F2 | a write touched something beyond this capability's quality lens or a decision | revert the out-of-scope write; /quality writes only the quality lens (and decisions) | autonomous |
| F3 | the lens carries non-gate content or the wrong shape | strip it back to the gates-only shape the schema requires | autonomous |
| F4 | an invented/ungrounded gate | drop it, or re-tie it to a profile target or an ICE constraint/failure; never keep an invented gate | autonomous |
| F5 | a vague or uncheckable gate | rewrite it as a concrete pass/fail with a value where the dimension needs one | autonomous |
| F6 | /quality read or depended on another lens | remove the dependency; /quality derives only from the ICE + profile (it is first) | autonomous |
| F7 | a material gate choice with no decision recorded | record the capability-level decision for the choice before persisting | autonomous |
| F8 | a non-lens/non-decision file changed, or an accepted decision was edited in place | restore it and re-apply only the quality lens (and the new decision), after a human confirms the restore | human |
| F9 | the lens or a decision fails v1 schema validation | re-emit the failing artifact to conform before the play completes | autonomous |
| F10 | gates were persisted before the checkpoint was approved | revert the premature write and re-present the checkpoint; persist only after the human approves | human |

## Pause and Resume

Steps run top to bottom. On entry, resolve config, run the readiness gate, resolve the
target capability from the play argument or the in-progress draft, check the status
marker, skip completed steps, reset any in-progress step to pending, and continue. The
pre-apply snapshot is captured at Step 4 (the gated apply step) and preserved on resume,
so the non-destructive comparison always diffs against true pre-apply state. A fresh
start with no marker runs everything and creates the marker at Step 1.
