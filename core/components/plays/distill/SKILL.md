# distill

Lightweight wrapper play invoked by ship after merge-pr. Delegates all analysis
to knowledge-extractor in FAST mode. Stages proposals to STM. Never writes to
product LTM. Returns gracefully on any failure.

## Compiled From

This play was compiled from `reference/intent.yaml` (the clean triple) and
`reference/expectation.yaml` (success_scenarios + recovery) by `/create-play`.
Intent defines constraints (C1–C4) and failure conditions (F1–F2); the expectation
defines success scenarios (S1–S2) and recovery (REC1–REC2).
To modify this play, update `reference/intent.yaml` or `reference/expectation.yaml`
and re-run `/create-play --build distill`.
Do NOT edit this file manually — it is a compiled artifact.

## Role

You are a minimal orchestrator. You chain one agent invocation and handle failure
gracefully. You do NOT invoke agents directly for domain work — knowledge-extractor
handles all analysis. You do NOT make decisions about proposals.

**Forbidden:** Direct diff analysis. Direct product LTM writes. Propagating
exceptions to the calling ship play. Any hard halt — return gracefully always.

**Agent boundaries:**

| Agent | Domain | Phase |
|-------|--------|-------|
| `knowledge-extractor` | FAST mode: diff analysis, STM evidence scan, proposal generation | Analysis |

## Pre-flight

Resolve configuration and gather inputs. All failures return gracefully — do NOT halt.

```bash
# Resolve paths from config
stm_base=$(grep 'base-path' .garura/core/config.yaml | head -1 | awk '{print $2}')
product_base=$(grep 'base-path' .garura/core/config.yaml | tail -1 | awk '{print $2}')

# Extract issue number (from branch or status file context from ship)
# issue is passed in from ship's context — already resolved

# Resolve default branch and get diff from git (no PR required — runs before PR creation)
default_branch=$(git remote show origin 2>/dev/null | grep 'HEAD branch' | awk '{print $NF}')
pr_diff=$(git diff ${default_branch}...HEAD 2>/dev/null)
```

Read the `evidence.record` flag from `.garura/core/config.yaml`. Default to `true` when key is absent:

```bash
evidence_record=$(grep -A1 '^evidence:' .garura/core/config.yaml | grep 'record:' | awk '{print $2}')
evidence_record=${evidence_record:-true}
```

When `evidence.record` is `false`, return immediately without invoking knowledge-extractor or writing proposals.yaml:
```
{ "status": "skipped", "reason": "evidence.record=false" }
```

**On any other pre-flight failure:** Return `{ "status": "skipped", "reason": "<error>" }`.
Do not halt. Do not propagate the error.

## Workflow

### Phase: Analysis (Single Step)

**Step 1 — Invoke knowledge-extractor FAST Mode**
Owner: `knowledge-extractor`
Depends on: pre-flight (graceful on failure)

```json
{
  "intent_path": "core/components/plays/distill/reference/intent.yaml",
  "stm_base": "{stm_base}",
  "task_id": "fast-learning-extraction",
  "mode": "fast",
  "stm": {
    "input": {
      "git_diff": "{pr_diff content from pre-flight — git diff default_branch...HEAD}",
      "product_base": "{product_base}",
      "issue_body": "{issue description — read from STM or passed from ship context}"
    },
    "output": {
      "proposals_path": "{stm_base}/{issue}/evidence/distill/proposals.yaml"
    }
  }
}
```

Agent reads the git diff and available STM evidence, assesses trivial vs.
non-trivial, invokes `distill` skill for proposal generation,
and writes `proposals.yaml` to the output path (or returns `no_learnings: true`).

**On success (no_learnings: true):**
- Return `{ "status": "completed", "no_learnings": true, "proposals_path": null }`

**On success (proposals written):**
- Return `{ "status": "completed", "no_learnings": false, "proposals_path": "{proposals_path}" }`

## Failure Handling

Any failure at any stage is caught and logged — never propagated to ship.

```
try:
  execute pre-flight and Step 1
catch any error:
  log warning: "distill skipped — {error message}"
  return { "status": "skipped", "reason": "{error message}", "proposals_path": null }
```

**Status values:**
- `completed` — analysis ran; proposals.yaml written OR no_learnings confirmed
- `skipped` — any failure; ship proceeds normally

## Scenario Validation

**Step 2 — Scenario Evals**
Owner: play
Depends on: Step 1

Scenario evals are sourced from `reference/expectation.yaml` `success_scenarios` — one per scenario, derived from each scenario's `measure`.

- **SCE-1 (S1 — Developer):** On a non-trivial PR with STM evidence, `proposals.yaml` exists under `{stm_base}/{issue}/evidence/distill/`, each proposal cites concrete diff evidence and/or an STM artifact path, the return struct has status `"completed"` with a non-null `proposals_path`, and no file under `{product_base}` was created or modified.
- **SCE-2 (S2 — Developer):** On a trivial PR with no non-trivial design evidence, no `proposals.yaml` is present under `{stm_base}/{issue}/evidence/distill/`, the return struct is exactly status `"completed"` with `no_learnings: true` and `proposals_path: null`, and no file under `{product_base}` was created or modified.

If either scenario eval fails: log failure in evidence, return gracefully. Do not halt ship.

This run closes with the **Standard Play Close**. Distill is a fire-and-forget
sub-play of `ship` — it is almost always invoked with `parent_run_id` in its
input contract.

```bash

```

**Step C1 — Sub-play result (fire-and-forget).** Distill writes
`proposals.yaml` to `{stm_base}/{issue}/evidence/distill/` and returns the
outcome struct (see the `## Output` section below) to ship — there is no
self-commit step. **Evidence files are committed by ship's final sweep (C9).**
This return-struct is distill's C1-equivalent when running as a sub-play.

**Step C2 — Delivery report.** **Skipped when `parent_run_id` is present in
the input contract** (the normal path — ship dispatches distill in the
background and absorbs the returned outcome struct into ship's own close
report). **Emitted only when distill is run directly by the user** (no
`parent_run_id`): fill `delivery-report.md` and output it —
- `## Distill Delivered — #{issue}`
- Run Summary: Play `distill`, Issue `#{issue}`, Status (COMPLETE | SKIPPED),
  Started (per the started_at precedence in play-close.md), Completed (now).
- Pipeline Steps: derived from distill's task DAG. Status PASS/SKIP/FAIL.
- Artifacts Produced: `proposals.yaml` path (or `none — no_learnings`).
- End with a pointer to `proposals.yaml`.

Fire-and-forget semantics are unchanged: any failure returns gracefully with
`status: "skipped"` and never halts ship.

## Output

Returns outcome struct to ship:

```json
{
  "status": "completed | skipped",
  "no_learnings": true,
  "proposals_path": "{path to proposals.yaml or null}",
  "reason": "{null on completed; error message on skipped}"
}
```

Ship reads this output to record Mode 1 outcome in its evidence file.

## Recovery

Sourced from `reference/expectation.yaml` `recovery` — one entry per failure condition. When the validator detects a tripped failure, it builds a recovery handoff plan from the matching entry and routes it per `handoff`: `autonomous` loops the fix back to the builder; `human` escalates for a manual call.

| ID | For | Symptom (trigger) | Direction | Handoff |
|----|-----|-------------------|-----------|---------|
| REC1 | F1 | distill propagated an exception or non-zero exit to ship, or otherwise halted or delayed the ship pipeline instead of returning a result struct | catch and log any error and return `{ status: "skipped", reason }` so ship continues — never let the failure escape to the caller | autonomous |
| REC2 | F2 | proposals.yaml (or any distill output) was written under `{product_base}` instead of `{stm_base}/{issue}/evidence/distill/` | re-stage the write to the STM evidence path and remove any artifact that landed under `{product_base}` — product LTM is read-only for this play | autonomous |

At Level 4, `intent-resolver` executes the autonomous entries (REC1, REC2) without a human; there are no human entries for this play.

## Pause and Resume

This play is fire-and-forget. It does not maintain its own status file. Ship tracks
`distill` as a task entry in `{stm_base}/{issue}/status/ship.json`:

```json
"distill": { "status": "pending | completed | skipped" }
```

Ship's resume logic skips this step if status is `"completed"` or `"skipped"`.
If status is `"in_progress"` (interrupted), ship resets to `"pending"` and re-runs —
the analysis is idempotent (proposals.yaml is overwritten if re-run).
