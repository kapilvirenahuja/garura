# Compiled Play Example — commit-code

This is the **target output format** for play-creator. It shows what a clean,
full compiled play looks like end to end. The play you generate should follow this
structure and carry every section shown here. (The example uses a Garura-style runtime
— config-resolved working base, structured agent contracts, a task DAG, and a standard
close. Mirror the *shape*; the concrete runtime details are whatever the target harness
provides.)

**Source:** compiled from a `commit-code` intent (C1–C7, F1–F4, S1–S2) using Structure
A (full checkpoint flow).

**Folder shape:** a compiled play with mechanical steps is a *folder* —
`commit-code/SKILL.md` plus `commit-code/scripts/` for the scripts its steps call. The
play below offloads its one mechanical step (Step 1) to a script; the rest is judgment and
orchestration, which stays in prose.

```markdown
---
name: commit-code
description: Commit changes grouped by concern with conventional messages and issue references. Use when you have uncommitted changes.

# commit-code

Commit all changed files grouped by concern, using conventional commit format with
issue references, and push to remote. Approval is required only when confidence is low.

## Compiled From

This play was compiled from the commit-code intent + expectation by play-creator.
Intent defines constraints (C1–C7) and failure conditions (F1–F4); the expectation
defines success scenarios (S1–S2) and one recovery entry per failure condition.
To modify this play, update the intent/expectation and recompile.
Do NOT edit this file manually — it is a compiled artifact.

## Role

You are the orchestrator. You own the workflow. You delegate domain tasks to agents
via structured contracts — never execute domain work directly.

**Forbidden:** direct git commands for analysis or commits; direct issue-tool commands.
All domain work goes through agents.

**Agent boundaries:**

| Agent | Domain | Phases |
|-------|--------|--------|
| `repo-orchestrator` | Git: analyze changes, create commits, push branch, draft brief | Preparation, Execution |
| `project-orchestrator` | Issues: fetch open issues, resolve issue mappings | Preparation |

## Pre-flight

| Check | Constraint | Action on Failure |
|-------|-----------|-------------------|
| Resolve working base from config | — | Hard halt |
| Branch guard (not on default branch) | C1 | Hard halt |
| Changes exist | C2 | Graceful exit |
| Sensitive file scan | C4 | Hard block |
| Open issues exist | C3 | Hard halt |

Resolve the working base once and hold it for every later path. Extract the issue
number from the branch name (e.g. `feature/95-slug` → 95).

**Resume check:** if a status marker for this play exists, resume — skip completed
steps, reset any in-progress step to pending, continue from the first incomplete one.

## Task DAG

Create ALL tasks immediately after resolving the working base — before any domain work.
The play owns this DAG; agents must not edit its top-level tasks.

```
[T1] Analyze Changes        blockedBy: []
[T2] Resolve Issues         blockedBy: [T1]
[T3] Create Commits         blockedBy: [T2]
[T4] Push Branch            blockedBy: [T3]
[T5] Scenario Validation    blockedBy: [T4]
[T6] Write Evidence         blockedBy: [T5]
```

Mark each task in-progress before its step and completed right after. No runtime
reordering. On resume, skip completed and reset in-progress to pending.

## Workflow

### Phase: Preparation

**Step 1 — Analyze Changes** · Owner: `repo-orchestrator` · Depends on: pre-flight
The mechanical part is offloaded: the step runs `scripts/group_changes.py`, which parses
the changeset against the sensitive-file patterns and emits the grouped analysis (one
group per concern, each file in exactly one group, sensitive files flagged). The agent
*decides* what to do with anything the script flags — it does not re-parse the diff by hand.
**Step 1 Eval:** groups are non-empty and each file is in exactly one group (F1); no
sensitive file staged without approval (C4).

**Step 2 — Resolve Issues** · Owner: `project-orchestrator` · Depends on: Step 1
The play dispatches a JSON contract; the subagent gathers context, invokes the issue-mapping
skill, and writes the result to disk — the contract moves only paths, never the data:

    {
      "task":    "map each change group to an open issue with a confidence score",
      "inputs":  { "analysis":  "<working>/analysis.json" },
      "outputs": { "issue_map": "<working>/issue-map.json" }
    }

It returns the same contract with `outputs.issue_map` confirmed on disk — not the mappings
inline.
**Step 2 Eval:** every group maps to an open issue with a confidence score (C3, F4).

### Phase: Checkpoint (skip when: all mappings high-confidence AND no sensitive files)

**Step 3 — Draft Brief** · Owner: `repo-orchestrator` · Depends on: Steps 1–2
Agent turns the STM artifacts into a human-readable brief; the play renders it.

**Step 4 — Human Review** · Owner: play · Depends on: Step 3
Present the brief → approve to proceed, or cancel to halt.

### Phase: Execution

**Step 5 — Create Commits** · Owner: `repo-orchestrator` · Depends on: Step 4 (or Step 2 if skipped)
**Critical:** the agent reads the Step 1–2 artifacts, never the Step 3 brief.
**Step 5 Eval:** every commit uses `type(scope): subject (#issue)` (C5, C6, F1–F3).

**Step 6 — Push Branch** · Owner: `repo-orchestrator` · Depends on: Step 5
**Step 6 Eval:** local HEAD matches remote after push (C7); push failure logs a
warning and does not halt.

### Phase: Scenario Validation

**Step 7 — Scenario Evals** · Owner: play · Depends on: Step 6
- **SCE-1 (S1 — Code reviewer):** each commit has a clear scope, traces to an issue,
  and uses conventional format — reviewable from the commit alone.
- **SCE-2 (S2 — Team lead):** all commits share a consistent format; per-issue progress
  is reportable from the log.

### Phase: Evidence & Close

**Step 8 — Close** · Owner: play · Depends on: Step 7
Emit the user-facing delivery report (run summary, pipeline steps from the DAG,
artifacts produced, optional next steps). Write the evidence file when recording is on;
otherwise note that it was skipped.

## Recovery

| For | Trigger | Direction | Handoff |
|-----|---------|-----------|---------|
| F1 | a file landed in two groups or none | re-run the analysis so every file is in exactly one group | autonomous |
| F2 | a commit message breaks conventional format | rewrite the offending messages to the convention | autonomous |
| F3 | a commit is missing its issue reference | amend to add the issue reference | autonomous |
| F4 | a group mapped to a closed or wrong issue | a human confirms the correct issue before committing | human |

## Pause and Resume

Steps run top to bottom. Resolve the issue from the branch name, check the status
marker, skip completed steps, reset in-progress to pending, continue. Fresh start with
no marker runs everything and creates the marker on step 1.
```
