# vision

Turn a business goal into the **seed** of the product model: one domain node, its
candidate capability nodes, a goals-only ICE per capability, and a directional draft
product profile. Everything is deliberately shallow and directional — /vision plants
the seed; the deeper work (full intent, context, expectations, concrete NFR targets,
functionalities, a firm profile) is done by /understand and /shape.

**Pipeline position: none.** /vision is a strategic, model-building play in the
shaping pipeline. It opens no delivery issue and cuts no branch, so the D2
pipeline-position rule injects neither a `start-change` head nor a close sequence.
It writes the persistent product model directly (additively).

## Role

You are the orchestrator. You own the workflow and the step order. You delegate the
domain work — KB grounding and seed authoring — to the `product-os-keeper` agent via
JSON contracts over files on disk, and you run the mechanical checks (schema/scope
validation, grounding, the additive persist) through bundled scripts. You never write
the product model YAML yourself, and you never persist anything before the human
approves the single checkpoint (C7).

**Forbidden:** hand-writing node/ICE/profile YAML; writing to `product_base` by any
route other than `scripts/apply_seed.py`; persisting the seed before Step 4 approval.

**Agent boundaries:**

| Agent | Domain | Skill it invokes | Phases |
|-------|--------|------------------|--------|
| `product-os-keeper` | Ground the goal in the KB; draft the directional seed (nodes + goals-only ICE + directional profile) | `search-kb`, `propose-kb-node`, `author-vision-seed` | Grounding, Draft |

`product-os-keeper` is the single **domain agent** this play uses (1 of the ≤5
budget). No utility agents are needed — git/issue machinery is absent (position none).

## Workflow

### Phase: Grounding

**Step 1 — Ground the goal** · Owner: `product-os-keeper` · Depends on: pre-flight
The agent does two things, in order: (1) route the goal to its **domain** with
`search-kb`, then (2) read that domain's full shelf through the KB router
(`python3 $KB shelf <domain>`) to **enumerate the candidate capability set** the
domain offers — breadth, not a single placement — keeping those a directional seed
should include. For any capability or whole domain the KB does not cover, it raises a
`propose-kb-node` proposal. It writes the grounding (domain + the candidate-capability
set, each with its KB shelf reference) and any proposals to disk:

    {
      "task":    "route the goal to its domain, then enumerate the candidate capability set from that domain's shelf; propose KB nodes for gaps",
      "inputs":  { "goal": "<the business goal>" },
      "outputs": { "grounding": "<working>/grounding.yaml",
                   "proposals_dir": "<working>/proposals/" }
    }

It returns the same contract with the output paths confirmed on disk — never the
routing data inline. The grounding must carry **one or more** candidate capabilities
(a domain seed is rarely a single capability). A net-new domain routes through
`propose-kb-node`'s own review; record that proposal as evidence and proceed — do not
block the seed on a KB-write approval (the single human gate is Step 4).

### Phase: Draft

**Step 2 — Draft the seed** · Owner: `product-os-keeper` · Depends on: Step 1
The agent invokes `author-vision-seed` to draft the directional seed into a draft
folder under STM — domain node, capability nodes (`status: proposed`), goals-only ICE
per capability, and a directional profile — plus a `seed-manifest.yaml`:

    {
      "task":    "draft the directional vision seed from the goal + grounding",
      "inputs":  { "goal": "<the business goal>",
                   "grounding": "<working>/grounding.yaml",
                   "proposals_dir": "<working>/proposals/",
                   "product_base": "<product_base>" },
      "outputs": { "draft_dir": "<working>/draft/",
                   "manifest":  "<working>/draft/seed-manifest.yaml" }
    }

The skill reads `product_base` **read-only** (to draft only absent nodes) and writes
the draft tree only — never the live model.

**Step 3 — Validate the draft** · Owner: play · Depends on: Step 2
Run the mechanical checks over the draft before showing it to the human:

```
python3 scripts/validate_seed.py --root <working>/draft/product-os
python3 scripts/grounding_check.py --manifest <working>/draft/seed-manifest.yaml \
        --proposals-dir <working>/proposals
```

**SE-1 (F1/C1):** `validate_seed.py` exits 0 — every drafted node, ICE, and profile
conforms to its v1 schema (required fields, valid type/status/state values).
**SE-2 (F2/C2):** no functionality node appears in the draft — depth stops at
capability (`validate_seed.py` counts `functionality: 0`).
**SE-3 (F2/C3):** every capability node is `status: proposed` and the profile is
`state: directional` — no `active`/`set`/`locked` (`validate_seed.py`).
**SE-4 (F2/C4):** every seed ICE has non-empty `intent.goals` and empty
constraints/failures, context, expectations, `nfr_needs`, `compliance_needs`
(`validate_seed.py`).
**SE-5 (F3/C5):** `grounding_check.py` exits 0 — every capability in the manifest
carries a KB shelf match or an existing KB-node proposal; none is ungrounded.
On any GAP, apply the matching recovery (REC1–REC3) and re-run before the checkpoint.

### Phase: Checkpoint (mandatory — never skipped, C7)

**Step 4 — Human review** · Owner: play · Depends on: Step 3
Present the proposed seed **inline** — the domain, each candidate capability with its
one-line summary and its seed goals, and the directional profile (shape + rough NFR
levels). This checkpoint is always presented and never skippable. Approve → continue
to persist; cancel → halt with nothing written to the model.
**SE-7 (F5/C7):** the seed is persisted only after this approval — Step 5 is the sole
writer to `product_base` and depends on this step; no product-model file exists before
Step 5 runs.

### Phase: Apply

**Step 5 — Persist the seed** · Owner: play · Depends on: Step 4
On approval, persist the draft additively. `apply_seed.py` is skip-if-exists: it
copies each draft artifact into the live model only when that path does not already
exist, and emits a `{written, skipped}` manifest — captured to disk so Step 6 can
validate exactly what this run wrote:

```
python3 scripts/apply_seed.py --draft <working>/draft --product-base <product_base> \
        > <working>/apply-manifest.json
```

**SE-6 (F4/C6):** the apply manifest's `written` list contains no path that already
existed (those appear only in `skipped`) — the skip-if-exists writer makes overwriting
an existing node/ICE/profile structurally impossible.

**Step 6 — Validate the model** · Owner: play · Depends on: Step 5
Re-validate **only the files this run wrote** — scoped to the apply manifest's
`written` list. The live model may already hold matured nodes (an `active` capability,
a functionality, a `set` profile from a prior /understand or /shape on another domain)
that /vision never touched and must not be judged by seed-only rules. Validating the
whole tree here would throw false failures on legitimately-progressed nodes — so the
post-apply check is scoped:

```
python3 scripts/validate_seed.py --root <product_base>/product-os \
        --apply-manifest <working>/apply-manifest.json
```

**SE-1 (F1/C1):** the node/ICE/profile this run persisted validate against the v1 schemas.
**SE-2 (F2/C2):** none of the files this run wrote is a functionality node.
**SE-3 (F2/C3):** the capabilities this run wrote are `proposed`; a profile this run
wrote is `directional`.
**SE-4 (F2/C4):** the seed ICE this run wrote remains goals-only.

### Phase: Scenario Validation

**Step 7 — Scenario evals** · Owner: play · Depends on: Step 6
- **SCE-1 (S1 — product strategist):** the seed this run persisted — a domain node, at
  least one `proposed` capability under it, a goals-only ICE per capability, and a
  `directional` profile — is schema-valid (`validate_seed.py` scoped to the apply
  manifest is clean) and present in the live model.
- **SCE-2 (S2 — architect):** every persisted capability traces to a KB shelf or a
  recorded proposal in the seed manifest (`grounding_check.py` is clean).
- **SCE-3 (S3 — product owner, non-destructive re-run):** on a re-run over an existing
  domain, the apply manifest's `skipped` list holds every pre-existing path and
  `written` holds only newly-added capabilities — existing files are untouched.
- **SCE-4 (S4 — reviewer):** the Step 4 checkpoint showed the domain, the capabilities
  with summaries and goals, and the directional profile inline, and no product-model
  file was written before that approval.

## Recovery

| For | Trigger | Direction | Handoff |
|-----|---------|-----------|---------|
| F1 | a written node/ICE/profile fails v1 schema validation | re-emit the failing artifact to conform to its schema before the play completes | autonomous |
| F2 | a functionality node, a `set`/`locked` profile, or context/expectations/nfr_needs/compliance_needs appears | strip the over-reach — drop the functionality, reset the profile to `directional`, remove the extra ICE sections — leaving only seed scope | autonomous |
| F3 | a capability has neither a KB shelf match nor a KB-node proposal | search the KB to ground it, or record a propose-kb-node proposal; never leave it ungrounded | autonomous |
| F4 | the content of an existing node/ICE/profile changed during the run | restore the prior content and re-apply only the additive seed for absent nodes, after a human confirms the restore | human |
| F5 | a product-model file was written before the checkpoint was approved | revert the premature write and re-present the checkpoint; persist only after the human approves | human |

## Pause and Resume

Steps run top to bottom. On entry, resolve config, derive `product_slug` from the goal
or the in-progress draft, check the status marker, skip completed steps, reset any
in-progress step to pending, and continue. A fresh start with no marker runs everything
and creates the marker at Step 1.
