# shape

Take one domain that /understand has firmed — capabilities enriched, profile `set` —
and select what to build: confirm the capabilities that stay (prune the rest), choose
the functionalities under each kept capability, and author each functionality's
build-unit ICE, the personas it serves, and the journeys those personas travel. /shape
reads the profile to judge fit but **never writes it** — a selection that needs more
than the box halts for the human to run /understand. One domain per run.

**Pipeline position: none.** /shape is a strategic, model-building play in the shaping
pipeline. It opens no delivery issue and cuts no branch, so the D2 pipeline-position
rule injects neither a `start-change` head nor a close sequence. It writes the
persistent product model directly.

## Role

You are the orchestrator. You own the workflow and the step order. You delegate the
domain work — the grounded selection bundle — to the `product-os-keeper` agent via a
JSON contract over files on disk, and you run the mechanical work (readiness gate,
bundle validation, the allowlisted persist, and the post-apply checks) through bundled
scripts. You never write the model YAML yourself, you never write the product profile,
and you never persist before the human approves the single checkpoint (C9).

**Forbidden:** hand-writing node/ICE/persona/journey/decision YAML; writing the product
profile by any route (it is /understand's artifact); reparenting, renaming, or deleting
nodes; persisting by any route other than `scripts/apply_shape.py`; persisting before
Step 3 approval.

## Workflow

### Phase: Draft

**Step 1 — Draft selection bundle** · Owner: `product-os-keeper` · Depends on: pre-flight
The persisted model does not record which KB shelf a domain came from, and the product
domain folder name (`order-management`) is not the KB shelf name (`ecommerce`). So the
agent first **recovers the shelf** with `search-kb`, then invokes `author-shape-bundle`
to draft the domain's selection bundle to STM — keep/prune per capability, selected
functionalities with build-unit ICE, personas, journeys, the decisions, AND the domain's
vertical slices (bundling functionalities into usable increments, each referencing its
ICE, plus the `_deferred` bucket), with stable ids:

    {
      "task":    "recover this domain's KB shelf via search-kb, then select what to build against the firmed profile + that shelf; draft the bundle AND compose the domain's vertical slices (every selected functionality in a slice or _deferred; slices reference ICE, carry no order/effort/depends_on)",
      "inputs":  { "domain_path":  "<product_base>/product-os/<domain>",
                   "profile_path": "<product_base>/product-os/profile.yaml" },
      "outputs": { "routing":      "<working>/routing.yaml",
                   "draft_dir":    "<working>/draft/",
                   "manifest":     "<working>/draft/shape-manifest.yaml",
                   "slices":       "<working>/draft/product-os/<domain>/slices/" }
    }

The agent passes the shelf `search-kb` resolved to `author-shape-bundle` as its
`kb_shelf` — the grounding is the shelf the router actually returned, not the product
domain name. It returns the contract with the output paths on disk — never inline
content. The skill reads the profile **read-only** and writes only the draft; an
out-of-box functionality is deferred to /understand and recorded in the manifest's
`deferred_out_of_box`.
**SE-1 (F1/C1):** the readiness gate passed at pre-flight — the profile is `set` and
the target capabilities have rich ICE; otherwise the run halted (REC1).

### Phase: Validate

**Step 2 — Validate the draft** · Owner: play · Depends on: Step 1
Run the bundle validator over the draft before the checkpoint:

```
python3 scripts/validate_shape.py --draft <working>/draft \
        --manifest <working>/draft/shape-manifest.yaml --product-base <product_base>
```

**SE-2 (F2/C4):** `validate_shape.py` is schema-clean — functionality nodes, ICE,
personas, journeys, and decisions carry their required v1 fields and valid enums.
**SE-3 (F4/C5):** tree integrity holds — every functionality has a parent, every ICE
persona ref resolves to a persona record, every journey references a real persona and
node.
**SE-4 (F3/C3):** grounding holds — every kept capability and selected functionality in
the manifest carries a KB shelf or a recorded proposal; none is invented.
**SE-5 (C2):** the bundle actually selects — the validator's counts show the
functionalities, personas, and journeys the selection calls for were drafted.
**SE-11 (F10/C12):** slice coverage holds — every selected functionality appears in a
slice or the `_deferred` bucket; `validate_shape.py` reports no unplaced functionality.
**SE-12 (F11/C11):** every slice references ICE by a resolvable `ice_ref` and embeds no
ICE body content — slices point at the ICE, never copy it.
**SE-13 (F12/C13):** no slice carries `order`, `effort`, or resolved `depends_on` —
/shape composes slices; planning them is /roadmap's.
On any GAP, apply REC2/REC3/REC4/REC10/REC11/REC12 and re-run before the checkpoint.

### Phase: Checkpoint (mandatory — never skipped, C9)

**Step 3 — Human review** · Owner: play · Depends on: Step 2
Present the domain's selection bundle **inline** — the kept capabilities, the pruned
capabilities (each with its reason), the selected functionalities, the personas, the
journeys, AND the vertical slices (each with its outcome and the functionalities it
bundles) plus the `_deferred` bucket — plus any `deferred_out_of_box` items routed to
/understand. Show the slices as the buildable increments, and say plainly that their
order, effort, and dependencies come from /roadmap, not here. This checkpoint is always
presented and never skippable. Approve → continue to persist;
cancel → halt with nothing written to the model.
**SE-9 (F8/C9):** the selection is persisted only after this approval — Step 4 is the
sole writer and depends on this step; no product-model file is written before Step 4.

### Phase: Apply

**Step 4 — Persist** · Owner: play · Depends on: Step 3
First **snapshot** the pre-run profile and each to-be-flipped capability node (so Step 5
can prove the profile was untouched and only `status` changed):

```
cp <product_base>/product-os/profile.yaml <working>/profile-before.yaml
mkdir -p <working>/caps-before   # copy each flipped capability's node.yaml as <capability-id>.yaml
```

Then persist on the fixed allowlist. `apply_shape.py` copies new records skip-if-exists
(stable ids ⇒ no duplicates on re-run) and flips capability `status` with a read-modify-
write that changes only that field. It is never handed the profile path, so it cannot
write the profile:

```
python3 scripts/apply_shape.py --draft <working>/draft --product-base <product_base> \
        --manifest <working>/draft/shape-manifest.yaml --out-manifest <working>/apply-manifest.json
```

**SE-8 (F7/C8):** the apply manifest's `written` set holds no path that already existed
(re-run additions only); a pruned capability is flipped to `deprecated`, never deleted.

**Step 5 — Verify persisted** · Owner: play · Depends on: Step 4
Check what landed against the snapshots and the manifests:

```
python3 scripts/check_shape.py --apply-manifest <working>/apply-manifest.json \
        --shape-manifest <working>/draft/shape-manifest.yaml \
        --profile-before <working>/profile-before.yaml --profile-after <product_base>/product-os/profile.yaml \
        --product-base <product_base> --caps-before-dir <working>/caps-before
```

**SE-6 (F5/C6):** the product profile is byte-identical before and after, and no profile
path appears in the written set — /shape never wrote the box.
**SE-7 (F6/C7):** each flipped capability node differs from its snapshot in the `status`
field only — no reparent, rename, or other out-of-scope structural mutation.
**SE-10 (F9/C10):** a decision record exists for every prune and every material
selection named in the manifest.

### Phase: Scenario Validation

**Step 6 — Scenario evals** · Owner: play · Depends on: Step 5
- **SCE-1 (S1 — product strategist, first selection):** kept capabilities are `active`,
  selected functionalities exist with build-unit ICE, personas and journeys are created,
  all schema-valid.
- **SCE-2 (S2 — architect, grounding):** every kept capability and selected
  functionality traces to a KB shelf or a recorded proposal in the manifest.
- **SCE-3 (S3 — product owner, prune is soft):** a pruned capability is `deprecated`
  with a recorded decision and still present — not deleted.
- **SCE-4 (S4 — reviewer, profile untouched):** the `profile.yaml` is byte-identical
  before and after and absent from the written set.
- **SCE-5 (S5 — product owner, re-run):** a second run duplicates no persona, journey,
  or functionality — the written set holds only genuinely new selections.
- **SCE-6 (S6 — QA engineer, the checkpoint):** the checkpoint showed kept/pruned
  capabilities, functionalities, personas, journeys, AND the slices + deferred bucket
  inline, before any write.
- **SCE-7 (S7 — delivery lead, slices):** each slice has a name, outcome, acceptance
  intent, and functionalities that reference ICE (resolvable `ice_ref`, no embedded ICE
  body) and carry no `order`/`effort`/`depends_on`.
- **SCE-8 (S8 — planner, full coverage):** the union of all slices' `functionality_ref`s
  and the `_deferred` bucket equals the selected functionalities — none unplaced.

## Recovery

| For | Trigger | Direction | Handoff |
|-----|---------|-----------|---------|
| F1 | the profile is `directional`, or a target capability lacks rich ICE, at start | halt and route to /understand to firm the profile and enrich the capabilities first | human |
| F2 | a written functionality, ICE, persona, journey, or decision fails v1 schema | re-emit the failing artifact to conform to its schema before the play completes | autonomous |
| F3 | a selected functionality or kept capability has no KB shelf match and no proposal | ground it against the KB, or record a propose-kb-node proposal; never keep an invented selection | autonomous |
| F4 | an orphaned functionality, a dangling persona ref, or a journey referencing a missing persona/node | repair the references before persisting — reparent under an active capability, create the persona, or fix the journey | autonomous |
| F5 | a write touched the product profile | revert the profile change; an out-of-box need routes to /understand, never to a /shape profile write | human |
| F6 | a structural mutation beyond scope (reparent, rename, create/delete a capability/domain, or a non-status capability edit) | revert the out-of-scope mutation; /shape creates functionalities and flips capability status only | autonomous |
| F7 | a hard delete, an in-place overwrite of an ICE/decision, or a duplicated persona/journey/functionality | restore the record, switch the prune to `deprecated`, and de-duplicate by stable id, after a human confirms the restore | human |
| F8 | the selection persisted with no checkpoint approval | revert the premature write and re-present the checkpoint; persist only after the human approves | human |
| F9 | a prune or material selection with no decision record | write the decision (ADR) for each prune and material selection before persisting | autonomous |
| F10 | a selected functionality is in neither a slice nor the `_deferred` bucket | place it — add it to a slice, or record it in `_deferred` with a reason — before persisting | autonomous |
| F11 | a slice copied ICE content, or references a functionality whose ICE does not resolve | replace copied content with an `ice_ref` path, or fix the reference to the real ICE; never duplicate ICE | autonomous |
| F12 | a slice carries `order`, `effort`, or resolved `depends_on` | strip those fields — /shape composes slices; /roadmap plans them | autonomous |
