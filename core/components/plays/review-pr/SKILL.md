**Confidence formula (C9, built-in):**
```
confidence = 0.25 * scan_coverage
           + 0.35 * citation_rate
           + 0.20 * intent_alignment
           + 0.20 * precedent_match
```
Weights sum to 1.0. `citation_rate` carries the highest weight because it directly enforces F1.

**Step 3 — Confidence & Routing**
Owner: `tech-designer` (domain)
Depends on: Step 2

```json
{
  "intent_path": "core/components/plays/review-pr/reference/intent.yaml",
  "stm_base": "{stm_base}",
  "stm": {
    "input": {
      "findings": "{stm_base}/{issue}/evidence/review-pr/findings.yaml",
      "context": "{stm_base}/{issue}/evidence/review-pr/context.yaml"
    },
    "output": {
      "confidence": "{stm_base}/{issue}/evidence/review-pr/confidence.yaml"
    }
  },
  "task_id": "confidence-routing",
  "config_inputs": {
    "confidence_threshold": ".garura/core/config.yaml#review-pr.confidence_threshold"
  }
}
```

Agent computes the confidence formula:
```
confidence = 0.25*scan_coverage + 0.35*citation_rate + 0.20*intent_alignment + 0.20*precedent_match
```
Where:
- `scan_coverage` is read from `findings.yaml` `meta.scan_coverage`.
- `citation_rate` = (findings with valid `standard_id`) / (total findings); if total findings == 0, citation_rate = 1.0.
- `intent_alignment` = heuristic 0.0–1.0 reflecting whether `findings` cluster in domains relevant to `intent_summary`. Default 0.5 if not derivable.
- `precedent_match` = fraction of findings whose `standard_id` appears in prior STM resolution traces for this repo. Default 0.5 if STM has no prior traces.

Routing decision:
- `block` iff `findings.counts.P1 > 0`
- `escalate` iff `confidence < confidence_threshold`
- `pass` otherwise

Emit `confidence.yaml`:
```yaml
confidence: 0.62
inputs: { scan_coverage: 0.83, citation_rate: 1.0, intent_alignment: 0.4, precedent_match: 0.5 }
threshold: 0.5
routing: block | escalate | pass
reasons: ["1 P1 finding (SEC-19)", "..."]
```

**Step 3 Eval — SE-5 (F5/F6):** `confidence.yaml` routing decision is `block` iff `findings.counts.P1 > 0`. Routing decision is `escalate` iff `confidence < confidence_threshold`. Otherwise `pass`.

```json
    "activity_window_days": ".garura/core/config.yaml#review-pr.activity_window_days"

```

Agent runs per changed path:
```bash

```
Aggregates committer emails across all `changed_paths`. Excludes the PR author (from `pr-meta.yaml`). Recency-weighted ranking:
- last 30 days × 3
- 30–60 days × 2
- 60–`activity_window_days` × 1

Selects top `reviewer_count`. Falls through to `review-pr.fallback_reviewers` if the candidate pool is short.

Emits `reviewers.yaml`:
```yaml

candidates_total: 5
selected_reviewers: ["alice@org", "bob@org", "carol@org"]
fallback_used: false
fallback_count: 0
author_excluded: "dave@org"
```

**Step 4 Evals:**
- **SE-6 (C6/F4):** `reviewers.yaml` `selected_reviewers` list does NOT contain the PR author and uses `git log` bounded by `activity_window_days` from config.
- **SE-7 (C7/F3):** When candidate count < `reviewer_count`, `reviewers.yaml` shows `fallback_used: true` and `selected_reviewers` is drawn from `review-pr.fallback_reviewers`. If `fallback_reviewers` is empty AND candidates short, play HALTs with structured failure F3.

## Recovery

Sourced from `reference/expectation.yaml` `recovery` — one entry per failure condition. When the validator detects a tripped failure, it builds a recovery handoff plan from the matching entry and routes it per `handoff`: `autonomous` loops the fix back to the builder; `human` escalates for a manual call.

| ID | For | Symptom (trigger) | Direction | Handoff |
|----|-----|-------------------|-----------|---------|
| REC1 | F1 | a finding has no cited standard ID, or one not in the PR severity taxonomy | drop or re-tag any finding whose standard ID is missing or unknown before the findings set is accepted | autonomous |
| REC2 | F2 | the play or quality-check-scoped read a file outside changed paths, or invoked /quality-check | re-scope the read set to the diff and re-run the diff-bounded skill; never run the full-repo check | autonomous |
| REC3 | F3 | reviewer pool is below reviewer_count AND fallback_reviewers is empty | a human must name reviewers or populate review-pr.fallback_reviewers — the builder cannot invent owners | human |
| REC4 | F4 | the PR author appears in the selected reviewer list | remove the author from the candidate set and re-select from the remaining committers | autonomous |
| REC5 | F5 | a P1 finding is present but routing is not "block" or the PR is not set to request-changes | recompute routing from the findings, force block, and set the PR to request-changes | autonomous |
| REC6 | F6 | confidence is below the threshold but no reviewers are requested | add the selected reviewers to the PR when confidence falls below the threshold | autonomous |
| REC7 | F7 | severity classification differs across two back-to-back runs on the same diff | re-source severity verbatim from the PR severity taxonomy row for each standard ID so two runs agree | autonomous |
| REC8 | F8 | the PR severity taxonomy file or any of the six review-pr config keys is missing | a human must supply the missing taxonomy file or config keys before any review runs | human |
| REC9 | F9 | the play dispatches more than 5 domain agents | collapse or drop dispatches so the domain-agent count is back within budget | autonomous |

At Level 4, `intent-resolver` executes the autonomous entries (REC1, REC2, REC4, REC5, REC6, REC7, REC9) without a human; the human entries (REC3, REC8) always escalate. Runtime recovery still follows the structured-failure protocol — read `domain_assessment.responsible_domain` to route the fix to the responsible agent, with a max of 2 retry cycles per agent before halting with full failure context.
