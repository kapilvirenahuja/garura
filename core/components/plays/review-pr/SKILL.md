# Review PR

Diff-scoped, deterministic PR review. Play, Structure A (checkpoint-capable, multi-agent, confidence-gated).

## Compiled From

This play was compiled from `reference/intent.yaml` by `/create-play`.
Intent defines constraints (C1–C10), failure conditions (F1–F9), and scenarios (S1–S4).
To modify this play, update intent.yaml and re-run `/create-play --build review-pr`.
Do NOT edit this file manually — it is a compiled artifact.

## Role

You are the orchestrator. You own the workflow. You delegate domain tasks to agents via JSON contracts — never execute domain work directly.

**Forbidden:** Direct `git`/`gh` commands, direct file reads outside the play's pre-flight, invoking `/quality-check` or any full-repo quality skill (C1, C5, C10).

**Agent boundaries:**

| Agent | Domain | Phases | Role type |
|---|---|---|---|
| `repo-orchestrator` | Pre-flight: fetch PR diff/paths/body via `gh pr view` + `gh pr diff` | Pre-flight | Utility (exempt) |
| `tech-designer` | Resolve standards set + bind intent summary; emit `context.yaml` | Step 1 Context Load | Domain |
| `quality-auditor` | Invoke `quality-check-scoped` against the diff; emit `findings.yaml` | Step 2 Scoped Quality Eval | Domain |

| `repo-orchestrator` | Post structured PR comment, request-changes/add-reviewer, evidence self-commit | Steps 5–6 | Utility (exempt) |

**Domain agent count:** 3 unique (`tech-designer`, `quality-auditor`, `repo-orchestrator-as-ranker`). Within L2 budget of 5 (C5).

**Forbidden play actions (C10):** Never invoke `create-pr` or modify any file under `core/components/plays/create-pr/`.

**Confidence formula (C9, built-in):**
```
confidence = 0.25 * scan_coverage
           + 0.35 * citation_rate
           + 0.20 * intent_alignment
           + 0.20 * precedent_match
```
Weights sum to 1.0. `citation_rate` carries the highest weight because it directly enforces F1.

**Step 3 — Confidence & Routing**
Owner: `tech-designer` (domain)
Depends on: Step 2

```json
{
  "intent_path": "core/components/plays/review-pr/reference/intent.yaml",
  "stm_base": "{stm_base}",
  "stm": {
    "input": {
      "findings": "{stm_base}/{issue}/evidence/review-pr/findings.yaml",
      "context": "{stm_base}/{issue}/evidence/review-pr/context.yaml"
    },
    "output": {
      "confidence": "{stm_base}/{issue}/evidence/review-pr/confidence.yaml"
    }
  },
  "task_id": "confidence-routing",
  "config_inputs": {
    "confidence_threshold": ".meridian/core/config.yaml#review-pr.confidence_threshold"
  }
}
```

Agent computes the confidence formula:
```
confidence = 0.25*scan_coverage + 0.35*citation_rate + 0.20*intent_alignment + 0.20*precedent_match
```
Where:
- `scan_coverage` is read from `findings.yaml` `meta.scan_coverage`.
- `citation_rate` = (findings with valid `standard_id`) / (total findings); if total findings == 0, citation_rate = 1.0.
- `intent_alignment` = heuristic 0.0–1.0 reflecting whether `findings` cluster in domains relevant to `intent_summary`. Default 0.5 if not derivable.
- `precedent_match` = fraction of findings whose `standard_id` appears in prior STM resolution traces for this repo. Default 0.5 if STM has no prior traces.

Routing decision:
- `block` iff `findings.counts.P1 > 0`
- `escalate` iff `confidence < confidence_threshold`
- `pass` otherwise

Emit `confidence.yaml`:
```yaml
confidence: 0.62
inputs: { scan_coverage: 0.83, citation_rate: 1.0, intent_alignment: 0.4, precedent_match: 0.5 }
threshold: 0.5
routing: block | escalate | pass
reasons: ["1 P1 finding (SEC-19)", "..."]
```

**Step 3 Eval — SE-5 (F5/F6):** `confidence.yaml` routing decision is `block` iff `findings.counts.P1 > 0`. Routing decision is `escalate` iff `confidence < confidence_threshold`. Otherwise `pass`.

**Step 4 — Reviewer Selection**
Owner: `repo-orchestrator` (domain — ranker)
Depends on: Step 3

```json
{
  "intent_path": "core/components/plays/review-pr/reference/intent.yaml",
  "stm_base": "{stm_base}",
  "stm": {
    "input": {
      "pr_meta": "{stm_base}/{issue}/evidence/review-pr/pr-meta.yaml"
    },
    "output": {
      "reviewers": "{stm_base}/{issue}/evidence/review-pr/reviewers.yaml"
    }
  },
  "task_id": "reviewer-selection",
  "config_inputs": {
    "reviewer_count": ".meridian/core/config.yaml#review-pr.reviewer_count",
    "fallback_reviewers": ".meridian/core/config.yaml#review-pr.fallback_reviewers",
    "activity_window_days": ".meridian/core/config.yaml#review-pr.activity_window_days"
  }
}
```

Agent runs per changed path:
```bash
git log --since="${activity_window_days} days ago" --format='%ae|%aI' -- "$path"
```
Aggregates committer emails across all `changed_paths`. Excludes the PR author (from `pr-meta.yaml`). Recency-weighted ranking:
- last 30 days × 3
- 30–60 days × 2
- 60–`activity_window_days` × 1

Selects top `reviewer_count`. Falls through to `review-pr.fallback_reviewers` if the candidate pool is short.

Emits `reviewers.yaml`:
```yaml
window_days: 90
candidates_total: 5
selected_reviewers: ["alice@org", "bob@org", "carol@org"]
fallback_used: false
fallback_count: 0
author_excluded: "dave@org"
```

**Step 4 Evals:**
- **SE-6 (C6/F4):** `reviewers.yaml` `selected_reviewers` list does NOT contain the PR author and uses `git log` bounded by `activity_window_days` from config.
- **SE-7 (C7/F3):** When candidate count < `reviewer_count`, `reviewers.yaml` shows `fallback_used: true` and `selected_reviewers` is drawn from `review-pr.fallback_reviewers`. If `fallback_reviewers` is empty AND candidates short, play HALTs with structured failure F3.

### Phase: Execution

**Step 5 — Post & Route**
Owner: `repo-orchestrator` (utility)
Depends on: Step 4 (or Step 4.5 if checkpoint ran)

```json
{
  "intent_path": "core/components/plays/review-pr/reference/intent.yaml",
  "stm_base": "{stm_base}",
  "stm": {
    "input": {
      "findings": "{stm_base}/{issue}/evidence/review-pr/findings.yaml",
      "confidence": "{stm_base}/{issue}/evidence/review-pr/confidence.yaml",
      "reviewers": "{stm_base}/{issue}/evidence/review-pr/reviewers.yaml",
      "pr_meta": "{stm_base}/{issue}/evidence/review-pr/pr-meta.yaml"
    },
    "output": {
      "comment_record": "{stm_base}/{issue}/evidence/review-pr/comment-record.yaml"
    }
  },
  "task_id": "post-and-route"
}
```

Agent constructs the structured comment using the template:
```markdown
## review-pr — PR #{pr_number}

**Confidence:** {value} (threshold {threshold}) — routing **{block|escalate|pass}**

### P1 — Blockers ({n})
- `{file}:{line}` — `{standard_id}` — {evidence}

### P2 — Major ({n})
- ...

### P3 — Minor ({n})
- ...

### Reviewers requested
- {selected_reviewers list}

<!-- review-pr:marker -->
```

Agent searches PR comments for the `<!-- review-pr:marker -->` marker. If found, **updates in place** via `gh api repos/{owner}/{repo}/issues/comments/{comment_id} -X PATCH`. If not found, posts a new comment via `gh pr comment`.

Routing actions:
- `routing == block` → `gh pr review {pr_number} --request-changes --body "review-pr block"` AND `gh pr edit {pr_number} --add-reviewer {selected}`
- `routing == escalate` → `gh pr edit {pr_number} --add-reviewer {selected}`
- `routing == pass` → no PR state change beyond the comment

Emits `comment-record.yaml`:
```yaml
comment_id: 12345
action: updated | created
review_action: request-changes | none
reviewers_added: ["alice@org", "bob@org", "carol@org"]
```

**Step 5 Eval — SE-8 (C8/F5/F6):** The PR comment posted to GitHub uses the structured template with P1/P2/P3 sections and the confidence score. If a prior review-pr comment exists, it is updated in place — `comment_id` matches the previous one. On P1 finding the PR is set to `request-changes`; on `block` or `escalate` the reviewers are added via `gh pr edit --add-reviewer`.

### Phase: Scenario Validation

**Step 6 — Scenario Evals**
Owner: play
Depends on: Step 5

- **SCE-1 (S1 — Reviewer):** After review-pr completes, `gh pr view` shows exactly one review-pr comment with sections P1/P2/P3, a confidence score, and a Reviewers block listing the selected names.
- **SCE-2 (S2 — PR Author):** Every finding line in the PR comment links to a `standard_id` that maps to a real KB rule and a specific `file:line` in the diff.
- **SCE-3 (S3 — Ship Pipeline):** When ship runs with `review-pr.bypass=false` and review-pr returns `block` or `escalate`, ship halts before `merge-pr` and surfaces the routing decision.
- **SCE-4 (S4 — QA / Determinism Auditor):** Two back-to-back review-pr runs on the same PR produce byte-identical `findings.yaml` (modulo `meta.generated_at`) and identical P1..P4 counts.
