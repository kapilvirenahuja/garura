# create-pr

Create a pull request on the user's configured git platform, linked to the originating issue. The PR body contains a dynamic quality checklist tailored to the actual changes, with evidence backing every item. Eval results are embedded in the PR as quality proof. Approval is confidence-gated: auto-submit when confidence is high, checkpoint for user decision when confidence is low.

## Pre-flight

Execute these checks before any domain work:

| Check | Constraint | Action on Failure |
|-------|-----------|-------------------|
| Resolve `stm_base` from `.meridian/core/config.yaml` | — | Hard halt |
| Branch guard | C8 | Hard halt |
| Working tree is clean | C3 | Hard halt — instruct user to commit first |
| Branch pushed to remote | C3 | Auto-push (infrastructure, not a halt) |
| Platform config exists | C1 | Hard halt — no platform configured |
| Platform reference file exists | C1 | Hard halt — unsupported platform |
| Open issues exist | F1 | Hard halt — no issue to link |
| Commits ahead of base | implicit | Graceful exit — nothing to PR |

```bash
stm_base=$(grep '^\s*base-path:' .meridian/core/config.yaml | head -1 | awk '{print $2}')
product_base=$(grep '^\s*base-path:' .meridian/core/config.yaml | tail -1 | awk '{print $2}')
branch=$(git branch --show-current)
default_branch=$(git remote show origin | grep 'HEAD branch' | awk '{print $NF}')
# C8: halt if branch == default_branch or branch == main or branch == master
# C3: git status --porcelain — halt if dirty
# C3: push check — auto-push if behind
git rev-parse --abbrev-ref --symbolic-full-name @{u} 2>/dev/null
git log @{u}..HEAD --oneline 2>/dev/null
# C1: platform config
platform=$(grep '^platform:' .meridian/core/config.yaml | awk '{print $2}')
# C1: platform reference exists
test -f "core/components/skills/submit-pr/reference/${platform}/pr.md"
# F1: open issues exist
# Commits ahead
git log ${default_branch}..HEAD --oneline

# Extract issue number from branch name (e.g., feature/95-some-slug -> 95)
issue=$(echo "$branch" | grep -oE '/[0-9]+' | tr -d '/')
```

**Resume check:** If `{stm_base}/{issue}/status/create-pr.json` exists, resume — skip completed steps, reset any `in_progress` to pending, continue from first incomplete.

**Steps 1-2 Evals:**

- **SE-2 (C4, F2):** Checklist items are change-specific — every item has a trigger reason tracing to actual file patterns in the diff. No generic items.
- **SE-3 (C5, F3):** Every checklist item has evidence backing — Verification Evidence table is complete with non-empty evidence for every item.
- **SE-4 (C7, F5):** Target branch confidence assessed. If high → checkpoint can be skipped. If less than high → checkpoint required.
- **SE-5 (C1):** Platform resolved from `.meridian/core/config.yaml`, not hardcoded.
- **SE-6 (C2):** No code modifications during analysis — git status unchanged after preparation.

### Phase: Checkpoint (skip when: all confidence signals are high AND no blocking issues)

The skip condition implements C7: human approval is required when any decision has less than high confidence. When all confidence signals are high, the play proceeds autonomously.

- Issue linkage

## Pause and Resume

Steps are in execution order — run top to bottom.

**Issue detection:** Extract issue number from branch name during pre-flight (e.g., `feature/95-slug` -> `95`). This is needed before the status file can be checked.

**Status file:** `{stm_base}/{issue}/status/create-pr.json`

```json
{
  "play": "create-pr",
  "issue": 95,
  "started_at": "2026-03-06T19:00:00+0530",
  "tasks": {
    "analyze-pr-readiness": { "status": "completed", "completed_at": "..." },
    "resolve-issue-linkage": { "status": "completed", "completed_at": "..." },
    "create-pr": { "status": "in_progress", "started_at": "..." }
  }
}
```

**Executor loop:**
```
resolve issue number from branch name
check for status file at {stm_base}/{issue}/status/create-pr.json

for each step in compiled order:
  if status file shows step "completed" -> skip
  if status file shows step "in_progress" -> reset to pending (may not have finished)
  mark step "in_progress" in status file
  execute step
  mark step "completed" in status file
```

**Fresh start:** No status file exists -> execute all steps, create status file on first step.
**Resume:** Status file exists -> skip completed, continue from first incomplete.
**Pause:** User terminates session. Status file reflects last completed step. Next invocation resumes.
