# build-arch

The architecture play in Meridian's 5-stage SDLC (Discover → Specify → Design → Build → Run). Named `build-arch` to reflect that we BUILD the architecture specification; actual code construction is `implement-epic`. Reads locked upstream artifacts from specify-product (under `{product_base}scope/` and `{product_base}specification/`) and design-exp (under `{product_base}experience/`) and produces `architecture.yaml` + `quality-standards.yaml` under `{product_base}architecture/`. No code, no test suites, no implementation work products.

| specify-product scope artifacts LOCKED at `{product_base}scope/` | C1 | Hard halt — "run specify-product first" |
| specify-product specification artifacts LOCKED at `{product_base}specification/` | C1 | Hard halt — "run specify-product first" |
| design-exp artifacts LOCKED at `{product_base}experience/` | C2 | Hard halt — "run design-exp first" |
| KB catalog consistency — invoke `validate-kb-extension` skill | — | Hard halt |
| `{product_base}architecture/` writable; scriber available | C10 | Hard halt |

```bash
stm_base=$(grep '^\s*base-path:' .meridian/core/config.yaml | awk '/stm:/,/base-path/' | tail -1 | awk '{print $2}')
product_base=$(grep -A1 '^product:' .meridian/core/config.yaml | grep 'base-path' | awk '{print $2}')

# C1: verify specify-product scope-stage artifacts exist (LOCKED status)
for f in scope.yaml enriched-capabilities.yaml; do

# C2: verify design-exp artifacts exist (LOCKED status)

```

Agent invokes `derive-architecture-spec` skill → produces `architecture.yaml` with stack, components, data model, API surface, deployment shape, integrations, and ADR log. Every technology is a named product; every decision cites a driver from the upstream artifacts.

**Step 1 Eval:**

- **SE-1 (F1):** The play does not complete with zero architectural components in architecture.yaml.
  - Artifact: `.meridian/product/architecture/architecture.yaml`
  - Pass: architecture.yaml exists and its `components_count` (from the derive-architecture-spec output contract) is >= 1.
  - Fail: architecture.yaml is missing, empty, or its `components_count == 0`.

**Step 2 — Validate architecture (Stage 1 validation)**
Owner: `tech-architect` (delegates to `validate-architecture-spec` skill)
Depends on: Step 1

```json
{
  "intent_path": "core/components/plays/build-arch/reference/intent.yaml",

      "ltm_architecture_path": "core/components/memory/knowledge/arch/"
    },
    "output": {
      "validation_path": "{product_base}architecture/validation-architecture.yaml"
    }
  },
  "task_id": "build-arch-stage-1-validate"
}
```

Stage-1 validation — checks architecture.yaml only (quality-standards not yet produced; `quality_standards_path: null`). Blocking: on `status: failed`, cycle back to Step 1 with error report. Max 2 retry cycles.

**Step 2 Evals:**

- **SE-2 (F2 / C3):** Every technology choice in architecture.yaml names a specific product or library — never a category ("a relational database", "a message queue", "a frontend framework").
  - Artifact: `.meridian/product/architecture/validation-architecture.yaml`
  - Pass: `validation_path.summary.by_category.category_term == 0`.
  - Fail: `validation_path.summary.by_category.category_term > 0` (one or more stack choices use a category term rather than a specific product name).

- **SE-3 (F3 / C4):** Every architectural decision carries rationale that cites at least one specific driver from the upstream artifacts: an epic constraint, a quality-profile target, a screen-inventory complexity hint, a flow-coverage requirement, or a compliance mandate.
  - Artifact: `.meridian/product/architecture/validation-architecture.yaml`
  - Pass: `validation_path.summary.by_category.uncited_decision == 0`.
  - Fail: `validation_path.summary.by_category.uncited_decision > 0` (one or more decisions lack a cited driver).

- **SE-4 (F4 / C5):** Every selected capability from specify-product.scope maps to at least one architectural component in architecture.yaml. Orphan capabilities are a validation failure.
  - Artifact: `.meridian/product/architecture/validation-architecture.yaml`
  - Pass: `validation_path.summary.by_category.orphan_capability == 0`.
  - Fail: `validation_path.summary.by_category.orphan_capability > 0` (one or more capabilities have no component mapping).

- **SE-5 (F5 / C6):** Every intent epic's performance constraint maps to a specific architectural mechanism in architecture.yaml (caching strategy, query pattern, infrastructure tier, etc.).
  - Artifact: `.meridian/product/architecture/validation-architecture.yaml`
  - Pass: `validation_path.summary.by_category.missing_performance_mapping == 0`.
  - Fail: `validation_path.summary.by_category.missing_performance_mapping > 0` (one or more performance targets are unmet).

- **SE-6 (F8 / C9):** Architectural Decision Records (ADRs) are produced for every non-obvious decision (at least two viable alternatives existed AND the choice narrows a future path). Each ADR carries the decision, alternatives considered, chosen option, and the upstream driver.
  - Artifact: `.meridian/product/architecture/validation-architecture.yaml`
  - Pass: `validation_path.summary.by_category.incomplete_adr == 0`.
  - Fail: `validation_path.summary.by_category.incomplete_adr > 0` (one or more non-obvious decisions ship without an ADR or with an ADR missing required fields).

### Phase: Scenario Validation

After both files pass validation and both checkpoints carry approval, run scenario evals against the full artifact set.

- **SCE-1 (S1 — Technical Architect):** The locked architecture.yaml carries every decision traced to a business driver — the architect can present the architectural direction to stakeholders without "we picked X because we like it".
  - Artifact: `.meridian/product/architecture/architecture.yaml`
  - Pass: Every decision in architecture.yaml (stack choices, components, ADRs, integrations) has a non-empty `rationale` field naming at least one specific driver from the upstream artifacts.
  - Fail: One or more decisions has an empty rationale or no cited driver.

- **SCE-2 (S2 — Engineering Manager):** quality-standards.yaml carries per-characteristic standards, named tooling, and concrete thresholds — the EM can set up CI/CD gates and code-review standards without re-deriving the thresholds from scratch.
  - Artifact: `.meridian/product/architecture/quality-standards.yaml`
  - Pass: Every standard under `qp_dimensions` has a non-empty `enforcement` field, a named `tooling` field, and a concrete `threshold` value an EM can copy directly into CI configuration or branch protection rules.
  - Fail: One or more standards lack `enforcement`, named `tooling`, or a concrete `threshold` value.

- **SCE-3 (S3 — Implementation Lead):** architecture.yaml components are mapped to the epic's capability and the epic's constraints section — the implementation lead can scope the implementation plan knowing which components are touched, what the performance budget is, and which standards apply.
  - Artifact: `.meridian/product/architecture/architecture.yaml`
  - Pass: Every component in `architecture.yaml.components` has a `capability` field referencing a capability ID from scope, a `performance_budget` field with source citation, and a `depends_on` field listing upstream components.
  - Fail: One or more components is missing the capability mapping, `performance_budget`, or `depends_on` field.

- **SCE-4 (S4 — Security Architect):** Every security constraint from the upstream epics is addressed by a specific architectural control (authentication, encryption, access control, audit logging) with a cited driver pointing to the epic's security constraint.
  - Artifact: `.meridian/product/architecture/architecture.yaml`
  - Pass: For every intent epic in `.meridian/product/scope/epics/*.yaml` that has a `constraints.security` field, there is at least one architectural control in architecture.yaml (within a component's rationale, an ADR, or an integration's security notes) that cites the epic's security constraint as its driver.
  - Fail: One or more epic security constraints has no corresponding architectural control with a cited driver.

- **SCE-5 (S5 — Product Manager at final review):** The Checkpoint 1 presentation contains stack names plus high-level summary counts and rationale — the PM can assess whether the chosen architecture fits the appetite and team skillset, and push back on over-engineering at Checkpoint 1.
  - Artifact: `.meridian/product/_checkpoints/build-arch/stage-1-architecture-*.md`
  - Pass: Checkpoint 1 artifact contains stack named choices, components count with capability coverage ratio, ADR count, and integration count — readable without engineering background.
  - Fail: Checkpoint 1 artifact is missing the summary counts or omits the stack named choices and ADR count.

- **SCE-6 (S6 — DevOps / Platform Engineer):** architecture.yaml deployment section + quality-standards.yaml operational standards together carry enough specifics for the DevOps engineer to plan infrastructure-as-code, observability, and on-call procedures without needing the architect to clarify intent.
  - Artifact: `.meridian/product/architecture/architecture.yaml` + `.meridian/product/architecture/quality-standards.yaml`
  - Pass: `architecture.yaml.deployment` section has `hosting`, `environments`, and `observability` sub-fields populated; `quality-standards.yaml` has a `qp_5_observability` dimension with named tooling, thresholds, and enforcement.
  - Fail: Deployment section is missing one or more of `hosting`, `environments`, `observability` OR `qp_5_observability` dimension is absent or incomplete.
