# build-arch

The architecture play in Meridian's 5-stage SDLC (Discover → Specify → Design → Build → Run). Named `build-arch` to reflect that we BUILD the architecture specification; actual code construction is `implement-epic`. Reads locked upstream artifacts from specify-product + design-exp and produces `architecture.yaml` + `quality-standards.yaml` under `.meridian/product/arch/`. No code, no test suites, no implementation work products.

## Compiled From

This play was compiled from `reference/intent.yaml` by `/create-play --build`.
Intent defines constraints (C1-C13), failure conditions (F1-F11), and scenarios (S1-S6).
To modify this play, update intent.yaml and re-run `/create-play --build build-arch`.
Do NOT edit this file manually — it is a compiled artifact.

## Role

You are the orchestrator. You own the workflow. You delegate domain tasks to agents via JSON contracts — never execute domain work directly.

**Forbidden:** Direct architecture derivation. Direct quality-standards authoring. Direct code/test-suite generation. All domain work goes through `tech-architect`, which invokes skills.

**Agent boundaries:**

| Agent | Domain | Phases |
|-------|--------|--------|
| `tech-architect` | Architecture derivation + quality-standards derivation + validation (reuses pre-existing agent with expanded skill pool) | All execution phases |
| `scriber` | Background writes of evidence, checkpoint, and status artifacts (utility — exempt) | All phases |
| `repo-orchestrator` | Self-commit of STM + product artifacts at play close (utility — exempt) | Close |

## Pre-flight

| Check | Constraint | Action on Failure |
|-------|-----------|-------------------|
| Resolve `stm_base` and `product_base` from `.meridian/core/config.yaml` | — | Hard halt |
| specify-product artifacts LOCKED at `{product_base}/product/` | C1 | Hard halt — "run specify-product first" |
| design-exp artifacts LOCKED at `{product_base}/ux/` | C2 | Hard halt — "run design-exp first" |
| KB catalog consistency — invoke `validate-kb-extension` | — | Hard halt |
| `{product_base}/arch/` writable; scriber available | C10 | Hard halt |

```bash
stm_base=$(grep '^\s*base-path:' .meridian/core/config.yaml | awk '/stm:/,/base-path/' | tail -1 | awk '{print $2}')
product_base=$(grep -A1 '^product:' .meridian/core/config.yaml | grep 'base-path' | awk '{print $2}')

# C1: verify specify-product artifacts exist AND status is LOCKED
for f in scope.yaml enriched-capabilities.yaml quality-profile.yaml; do
  test -f "${product_base}product/$f" || halt "missing $f"
done
test -d "${product_base}product/epics" || halt "missing epics dir"

# C2: verify design-exp artifacts exist AND status is LOCKED
for f in personas.md design-spec.md; do
  test -f "${product_base}ux/$f" || halt "missing $f — run design-exp first"
done
test -d "${product_base}ux/screens" || halt "missing screens dir"
test -d "${product_base}ux/flows" || halt "missing flows dir"

# Invoke validate-kb-extension via Skill tool
```

**Resume check:** If `{product_base}arch/_status/build-arch.json` exists, resume — skip completed stages, reset in_progress to pending.

### Phase: Scenario Validation

After both files pass validation, run scenario evals against the full artifact set.

**SCE-1 (S1 — Technical Architect):** Every decision in `architecture.yaml` has a `rationale` field naming at least one specific driver from the upstream artifacts (epic ID, quality-profile characteristic, or compliance mandate).

**SCE-2 (S2 — Engineering Manager):** `quality-standards.yaml` contains CI/CD-gate-ready standards with `enforcement` fields. An EM can copy the threshold values directly into branch protection rules without reinterpretation.

**SCE-3 (S3 — Implementation Lead):** `architecture.yaml.components` has a per-capability mapping. An implementation lead picking up EPIC-user-login-001 can look up which component(s) own the work and find the component's performance_budget + depends_on.

**SCE-4 (S4 — Security Architect):** For every intent epic with a `constraints.security` field, there is an architectural control in `architecture.yaml` (within a component's rationale, an ADR, or an integration's security notes) that cites the epic's security constraint as its driver.

**SCE-5 (S5 — Product Manager at final review):** Checkpoint 2 presentation is readable without deep engineering background — summary counts, stack names, and high-level rationale. PM can Tether / Orbit / Vanish based on appetite + cost intuition.

**SCE-6 (S6 — DevOps / Platform Engineer):** `architecture.yaml.deployment` section + `quality-standards.yaml.qp_5_observability` together contain enough specifics (hosting platform, IaC approach, observability stack, alerting targets) to plan infrastructure-as-code without needing the architect to clarify intent.
