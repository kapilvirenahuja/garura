# start-feature-planning

Resolve or create an issue, produce IDD-aware planning artifacts (spec, verification, task graph), create a feature branch, and deliver a ready-to-execute task graph — stopping before implementation. All planning via Plan sub-agent, never via interactive plan mode.

## Compiled From

This play was compiled from `reference/intent.yaml` by `/create-play`.
Intent defines constraints (C1-C10), failure conditions (F1-F4), and scenarios (S1-S4).
To modify this play, update intent.yaml and re-run `/create-play --build start-feature-planning`.
Do NOT edit this file manually — it is a compiled artifact.

## Role

You are the orchestrator. You own the workflow. You delegate domain tasks to agents via JSON contracts — never execute domain work directly.

**Forbidden:** `Bash`, `Grep`, `Glob`, `Edit`, `EnterPlanMode`, `ExitPlanMode`, or any direct git/gh commands (C3, C8).

**CRITICAL: You MUST NOT call `EnterPlanMode` or `ExitPlanMode`. Zero exceptions. All planning is via the Plan sub-agent (Agent tool with `subagent_type: "Plan"`).**

**Agent boundaries:**

| Agent | Domain | Phases |
|-------|--------|--------|
| `project-orchestrator` | Issues: resolve or create issue, type hint derivation | Preparation — Resolve Issue |
| `repo-orchestrator` | Git: create branch, push to origin, evidence self-commit | Preparation — Create Branch, Evidence |
| Plan sub-agent | Technical analysis: codebase exploration, design, task breakdown | Preparation — Deep Analysis |

Plan sub-agent is a Claude built-in tool (`subagent_type: "Plan"`), exempt from agent limits (C9).

## Workflow

### Phase: Preparation — Resolve Issue

**Step 1 — Resolve or Create Issue**
Owner: `project-orchestrator`
Depends on: pre-flight

```json

  "stm_base": "{stm_base}",
  "stm": {
    "input": {

    },
    "output": {
      "issue_resolution": "{stm_base}/_pending/start-feature-planning/issue-resolution.yaml"
    }
  },
  "task_id": "resolve-issue"
}
```

Write `args.yaml` to STM before dispatch: `{ input_type: "issue_number|title", value: "<value>", parent: <number or null> }`.

Agent reads intent.yaml, invokes `manage-issue` (resolve_or_create). If `--parent` provided, attaches as sub-issue.

**Expected output artifact (`issue-resolution.yaml`):**

```yaml
issue:
  number: {int}
  title: "{title}"
  body: "{body}"
  url: "{url}"
  state: "open"
  created: true|false
  type_hint: "{feature|fix|hotfix|refactor|docs|chore|null}"
```

If `type_hint` is null → default to `feature/` (C10).

Derive branch name: `{type}/{issue_number}-{slug}` — slug is lowercase, hyphens, max 40 chars.

**Step 2 Eval:**

**Orchestrator initializes STM directory** at `{stm_base}/{issue}/` with subdirectories: `spec/`, `design/`, `evidence/`, `delivery/`, `checkpoint/`, `planning/`.

Once issue number is known, relocate pending artifacts: `mv {stm_base}/_pending/start-feature-planning/* → {stm_base}/{issue}/evidence/start-feature-planning/`.

**Risks:** {risk -> mitigation -> severity}
