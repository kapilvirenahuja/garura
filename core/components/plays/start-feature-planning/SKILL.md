# start-feature-planning

Resolve or create an issue, produce IDD-aware planning artifacts (spec, verification, task graph), create a feature branch, and deliver a ready-to-execute task graph — stopping before implementation. All planning via Plan sub-agent, never via interactive plan mode.

## Recovery

Sourced from `reference/expectation.yaml` `recovery` — one entry per failure condition. When the validator detects a tripped failure, it builds a recovery handoff plan from the matching entry and routes it per `handoff`: `autonomous` loops the fix back to the builder; `human` escalates for a manual call.

| ID | For | Symptom (trigger) | Direction | Handoff |
|----|-----|-------------------|-----------|---------|
| REC1 | F1 | planning artifacts empty / no actionable design — sub-agent returned no meaningful output for the intent | re-run the planning sub-agent with the full intent context so it produces a non-empty, design-bearing output | autonomous |
| REC2 | F2 | plan rejected at the gate (Vanish) but the remote feature branch still exists on origin | delete the lingering feature branch from origin, treating an already-deleted remote as success | autonomous |
| REC3 | F3 | planning output missing one or more of SPEC, VERIFY, or TASKS | re-run the planning sub-agent and re-parse its output until every required section is produced | autonomous |
| REC4 | F4 | feature branch absent on origin after the branch-creation step completes | re-create and push the branch to origin so the ready-to-execute state holds | autonomous |

At Level 4, `intent-resolver` executes the autonomous entries (REC1, REC2, REC3, REC4) without a human; there are no human entries.

When an agent returns a structured failure during a step (per `structured-failure-protocol.md`): read `domain_assessment.responsible_domain` to identify the responsible agent, invoke it with fix context plus the original intent, max 2 retry cycles per agent, then HALT with full failure context.
