# implement-epic

Given a locked execution plan entry and a defined feature, produce a working implementation through an eval-driven TDD loop. Context isolation is structural: the test-writer receives only behavioral specifications (TEST-CONTEXT.md), the code-builder receives CONTEXT.md plus read-only test files, the eval generator reads only specs, the judge reads only evals. An engineering-manager certifies QP compliance after quality gates pass. Quality gate FAIL is a hard stop — the judge does not execute while gates are failing. Remediation draws from all three sources: judge failures, quality gate failures, and QP threshold failures. Eval count stability is audited across fix iterations. A feature-steward generates manual test scenarios. The process produces committed code, a judge report, quality report, EM certification, and manual test scenarios.

## Role

You are the orchestrator. You own the workflow. You delegate domain tasks to agents via JSON contracts — never execute domain work directly.

**Forbidden:** Direct code implementation. Direct test writing. Direct eval generation. Direct quality checks. All domain work goes through agents. The orchestrator's only direct operations are: reading status files, writing status files, writing checkpoints/evidence, updating CLAUDE.md pointers, building TEST-CONTEXT.md, recording test file checksums, auditing eval counts, and mediating information flow between agents (with filtering per C4/C5/C6/C10/C17).

**Agent boundaries:**

|-------|------|--------|----------|-----------------|--------|--------|

| `eval-generator` | Eval Generator | Generate encrypted evals from specs | features.yaml behaviors, scenarios.yaml verification scenarios, feature success/failure, plan.yaml exit gate | Implementation code, builder prompts, prior evals, architecture, vision | Preparation + Fix Loop | Domain (1 call per invocation) |

**Context isolation invariants (C4/C5/C6/C17):** The orchestrator is the ONLY entity that touches multiple agent outputs. When passing information between agents, the orchestrator MUST filter. Specifically:
- **test-writer receives:** TEST-CONTEXT.md ONLY — scope descriptions (no file paths), acceptance criteria, test framework name. Never CONTEXT.md "Files You Own", never file paths, never architecture or tech decisions (C17).
- **code-builder receives:** CONTEXT.md + test file paths marked read-only. Never evals, never TEST-CONTEXT.md, never judge report content (C5).
- **Judge receives:** Encrypted evals + project root only. Never builder prompts, never quality results, never EM report (C6).
- **Judge failures → Builder:** Orchestrator extracts remediation instructions only (C10). Strips eval IDs, eval text, pass criteria, raw scores.
- No other cross-agent information flow is permitted.

**CRITICAL — Sequential gate chain (C19, C21):**
```
quality-auditor → [HARD GATE: overall PASS?] → engineering-manager → [GATE: CERTIFIED?] → judge
```
If quality-auditor reports FAIL → do NOT call EM or judge. Enter fix loop.
If EM reports BLOCKED → do NOT call judge. Enter fix loop.
Steps 7, 7b, and 8 are strictly sequential — NOT parallel.

**Path resolution:**
All `.meridian/` paths are relative to the project root. STM artifacts go to `.meridian/project/issues/{issue}/`. Product artifacts live under `.meridian/product/{phase}/`. Eval files go OUTSIDE the repo tree (C8).

**Step 3 — Capture Quality Vision Gates**
Owner: orchestrator (direct — reads quality-standards.yaml and LTM, creates enriched gates)
Depends on: pre-flight

**QP Derivation (C15, C20, F19):** After detecting the toolchain, check if `.meridian/product/architecture/quality-standards.yaml` exists. If it does, translate QP dimension levels to concrete thresholds using the QP Translation Table below. Produce an enriched `quality-vision-gates.yaml` with non-null thresholds. If `quality-standards.yaml` is absent, fall back to generic toolchain detection (backward compatibility).

**QP Translation Table:**

| QP Dimension | Level | Gate Value |
|-------------|-------|------------|
| QP-1 (Testing Depth) | 1 | min_coverage: null (no gate) |
| QP-1 (Testing Depth) | 2 | min_coverage: 40 |
| QP-1 (Testing Depth) | 3 | min_coverage: 70 |
| QP-1 (Testing Depth) | 4 | min_coverage: 80 |
| QP-1 (Testing Depth) | 5 | min_coverage: 90 |
| QP-2 (Code Quality) | 1 | no gate |
| QP-2 (Code Quality) | 2 | max_warnings: 0 (errors only), no static analysis |
| QP-2 (Code Quality) | 3 | max_warnings: 0, static_analysis: required (tool from QS), max_cyclomatic: 15 |
| QP-2 (Code Quality) | 4 | max_warnings: 0, static_analysis: required + arch rules, max_cyclomatic: 10 |
| QP-2 (Code Quality) | 5 | max_warnings: 0, static_analysis: required + conformance, max_cyclomatic: 8 |
| QP-3 (Documentation) | 1 | no gate |
| QP-3 (Documentation) | 2 | README exists, public API JSDoc/docstrings present |
| QP-3 (Documentation) | 3 | API docs auto-generated, architecture overview exists |
| QP-7 (Security) | 1 | no gate |
| QP-7 (Security) | 2 | dependency_scan_command runs, zero critical vulnerabilities |
| QP-7 (Security) | 3 | Level 2 + sast_command runs, zero critical findings |

Note: If `quality-standards.yaml` `standards.testing.coverage_target` provides a numeric value, it takes precedence over the QP-1 table default.

**QP-4 (CI/CD), QP-5 (Observability), QP-6 (Accessibility)** do not produce implement-time gates — they are infrastructure/process concerns validated at deployment. They are recorded in the EM certification for traceability only.

```yaml
# {quality_gates_path} — enriched gates when quality-standards.yaml exists
quality_gates:
  build: { required: true, command: "{project build command}" }
  typecheck: { required: true, command: "{project typecheck command}" }
  lint:
    required: true
    command: "{project lint command}"
    max_warnings: 0
    static_analysis_tool: "{from QS if QP-2 >= 3}"
    static_analysis_command: "{from QS if QP-2 >= 3}"
  unit_tests:
    required: true
    command: "{project test command}"
    min_coverage: "{from QP-1 translation — non-null when QP-1 >= 2}"
  complexity:
    required: "{true if QP-2 >= 3}"
    max_cyclomatic: "{from QP-2 translation}"
  security:
    required: "{true if QP-7 >= 2}"
    dependency_scan_command: "{from QS}"
    sast_command: "{from QS if QP-7 >= 3}"
  documentation:
    required: "{true if QP-3 >= 2}"
    api_doc_check: "{from QS}"
source:
  quality_standards_path: "{quality_standards_path or 'absent — toolchain defaults used'}"

```

Detect the project's toolchain from existing config files (package.json → npm, pyproject.toml → pytest, Makefile → make, Cargo.toml → cargo, etc.) and populate commands accordingly.

**Issue detection:** From `--issue` argument or branch name during pre-flight.

**Status file:** `{stm_dir}/status/implement-epic.json`

```json
  "started_at": "{timestamp}",
  "eval_dir": "{eval storage path outside repo}",
  "original_eval_count": "{count from initial manifest}",
  "fix_iteration": 0,
  "tasks": {
    "build-context":           { "status": "completed", "completed_at": "..." },
    "update-claude-md":        { "status": "completed", "completed_at": "..." },
    "capture-quality-gates":   { "status": "completed", "completed_at": "..." },
    "build-test-context":      { "status": "completed", "completed_at": "..." },
    "generate-evals":          { "status": "completed", "completed_at": "..." },
    "write-tests":             { "status": "in_progress", "started_at": "..." },
    "build-epic":              { "status": "pending" },
    "quality-gate":            { "status": "pending" },
    "em-certification":        { "status": "pending" },
    "judge-evals":             { "status": "pending" },
    "fix-loop":                { "status": "pending", "iterations_completed": 0 },
    "test-scenarios":          { "status": "pending" },
    "commit-push":             { "status": "pending" },
    "cleanup":                 { "status": "pending" },
    "report":                  { "status": "pending" }
  }

```

**Executor loop:**
```
parse --epic, --phase, --issue arguments
resolve issue number
check for status file at {stm_dir}/status/implement-epic.json

if file exists (resume):
  restore eval_dir, original_eval_count, and fix_iteration from status
  for each step in compiled order:
    if status == "completed" → skip
    if status == "in_progress" → reset to pending
    mark step "in_progress"
    execute step
    mark step "completed"
  for fix loop:
    restore iterations_completed
    continue from current iteration

if file absent (fresh start):
  execute all steps
  create status file on Step 1
```

**Fresh start:** No status file → execute all steps, create on first step.
**Resume:** Status file exists → skip completed, continue from first incomplete.
**Fix loop resume:** `fix_iteration` field tracks current iteration count. On resume, restore count and continue.

1. Read the error from the returned contract.
2. For build failures (F1): check if the error is in the builder's scope or a pre-existing issue.
3. For eval generation failures: retry once with the same specs. After 2 failures, halt.
4. For judge failures: verify encrypted file is readable. If corrupted, regenerate evals and retry.
5. For quality gate failures: these feed into the fix loop — they are expected during iteration.
6. For EM certification failures (agent returned `failed`, not `BLOCKED`): check error type. If `quality_standards_missing`, verify path and retry. If `quality_report_missing`, check quality-auditor completed successfully.
7. For test-writer failures: retry once. If failure persists, halt — test authorship cannot be skipped.
8. For checksum mismatch (F16): halt immediately. The builder modified test files — this is a hard integrity violation. Do not enter fix loop. Present F16 to user.

**Pre-flight failures (C1, C2, C3, C12) are not recoverable** — hard halt with clear error message.

**F2 (>50% failure rate on first run):** This suggests the feature scope is too ambitious or the builder received insufficient context. Halt and present to user — do not enter the fix loop. The orchestrator presents: "First-run eval pass rate is below 50%. The feature may need restructuring before re-attempting implementation."
