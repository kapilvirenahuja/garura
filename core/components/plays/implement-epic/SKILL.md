# implement-epic

Given a locked execution plan entry and a defined feature, produce a working implementation through an eval-driven TDD loop. Context isolation is structural: the test-writer receives only behavioral specifications (TEST-CONTEXT.md), the code-builder receives CONTEXT.md plus read-only test files, the eval generator reads only specs, the judge reads only evals. The quality-auditor handles both quality gates AND QP certification in a single step — no separate engineering-manager agent. Eval-generator is scoped to unit test coverage completeness: "did you test all behaviors?" Judge is scoped to developer self-review: "are unit tests genuinely covering the spec?" System-level verification (e2e, integration against real services, manual test scenarios) is handled by validate-epic, not this play. When failures occur, remediation draws from two sources: quality-auditor failures (including QP threshold failures) and judge failures. Eval count stability is audited across fix iterations. Mock infrastructure is set up from tech.yaml before test authorship. An integration test pass verifies cross-scope-item behavior after all scope items complete. The process produces committed code, a judge report, and a quality report with QP certification.

**Forbidden:** Direct code implementation. Direct test writing. Direct eval generation. Direct quality checks. All domain work goes through agents. **Context isolation invariants (C4/C5/C6/C17):** The orchestrator is the ONLY entity that touches multiple agent outputs. When passing information between agents, the orchestrator MUST filter. Specifically:
- **test-writer receives:** TEST-CONTEXT.md ONLY — scope descriptions (no file paths), acceptance criteria, test framework name. Never CONTEXT.md "Files You Own", never file paths, never architecture or tech decisions (C17).
- **code-builder receives:** CONTEXT.md + test file paths marked read-only. Never evals, never TEST-CONTEXT.md, never judge report content (C5).
- **Judge receives:** Encrypted evals + project root only. Never builder prompts, never quality results (C6).
- **Judge failures → Builder:** Orchestrator extracts remediation instructions only (C10). Strips eval IDs, eval text, pass criteria, raw scores.
- No other cross-agent information flow is permitted.

**CRITICAL — Simplified gate chain (C20):**
```
quality-auditor → [HARD GATE: overall PASS AND qp_certification.overall CERTIFIED?] → judge
```
If quality-auditor reports overall FAIL OR qp_certification.overall BLOCKED → do NOT call judge. Enter fix loop.
One gate, not two. Steps 7 and 8 are strictly sequential — NOT parallel.

## Recovery

When an agent returns `status: "failed"`:

1. Read the error from the returned contract.
2. For build failures (F1): check if the error is in the builder's scope or a pre-existing issue.
3. For eval generation failures: retry once with the same specs. After 2 failures, halt.
4. For judge failures: verify encrypted file is readable. If corrupted, regenerate evals and retry.
5. For quality gate failures (including QP certification): these feed into the fix loop — they are expected during iteration.
6. For mock setup failures (F20): halt immediately. Mock infrastructure is a hard prerequisite. Present error to user with tech.yaml mock_strategy section for debugging.
7. For test-writer failures: retry once. If failure persists, halt — test authorship cannot be skipped.
8. For checksum mismatch (F15): halt immediately. The builder modified test files — this is a hard integrity violation. Do not enter fix loop. Present F15 to user.
9. For integration test failures (F21): route to the fix loop. These are expected failures that the remediation process handles.

**Pre-flight failures (C1, C2, C3, C12) are not recoverable** — hard halt with clear error message.

**F2 (>50% failure rate on first run):** This suggests the feature scope is too ambitious or the builder received insufficient context. Halt and present to user — do not enter the fix loop. The orchestrator presents: "First-run eval pass rate is below 50%. The feature may need restructuring before re-attempting implementation."
