# garura:enrich

Terminal stage of the Learning Pipeline. Reads native proposals staged by `distill`, `reap`, `codify`, or `decode` for one or more issues, normalizes each into a reconciliation file, surfaces them to the reviewer at tier-appropriate granularity, applies approved entries to product LTM, promotes approved Tier 1 ADR drafts into the ADR archive, and archives the processed issue STM. Two invocation modes: single-issue (`/garura:enrich <issue>`) and sweep (`/garura:enrich`, processes every active issue under the issues area excluding the archived and pending subtrees). Idempotency comes from archival ‚Äî once an issue is archived, re-running finds no source to re-apply.

**Step 2 Eval ‚Äî SE-11 (C2, F1):**

**Step 2 Eval ‚Äî SE-12 (C3):**

**Step 2 Eval ‚Äî SE-1 (F1):**

### Phase: Checkpoint

**Step 3 ‚Äî Reviewer Approval Checkpoint**
Owner: play
Depends on: Step 2
Task: `TaskUpdate [T3] ‚Üí in_progress` before. `TaskUpdate [T3] ‚Üí completed` after every entry has `approval_status` set to `approved` or `rejected`.

Present proposals to the reviewer at tier-appropriate granularity:

- **Tier 1:** for each entry, present individually. Show `target_path`, the change block, `taxonomy_justification` (if present), and any `adr_draft_path` content. Collect Tether (approve) / Vanish (reject) per entry. Tether on a T1 entry is the explicit recorded approval the play stakes its Tier 1 writes on.
- **Tier 2 / Tier 3:** present as one batch with diffs. Default action is approve-batch; the reviewer may opt into per-proposal review for any entry by name. Tether on the batch records `approval_status: approved` for every batch entry the reviewer did not explicitly reject; entries called out for rejection are recorded as `rejected`.

After the reviewer interaction, write the updated `approval_status` values back into `{stm_base}/{issue}/evidence/enrich/reconciliation-proposals.yaml`. No entry may remain `pending` once this step completes.

```
## Reviewer checkpoint ‚Äî issue #{issue} (source: {detected_source})

  Reconciliation file: {stm_base}/{issue}/evidence/enrich/reconciliation-proposals.yaml
  Tier 1 entries: {n}
  Tier 2 entries: {n} (batch)
  Tier 3 entries: {n} (batch)
  Rejected at normalize: {n} ({list of reasons})

---

Proceeding through Tier 1 entries individually.
```

**On Vanish for the entire issue:** mark all `pending` entries as `rejected` with reason `reviewer_vanish_all`, halt the issue's DAG before Step 4. Issue is NOT archived; status records `halted`. Sweep mode continues to the next issue.

**Step 3 Eval ‚Äî SE-7 (F7):**
In the updated reconciliation file, every entry's `approval_status` is `approved` or `rejected`. No entry has `approval_status: pending`.

## Scenario Validation

Run these checks against the produced artifacts after Step 6 completes for the current issue (single mode) or after the sweep finishes (sweep mode).

**SCE-1 (S1 ‚Äî Repository maintainer, mixed-tier single issue):**

Product LTM artifacts named in approved entries reflect those changes. `{stm_base}/{issue}/` does not exist; `{stm_archive}/{YYYY-MM}/{issue}/` does.

**SCE-2 (S2 ‚Äî Repository maintainer, sweep across active issues):**
For each issue directory that existed in the active issues area at run start (excluding archived and pending subtrees), the corresponding approved changes are present in product LTM and the issue is now under the archived subtree. Archived and pending entries that existed before the run are bit-identical after.

**SCE-3 (S3 ‚Äî Repository maintainer, re-invoke on already-archived issue):**
Re-invocation completes without error. Product LTM artifacts and `docs/adr/` are bit-identical to their pre-re-invocation state.

**SCE-4 (S4 ‚Äî Architecture reviewer, T1 ADR promotion):**
For the approved Tier 1 proposal carrying an ADR draft, a new ADR file `docs/adr/NNNN-{slug}.md` exists whose `NNNN` is one greater than the maximum ADR number that existed before the run, whose body matches the draft content, and whose frontmatter records `sourced_from` equal to the proposal's id. Any in-place LTM edit the proposal entailed is also present in the named target.

**SCE-5 (S5 ‚Äî Repository maintainer, taxonomy-missing entry):**
The archived reconciliation file lists the offending entry in `rejected[]` with reason `missing_taxonomy`. Valid entries appear with `approval_status: approved` or `rejected` and corresponding writes (where approved) are present in product LTM. No `pending` entries remain.

**SCE-6 (S6 ‚Äî Learning pipeline auditor, post-sweep):**
Every archived issue contains a reconciliation file with each entry's final `approval_status`, an enrichment report listing every applied write with its target path, and references to any ADRs produced during the run.

## Pause and Resume

Steps are in execution order ‚Äî run top to bottom per issue.

**Issue detection:**
- Single mode: `<issue>` is provided as input when the user invokes `/garura:enrich <issue>`.
- Sweep mode: no argument; the play enumerates active issues at pre-flight time and processes each in turn.

**Status file (per issue):** `{stm_base}/{issue}/status/enrich.json` while the issue is active; moves to `{stm_archive}/{YYYY-MM}/{issue}/status/enrich.json` after Step 5.

```json
{
  "play": "enrich",
  "issue": "{issue}",
  "started_at": "{ISO-8601}",
  "tasks": {
    "resolve-source": { "status": "completed", "completed_at": "..." },
    "normalize": { "status": "completed", "completed_at": "..." },
    "review": { "status": "in_progress", "started_at": "..." },
    "apply-and-promote": { "status": "pending" },
    "archive": { "status": "pending" },
    "evidence-commit": { "status": "pending" }
  }
}
```

**Executor loop:**
```
for each target issue (single mode = [issue_arg]; sweep mode = enumerated list):
  if {stm_archive}/{YYYY-MM}/{issue}/status/enrich.json shows "completed" ‚Üí skip issue
  check for status file at {stm_base}/{issue}/status/enrich.json
  for each step in compiled order:
    if step "completed" ‚Üí skip
    if step "in_progress" ‚Üí reset to pending (may not have finished)
    mark step "in_progress"
    execute step
    mark step "completed"
```

**Fresh start:** No status file ‚Üí execute all steps, create status file on first step.
**Resume:** Status file exists ‚Üí skip completed, continue from first incomplete.
**Pause:** User terminates session. Status file reflects last completed step. Next invocation resumes.

**Sweep failure isolation:** A failure on one issue's DAG does NOT halt the sweep. The play records the failure in that issue's evidence, leaves the issue in the active area (not archived), and continues with the next enumerated issue.
