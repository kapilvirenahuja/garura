# garura:enrich

Terminal stage of the Learning Pipeline. Reads native proposals staged by `distill`, `reap`, `codify`, or `decode` for one or more issues, normalizes each into a reconciliation file, surfaces them to the reviewer at tier-appropriate granularity, applies approved entries to product LTM, promotes approved Tier 1 ADR drafts into the ADR archive, and archives the processed issue STM. Two invocation modes: single-issue (`/garura:enrich <issue>`) and sweep (`/garura:enrich`, processes every active issue under the issues area excluding the archived and pending subtrees). Idempotency comes from archival — once an issue is archived, re-running finds no source to re-apply.

## Task DAG

The DAG is per-issue. In single mode the play runs the DAG once. In sweep mode the play runs the DAG once per enumerated issue, in serial — failure on one issue does NOT halt processing of the others.

Create these tasks for the current issue immediately after pre-flight resolves the target. The play owns this DAG. Agents must NOT update play-level tasks (they may `TaskCreate` discovered sub-work with `addBlockedBy`).

### Phase: Checkpoint

**Step 3 — Reviewer Approval Checkpoint**
Owner: play
Depends on: Step 2
Task: `TaskUpdate [T3] → in_progress` before. `TaskUpdate [T3] → completed` after every entry has `approval_status` set to `approved` or `rejected`.

Present proposals to the reviewer at tier-appropriate granularity:

- **Tier 1:** for each entry, present individually. Show `target_path`, the change block, `taxonomy_justification` (if present), and any `adr_draft_path` content. Collect Tether (approve) / Vanish (reject) per entry. Tether on a T1 entry is the explicit recorded approval the play stakes its Tier 1 writes on.
- **Tier 2 / Tier 3:** present as one batch with diffs. Default action is approve-batch; the reviewer may opt into per-proposal review for any entry by name. Tether on the batch records `approval_status: approved` for every batch entry the reviewer did not explicitly reject; entries called out for rejection are recorded as `rejected`.

After the reviewer interaction, write the updated `approval_status` values back into `{stm_base}/{issue}/evidence/enrich/reconciliation-proposals.yaml`. No entry may remain `pending` once this step completes.

```
## Reviewer checkpoint — issue #{issue} (source: {detected_source})

  Reconciliation file: {stm_base}/{issue}/evidence/enrich/reconciliation-proposals.yaml
  Tier 1 entries: {n}
  Tier 2 entries: {n} (batch)
  Tier 3 entries: {n} (batch)
  Rejected at normalize: {n} ({list of reasons})

---

Proceeding through Tier 1 entries individually.
```

**On Vanish for the entire issue:** mark all `pending` entries as `rejected` with reason `reviewer_vanish_all`, halt the issue's DAG before Step 4. Issue is NOT archived; status records `halted`. Sweep mode continues to the next issue.

**Step 3 Gate (C4 — approval completeness):**
In the updated reconciliation file, every entry's `approval_status` is `approved` or `rejected`. No entry has `approval_status: pending`. This is the reviewer-completeness precondition; the archive-time form of F7 (a confirmed write must have landed for every approved entry) is evaluated at Step 5.

## Recovery

Sourced from `reference/expectation.yaml` `recovery` — one entry per failure condition. When the validator detects a tripped failure, it builds a recovery handoff plan from the matching entry and routes it per `handoff`: `autonomous` loops the fix back to the builder; `human` escalates for a manual call. enrich is the LTM write boundary — its writes are outward-facing, so unwinding a bad LTM/ADR write or supplying a missing authority routes to a human, while the mechanical STM/archival and promotion failures route to autonomous.

| ID | For | Symptom (trigger) | Direction | Handoff |
|----|-----|-------------------|-----------|---------|
| REC1 | F1 | product LTM holds content from a taxonomy-incomplete proposal | reject taxonomy-incomplete proposals before any write; the content already in outward-facing LTM needs a human call to unwind | human |
| REC2 | F2 | a write landed outside `.garura/product/` or `docs/adr/` | every write must stay within those two roots — an out-of-boundary write touches a foreign root and needs a human call to reverse | human |
| REC3 | F3 | the same proposal's content is applied more than once across product LTM | each proposal must appear at most once — deduplicating an outward-facing LTM edit needs a human call | human |
| REC4 | F4 | Tier 1 content reached LTM or the ADR archive with no recorded approval | no Tier 1 content lands without a recorded approval — the missing approval is an authority the builder lacks, so it escalates | human |
| REC5 | F5 | an approved Tier 1 ADR draft completed with no corresponding new ADR file | promote the missing draft into the ADR archive at the next sequential number | autonomous |
| REC6 | F6 | a successfully processed issue's STM lingers in the active area | move the issue STM directory into the archived subtree | autonomous |
| REC7 | F7 | an issue was archived while a proposal lacked a confirmed successful write to product LTM and was not explicitly rejected | restore the issue to the active area, ensure every approved proposal's write actually lands or is explicitly rejected, then re-archive | autonomous |
| REC8 | F8 | a malformed proposal left a partial product LTM write | a malformed proposal must produce no write — the partial outward-facing write needs a human call to unwind | human |
| REC9 | F9 | sweep processed a directory under the archived or pending subtrees | exclude the archived and pending subtrees from enumeration and re-run | autonomous |

At Level 4, `intent-resolver` executes the autonomous entries (REC5, REC6, REC7, REC9) without a human; the human entries (REC1, REC2, REC3, REC4, REC8) always escalate.

```
for each target issue (single mode = [issue_arg]; sweep mode = enumerated list):
  if {stm_archive}/{YYYY-MM}/{issue}/status/enrich.json shows "completed" → skip issue
  check for status file at {stm_base}/{issue}/status/enrich.json
  for each step in compiled order:
    if step "completed" → skip
    if step "in_progress" → reset to pending (may not have finished)
    mark step "in_progress"
    execute step
    mark step "completed"
```

**Fresh start:** No status file → execute all steps, create status file on first step.
**Resume:** Status file exists → skip completed, continue from first incomplete.
**Pause:** User terminates session. Status file reflects last completed step. Next invocation resumes.

**Sweep failure isolation:** A failure on one issue's DAG does NOT halt the sweep. The play records the failure in that issue's evidence, leaves the issue in the active area (not archived), and continues with the next enumerated issue.
