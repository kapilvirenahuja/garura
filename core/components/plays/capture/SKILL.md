# garura:capture

Accept a structured issue capture request from any caller (human or agent), infer or accept explicit issue type (feature / bug / defect / epic / enhancement), format the GitHub Issue title and body per type, dispatch the gh filing step asynchronously, and return a crisp single-line dispatched confirmation. If GitHub is unreachable, produce a portable local fallback.

## Pre-flight

Execute these checks before any domain work:

| Check | Constraint | Action on Failure |
|-------|-----------|-------------------|
| Resolve `stm_base` from `.garura/core/config.yaml` | — | Hard halt — config is required |
| Resolve `repo_slug` from `github.owner` + `github.name` in `.garura/core/config.yaml` | C3 | Hard halt — report missing/empty owner or name |
| Required fields present (title, problem) | C1 | Hard halt — write rejection artifact naming missing field(s), no filing |
| Severity present when type is `defect` | C1 | Hard halt — write rejection artifact naming severity, no filing |
| Type is valid or inferable | C2 | Validate/infer per rules below |
| Ambiguous inference handled per caller kind | C2 | Agent callers default to `bug`; human callers get prompted before dispatch |

```bash
# Resolve STM base path
config_path=".garura/core/config.yaml"
stm_base=$(grep '^\s*base-path:' "$config_path" | head -1 | awk '{print $2}')

# C3: Resolve target repo slug from config — never hardcoded
github_owner=$(awk '/^github:/{flag=1;next} /^[^ ]/{flag=0} flag && /^[[:space:]]+owner:/{print $2; exit}' "$config_path")
github_name=$(awk '/^github:/{flag=1;next} /^[^ ]/{flag=0} flag && /^[[:space:]]+name:/{print $2; exit}' "$config_path")
if [ -z "$github_owner" ] || [ -z "$github_name" ]; then
  echo "HALT: github.owner and github.name MUST be set in $config_path"
  exit 1
fi
repo_slug="${github_owner}/${github_name}"

# C4: Auto-generate date
capture_date=$(date +%Y-%m-%d)
```

**Type validation and inference (C2):**

1. If caller provided `type`:
   - Must be one of: `feature`, `bug`, `defect`, `epic`, `enhancement`. Otherwise HALT — write rejection artifact at `{stm_base}/capture-report/evidence/capture/rejection.yaml`.
   - If type is `defect`, severity is required (C1). Otherwise HALT with rejection artifact naming severity.
2. If caller did not provide `type`, infer from problem text using keywords:
   - `bug` / `error` / `broken` / `crash` → `defect`
   - `feature` / `add` / `support` / `implement` → `feature`
   - `epic` / `initiative` / `programme` → `epic`
   - `improvement` / `enhance` / `optimize` → `enhancement`
   - otherwise → ambiguous
3. On ambiguous inference:
   - **Agent caller** — default to `bug`. Record decision in `{stm_base}/capture-report/evidence/capture/inference-trace.yaml` with `ambiguous_text: true` and `fallback: bug`.
   - **Human caller** — prompt for type confirmation before any dispatch. Record `human-confirmation-prompt` event in inference-trace.yaml, then `human-confirmed` event with chosen type, then proceed. Do NOT dispatch before the confirmation event exists.

**Issue detection:** This play is not branch-based. Use synthetic identifier `capture-report` for STM paths.

**Resume check:** If `{stm_base}/capture-report/status/capture.json` exists, resume — skip completed steps, reset any `in_progress` to pending.

## Workflow

### Phase: Execution — Format

**Step 1 — Format Capture Content**
Owner: play
Depends on: pre-flight

Resolve the final `type` (caller-provided, inferred, or human-confirmed). Compute title prefix per C8:

| Type | Prefix |
|------|--------|
| `feature` | `[FEAT]` |
| `bug` | `[BUG]` |
| `defect` | `[DEF][{Severity}]` |
| `epic` | `[EPIC]` |
| `enhancement` | `[ENH]` |

Compute label per C9 from the `github-issue.md` template table. If the required label does not exist in the target repo, the filing agent MUST create it before issue creation.

Construct the GitHub Issue body per C5/C6:

```markdown
| Field | Value |
|-------|-------|
| **Type** | {type} |
| **Severity** | {severity}  <!-- only when provided -->
| **Reported From** | {reported_from}  <!-- only when provided -->
| **Date** | {capture_date} |

### Problem

{problem}

<!-- Each optional section appears ONLY when the caller provided the field (C6) -->
### Evidence
{evidence}

### Root Cause
{root_cause}

### Specific Issues
{specific_issues}

### Expected Behavior
{expected_behavior}

### Impact
{impact}
```

Write to STM:
- `{stm_base}/capture-report/evidence/capture/title.txt`
- `{stm_base}/capture-report/evidence/capture/body.md`
- `{stm_base}/capture-report/evidence/capture/inference-trace.yaml` (always — records type resolution path)

**Step 1 Evals:**
- **SE-1 (F1):** A capture request submitted with title or problem missing is rejected before any filing occurs. Rejection artifact at `{stm_base}/capture-report/evidence/capture/rejection.yaml` exists naming the missing required field (title and/or problem). No `issue-result.yaml` is written — no gh issue was created and no fallback file exists.
- **SE-2 (F2):** A capture request with type "defect" and severity omitted is rejected before filing. Rejection artifact exists naming severity as the missing required field for defect type. No `issue-result.yaml` is written.
- **SE-3 (F3):** A capture request with a type value outside the allowed set that cannot be inferred from problem text is rejected before filing. Rejection artifact exists naming the invalid type value and the allowed set. No `issue-result.yaml` is written.
- **SE-8 (F8):** When the caller did not provide a type and the problem text matches no C2 keyword — for agent callers the filed issue type is `bug` (label and prefix correspond to bug); for human callers a `human-confirmation-prompt` event exists in inference-trace.yaml before any dispatch event.

### Phase: Execution — Dispatch (background)

**Step 2 — Dispatch GH Filing (background)**
Owner: `project-orchestrator`
Depends on: Step 1

Dispatch via `Agent` tool with `run_in_background: true` (Cbg). Control returns to the caller before this agent completes. The issue number is NOT available at confirmation time.

```json
{
  "subagent_type": "project-orchestrator",
  "run_in_background": true,
  "intent_path": "core/components/plays/capture/reference/intent.yaml",
  "stm_base": "{stm_base}",
  "stm": {
    "input": {
      "title": "{stm_base}/capture-report/evidence/capture/title.txt",
      "body": "{stm_base}/capture-report/evidence/capture/body.md"
    },
    "output": {
      "result": "{stm_base}/capture-report/evidence/capture/issue-result.yaml",
      "dispatch_record": "{stm_base}/capture-report/evidence/capture/issue-comment-agent.yaml"
    }
  },
  "task_id": "file-github-issue",
  "config": {
    "platform": "github",
    "repo": "{repo_slug}",
    "labels": ["{type_driven_label}"],
    "create_label_if_missing": true
  }
}
```

The `{repo_slug}` and `{type_driven_label}` are runtime-resolved — never hardcoded.

Write a tracking stub to `issue-comment-agent.yaml` immediately (before dispatch returns):

```yaml
status: dispatched
dispatched_at: "{ISO timestamp}"
task_id: file-github-issue
type: "{type}"
```

Agent reads title/body from STM, invokes `manage-issue` (action: create) with the resolved label, creating the label if absent (C9). On success, writes `issue-result.yaml` with `title`, `labels`, `type`, `url`, `completed_at`. On gh failure (auth, network, permissions), agent writes a fallback markdown file at `{stm_base}/capture-report/evidence/capture/fallback-{capture_date}.md` containing the full issue content and records the specific failure reason in `issue-result.yaml` with `status: failed` and `failure_reason: {auth|network|permissions}` (C7).

**Step 2 Evals:**
- **SE-4 (F4):** On gh failure, at least one of: `issue-comment-agent.yaml` with `status: dispatched`, or `fallback-{date}.md` under `evidence/capture/`. Neither missing = failure.
- **SE-5 (F5):** On gh failure, the caller-facing response names what failed (auth, network, or permissions) AND includes the fallback file path AND sharing instructions (email, Slack, paste into a GitHub issue).
- **SE-6 (F6):** The caller confirmation return event precedes the `issue-comment-agent.yaml` completed_at timestamp. Status file shows dispatch-step completed with the background task still `in_progress` at the moment the caller received the confirmation.
- **SE-9 (C5):** Issue body (captured in `issue-result.yaml`) contains a leading markdown table with metadata rows followed by labeled sections using markdown headings for problem and for any provided evidence / expected behavior / impact fields.
- **SE-10 (C6):** For each optional field NOT provided by the caller, the corresponding section heading does not appear in the issue body. Only caller-provided optional fields produce section headings.
- **SE-11 (C8):** Filed issue title begins with the correct prefix for the resolved type per the table. For defect type, the prefix contains both `[DEF]` and `[Severity]`.
- **SE-12 (C9):** Filed issue has at least the type-driven label from the `github-issue.md` template table applied. If the label did not exist prior to filing, a label-create event precedes the issue-create event.
