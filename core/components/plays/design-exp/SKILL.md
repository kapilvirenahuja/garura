# design-exp

The DESIGN-stage play in Meridian's 5-stage SDLC (Discover → Specify → Design → Build → Run). Reads locked specify-product output and produces the UX layer: personas, screens, flows, wireframes, interaction patterns, consolidated design spec. Shallow or incomplete output is structurally impossible — the play halts and cycles back on any validation failure. Visual design (colors, typography, spacing) is explicitly out of scope.

Intent defines constraints (C1-C13), failure conditions (F1-F12), and scenarios (S1-S7).
To modify this play, update intent.yaml and re-run `/create-play --rebake design-exp`.

## Role

You are the orchestrator. You own the workflow. You delegate domain tasks to agents via JSON contracts — never execute domain work directly.

**Forbidden:** Direct screen authoring. Direct wireframe generation. Direct persona creation. Visual-design decisions (colors, typography, spacing scales). All domain work goes through the `designer` agent, which invokes skills.

**Agent boundaries:**

| Agent | Domain | Phases |
|-------|--------|--------|
| `designer` | UX design — personas, screens, flows, wireframes, design spec | All design phases (Stages 1-6) |
| `judge` | Screen schema + coverage validation (Mode 2) | Execution (post-screen-inventory, post-flows) |
| `scriber` | Background writes of evidence, checkpoint, and status artifacts | All phases (utility, exempt from domain count) |
| `repo-orchestrator` | Self-commit of STM + product artifacts at play close | Close (utility, exempt) |

### Phase: Scenario Validation

After all steps complete, run the scenario evals.

**SCE-1 (S1 — Developer):** Given the screen inventory with all state entries and the wireframe for a specific screen, the developer can build the screen layout directly from the specification — components, data fields, layout pattern, and actions are all explicit — without asking the designer for clarification. Every screen MD file contains a ## Wireframe section with an explicit layout pattern, named components, data fields, and actions per state; a developer reading any screen file can build the layout without external clarification.

**SCE-2 (S2 — Designer (external, receiving the spec)):** Given the personas.md file with JTBD persona profiles and capability mapping, the external designer can validate that every target user type is represented, and can add visual design (colors, type, spacing) on top without re-deriving the interaction structure. personas.md uses JTBD format throughout; every persona has at least one 'When [situation], I want [motivation], so I can [outcome]' job story; every persona maps to at least one capability from the scope; the Capability → Persona Map covers all scope capabilities.

**SCE-3 (S3 — Product Manager):** Given the user flow diagrams for both success and recovery paths, the PM can trace each persona's journey through the product and verify that every edge case raised in the intent epics has a corresponding flow. For every intent epic success_scenario and failure_scenario, at least one flow MD file in flows/ references the scenario via its frontmatter source_scenario field; recovery flows exist for every failure scenario.

**SCE-4 (S4 — Technical Architect):** Given the screen inventory with state counts, data requirements, and action lists, the Technical Architect can estimate frontend complexity (state management needs, API surface, routing depth) without needing to see visual mockups. Every screen MD file discloses its state count, data requirements per state, and action lists per state in the ## States section; navigation entry and exit points are listed in the ## Navigation section.

**SCE-5 (S5 — Accessibility Lead):** Given the wireframes and accessibility requirements per screen, the Accessibility Lead can verify WCAG conformance plans for every interactive screen and flag any screen whose requirements fall below the product's accessibility quality target. Every interactive screen's ## Accessibility section references a specific WCAG level that matches or exceeds the product's quality-profile.interaction_capability target.

**SCE-6 (S6 — Developer):** Given the interaction pattern specifications, the developer can implement form validation, loading indicators, error messaging, and empty states consistently across the product without re-inventing patterns per screen. design-spec.md section 5 'Interaction Patterns' contains a product-wide table with a single decision per pattern (form validation, loading indicators, error placement, empty states, transitions); decisions are applied consistently across listed screens.

**SCE-7 (S7 — Product Manager (at final review)):** Given the consolidated design-spec.md linking personas, screens, flows, wireframes, and interaction patterns, the PM can approve the full design package in one Tether decision, or Orbit with feedback to a specific stage without losing the context of the unchanged stages. design-spec.md contains all sections 1-7 (Scope, Personas, Screen Inventory, User Flows, Interaction Patterns, Accessibility Summary, Handoff Notes); section 7 Handoff Notes points at downstream consumers (build-arch, implement-epic) and explicitly notes that visual design is out of scope; the PM can approve the full package at Checkpoint 3 with a single Tether.
