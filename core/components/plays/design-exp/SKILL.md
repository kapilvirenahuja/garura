# design-exp

The DESIGN-stage play in Meridian's 5-stage SDLC (Discover → Specify → Design → Build → Run). Reads locked specify-product output and produces the UX layer: personas narrowed by mvp-recommendation, screens per user-surface capability with visual-first wireframes that a reader sees at a glance, flows for success + recovery paths, and a consolidated design spec. Shallow or incomplete output is structurally impossible — the play halts and cycles back on any validation failure. Visual design (colors, typography, spacing) is explicitly out of scope.

## Compiled From

This play was compiled from `reference/intent.yaml` by `/create-play --rebuild design-exp`.
Intent defines constraints (C1-C18), failure conditions (F1-F17), and scenarios (S1-S7).
To modify this play, update `reference/intent.yaml` and re-run `/create-play --rebuild design-exp`.
Do NOT edit this file manually — it is a compiled artifact.

## Role

You are the orchestrator. You own the workflow. You delegate domain tasks to agents via JSON contracts — never execute domain work directly.

**Forbidden:** Direct screen authoring. Direct wireframe generation. Direct persona creation. Visual-design decisions (colors, typography, spacing scales). Passing `ltm_domain_taxonomy_path` or any direct reference to `core/components/memory/knowledge/domain/` as a skill input — per C14 / rules/product.md Rule 15 (Pull-to-Product Principle), all domain content is read from `{product_base}research/` only. Writing screen MD files in any section order other than visual-first (`## Wireframe` → `## Purpose` → `## Personas` → `## States` → `## Navigation` → `## Accessibility` → `## Layout Spec`) — per C16, every screen file is visual-first so a reader opens the file and SEES the layout at a glance before reading any prose. **Silently committing any inferred decision produced by a downstream skill without running the tiered Decision Surfacing flow** (C17 / F16) — high-confidence auto-resolution is a blocking failure, not a shortcut. **Dispatching any skill that produces inferred decisions without a `decision_manifest_path` in its JSON contract, or continuing past a skill whose manifest is missing/malformed** (C18 / F17). All domain work goes through the `designer` agent, which invokes skills.

**Decision Surfacing contract (C17 / C18 / F16 / F17):** Every skill dispatch in this play that produces inferred decisions — `synthesize-personas`, `generate-screen-inventory`, `map-user-flows`, `generate-wireframes`, `compile-design-spec` — MUST receive a `decision_manifest_path` in its JSON contract and MUST write a `decision-manifest.yaml` at that path before returning. After each such skill completes, the orchestrator runs the **tiered Decision Surfacing flow** BEFORE any downstream skill consumes the skill's primary artifact: (1) Load the manifest; (2) group entries by `tier`; (3) present HIGH as a single batch confirmation ("Tether all N high-confidence decisions, or Orbit any single one to reopen"); (4) present MID as a batch with explicit questions per decision ("Tether the batch, or drill into individual decisions one-by-one"); (5) present LOW one-by-one with full Q&A depth matching configure-capabilities (grounding_source, recommendation, alternatives_considered, user_response per decision); (6) write `user_response` (accept | override | orbit) + `user_response_detail` back into the manifest for every entry; (7) only then advance to the next step. A skill whose manifest is missing, omits any required field (tier, grounding_source, recommendation, alternatives_considered), or whose `user_response` is not populated for all entries after the surfacing flow is a structural failure per F17 and halts the play.

### Phase: Execution — Wireframe Generation

**Step 8 — Generate wireframes**
Owner: `designer`
Depends on: Step 7 (passed)

Domain content for layout / component hints is read from `{product_base}research/` (C14).

**Visual-first two-block rewrite (C16):** generate-wireframes does NOT simply append to each screen file. It performs a TWO-BLOCK REWRITE:

1. **Visual wireframe block (top of file):** Reads each screen's existing `## States` section and composes one Unicode/ASCII box-drawing block per state. The skill then **replaces the `## Wireframe` placeholder** that Stage 3 wrote at the top of the file with the real visual block. Each block is a fenced `text` code block using ASCII-safe Unicode box-drawing characters (`┌ ─ ┐ │ └ ┘ ├ ┤ ┬ ┴ ┼`), every line under 80 columns, every box a true rectangle, and **no visual-design elements** (no hex codes, no pixel/rem values, no font-family names — those are downstream visual-design concerns).

2. **Layout Spec block (bottom of file):** Composes the machine-parseable detailed spec — layout pattern name, grid/regions, component inventory table, per-state spec (layout, components, data, actions, transitions), interaction patterns, data binding, and accessibility wireframe notes. The skill **appends this as a NEW `## Layout Spec` section** at the BOTTOM of the file, after `## Accessibility`. Downstream skills (e.g., build-arch) consume this section.

The required body section order at the end of Stage 8 is: frontmatter → H1 → `## Wireframe` (real ASCII per state) → `## Purpose` → `## Personas` → `## States` → `## Navigation` → `## Accessibility` → `## Layout Spec`. This satisfies the "complete shape" half of C16. A reader opening any screen file SEES the layout at a glance (top of file), reads the prose annotation in the middle, and consults the machine-spec at the bottom. Idempotent on re-runs: if `## Wireframe` already contains real ASCII and `## Layout Spec` is present, the skill regenerates both from the current `## States` content and rewrites in place.
