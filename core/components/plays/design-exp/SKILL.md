# design-exp

The DESIGN-stage play in Meridian's 5-stage SDLC (Discover → Specify → Design → Build → Run). Reads locked specify-product output and produces the UX layer: personas, screens, flows, wireframes, interaction patterns, consolidated design spec. Shallow or incomplete output is structurally impossible — the play halts and cycles back on any validation failure. Visual design (colors, typography, spacing) is explicitly out of scope.

## Compiled From

This play was compiled from `reference/intent.yaml` by `/create-play --build`.
Intent defines constraints (C1-C13), failure conditions (F1-F12), and scenarios (S1-S7).
To modify this play, update intent.yaml and re-run `/create-play --build design-exp`.
Do NOT edit this file manually — it is a compiled artifact.

## Role

You are the orchestrator. You own the workflow. You delegate domain tasks to agents via JSON contracts — never execute domain work directly.

**Forbidden:** Direct screen authoring. Direct wireframe generation. Direct persona creation. Visual-design decisions (colors, typography, spacing scales). All domain work goes through the `designer` agent, which invokes skills.

**Agent boundaries:**

| Agent | Domain | Phases |
|-------|--------|--------|
| `designer` | UX design — personas, screens, flows, wireframes, design spec | All design phases (Stages 1-6) |
| `judge` | Screen schema + coverage validation (Mode 2) | Execution (post-screen-inventory, post-flows) |
| `scriber` | Background writes of evidence, checkpoint, and status artifacts | All phases (utility, exempt from domain count) |
| `repo-orchestrator` | Self-commit of STM + product artifacts at play close | Close (utility, exempt) |

## Pre-flight

Execute these checks before any domain work:

| Check | Constraint | Action on Failure |
|-------|-----------|-------------------|
| Resolve `stm_base` and `product_base` from `.meridian/core/config.yaml` | — | Hard halt — config required |
| specify-product artifacts exist and are LOCKED: `scope.yaml`, `enriched-capabilities.yaml`, `epics/*.yaml`, `quality-profile.yaml` under `{product_base}/product/` | C1 | Hard halt — "specify-product must complete before design-exp starts" |
| KB catalog consistency — invoke `validate-kb-extension` skill | C2 | Hard halt — print validator report, block until fixed |
| `{product_base}/ux/` writable; scriber agent reachable | C10, C11 | Hard halt — infrastructure check |

```bash
stm_base=$(grep '^\s*base-path:' .meridian/core/config.yaml | awk '/stm:/,/base-path/' | tail -1 | awk '{print $2}')
product_base=$(grep -A1 '^product:' .meridian/core/config.yaml | grep 'base-path' | awk '{print $2}')

# C1: verify specify-product artifacts exist AND status is LOCKED
for f in scope.yaml enriched-capabilities.yaml quality-profile.yaml; do
  test -f "${product_base}product/$f" || halt "missing $f — run specify-product first"
done
test -d "${product_base}product/epics" || halt "missing epics directory"
# parse each .yaml for `status: LOCKED` — halt if any DRAFT

# C2: invoke validate-kb-extension via Skill tool — abort on status: failed

# C10, C11: test write access to ${product_base}ux/ and scriber availability
```

**Resume check:** If `{product_base}ux/_status/design-exp.json` exists, resume — skip completed stages, reset any `in_progress` to pending, continue from first incomplete.

### Phase: Scenario Validation

After all steps complete, run the scenario evals.

**SCE-1 (S1 — Developer):** Every screen MD file contains a `## Wireframe` section with explicit layout pattern, named components, data fields, and actions per state. A developer reading any screen file can build the layout without external clarification.

**SCE-2 (S2 — External Designer):** `personas.md` uses JTBD format throughout. Every persona has at least one job story in the `When [situation], I want to [motivation], so I can [outcome]` structure, and every persona maps to at least one capability from the scope.

**SCE-3 (S3 — Product Manager):** For every intent epic success and failure scenario, there is at least one flow MD file in `flows/` that references the scenario via its frontmatter `source_scenario` field.

**SCE-4 (S4 — Technical Architect):** The screen inventory discloses state counts, data requirements, and action lists for every screen. The Technical Architect can estimate frontend complexity (state management needs, API surface, routing depth) without visual mockups.

**SCE-5 (S5 — Accessibility Lead):** Every interactive screen's `## Accessibility` section references a specific WCAG level that matches or exceeds the product's `quality-profile.interaction_capability` target.

**SCE-6 (S6 — Developer, interaction patterns):** `design-spec.md` section 5 "Interaction Patterns" contains a product-wide table with a single decision per pattern (form validation, loading indicators, error placement, empty states, transitions). A developer can implement consistently without cross-referencing individual screens.

**SCE-7 (S7 — Product Manager at final review):** `design-spec.md` section 7 "Handoff Notes" points at the downstream consumers (build-arch, implement-epic) and explicitly notes that visual design is out of scope. The PM can approve the full package at Checkpoint 3 with a single Tether.
