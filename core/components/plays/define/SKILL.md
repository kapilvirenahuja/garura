# define

The SCOPE-stage play in Meridian's product-configuration flow. Takes a feature idea (free-text description OR a FEAT-ID referencing an existing entry in the product catalog) and produces: a feature row in `features.yaml` with an `issue` field linking to the anchoring GH issue, any new capability/domain extensions, an intent epic under the product epics directory that passes `validate-intent-epics`, and STM evidence recording the catalog-match, constraint-fit, grounding, and decision history for the scoping session. The play is the single doorway from raw feature idea to `/prepare`-ready scope.

## Compiled From

This play was compiled from `reference/intent.yaml` by `/create-play --build define`.
Intent defines constraints (C1-C16), failure conditions (F1-F10), and scenarios (S1-S6).
To modify this play, update `intent.yaml` and re-run `/create-play --rebuild define`.
Do NOT edit this file manually — it is a compiled artifact.

## Role

You are the orchestrator. You own the workflow. You delegate domain tasks to agents via JSON contracts — never execute domain work directly (C13).

**Forbidden:** Direct `gh` or `git` invocations. Direct feature-catalog authoring. Direct epic drafting. Direct `gh issue` CRUD. All domain work goes through agents, which invoke skills.

**Agent boundaries:**

| Agent | Domain | Phases |
|-------|--------|--------|
| `project-orchestrator` | GH issue resolve/create/close (via `manage-issue` skill) | Phase 0, Phase 5 (on Vanish) |
| `product-keeper` | Catalog match, constraint fit, KB/LTM grounding, product-spec updates, epic generate/validate | Phases 2, 3, 4, 6, 7 (and gap-interview re-validate in Phase 8) |
| `repo-orchestrator` | STM + product-LTM evidence commit at close (utility — exempt from domain-agent budget) | Phase 10 |

Phases 1, 5, 8, 9 are play-owned (user-facing interview and checkpoint surface).

## Pause and Resume

Phases execute top-to-bottom. At any point, the play can be interrupted and resumed.

**Status file:** `{stm_base}{issue}/status/define.json`

```json
{
  "play": "define",
  "issue": "{issue}",
  "started_at": "2026-04-20T10:00:00+0530",
  "phases": {
    "phase-0-issue": { "status": "completed", "completed_at": "..." },
    "phase-1-intake": { "status": "completed", "completed_at": "..." },
    "phase-2-catalog-match": { "status": "completed", "completed_at": "..." },
    "phase-3-constraint-fit": { "status": "in_progress", "started_at": "..." },
    "phase-4-grounding": { "status": "pending" },
    "phase-5-placement-checkpoint": { "status": "pending" },
    "phase-6-spec-update": { "status": "pending" },
    "phase-7-epic": { "status": "pending" },
    "phase-8-gap-interview": { "status": "pending" },
    "phase-9-epic-checkpoint": { "status": "pending" },
    "phase-10-evidence-commit": { "status": "pending" }
  }
}
```

**Resume logic:**

```
read status file
for each phase in compiled order:
  if status == "completed" -> skip
  if status == "in_progress" -> reset to "pending", re-execute
  if status == "pending" -> execute
  write status update
```

Phase 0 is idempotent — on resume with a completed status, the issue number is read from `issue-resolution.yaml` and reused; no new issue is created.
