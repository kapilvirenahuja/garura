# meridian:fix-it

Given an existing open issue describing a defect, perform thorough root cause analysis tracing symptoms to the specific code and logic that is wrong, design a fix with alternatives considered and risk assessment, present RCA findings and fix design at a single human checkpoint for approval, then upon approval implement the fix and ship it (commit, PR, merge) fully autonomously. The end state is: fix merged to main, feature branch deleted, issue traceable in commit and PR history.

## Pre-flight

Execute these checks before any domain work:

| Check | Constraint | Action on Failure |
|-------|-----------|-------------------|
| Resolve `stm_base` from `.garura/core/config.yaml` | — | Hard halt — config is required |
| Resolve `product_base` from `.garura/core/config.yaml` | C11 | Soft — set `has_product_ltm=false`, continue without product LTM |
| Issue number resolvable | C1 | Hard halt — "No issue provided. Provide an issue number or run from a fix/ branch." |
| Issue exists and is open | C1 | Hard halt — "Issue #{n} does not exist or is not open." |
| Current branch guard | C2 | Create fix/{issue}-{slug} branch via repo-orchestrator |

```bash
# Resolve STM base path
stm_base=$(grep '^\s*base-path:' .garura/core/config.yaml | head -1 | awk '{print $2}')

# C11: Resolve product base path for LTM grounding
product_base=$(grep -A1 '^product:' .garura/core/config.yaml | grep 'base-path' | awk '{print $2}')
has_product_ltm=false
if [ -d "${product_base}architecture/" ]; then
  has_product_ltm=true
fi

# C1: Extract issue number from branch name or arguments
branch=$(git branch --show-current)
issue=$(echo "$branch" | grep -oE '/[0-9]+' | tr -d '/')
# If no issue from branch, expect it from user input
# If no issue at all → hard halt

# C2: Branch guard — determine if on main/master/default
default_branch=$(git remote show origin | grep 'HEAD branch' | awk '{print $NF}')
# If on default branch → need to create fix/ branch
```

**Resume check:** If `{stm_base}/{issue}/status/fix-it.json` exists, resume — skip completed steps, reset any `in_progress` to pending, continue from first incomplete.

**LTM context assembly (C11):** The `ltm_context` block grounds tech-designer's RCA and design in the product's architecture and organizational knowledge via the R1-R4 resolution protocol (`standards/rules/resolution.md`).

- `project_base` points to the product's locked architecture directory. **When `has_product_ltm` is false** (no product architecture artifacts exist), omit `project_base` from the contract — tech-designer resolves against core LTM only (R3-R4).
- `locked_artifacts` is populated by scanning `{product_base}architecture/` for artifacts with `status: LOCKED`. If none are locked, pass an empty list — all project artifacts are treated as DRAFT (advisory).
- The resolution trace output at `stm.output.resolution_trace` records which knowledge layer answered each domain question during RCA. This trace feeds downstream plays (capture-learning, check-drift).

Agent reads intent.yaml — applies constraints C3 (RCA depth), C4 (alternatives), C5 (risk assessment), C11 (LTM grounding). Reads the issue from STM. Follows R1-R4 resolution protocol before analysis. Performs:

1. **Root Cause Analysis** — trace from symptom to specific root cause: the file, the logic, and why it is wrong (C3). Not a restatement of the issue title.
2. **Blast Radius** — affected files map with roles and required changes.
3. **Fix Design** — chosen strategy with at least one alternative considered and reason rejected (C4).
4. **Risk Assessment** — risks with severity levels for each (C5).
5. **Execution Plan** — self-sufficient steps for code-builder to implement.

Writes RCA and design artifacts to STM.

**Step 3 Evals:**
- **SE-2 (F2/C3):** File `{stm_base}/{issue}/evidence/fix-it/rca.yaml` exists and its `root_cause` field is non-empty, names at least one specific file path, identifies the specific logic or code location within that file, and explains why it is wrong — the value must be textually distinct from the `title` and `body` fields in `{stm_base}/{issue}/evidence/fix-it/issue-read.yaml`.
- **SE-3 (F3/C4):** File `{stm_base}/{issue}/evidence/fix-it/design.yaml` exists and its `alternatives_considered` field is a non-empty list where each entry contains both an option name/description and a `rejection_reason` field that is non-empty.
- **SE-4 (C5):** File `{stm_base}/{issue}/evidence/fix-it/design.yaml` contains a `risks` list with at least one entry. Every entry in the list has a non-empty `severity` field (e.g., `low`, `medium`, `high`, `critical` or equivalent).
- **SE-5 (C11/F8):** When the tech-designer contract included a `ltm_context` block, file `{stm_base}/{issue}/evidence/fix-it/resolution-trace.yaml` exists and contains a non-empty list of resolution entries. Each entry must include a `resolved_from` field. When `has_product_ltm` is true, at least one entry must have `resolved_from` of `"project"` or `"core"` (not every entry may be `"llm"`). When `has_product_ltm` is false, the trace must contain at least one entry from `"core"` or must document `"llm"` fallback explicitly with a rationale field.

If RCA is inconclusive (tech-designer returns structured failure) → HALT: "RCA inconclusive — cannot proceed to fix design."

**Executor loop:**
```
resolve issue number from branch name or user input
check for status file at {stm_base}/{issue}/status/fix-it.json

for each step in compiled order:
  if status file shows step "completed" -> skip
  if status file shows step "in_progress" -> reset to pending
  mark step "in_progress" in status file
  execute step
  mark step "completed" in status file
```

**Resume:** Status file exists → skip completed, continue from first incomplete.
**Pause:** User terminates session. Status file reflects last completed step.
