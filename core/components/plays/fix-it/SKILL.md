# fix-it

Given an existing open issue describing a defect, perform thorough root cause analysis tracing symptoms to the specific code and logic that is wrong, design a fix with alternatives considered, present RCA findings and fix design at a single human checkpoint for approval, then upon approval implement the fix (with independent verification by quality-auditor) and ship it (commit, PR, merge) fully autonomously. The end state is: fix merged to main, feature branch deleted, issue traceable in commit and PR history.

## Pre-flight

Execute these checks before any domain work:

## Recovery

Sourced from `reference/expectation.yaml` `recovery` — one entry per failure condition. When the validator detects a tripped failure, it builds a recovery handoff plan from the matching entry and routes it per `handoff`: `autonomous` loops the fix back to the builder; `human` escalates for a manual call.

| ID | For | Symptom (trigger) | Direction | Handoff |
|----|-----|-------------------|-----------|---------|
| REC1 | F1 | implementation began (report exists or implement-fix in_progress) while approval-gate is not a completed Tether preceding it | hold for the single human Tether/Vanish before any implementation step runs | human |
| REC2 | F2 | rca.yaml missing, root_cause empty, or root_cause restates the issue title/body | re-run the RCA to trace symptom → specific file, logic, and why it is wrong, distinct from the issue text | autonomous |
| REC3 | F3 | design.yaml missing or alternatives_considered empty / lacking a rejection_reason | re-run the design step to add at least one alternative with the reason it was rejected | autonomous |
| REC4 | F4 | commits exist but no PR, or a PR exists but was not merged — intermediate post-ship state | resume the ship pipeline from the first incomplete step to a fully merged, branch-deleted state | autonomous |
| REC5 | F5 | the issue does not exist on GitHub or is not open when the play starts | escalate for the human to provide or reopen a valid open issue before the play proceeds | human |
| REC6 | F6 | files_modified lists a file absent from design.yaml.affected_files with no deviation_justification | restrict the change to the mapped files or record the deviation justification before shipping | autonomous |
| REC7 | F7 | the code-builder contract carries a checkpoint brief path, HTML path, or inline checkpoint markdown | rebuild the contract with only the STM design paths plus the regression-test path, stripping all brief content | autonomous |
| REC8 | F8 | ltm_context was provided but resolution-trace.yaml is missing, empty, or has zero entries | re-run the RCA to emit a non-empty resolution trace recording the layer that answered each domain question | autonomous |
| REC9 | F9 | a standalone HTML artifact (or briefs/ directory) exists under evidence/fix-it/ after checkpoint prep | remove the HTML artifact and any briefs/ directory so the evidence directory holds none | autonomous |
| REC10 | F10 | Tether received but issue-comment-agent.yaml not written (or status: pending) before Step 6 | write the dispatched tracking stub with a dispatched_at timestamp before implement-fix goes in_progress | autonomous |

At Level 4, `intent-resolver` executes the autonomous entries (REC2, REC3, REC4, REC6, REC7, REC8, REC9, REC10) without a human; the human entries (REC1, REC5) always escalate. When an agent returns a structured failure (per `structured-failure-protocol.md`), read `domain_assessment.responsible_domain`, invoke the responsible agent with fix context plus the original intent, cap at 2 retry cycles per agent then HALT — while the Step 6 + Step 6b retry loop is governed separately by C19 (cap=2 with tech-designer re-plan on exhaustion; warn-and-continue, never halt, never escalate).

## Pause and Resume

Steps are in execution order — run top to bottom.

**Issue detection:** Extract issue number from branch name during pre-flight (e.g., `fix/42-broken-login` -> `42`). If not on a fix/ branch, issue comes from user input.

**Status file:** `{stm_base}/{issue}/status/fix-it.json`

```json
{
  "play": "fix-it",
  "issue": "{issue}",
  "started_at": "{ISO timestamp}",
  "tasks": {
    "validate-issue": { "status": "completed", "completed_at": "..." },
    "create-branch": { "status": "completed", "completed_at": "..." },
    "rca-and-design": { "status": "in_progress", "started_at": "..." },
    "prepare-checkpoint": { "status": "pending" },
    "issue-comment-dispatch": { "status": "pending" },
    "approval-gate": { "status": "pending" },
    "implement-fix": { "status": "pending" },
    "verify-fix": { "status": "pending" },
    "ship": { "status": "pending" },
    "scenario-evals": { "status": "pending" },
    "evidence-report": { "status": "pending" }
  },
  "task_list": [
    {
      "id": "task-1",
      "description": "Apply fix per design.yaml step 1",
      "status": "pending",
      "retry_count": 0,
      "regression_test_path": "{stm_base}/{issue}/evidence/fix-it/regression-test.yaml"
    }
  ]
}
```

**`task_list` ownership (C18):** The play is the sole writer of `task_list`. Agents may propose additions via their output contracts but MUST NOT mutate `task_list` directly. `task_list` is initialized from RCA output before code-builder is invoked. Each entry contains: `id`, `description`, `status` (`pending | in_progress | completed | failed`), `retry_count`, and `regression_test_path`.

**Executor loop:**
```
resolve issue number from branch name or user input
check for status file at {stm_base}/{issue}/status/fix-it.json

for each step in compiled order:
  if status file shows step "completed" -> skip
  if status file shows step "in_progress" -> reset to pending
  mark step "in_progress" in status file
  execute step
  mark step "completed" in status file
```

**Fresh start:** No status file → execute all steps, create status file on first step.
**Resume:** Status file exists → skip completed, continue from first incomplete.
**Pause:** User terminates session. Status file reflects last completed step.
