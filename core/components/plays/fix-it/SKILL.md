# meridian:fix-it

Given an existing open issue describing a defect, perform thorough root cause analysis tracing symptoms to the specific code and logic that is wrong, design a fix with alternatives considered and risk assessment, present RCA findings and fix design at a single human checkpoint for approval, then upon approval implement the fix and ship it (commit, PR, merge) fully autonomously. The end state is: fix merged to main, feature branch deleted, issue traceable in commit and PR history.

Execute these checks before any domain work:

|-------|-----------|-------------------|

| Issue number resolvable | C1 | Hard halt — "No issue provided. Provide an issue number or run from a fix/ branch." |
| Issue exists and is open | C1 | Hard halt — "Issue #{n} does not exist or is not open." |
| Current branch guard | C2 | Create fix/{issue}-{slug} branch via repo-orchestrator |

```bash

# C1: Extract issue number from branch name or arguments
branch=$(git branch --show-current)
issue=$(echo "$branch" | grep -oE '/[0-9]+' | tr -d '/')
# If no issue from branch, expect it from user input
# If no issue at all → hard halt

# C2: Branch guard — determine if on main/master/default
default_branch=$(git remote show origin | grep 'HEAD branch' | awk '{print $NF}')
# If on default branch → need to create fix/ branch
```

**Resume check:** If `{stm_base}/{issue}/status/fix-it.json` exists, resume — skip completed steps, reset any `in_progress` to pending, continue from first incomplete.


Agent reads intent.yaml — applies constraints C3 (RCA depth), C4 (alternatives), C5 (risk assessment). Reads the issue from STM. Performs:

1. **Root Cause Analysis** — trace from symptom to specific root cause: the file, the logic, and why it is wrong (C3). Not a restatement of the issue title.
2. **Blast Radius** — affected files map with roles and required changes.
3. **Fix Design** — chosen strategy with at least one alternative considered and reason rejected (C4).
4. **Risk Assessment** — risks with severity levels for each (C5).
5. **Execution Plan** — self-sufficient steps for code-builder to implement.

Writes RCA and design artifacts to STM.

**Step 3 Evals:**
- **SE-2 (F2/C3):** RCA artifact contains root_cause field that identifies a specific file, the specific logic within that file, and an explanation of why it is wrong — distinct from the issue title and problem description.
- **SE-3 (F3/C4):** Design artifact contains alternatives_considered with at least one entry, and each entry includes the reason it was rejected.
- **SE-8 (C5):** Design artifact contains a risks section where each identified risk has an associated severity level.

If RCA is inconclusive (tech-designer returns structured failure) → HALT: "RCA inconclusive — cannot proceed to fix design."

## Pause and Resume

Steps are in execution order — run top to bottom.

**Issue detection:** Extract issue number from branch name during pre-flight (e.g., `fix/42-broken-login` -> `42`). If not on a fix/ branch, issue comes from user input.

```json

  "issue": "{issue}",
  "started_at": "{ISO timestamp}",
  "tasks": {
    "validate-issue": { "status": "completed", "completed_at": "..." },
    "create-branch": { "status": "completed", "completed_at": "..." },
    "rca-and-design": { "status": "in_progress", "started_at": "..." },
    "generate-brief": { "status": "pending" },
    "approval-gate": { "status": "pending" },
    "implement-fix": { "status": "pending" },
    "ship": { "status": "pending" },
    "scenario-evals": { "status": "pending" },
    "evidence-report": { "status": "pending" }
  }
}
```

**Executor loop:**
```
resolve issue number from branch name or user input
check for status file at {stm_base}/{issue}/status/fix-it.json

for each step in compiled order:
  if status file shows step "completed" -> skip
  if status file shows step "in_progress" -> reset to pending
  mark step "in_progress" in status file
  execute step
  mark step "completed" in status file
```

**Fresh start:** No status file → execute all steps, create status file on first step.
**Resume:** Status file exists → skip completed, continue from first incomplete.
**Pause:** User terminates session. Status file reflects last completed step.
