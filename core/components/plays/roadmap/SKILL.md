# roadmap

Take the features /shape has created and decide what to build first. /roadmap ranks the
`active` and `proposed` features into a build sequence: every active feature is
sequenced ahead of every proposed one, and within each tier features are ordered by
value (read from their ICE goals and outcomes) and by their dependencies. It writes that
sequence as a single number — `priority` — on each feature node, and touches nothing
else. One run ranks the whole comparable set, because priority only means something by
comparison. Convention: a lower priority number means sooner, so active features take
the smaller numbers.

**Pipeline position: none.** /roadmap is a strategic, model-building play in the shaping
pipeline. It opens no delivery issue and cuts no branch, so the D2 pipeline-position
rule injects neither a `start-change` head nor a close sequence. It writes the
persistent product model directly. It runs after /shape, since features must exist
before they can be ranked.

## Role

You are the orchestrator. You own the workflow and the step order. You delegate the one
piece of judgment — ordering the features by value — to the `product-os-keeper` agent
via a JSON contract over files on disk, and you run every mechanical part (the snapshot,
the ranking math, the allowlisted persist, and the post-apply checks) through bundled
scripts. You never write the model YAML yourself, you never assign priorities by hand,

## Workflow

### Phase: Snapshot

**Step 1 — Capture snapshot** · Owner: play · Depends on: pre-flight
Take the pre-apply "before" picture — the model state the ranking reads from and the
verify step compares against. This is a **gated step**, so on resume it is skipped and
the pre-apply snapshot is preserved (the pre-flight probe never writes it):

```
python3 scripts/snapshot_model.py --product-base <product_base> --out <working>/snapshot.json
```

The capture holds every node (id, type, status, depends_on, ice_ref, ice_present,
priority, the full parsed node) and a path→hash census of every file under product-os.
**SE (no eval here):** Step 1 only records state; its content is exercised by Step 3's
ranking and proven by Step 6's non-destructive check.

### Phase: Draft

**Step 2 — Draft value ordering** · Owner: `product-os-keeper` · Depends on: Step 1
The agent invokes `author-roadmap` to read each active/proposed feature's ICE goals and
outcomes and order the features by value within each tier (active, proposed,
independently), with a one-line reason per feature, and to list any feature with no ICE
as un-rankable. It drafts `value-order.yaml` to STM:

    {
      "task":    "order the active and proposed features by value from their ICE goals + outcomes; one reason each; list no-ICE features as un-rankable",
      "inputs":  { "snapshot_path": "<working>/snapshot.json",
                   "product_base":  "<product_base>" },
      "outputs": { "value_order":   "<working>/value-order.yaml" }
    }

The skill reads ICE **read-only** and writes only the draft. It assigns no priority
numbers and does not resolve dependencies or the tier rule — those are Step 3's
deterministic job. It returns the contract with the output path on disk — never inline
content.
**SE (no eval here):** Step 2 produces only a value preference; its correctness is
proven downstream by the ranking checks in Step 3 and the persisted checks in Step 6.

### Phase: Compute

**Step 3 — Compute ranking** · Owner: play · Depends on: Step 2
Turn the value preference into a coherent, dependency-correct ranking:

```
python3 scripts/compute_ranking.py --snapshot <working>/snapshot.json \
        --value-order <working>/value-order.yaml --out <working>/ranking.json
```

`compute_ranking.py` partitions the features (C1/C5), applies the two-tier rule (active
block before proposed block, C2), topologically sorts within each tier so dependencies
precede dependents with the value order as tiebreak (C4), records cross-tier
active→proposed dependencies as anomalies, and assigns the distinct integer priorities
1..N (C6).
**SE-1 (F5/C1):** the ranking assigns a priority only to active/proposed features — the
`ranking` list contains no capability, domain, or deprecated-feature id; such ids appear
only in `clear` if they held a stale priority.
**SE-2 (F2/C2):** the two-tier rule holds in the computed ranking — every active
feature's priority is smaller than every proposed feature's.
**SE-4 (F3/C4):** within each tier no feature precedes one it depends_on; every
active→proposed dependency is recorded in `anomalies.cross_tier_deps`, not reordered
across the boundary.
**SE-5 (F4/C5):** every active/proposed feature with no ICE is in `unrankable` and gets
no priority.
**SE-6 (F6/C6):** the script exits 0 (coherent) — priorities are distinct integers 1..N
and `anomalies.cycles` is empty; on a non-zero exit (cycle or a feature missing from the
value order) apply REC2/REC3/REC6 and re-run before the checkpoint.

### Phase: Checkpoint (mandatory — never skipped, C9)

**Step 4 — Human review** · Owner: play · Depends on: Step 3
Present the proposed ranking **inline** — the active tier in order and the proposed tier
in order, each feature with its priority number and its one-line value reason, plus the
un-rankable features and any cross-tier dependency anomalies (an active feature that
depends on a still-proposed one) for the human to resolve. This checkpoint is always
presented and never skippable. Approve → continue to persist; cancel → halt with no
priority written to the model.
**SE-9 (F9/C9):** priorities are persisted only after this approval — Step 5 is the sole
writer and depends on this step; no node's `priority` changes before Step 5.

### Phase: Apply

**Step 5 — Apply priorities** · Owner: play · Depends on: Step 4
Persist on the fixed allowlist. `apply_roadmap.py` is handed only the ranking and the
snapshot; it does a read-modify-write that changes only `node.priority` on each ranked
feature (and resets to null any feature in `clear`), preserving every other field, and
cannot reach any other node, ICE, the profile, lenses, personas, journeys, or decisions:

```
python3 scripts/apply_roadmap.py --ranking <working>/ranking.json \
        --snapshot <working>/snapshot.json --product-base <product_base> \
        --out-manifest <working>/apply-manifest.json
```

**SE-3 (F1/C3):** the apply manifest's `changes` touch only the `priority` field on
feature nodes — no other field and no other artifact (node/ICE/profile/lens/persona/
journey/decision) appears.

**Step 6 — Verify persisted** · Owner: play · Depends on: Step 5
Prove the persisted model against the snapshot and the ranking:

```
python3 scripts/validate_roadmap.py --snapshot <working>/snapshot.json \
        --ranking <working>/ranking.json --apply-manifest <working>/apply-manifest.json \
        --product-base <product_base>
```

**SE-7 (F7/C7):** non-destructive — every file under product-os is byte-identical to the
snapshot except feature node files whose only changed field is `priority`; no file added
or removed, no non-node file changed.
**SE-8 (F8/C8):** every feature node still validates against product-os v1 — required
fields present and `priority` an integer or null.
(SE-1, SE-2, SE-4, SE-5 are re-asserted here against the persisted model: no
cap/domain/deprecated priority, tier rule, intra-tier dependencies, un-rankable null.)
On any GAP, apply the matching recovery (REC1–REC8) and re-run.

### Phase: Scenario Validation

**Step 7 — Scenario evals** · Owner: play · Depends on: Step 6
- **SCE-1 (S1 — product strategist, first ranking):** every active/proposed feature has
  a non-null integer priority; every domain, capability, and deprecated feature has
  `priority: null`; every written node validates product-os v1.
- **SCE-2 (S2 — product owner, tiers hold):** the largest priority among active features
  is smaller than the smallest priority among proposed features.
- **SCE-3 (S3 — architect, dependencies):** for every intra-tier `A depends_on B`,
  priority(B) ≤ priority(A); every active→proposed dependency is named in the
  checkpoint's anomaly list.
- **SCE-4 (S4 — QA engineer, un-rankable):** a feature with a null `ice_ref` / no ICE is
  in the un-rankable report and its `priority` is still null.
- **SCE-5 (S5 — product owner, re-run non-destructive):** on an unchanged model every
  product-model file is byte-identical before and after; on a changed model the only
  field differing in any node file is `priority`, and no non-node file changed.
- **SCE-6 (S6 — reviewer, the checkpoint):** the checkpoint showed the active tier
  ordered, the proposed tier ordered, each item's reasoning, and the anomaly list,
  before any write.

## Recovery

| For | Trigger | Direction | Handoff |
|-----|---------|-----------|---------|
| F1 | a write touched something other than a feature node's priority | revert the out-of-scope write; /roadmap writes only the priority field on feature nodes | autonomous |
| F2 | a proposed feature's priority beats an active one's | recompute the ranking with the tier rule enforced — all active before all proposed — before persisting | autonomous |
| F3 | within a tier, a feature is ranked ahead of a feature it depends_on | re-sort that tier so dependencies precede dependents; a cross-tier active→proposed conflict is surfaced at the checkpoint, never resolved across the boundary | autonomous |
| F4 | a feature with no ICE was given a priority | drop the guessed priority and move the feature to the un-rankable report | autonomous |
| F5 | a capability, domain, or deprecated feature received a priority | clear that priority back to null; only active/proposed features are ranked | autonomous |
| F6 | the ordering is invalid (a ranked feature has no integer priority, or the values are incoherent) | recompute a complete, coherent integer ranking over the ranked set before persisting | autonomous |
| F7 | a node field other than priority changed, or a non-node product-model file changed | restore the changed content and re-apply only the priority updates, after a human confirms the restore | human |
| F8 | a written node fails product-os v1 schema validation | re-emit the failing node to conform to its schema before the play completes | autonomous |
| F9 | a priority was persisted before the checkpoint was approved | revert the premature write and re-present the checkpoint; persist only after the human approves | human |

## Pause and Resume

Steps run top to bottom. On entry, resolve config, run the write-free pre-flight probe,
check the status marker (`{stm_base}_shaping/roadmap/status.json`), skip completed steps,
reset any in-progress step to pending, and continue. A fresh start with no marker runs
everything and creates the marker at Step 1. The pre-apply snapshot is captured once, at
Step 1, and preserved on resume (Step 1 is skipped once complete) — so the
non-destructive comparison always diffs against true pre-apply state, never against an
already-applied model. A genuinely new run (no marker) re-captures a fresh snapshot.
