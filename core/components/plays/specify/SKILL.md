# specify

The SPECIFY-stage play in Garura's 5-stage SDLC (Discover → Specify → Design → Build → Run). Transforms a product idea plus project profile into a complete product specification: market intelligence, selected capabilities traced to the KB catalog, an MVP recommendation that narrows the capability walk, enriched intent epics with quantified constraints, and an aggregated quality profile. Shallow output is structurally impossible — the play halts and cycles back on any validation failure.

## Compiled From

This play was compiled from `reference/intent.yaml` (the clean triple) and
`reference/expectation.yaml` (success_scenarios + recovery) by `/create-play`.
Intent defines constraints (C1-C21) and failure conditions (F1-F15); the expectation
defines success scenarios (S1-S8) and recovery (REC1-REC15).
To modify this play, update `reference/intent.yaml` or `reference/expectation.yaml`
and re-run `/create-play --build specify`.
Do NOT edit this file manually — it is a compiled artifact.

## Role

You are the orchestrator. You own the workflow. You delegate domain tasks to agents via JSON contracts — never execute domain work directly.

**Forbidden:** Direct KB reading for feature selection. Direct epic authoring. Direct quality-profile computation. All domain work goes through agents, which invoke skills.

**Decision Surfacing Discipline (C18 — structural):** Every skill that emits inferred decisions writes a `decision-manifest.yaml` alongside its primary artifact. After each such skill returns, the orchestrator — BEFORE dispatching the next downstream skill — walks the manifest and drives the tiered surfacing flow:

- **HIGH tier decisions** → batch confirmation. Present the list with grounding source per item. User may Tether the whole batch in one keystroke, or Orbit any single row to reopen it for one-by-one review.
- **MID tier decisions** → batch presentation with explicit per-decision questions and recommendations. User may Tether the batch, or drill one-by-one.
- **LOW tier decisions** → one-by-one only, NEVER batched. Each low-confidence decision is its own Tether/Vanish/Orbit exchange.

High confidence is permission to batch — never permission to skip user review. `configure-capabilities` is the canonical exemplar for one-by-one Q&A depth. Silent commits (including high-confidence auto-resolution and silent KB-single-candidate selection) are blocking failures per F14. A skill that produces inferred decisions and writes no manifest is a structural failure per F15.

## Pre-flight

Execute these checks before any domain work:

| Check | Constraint | Action on Failure |
|-------|-----------|-------------------|
| Resolve `stm_base` and `product_base` from `.garura/core/config.yaml` | — | Hard halt — config required |
| Product description has >=5 meaningful words | C1 | Hard halt — "Product description too vague; please elaborate" |
| Project profile is complete (security_level, industry, audience, timeline, scale, compliance, team_size, delivery_ambition) | C2 | Interactive collection loop — pause until complete |
| KB catalog consistency — invoke `validate-kb-extension` skill | C3 | Hard halt — print validator report, block until fixed |
| `.garura/product/` writable; scriber agent reachable | C10, C11 | Hard halt — infrastructure check |
| `scope/mvp-recommendation.md` exists and is non-empty (gate for Stage 3 — see rules/product.md Rule 13) | C15 | Hard halt — "Author scope/mvp-recommendation.md before proceeding to Stage 3 — see rules/product.md Rule 13 and Defect 6 for the artifact shape" |

```bash
stm_base=$(grep '^\s*base-path:' .garura/core/config.yaml | awk '/stm:/,/base-path/' | tail -1 | awk '{print $2}')
product_base=$(grep -A1 '^product:' .garura/core/config.yaml | grep 'base-path' | awk '{print $2}')

# C1: count meaningful words in product description (excluding filler)
word_count=$(echo "$product_idea" | tr -s ' ' '\n' | grep -v -E '^(a|an|the|of|for|to|in|on|with|and|or|but|is|are|am)$' | wc -l)
# halt if word_count < 5

# C2: profile schema check — all required fields present
# interactive collection if missing

# C3: invoke validate-kb-extension via Skill tool — abort on status: failed

# C10, C11: test write access to .garura/product/ and scriber availability

# C15: Stage-3 pre-flight — run before Step 6 (Configure capabilities).
# The MVP recommendation artifact is authored at Stage 2.75 (play-owned verify
# step) between Checkpoint 2 (domain review) and Stage 3 (capability
# configuration). It gates Stage 3 and cannot be bypassed.
test -s "${product_base}/scope/mvp-recommendation.md" || {
  echo "HALT (C15): ${product_base}/scope/mvp-recommendation.md is missing or empty."
  echo "See rules/product.md Rule 13 and Defect 6 for the expected artifact shape."
  exit 1
}
```

**Resume check:** If `{product_base}/_status/specify.json` exists, resume — skip completed stages, reset any `in_progress` to pending, continue from first incomplete.

**Pull-to-product copy (Rule 15 / C17 / Defect 8):** after `domain-selection.yaml` is written, for EACH selected domain:

1. Check whether a canonical KB file exists at `core/components/memory/knowledge/domain/{domain}.md`.
2. If YES (domain is KB-sourced): copy the KB file to `{product_base}/research/{domain}.md` and prepend a provenance header as an HTML comment block at the top of the copied file:
   ```
   <!-- Provenance
   origin: kb
   kb_source_path: core/components/memory/knowledge/domain/{domain}.md
   copied_at: <ISO-8601 timestamp>
   kb_sha_at_copy: <shasum -a 256 of the KB file>
   editable: false
   -->
   ```

3. If NO (domain does NOT exist in KB): leave the domain for the DD-1 research-then-promote flow. That flow authors `{product_base}/research/{domain}.md` with `origin: stm_research`, `editable: true`, and the 9 required sections per `kb-extension-conventions.md`. `configure-capabilities` treats both origins through the same read path — the provenance header is how downstream consumers know whether to treat the file as frozen or editable.

Missing or stale copies at `{product_base}/research/{domain}.md` cause `configure-capabilities` to halt at Stage 3 with `what_failed: missing_domain_in_product_research`. Re-running Step 3 is the canonical recovery — this step owns population of the product research folder for KB-sourced domains.

| ID | For | Symptom (trigger) | Direction | Handoff |
|----|-----|-------------------|-----------|---------|
| REC1 | F1 | zero intent epics in the output directory | re-run epic generation so every selected capability yields an epic file | autonomous |
| REC2 | F2 | an epic mandatory field is empty, null, whitespace, or placeholder | re-generate the affected epics so no mandatory field is empty or placeholder | autonomous |
| REC3 | F3 | a constraint field carries unquantified prose (fast/secure/reliable) | re-generate constraints with a number+unit, named standard/control, WCAG level, or specific regulation | autonomous |
| REC4 | F4 | an epic has fewer than two success or fewer than two failure scenarios | re-generate the affected epics to meet the minimum scenario counts | autonomous |
| REC5 | F5 | an epic's kb_source.capability points at a non-existent feature ID | re-derive traceability so kb_source.capability resolves to a real feature | autonomous |
| REC6 | F6 | scope.selected_capabilities has an entry that is not a real feature ID | re-run capability configuration so every selection resolves to an existing feature | autonomous |
