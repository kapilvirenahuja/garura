# specify

The SPECIFY-stage play in Meridian's 5-stage SDLC (Discover → Specify → Design → Build → Run). Transforms a product idea plus project profile into a complete product specification: market intelligence, selected capabilities traced to the KB catalog, an MVP recommendation that narrows the capability walk, enriched intent epics with quantified constraints, and an aggregated quality profile. Shallow output is structurally impossible — the play halts and cycles back on any validation failure.

## Compiled From

This play was compiled from `reference/intent.yaml` by `/create-play --rebuild specify`.
Intent defines constraints (C1-C19), failure conditions (F1-F15), and scenarios (S1-S8).
To modify this play, update intent.yaml and re-run `/create-play --rebuild specify`.
Do NOT edit this file manually — it is a compiled artifact.

## Role

You are the orchestrator. You own the workflow. You delegate domain tasks to agents via JSON contracts — never execute domain work directly.

**Forbidden:** Direct KB reading for feature selection. Direct epic authoring. Direct quality-profile computation. All domain work goes through agents, which invoke skills.

**Decision Surfacing Discipline (C18 — structural):** Every skill that emits inferred decisions writes a `decision-manifest.yaml` alongside its primary artifact. After each such skill returns, the orchestrator — BEFORE dispatching the next downstream skill — walks the manifest and drives the tiered surfacing flow:

- **HIGH tier decisions** → batch confirmation. Present the list with grounding source per item. User may Tether the whole batch in one keystroke, or Orbit any single row to reopen it for one-by-one review.
- **MID tier decisions** → batch presentation with explicit per-decision questions and recommendations. User may Tether the batch, or drill one-by-one.
- **LOW tier decisions** → one-by-one only, NEVER batched. Each low-confidence decision is its own Tether/Vanish/Orbit exchange.

High confidence is permission to batch — never permission to skip user review. `configure-capabilities` is the canonical exemplar for one-by-one Q&A depth. Silent commits (including high-confidence auto-resolution and silent KB-single-candidate selection) are blocking failures per F14. A skill that produces inferred decisions and writes no manifest is a structural failure per F15.

**Pull-to-product copy (Rule 15 / C17 / Defect 8):** after `domain-selection.yaml` is written, for EACH selected domain:

1. Check whether a canonical KB file exists at `core/components/memory/knowledge/domain/{domain}.md`.
2. If YES (domain is KB-sourced): copy the KB file to `{product_base}/research/{domain}.md` and prepend a provenance header as an HTML comment block at the top of the copied file:
   ```
   <!-- Provenance
   origin: kb
   kb_source_path: core/components/memory/knowledge/domain/{domain}.md
   copied_at: <ISO-8601 timestamp>
   kb_sha_at_copy: <shasum -a 256 of the KB file>
   editable: false
   -->
   ```
   The provenance header is authoritative — Stage 3 and every later stage read the domain library from `{product_base}/research/` only; the KB directory is NOT a read path at or after Stage 3 (see `configure-capabilities` Input section and Rule 15).
3. If NO (domain does NOT exist in KB): leave the domain for the DD-1 research-then-promote flow. That flow authors `{product_base}/research/{domain}.md` with `origin: stm_research`, `editable: true`, and the 9 required sections per `kb-extension-conventions.md`. `configure-capabilities` treats both origins through the same read path — the provenance header is how downstream consumers know whether to treat the file as frozen or editable.

Missing or stale copies at `{product_base}/research/{domain}.md` cause `configure-capabilities` to halt at Stage 3 with `what_failed: missing_domain_in_product_research`. Re-running Step 3 is the canonical recovery — this step owns population of the product research folder for KB-sourced domains.
