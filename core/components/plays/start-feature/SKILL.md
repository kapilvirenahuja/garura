# start-feature

Given any starting state — uncommitted changes, an issue number, or a descriptive title — produce a ready-to-work environment: a GitHub issue tracking the work, a feature branch checked out by convention, and an initialized STM directory. Single flow, no modes. Uncommitted changes always preserved.

## Role

You are the orchestrator. You own the workflow. You delegate domain tasks to agents via JSON contracts — never execute domain work directly.

**Forbidden:** Direct `git` commands for branch creation/switching. Direct `gh` commands for issue operations. All domain work goes through agents (C9).

**Agent boundaries:**

| Agent | Domain | Phases |

| `project-orchestrator` | Issues: resolve issues, create issues, confidence scoring | Execution — Resolve Issue |

## Pre-flight

Execute these checks before any domain work:

| Check | Constraint | Action on Failure |
|-------|-----------|-------------------|
| Resolve `stm_base` from `.garura/core/config.yaml` | — | Hard halt — config is required |
| Git repository present | — | Hard halt — not a git repo |
| Platform config exists (`platform: github`) | — | Hard halt — no platform configured |
| Capture `git status --porcelain` snapshot | C2 | Store as pre-play state for F4 eval |
| Parse input arguments | C1 | Determine input pattern (no args, issue number, title) |

```bash
# Resolve STM base path
stm_base=$(grep '^\s*base-path:' .garura/core/config.yaml | awk '{print $2}')

# Git repo check
git rev-parse --is-inside-work-tree

# Platform config
grep '^platform:' .garura/core/config.yaml

# Snapshot changed files (for F4 eval later)
pre_state=$(git status --porcelain)

# Extract issue from branch name if on a feature branch (for resume detection)
branch=$(git branch --show-current)
issue=$(echo "$branch" | grep -oE '/[0-9]+' | tr -d '/')
```

No branch guard. The play works from any branch (main, develop, feature/*).

**Input parsing** — the play accepts three input patterns (C1 — single flow, no modes):

| Input | Detection | Action |
|-------|-----------|--------|
| No args | No arguments provided | Capture changed files → resolve issue from changes |
| `42` or `#42` | Numeric or `#`-prefixed numeric | Use as issue number directly |
| `"Add OAuth login"` | Non-numeric string | Use as issue title for creation |

All three converge on the same downstream flow: issue resolved → branch created → STM initialized.

**Resume check:** If `{stm_base}/{issue}/status/start-feature.json` exists, resume — skip completed steps, reset any `in_progress` to pending, continue from first incomplete.

Agent reads intent.yaml for constraints C4 (open issues only), C5 (ambiguity resolution), C6 (confidence-gated), C10 (issue body ~40 words). Invokes `manage-issue` (resolve_or_create) or `resolve-issues` (for inference from changes) as needed.

**Expected output artifact (`issue-resolution.yaml`):**

```yaml
issue:
  number: {int}
  title: "{title}"
  url: "{url}"
  state: "open"
  created: true|false
  type_hint: "{feature|fix|hotfix|refactor|docs|chore|null}"

  source: "changed_files|explicit_number|explicit_title"
  confidence: "high|medium|low"
  candidates_considered: {int}
  ambiguity_resolved_by_user: true|false
```

Note: Uses `{stm_base}/_pending/start-feature/` for pre-issue artifacts. Once issue number is known, the play moves artifacts to `{stm_base}/{issue}/evidence/start-feature/`.

**Ambiguity handling (C5):** If the agent returns with `status: "blocked"` and ambiguity needs resolution, the play presents candidates to the user, collects their choice, and re-invokes the agent with the selected issue number.

**Step 2 Eval:**

| Eval | Failure Condition | Check |
|------|-------------------|-------|
| SE-1 | F1 | Issue number is non-null and issue exists on GitHub as OPEN |
| SE-2 | F5 | If inference confidence is "low", play must have halted — not proceeded |
| SE-3 | F6 | If existing open issue matched, no new issue was created |
| SE-4 | F7 | If best candidate was closed, user confirmation was obtained before reopen |
| SE-5 | F8 | If multiple candidates with no clear winner, user was asked to pick |
| SE-6 | C10 | If new issue created, body contains ~25-60 word summary |

If SE-2 triggers (low confidence), halt play. Do not proceed to Step 3.

## Recovery

Sourced from `reference/expectation.yaml` `recovery` — one entry per failure condition. When the validator detects a tripped failure, it builds a recovery handoff plan from the matching entry and routes it per `handoff`: `autonomous` loops the fix back to the builder; `human` escalates for a manual call.

| ID | For | Symptom (trigger) | Direction | Handoff |
|----|-----|-------------------|-----------|---------|
| REC1 | F1 | feature branch produced but no associated GitHub issue recorded in issue-resolution.yaml | re-run issue resolution against the current input so an issue is resolved or created and recorded | autonomous |
| REC2 | F2 | checked-out branch does not follow {type}/{issue_number}-{slug} | rename the branch to the convention using the resolved issue's type, number, and slug | autonomous |
| REC3 | F3 | no STM directory at {stm_base}/{issue}/ on completion | create the missing directory | autonomous |
| REC4 | F4 | files in the pre-play snapshot no longer present in the working tree | lost working-tree state the builder cannot regenerate — a user must decide how to recover before proceeding | human |
| REC5 | F5 | a branch and issue were produced from a low-confidence inference without halting | the weak signal needs a user to confirm or pick the correct issue before any branch or issue is created | human |
| REC6 | F6 | a new issue created even though an existing open issue already matched | reconciling a duplicate against the real matching issue is external-entity judgment — confirm the correct issue and discard the duplicate | human |
| REC7 | F7 | matched or created against a closed issue without confirmation to reopen | reopening is an authority the builder lacks — escalate for the user's go-ahead before proceeding | human |
| REC8 | F8 | multiple candidates with no clear winner, proceeded without asking | present the candidates and let the user pick the correct issue before proceeding | human |

At Level 4, `intent-resolver` executes the autonomous entries (REC1, REC2, REC3) without a human; the human entries (REC4, REC5, REC6, REC7, REC8) always escalate.

## Pause and Resume

Steps are in execution order — run top to bottom.

**Issue detection:** Extract issue number from branch name during pre-flight (e.g., `feature/95-slug` → `95`). If on a feature branch with an issue number, check for status file before starting.

**Status file:** `{stm_base}/{issue}/status/start-feature.json`

```json
{
  "play": "start-feature",
  "issue": 95,
  "started_at": "2026-03-06T10:00:00+0530",
  "tasks": {
    "write-input": { "status": "completed", "completed_at": "..." },
    "resolve-issue": { "status": "completed", "completed_at": "..." },
    "relocate-stm": { "status": "completed", "completed_at": "..." },
    "create-branch": { "status": "in_progress", "started_at": "..." },
    "init-stm-dir": { "status": "pending" },
    "scenario-evals": { "status": "pending" },
    "write-evidence": { "status": "pending" }
  }
}
```

**Executor loop:**
```
resolve issue number from branch name or pending artifacts
check for status file at {stm_base}/{issue}/status/start-feature.json

for each step in compiled order:
  if status file shows step "completed" → skip
  if status file shows step "in_progress" → reset to pending (may not have finished)
  mark step "in_progress" in status file
  execute step
  mark step "completed" in status file
```

**Fresh start:** No status file exists → execute all steps, create status file on first step.
**Resume:** Status file exists → skip completed, continue from first incomplete.
**Pause:** User terminates session. Status file reflects last completed step. Next invocation resumes.
