# understand

Take a capability that /vision seeded — shallow, goals only — and deepen it into a
**rich ICE**: full intent (goals, constraints, failures), context (persona, systems,
scope), expectations (outcomes), and concrete NFR + compliance needs, grounded in the
capability's KB shelf. Then roll those needs into the product profile. On a still
`directional` box the roll-up establishes the box and firms it to `set`; on an
already-`set` box, a need that exceeds the committed ceiling halts for a human
decision that moves the box and records why. One capability per run.

**Pipeline position: none.** /understand is a MIDDLE play of the strategy pipeline (vision → understand → shape → roadmap): it expects to run on the branch /vision already started, injects no `start-change` head and no close sequence, stops when its work is done, and leaves the branch as-is for the next play to pick up. The close belongs to /roadmap. It writes the persistent product model directly, on the already-started branch. (#437)

## Workflow

### Phase: Enrich

**Step 1 — Enrich the ICE** · Owner: `product-os-keeper` · Depends on: pre-flight
The persisted model does **not** record which KB shelf a capability came from
(/vision kept that only in STM, and the product domain name is not the KB shelf name).
So the agent first **re-routes** the capability to its KB shelf with `search-kb`, then
invokes `enrich-capability-ice` to deepen the seed ICE into a rich ICE grounded in that
recovered shelf, emitting the per-dimension required levels the roll-up consumes — all
to a draft folder under STM:

    {
      "task":    "re-route this capability to its KB shelf, then enrich its seed ICE into a rich ICE grounded in that shelf; emit implied NFR levels",
      "inputs":  { "capability_path": "<product_base>/product-os/.../<capability>",
                   "seed_ice_path":   "<product_base>/product-os/.../<capability>/ice.yaml" },
      "outputs": { "routing":         "<working>/routing.yaml",
                   "draft_dir":       "<working>/draft/",
                   "enriched_ice":    "<working>/draft/ice.yaml",
                   "enrich_manifest": "<working>/draft/enrich-manifest.yaml" }
    }

The agent passes the shelf `search-kb` returned to `enrich-capability-ice` as its
`kb_domain`, so the grounding is the shelf the router actually resolved — not a guess.
It returns the contract with the output paths on disk — never inline content. The skill
reads the live model **read-only** and writes only the draft.
**SE-1 (F1/C1):** the target capability node and its seed ICE were resolved and loaded
— an absent capability halted the run at pre-flight (REC1).
**SE-2 (C5):** the `enrich-manifest.yaml` records `grounded_in`, and it matches the
shelf `search-kb` routed to in `routing.yaml` — the enrichment is grounded in the
recovered shelf, not an invented label.

### Phase: Validate + Roll up

**Step 2 — Validate the draft** · Owner: play · Depends on: Step 1
Run the ICE validator over the draft before the checkpoint:

```
python3 scripts/validate_ice.py --ice <working>/draft/ice.yaml
```

**SE-3 (F4/C3):** `validate_ice.py` exits 0 for completeness — intent (goals,
constraints, failures), context (persona, systems, scope), expectations.outcomes, and
`nfr_needs` are all non-empty.
**SE-4 (F5/C4):** every `nfr_needs.<dimension>` is concrete — a measurable target with
a gate, not a vague adjective.
**SE-5 (F2):** the enriched ICE parses and carries the ice v1 shape.
On any GAP, apply REC4/REC5 and re-run before the checkpoint.

**Step 3 — Roll up the box** · Owner: play · Depends on: Step 1
Roll the implied levels into the profile, monotonic-up. The script branches on box
state: a `directional` box is established and firmed to `set` (no moves); a `set` box
yields an out-of-box move for any dimension or compliance regime the need exceeds:

```
python3 scripts/rollup_box.py --profile <product_base>/product-os/profile.yaml \
        --enrich-manifest <working>/draft/enrich-manifest.yaml \
        --out-profile <working>/draft/proposed-profile.yaml \
        --out-report <working>/rollup.json
```

**SE-6 (F6/C6):** the roll-up report shows `state_after: set` and never lowers a
dimension (the proposed level on each dimension is the max of the current box and the
need). `establishing: true` on a `directional` box yields no box-moves.

*Note — partial gates are expected.* On the first firm-up, dimensions no capability has
touched yet keep their directional level and may have empty gates; the profile's
`quality_vision.gates` fills in as more capabilities are understood. This is incremental
firm-up, not a defect.

### Phase: Checkpoint (mandatory — never skipped, C8)

**Step 4 — Human review** · Owner: play · Depends on: Steps 2–3
Present the enriched ICE and the profile changes **inline**. For each box-move in
`rollup.json`, render its **own explicit line item** — the dimension, the from→to
levels, and the product-level decision (ADR) it will create — distinct from the ICE
approval. This checkpoint is always presented and never skippable. Approve → continue
to persist; cancel → halt with nothing written to the model.
**SE-7 (F7/C7):** when the box is `set` and `out_of_box: true`, each move is approved
here before any write — Step 5 is the sole writer and depends on this step, so an
out-of-box need is never persisted without approval.
**SE-8 (C8):** every box-move in the run appears as its own line item (dimension,
from→to, the ADR), separate from the ICE approval — a box-move cannot be rubber-stamped
inside a wall of ICE detail.

### Phase: Apply

**Step 5 — Persist** · Owner: play · Depends on: Step 4
On approval, persist on a fixed allowlist. `apply_understand.py` writes only the paths
it is handed — the target ICE (the enrichment — this one is overwritten), the firmed
profile, and one decision per approved box-move. It never tree-walks, so it cannot
clobber other capabilities by enumeration; Step 6 confirms the written set. (This is a
weaker guarantee than /vision's skip-if-exists writer — a wrong-but-consistent target
path would pass — so Step 6 is a real check, not a formality.)

*Convention (pinned for /shape and /realize to follow):* product-level box-move
decisions live at `{product_base}/product-os/decisions/` (top-level, since a box move is
product-wide); per-capability decisions live under `{capability}/decisions/`.

```
python3 scripts/apply_understand.py \
    --enriched-ice <working>/draft/ice.yaml --target-ice <product_base>/product-os/.../<capability>/ice.yaml \
    --proposed-profile <working>/draft/proposed-profile.yaml --profile-dest <product_base>/product-os/profile.yaml \
    --rollup-report <working>/rollup.json --decisions-dir <product_base>/product-os/decisions \
    --capability-ref <capability id> --decided-by /understand --date "$(date -u +%Y-%m-%d)" \
    --out-manifest <working>/apply-manifest.json
```

**Step 6 — Verify persisted** · Owner: play · Depends on: Step 5
Check what landed against the before/after profiles and the written decisions. Pass the
pre-run profile (snapshot it before Step 5) as `--profile-before`:

```
python3 scripts/check_apply.py --manifest <working>/apply-manifest.json \
        --profile-before <working>/profile-before.yaml --profile-after <product_base>/product-os/profile.yaml \
        --target-ice <product_base>/product-os/.../<capability>/ice.yaml \
        --decisions-dir <product_base>/product-os/decisions
```

**SE-9 (F3/C2):** the written set holds no `node.yaml` — structure is unchanged.
**SE-10 (F9/C9):** the written set is confined to the target capability's ICE, the
profile, and decisions — no other capability's ICE appears.
**SE-11 (F8/C8):** there is one product-level decision per box-move, each carrying its
dimension, its from→to, and status `accepted`.
**SE-12 (F6):** the persisted profile is `state: set` and no dimension was lowered.
**SE-13 (F2):** the persisted profile and decisions conform to their v1 schemas.

### Phase: Scenario Validation

**Step 7 — Scenario evals** · Owner: play · Depends on: Step 6
- **SCE-1 (S1 — product strategist, first firm-up):** on a `directional` box, the ICE
  is enriched complete and the profile is established and firmed to `set` with no
  decision written this run.
- **SCE-2 (S2 — architect, out-of-box move):** on a `set` box exceeded on a dimension,
  the run halted at the checkpoint, and on approval the box rose with a product-level
  `accepted` decision recording the from→to.
- **SCE-3 (S3 — product owner, inside the box):** on a `set` box the need fits, the run
  wrote zero decisions, every dimension level is unchanged, and the ICE is enriched.
- **SCE-4 (S4 — reviewer, non-destructive):** only the target capability's ICE changed
  and every profile dimension is greater-than-or-equal to its pre-run level.
- **SCE-5 (S5 — QA engineer, the checkpoint):** each box-move appeared as its own line
  item (dimension, from→to, ADR) and no product-model file was written before approval.

## Recovery

| For | Trigger | Direction | Handoff |
|-----|---------|-----------|---------|
| F1 | the target capability node or its ICE is absent at start | halt and ask for a valid existing capability (run /vision first to seed it) before proceeding | human |
| F2 | a written ICE, profile, or decision fails v1 schema validation | re-emit the failing artifact to conform to its schema before the play completes | autonomous |
| F3 | a node was created, renamed, reparented, a functionality added, or a status flipped | revert the structural change — /understand touches only ICE, profile, and decisions | autonomous |
| F4 | a required ICE section is empty after enrichment | re-run enrichment to fill the missing section from the capability's shelf and seed goals | autonomous |
| F5 | an NFR need has no measurable value or no gate | re-draft that need as a concrete target with how it is checked | autonomous |
| F6 | a profile dimension was lowered, or the box firmed to a state other than `set` | recompute the roll-up monotonic-up (max per dimension) and firm to `set` | autonomous |
| F7 | an out-of-box need against a `set` box was persisted with no approved decision | revert the box change, surface the move at the checkpoint, persist only after a human approves it | human |
| F8 | a box-move lacks its decision record, or a decision omits its dimension or from→to | write or complete the product-level decision for each move with its dimension and from→to before persisting | autonomous |
| F9 | an ICE file other than the target capability's was written | restore the other capability's ICE and re-run writing only the target's ICE, after a human confirms the restore | human |
