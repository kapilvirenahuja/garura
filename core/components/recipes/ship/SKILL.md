# ship

Deliver all uncommitted work on the current feature branch to the main branch in a single automated pipeline: commit all changes, create and submit a pull request, merge the PR, switch to main, pull latest, and delete the feature branch. No human approvals at any stage — all decisions auto-proceed. The end state is: local checkout on main, main up to date with remote, feature branch gone, and all work reachable from main's history.

## Pre-flight

Execute these checks before any sub-recipe invocation:

| Check | Constraint | Action on Failure |
|-------|-----------|-------------------|
| Resolve `stm_base` from `core/config.yaml` | — | Hard halt — config is required |
| Current branch is not main/master/default | C1 | Hard halt with message |
| Changed files exist (staged, unstaged, untracked) | implicit | Graceful exit — nothing to ship |
| Platform config exists | implicit | Hard halt — no platform configured |

```bash
# Resolve STM base path
stm_base=$(grep 'base-path' core/config.yaml | head -1 | awk '{print $2}')

# Extract issue number from branch name
branch=$(git branch --show-current)
issue=$(echo "$branch" | grep -oE '/[0-9]+' | tr -d '/')

# C1 — branch guard
default_branch=$(git remote show origin | grep 'HEAD branch' | awk '{print $NF}')
# Halt if branch == default_branch or branch is "main" or "master"

# Changes exist
git status --porcelain
# Graceful exit if empty

# Platform config
grep '^platform:' core/config.yaml
# Halt if platform is unset
```

Ship's pre-flight is lighter than individual sub-recipe pre-flights because each sub-recipe runs its own. Ship validates only cross-cutting concerns.

**Resume check:** If `{stm_base}/{issue}/status/ship.json` exists, resume — skip completed steps, reset any `in_progress` to pending, continue from first incomplete.

**Step 4 — Scenario Evals**

## Pause and Resume

Steps are in execution order — run top to bottom.

**Issue detection:** Extract issue number from branch name during pre-flight (e.g., `feature/95-slug` → `95`).

**Status file:** `{stm_base}/{issue}/status/ship.json`

```json
{
  "recipe": "ship",
  "issue": "{issue}",
  "started_at": "{ISO timestamp}",
  "feature_branch": "{branch name before merge}",
  "tasks": {
    "commit-code": { "status": "completed", "completed_at": "..." },
    "create-pr": { "status": "in_progress", "started_at": "..." },
    "merge-pr": { "status": "pending" },
    "scenario-evals": { "status": "pending" },
    "write-evidence": { "status": "pending" }
  }
}
```

**Executor loop:**
```
resolve issue number from branch name
check for status file at {stm_base}/{issue}/status/ship.json

for each step in compiled order:
  if status file shows step "completed" → skip
  if status file shows step "in_progress" → reset to pending (may not have finished)
  mark step "in_progress" in status file
  execute step
  mark step "completed" in status file
```

**Fresh start:** No status file → execute all steps, create status file on first step.
**Resume:** Status file exists → skip completed sub-recipes, continue from first incomplete.
**Important:** `feature_branch` is stored at start because after merge-pr the branch no longer exists — needed for scenario evals.
