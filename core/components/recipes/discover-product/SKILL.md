# discover-product

Given a product idea or problem statement, discover the market opportunity, draft a product vision with strategic goals and market positioning, validate the vision for completeness, and lock it when ready. The recipe runs as a single invocation with inline checkpoints. It produces a vision artifact (product.yaml), an interactive HTML brief (product-brief.html) for human review, validates completeness, and locks the vision — all in one flow. All product artifacts are stored in project-scoped product STM. Supports both market-facing products and internal libraries.

## Pre-flight

Execute these checks before any domain work.

| Check | Constraint | Action on Failure |
|-------|-----------|-------------------|
| Intent text provided and >5 meaningful words | C1 | Hard halt |
| Derive slug from intent text | — | Hard halt if empty |
| Check if `{product_base}/{slug}/product.yaml` exists with status LOCKED | F7 | Hard halt |
| Check for status file for resume | — | Resume if found |

```
product_base = ".meridian/project/product"
intent = "{user-provided intent text}"

# C1: halt if empty or word count <= 5
word_count = count_words(intent)
if word_count <= 5:
  halt("Intent must contain more than 5 meaningful words")

# Derive slug from intent
slug = slugify(intent)

# F7: halt if already locked
if file_exists("{product_base}/{slug}/product.yaml"):
  product = read_yaml("{product_base}/{slug}/product.yaml")
  if product.status == "LOCKED":
    halt("Product {slug} is already LOCKED. Re-discovery not permitted. (F7)")

# Resume check
status_path = "{product_base}/{slug}/status/discover-product.json"
if file_exists(status_path):
  resume from first incomplete step (see Pause and Resume)
```

## Workflow

10 steps, single flow. One invocation: assess → discover → draft → brief → checkpoint → validate → checkpoint → lock → evidence → summary.

### Step 1 — Assess Intent

Owner: recipe (no agent call)
Depends on: pre-flight

Before any agent calls, assess the intent to determine product type (C13) and whether opportunity discovery should be skipped (C12).

**Three-signal assessment (C12):**

Evaluate the intent against three signals:
1. The intent names specific consumers rather than market segments
2. The intent describes a technical interface boundary rather than a user problem
3. No market-facing end users are identifiable

```
product_type = "product"    # default
skip_opportunity = false

if (intent names specific consumers, NOT market segments)
   AND (intent describes technical interface boundary, NOT user problem)
   AND (no market-facing end users identifiable):
  product_type = "library"
  skip_opportunity = true

# C13: Record product_type for product.yaml type field
# C12: Record skip_opportunity for Step 2 conditional
```

**When `skip_opportunity = true`:**
- Skip Step 2 entirely
- Pass `raw_intent` directly to Step 3 (draft-vision) instead of `market_context`
- C5 domain agent budget: 2 base (draft + validate) plus up to 2 cycle-back = 4 max

**When `skip_opportunity = false`:**
- Proceed to Step 2 normally
- C5 domain agent budget: 3 base (discover + draft + validate) plus up to 1 cycle-back = 4 max

Owner: recipe (no agent call)

- `Vanish`/`vanish` -> cycle-back per C9:

**Cycle-back logic (C9):**
1. Read iteration_count from status file
2. Check against type-dependent limit:
   - Product type: max 1 cycle-back iteration
   - Library type: max 2 cycle-back iterations
3. If within limit: increment iteration_count, re-invoke `product-strategist` with `draft-product-vision` skill + feedback from validation issues. Return to Step 3 and continue flow from there.
4. If limit reached: halt with F5 — "Maximum cycle-back iterations reached. Re-invoke the recipe to start fresh."

## Evidence & Close

Evidence pattern per ADR 012:

1. Recipe writes evidence and checkpoint artifacts to STM
2. Recipe presents summary to user
3. Recipe invokes `repo-orchestrator` with targeted self-commit:
   - Explicit file list (only files this recipe produced)
   - Commit message: `chore(stm): record discover-product evidence for {slug}`
   - No analysis, no grouping — just stage and commit
4. Non-blocking: if self-commit fails, recipe logs warning and exits successfully

## Recovery

When an agent returns structured failure:
1. Read `domain_assessment.responsible_domain`
2. Invoke responsible agent with fix context + original intent
3. Max 2 retries per step. After that, halt with full failure context.

For retries, add to contract:
```json
{
  "retry": {
    "previous_failure": "{what_failed}",
    "fix_applied": "{what was done}",
    "attempt": 2
  }
}
```

**Pre-flight failures (C1, F7) are not recoverable** — hard halt.

**Status file:** `{product_base}/{slug}/status/discover-product.json`

```json
  "slug": "{slug}",
  "started_at": "2026-03-23T10:00:00+0530",
  "iteration_count": 0,
  "intent_assessment": {
    "product_type": "product",
    "skip_opportunity": false
  },
  "tasks": {
    "assess-intent": { "status": "completed", "completed_at": "..." },
    "discover-opportunity": { "status": "completed", "completed_at": "..." },
    "draft-vision": { "status": "completed", "completed_at": "..." },
    "generate-brief": { "status": "in_progress", "started_at": "..." },
    "checkpoint-brief": { "status": "pending" },
    "validate-vision": { "status": "pending" },
    "checkpoint-validation": { "status": "pending" },
    "lock-vision": { "status": "pending" },
    "evidence-self-commit": { "status": "pending" },
    "summary": { "status": "pending" }
  }
}
```

**Executor loop:**
```
derive slug from intent text
check for status file at {product_base}/{slug}/status/discover-product.json

for each step in compiled order:
  if status file shows step "completed" -> skip
  if status file shows step "in_progress" -> reset to pending
  mark step "in_progress" in status file
  execute step
  mark step "completed" in status file
```

**Fresh start:** No status file -> execute all steps, create status file on first step.
**Resume:** Status file exists -> skip completed, reset any in_progress to pending, continue from first incomplete step.
