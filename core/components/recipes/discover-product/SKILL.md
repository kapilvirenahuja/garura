# discover-product

Given a product idea or problem statement, discover the market opportunity, draft a product vision with strategic goals and market positioning, validate the vision for completeness, and lock it when ready. The recipe operates in three phases (DRAFT, VALIDATE, LOCK) invoked independently. DRAFT produces a `product.yaml` and an interactive `product-brief.html` for human review at a checkpoint. VALIDATE checks vision completeness with cycle-back support. LOCK finalizes the product artifact for downstream planning. All product artifacts are stored in project-scoped product STM.

## Workflow: DRAFT Phase

Structure A: Pre-flight -> Intent Assessment -> Preparation -> Checkpoint -> Evidence

### Intent Assessment (C12, C13)

Before any agent calls, the recipe assesses the intent to determine:
1. **Product type** (C13): "product" or "library"
2. **Opportunity skip** (C12): Whether to skip Step 1 (discover-opportunity)

**Assessment logic:**

Evaluate the intent against C12's three signals:
1. The intent names specific consumers rather than market segments
2. The intent describes a technical interface boundary rather than a user problem
3. No market-facing end users are identifiable

```
intent_type = "product"  # default
skip_opportunity = false

if (intent names specific consumers, NOT market segments)
   AND (intent describes technical interface boundary, NOT user problem)
   AND (no market-facing end users identifiable):
  intent_type = "library"
  skip_opportunity = true
fi

# C13: Record product_type for product.yaml type field
product_type = intent_type
```

**When `skip_opportunity = true`:**
- Skip Step 1 entirely
- Pass `raw_intent` directly to Step 2 (draft-vision) instead of `market_context`
- C5 agent budget becomes 2 (not 3) for DRAFT phase

**When `skip_opportunity = false`:**
- Proceed to Step 1 normally
- C5 agent budget is 3 for DRAFT phase

## Scenario Validation

Run after the active phase's workflow completes, before evidence.

**DRAFT scenarios:**
- **SCE-1 (S1):** The vision artifact and interactive HTML brief produced by the DRAFT phase enable evaluation of whether the strategic goals and positioning accurately capture the user's intent — including for internal libraries where market sections are absent and the brief renders technical context instead. Pass: product-brief.html contains strategic goals and positioning derived from the original intent. For type "product", market positioning sections are present. For type "library", market sections are absent and technical context renders instead. The brief is navigable via tabbed layout for human review. Fail: The brief does not render strategic goals or positioning, OR a type "library" brief renders market sections that should be absent, OR the brief is not navigable for human evaluation.

**VALIDATE scenarios:**
- **SCE-3 (S3):** The validation results show completeness score, checklist status, and any issues by severity — enabling determination of whether the vision is ready to lock or identifying exactly which sections need strengthening. Pass: Validation output includes a completeness_score, a checklist with per-item status, and issues categorized by severity — a Product Manager can determine readiness to lock or identify specific sections needing improvement. Fail: Validation output is missing completeness_score, checklist, or issue severity — a Product Manager cannot determine readiness or identify which sections need strengthening.

**LOCK scenarios:**
- **SCE-2 (S2):** A LOCKED vision artifact can be used as authoritative input for roadmap planning — strategic goals, success metrics, and scope boundaries are defined clearly enough to derive engineering milestones without ambiguity, regardless of product type. Pass: product.yaml has status LOCKED, contains strategic_goals with 3+ entries each having an ID, contains success_metrics or measurable outcomes, and defines scope boundaries — sufficient to derive engineering milestones without ambiguity. Fail: product.yaml is not LOCKED, OR strategic goals lack IDs or have fewer than 3 entries, OR success metrics are missing, OR scope boundaries are undefined — engineering milestones cannot be derived without ambiguity.
- **SCE-4 (S4):** A LOCKED vision artifact for an internal library with type "library" can be used as authoritative input for roadmap planning — strategic goals with IDs, value proposition, and assumptions are defined clearly enough to derive engineering milestones, even though market analysis sections are absent. Pass: product.yaml has status LOCKED, type "library", strategic_goals with IDs (3+ entries), a value_proposition field, and an assumptions field — sufficient to derive engineering milestones even without market analysis sections. Fail: product.yaml type is not "library", OR status is not LOCKED, OR strategic goals lack IDs, OR value_proposition is missing, OR assumptions are missing — engineering milestones cannot be derived for the internal library.

```json
    "skip_opportunity": false
  },
  "tasks": {
    "draft-discover-opportunity": { "status": "completed", "completed_at": "..." },
    "draft-vision": { "status": "completed", "completed_at": "..." },
    "draft-generate-brief": { "status": "in_progress", "started_at": "..." },
    "draft-checkpoint": { "status": "pending" },
    "draft-evidence": { "status": "pending" },
    "validate-vision": { "status": "pending" },
    "validate-checkpoint": { "status": "pending" },
    "validate-evidence": { "status": "pending" },
    "lock-vision": { "status": "pending" },
    "lock-evidence": { "status": "pending" }
  }
}
```

**Executor loop:**
```
determine slug from input (intent text or artifact path)
check for status file at {product_base}/{slug}/status/discover-product.json

for each step in active phase's compiled order:
  if status file shows step "completed" -> skip
  if status file shows step "in_progress" -> reset to pending
  mark step "in_progress" in status file
  execute step
  mark step "completed" in status file
```

**Fresh start:** No status file -> execute all steps, create on first step.
**Resume:** Status file exists -> skip completed, continue from first incomplete in active phase.
**Cross-phase:** VALIDATE and LOCK phases read but do not reset DRAFT task statuses. Each phase manages only its own tasks.
