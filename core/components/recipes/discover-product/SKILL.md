# discover-product

Given a product idea or problem statement, discover the market opportunity, draft a product vision with strategic goals and market positioning, validate the vision for completeness, and lock it when ready. The recipe runs as a single invocation with inline checkpoints. It produces a vision artifact (product.yaml), an interactive HTML brief (product-brief.html) for human review, validates completeness, and locks the vision — all in one flow. All product artifacts are stored in project-scoped product STM. Supports both market-facing products and internal libraries.

| Check if `{product_base}/discovery/product.yaml` exists with status LOCKED | F7 | Hard halt |
| Check for status file for resume | — | Resume if found |

```

# C1: halt if empty or word count <= 5

if file_exists(status_path):
  resume from first incomplete step (see Pause and Resume)
```

## Workflow

12 steps, single flow. One invocation: assess → discover → draft → derive-profiles → brief → checkpoint → validate → checkpoint → lock → evidence → summary.

### Step 1 — Assess Intent

Owner: recipe (no agent call)
Depends on: pre-flight

Before any agent calls, assess the intent to determine product type (C13) and whether opportunity discovery should be skipped (C12).

**Three-signal assessment (C12):**

Evaluate the intent against three signals:
1. The intent names specific consumers rather than market segments
2. The intent describes a technical interface boundary rather than a user problem
3. No market-facing end users are identifiable

```
product_type = "product"    # default
skip_opportunity = false

if (intent names specific consumers, NOT market segments)
   AND (intent describes technical interface boundary, NOT user problem)
   AND (no market-facing end users identifiable):
  product_type = "library"
  skip_opportunity = true

# C13: Record product_type for product.yaml type field
# C12: Record skip_opportunity for Step 2 conditional
```

**When `skip_opportunity = true`:**
- Skip Step 2 entirely
- Pass `raw_intent` directly to Step 3 (draft-vision) instead of `market_context`
- C5 domain agent budget: 2 base (draft + validate) plus up to 2 cycle-back = 4 max

**When `skip_opportunity = false`:**
- Proceed to Step 2 normally
- C5 domain agent budget: 3 base (discover + draft + validate) plus up to 1 cycle-back = 4 max

### Step 3a — Reverse-Coverage Check (Context-Isolated)

Owner: `judge`
Depends on: Step 3

**Context isolation:** The judge receives ONLY the product.yaml path and the problem_statement (from config). It does NOT receive market context artifacts, product-strategist drafting notes, domain research, or any intermediate reasoning.

```json

  "intent_path": "core/components/recipes/discover-product/reference/intent.yaml",
  "stm_base": ".meridian/product/",
  "mode": "check-input-output-coverage",
  "artifact_paths": {
    "input_path": "{product_base}/discovery/product.yaml",
    "output_path": "{product_base}/discovery/product.yaml"
  },
  "check_type": "problem-to-vision",
  "config": {
    "problem_statement": "{original intent text from pre-flight}",
    "market_context_path": "{product_base}/discovery/market-context.yaml (if Step 2 ran, null otherwise)"
  },
  "stm": {
    "output": {
      "coverage_check_path": "{product_base}/discovery/coverage-check.yaml"
    }
  },
  "task_id": "check-coverage"
}
```

The judge extracts facets from the problem_statement and market context (target users, risks, differentiators, problem dimensions) and verifies that product.yaml addresses each:
- **covered**: strategic goal, value proposition, or scope element traces to this input facet
- **partial**: facet acknowledged but not reflected in strategic goals or metrics
- **dropped**: input facet has no representation in product.yaml
- **drifted**: product.yaml contains strategic goals or scope that don't trace to any input facet

**On dropped elements:** If any core facet of the problem_statement is completely dropped, log in evidence and flag for the user at the next checkpoint (Step 5) — this is not a hard halt for discover-product because the vision is still in DRAFT.

Update status file: `check-coverage → completed`.

**Step 3a Evals:**
- **SE-15:** Read `stm.coverage_check_path` — verify the coverage report exists with a valid summary. PASS if total_input_elements > 0 and all fields are populated. FAIL if report is empty or malformed.
- **SE-16:** Every facet of the problem_statement has at least `partial` coverage in product.yaml. PASS if no core problem facet is `dropped`. FAIL if any facet from the problem_statement is completely dropped — flag for user attention at checkpoint.

### Step 7 — Pre-Lock Resolution Gate

Owner: recipe (no agent call)
Depends on: Step 6

Read validation_result from Step 6. Scan product.yaml for placeholder values.

**7a. Collect unresolved items:**

```
validation = read_yaml(validation_result_path)
product = read_yaml("{product_base}/discovery/product.yaml")

blockers = [i for i in validation.issues if i.severity == "blocker"]

# C14: Scan for placeholder field values

# Returns [{field_path, current_value}] for each matching field

```

**Branch A — Clean path (no unresolved items):**

If `validation.ready_for_lock == true` AND `unresolved_items` is empty (zero blockers, zero warnings, zero placeholders):
- Auto-proceed. Show validation summary inline (completeness score, checklist).
- No user pause required. Continue to Step 8.

**Branch B — Warnings and/or placeholders (no blockers):**

If `validation.ready_for_lock == true` AND `blockers` is empty AND (`warnings` non-empty OR `placeholders` non-empty):
- Write checkpoint artifact to `{product_base}/checkpoints/discover-product/{YYYYMMDD-HHMMSS}.md` with Status: `PENDING_APPROVAL`.
