# briefs

Scan a product directory for YAML design artifacts, checksum each one against stored values, regenerate the corresponding HTML brief for any that changed, and rebuild hub.html. Context-aware — when run from an epic branch, scope regeneration to that epic's artifacts plus product-level artifacts; when on main or with no epic context, scan product-level artifacts and all epics with YAMLs on disk. No flags, no arguments — derives everything from the execution context.

## Role

You are the orchestrator. You own context resolution, checksum comparison, and delegation to doc-builder. You never write HTML directly.

**Forbidden:** Direct HTML generation. Direct brief writing. Direct hub.html writing. All brief and hub work goes through doc-builder.

## Workflow

### Phase: Context Resolution

**Step 1 — Resolve Product and Epic Scope**
Owner: recipe
Depends on: pre-flight

1. Read product directory at `{product_base}/{slug}/`
2. Glob for product-level YAMLs: `product.yaml`, `roadmap.yaml`
3. Glob for epic directories: `epics/*/` — for each, glob for `*.yaml`
4. If epic context was resolved in pre-flight:
   - Scope = product-level YAMLs + `epics/{epic_id}/*.yaml`
   - (C7: product-level always included)
5. If no epic context (main branch or unresolvable):
   - Scope = product-level YAMLs + all `epics/*/*.yaml` found on disk
6. If no YAML artifacts found in scope → graceful halt (F4)

Record the full list of YAML paths in scope with their relative keys (e.g., `product.yaml`, `epics/E1/features.yaml`).

---

### Phase: Checksum Comparison

**Step 2 — Compute and Compare Checksums**
Owner: recipe
Depends on: Step 1

1. Read `{product_base}/{slug}/briefs/.checksums.json` — create empty `{}` if missing (C6)
2. For each YAML in scope:
   a. Compute sha256 of the file content
   b. Look up the relative key in .checksums.json
   c. If checksum differs or key is missing → add to `changed` list
   d. If checksum matches → skip (C2)
3. Build the `changed` list with artifact names grouped by scope:
   - `product_changed`: list of product-level artifact names (e.g., `["product", "roadmap"]`)
   - `epic_changed`: map of `{epic_id: [artifact_names]}` (e.g., `{"E1": ["features", "architecture"]}`)
4. If `changed` is empty → report "All briefs up to date." with hub path, exit (S4)

**Step 2 Evals:**

- **SE-10 (C2):** Staleness is determined solely by comparing a cryptographic checksum of each YAML artifact against the stored checksum from the previous run. A brief is regenerated if and only if its source YAML checksum differs from the stored value or no stored value exists. Pass: The regeneration list exactly matches the set of YAMLs whose sha256 differs from .checksums.json. No other criteria (mtime, file size) influence the decision. Fail: Regeneration decision uses criteria other than checksum comparison, or a YAML with changed checksum is not regenerated, or a YAML with unchanged checksum is regenerated.

- **SE-2 (F2):** No YAML artifact whose checksum matched the stored value had its brief regenerated. Pass: The doc-builder contract's briefs_requested list contains only artifacts whose checksums differed from stored values. Unchanged artifacts are absent from briefs_requested. Fail: A YAML artifact whose checksum has NOT changed appears in the briefs_requested list sent to doc-builder.

- **SE-7 (F7):** On an epic branch, only the scoped epic's artifacts and product-level artifacts are in the regeneration scope. Pass: The briefs_requested list sent to doc-builder contains only artifacts from the resolved epic and product-level artifacts. No artifacts from other epics appear. Fail: Artifacts from unrelated epics are regenerated when only the scoped epic's artifacts and product-level artifacts should be in scope.

- **SE-13 (C7):** When epic context is resolved, product-level artifacts are always included in the scan alongside the scoped epic's artifacts. Pass: The scan scope includes product.yaml and roadmap.yaml (if they exist) regardless of which epic is in context. Fail: Product-level artifacts are excluded from the scan when epic context is resolved.

### Phase: Execution

**Step 3 — Regenerate Changed Briefs**
Owner: `doc-builder`
Depends on: Step 2 (changed list non-empty)

**3a — Product-level briefs** (if any product-level artifacts changed):

```json
{
  "intent_path": "core/components/recipes/briefs/reference/intent.yaml",
  "stm_base": "{product_base}",
  "artifact_base": "{product_base}/{slug}",
  "slug": "{slug}",
  "briefs_requested": ["{product_changed list}"],
  "stm": {
    "input": {
      "product_yaml_path": "{product_base}/{slug}/product.yaml",
      "roadmap_yaml_path": "{product_base}/{slug}/roadmap.yaml"
    }
  },
  "task_id": "regenerate-product-briefs",
  "config": {
    "product_slug": "{slug}",
    "phase": "REVIEW"
  }
}
```

**3b — Epic-scoped briefs** (one call per epic with changes):

For each `{epic_id}` in `epic_changed`:

```json
{
  "intent_path": "core/components/recipes/briefs/reference/intent.yaml",
  "stm_base": "{product_base}",
  "artifact_base": "{product_base}/{slug}/epics/{epic_id}",
  "slug": "{slug}",
  "briefs_requested": ["{epic_changed[epic_id] list}"],
  "stm": {
    "input": {
      "features_yaml_path": "{product_base}/{slug}/epics/{epic_id}/features.yaml",
      "architecture_yaml_path": "{product_base}/{slug}/epics/{epic_id}/architecture.yaml",
      "tech_yaml_path": "{product_base}/{slug}/epics/{epic_id}/tech.yaml",
      "scenarios_yaml_path": "{product_base}/{slug}/epics/{epic_id}/scenarios.yaml",
      "plan_yaml_path": "{product_base}/{slug}/epics/{epic_id}/plan.yaml"
    }
  },
  "task_id": "regenerate-epic-{epic_id}-briefs",
  "config": {
    "product_slug": "{slug}",
    "epic_id": "{epic_id}",
    "phase": "REVIEW"
  }
}
```

Only include existing YAML paths in `stm.input` — skip paths for YAMLs that don't exist on disk.

Doc-builder handles: skill routing, output path computation under `briefs/`, hub.html regeneration after each call.

**Step 3 Evals:**

- **SE-1 (F1):** Every YAML artifact whose checksum changed has a corresponding regenerated HTML brief under briefs/. Pass: For each YAML in the regeneration list, a brief file exists at the expected briefs/ path with a modification time after the recipe started. Fail: A YAML artifact whose checksum changed has no corresponding brief under briefs/ after the recipe completes.

- **SE-3 (F3):** No HTML files exist outside the briefs/ subdirectory after completion. Pass: All brief paths returned by doc-builder are under {artifact_base}/briefs/. No *-brief.html or hub.html exists at {artifact_base}/ root level. Fail: HTML files exist outside the briefs/ subdirectory of the product directory after the recipe completes.

- **SE-11 (C5):** All generated briefs and hub.html reside under the briefs/ subdirectory. No HTML output is written outside that directory. Pass: Every path in doc-builder's output contract briefs_written list is under {artifact_base}/briefs/. hub_path is under briefs/. Fail: Any brief or hub.html path is outside the briefs/ subdirectory.

- **SE-5 (F5):** Hub.html reflects all current briefs — no missing entries for briefs that exist, no entries for briefs that do not exist. Pass: Hub.html contains a card/link for every brief file that exists under briefs/. No card/link points to a non-existent brief. Fail: Hub.html has missing entries for briefs that exist, or entries for briefs that do not exist.

- **SE-8 (F8):** All generated briefs and hub.html use the Phoenix Design System. Pass: Brief HTML files contain Phoenix CSS tokens (--bg-primary, --color-air, --color-water, --shadow, --glow, etc.) with warm slate backgrounds and cyan/orange accents. No brief uses browser default styling. Fail: Any generated brief or hub.html uses default unstyled HTML instead of the Phoenix Design System.

- **SE-9 (F9):** Hub.html reflects all epics that have YAML artifacts on disk. Pass: Hub.html contains a section for every epic directory that has at least one YAML artifact. No section exists for epic directories with no YAML artifacts. Fail: An epic with artifacts is missing from hub.html, or an epic with no artifacts appears in hub.html.

- **SE-14 (C9):** Hub.html renders a product-level section followed by per-epic sections. Epics with no YAMLs on disk are omitted. Pass: Hub.html contains a product-level section with product.yaml and roadmap.yaml cards, plus a distinct section per epic that has YAML artifacts. Epic directories with no YAMLs have no section. Fail: Hub.html lacks product-level section, or lacks per-epic sections, or includes sections for epics with no YAML artifacts.

### Phase: Finalize

**Step 4 — Update Checksums**
Owner: recipe
Depends on: Step 3

1. For each YAML that was in scope (changed or not), write its current sha256 to the checksums map
2. Write `{product_base}/{slug}/briefs/.checksums.json` with the complete map
3. Verify: re-read the file and confirm all entries match current YAML state (SE-6)

**Step 4 Evals:**

- **SE-6 (F6):** Stored checksums match actual YAML checksums after completion. Pass: .checksums.json contains an entry for every YAML artifact in scope, and each stored checksum matches sha256 of the current YAML file. Fail: Stored checksums after completion do not match actual current checksums of the YAML artifacts on disk.

- **SE-12 (C6):** Checksums are persisted to a checksums file within the briefs/ directory after the recipe completes. Pass: .checksums.json exists at {artifact_base}/briefs/.checksums.json and contains an entry for every YAML artifact that was in scope. Fail: .checksums.json is missing, empty, or does not contain entries for all in-scope YAML artifacts.

This recipe is lightweight and idempotent — no pause/resume needed. Re-running `/briefs` re-checks all checksums and only regenerates what's stale. The checksums file IS the resume state.
