# implement-epic

Given a locked LLD phase and a defined epic, produce a working implementation through an eval-driven TDD loop. Context isolation is structural: the eval generator reads only specs, the builder reads only implementation context, the judge reads only evals. A quality agent independently verifies standards. If the judge reports failures, the builder fixes issues and is re-evaluated with freshly generated evals. The process produces committed code, a judge report, quality report, and manual test scenarios.

## Role

You are the orchestrator. You own the workflow. You delegate domain tasks to agents via JSON contracts — never execute domain work directly.

**Forbidden:** Direct code implementation. Direct test writing. Direct eval generation. Direct quality checks. All domain work goes through agents. The orchestrator's only direct operations are: reading status files, writing status files, writing checkpoints/evidence, updating CLAUDE.md pointers, and mediating information flow between agents (with filtering per C4/C5/C6/C10).

**Agent boundaries:**

| Agent | Role | Domain | Receives | Does NOT Receive | Phases | Budget |
|-------|------|--------|----------|-----------------|--------|--------|
| `tech-designer` | Context Builder | Distill LLD + tech approach into CONTEXT.md | LLD phase, tech approach, CLAUDE.md | Evals, scenarios, product spec | Preparation | Domain (1 call) |
| `eval-generator` | Eval Generator | Generate encrypted evals from specs | Product-spec behaviors, verification scenarios, epic success/failure, LLD exit gate | Implementation code, builder prompts, prior evals, tech approach, vision | Preparation | Domain (1 call per iteration) |
| `code-builder` | Builder | TDD implementation of scope items | CONTEXT.md only (contains distilled LLD + tech + conventions) | Evals, eval IDs, judge reports, pass criteria, scenarios, product spec | Execution | Domain (1 call per iteration) |
| `quality-auditor` | Quality Auditor | Lint, test, type check, build, vision gates | Implemented code, quality vision gates | Evals, builder prompts, judge reports, eval-generator output | Execution | Domain (1 call per iteration) |
| `judge` | Judge | Decrypt evals, run checks, report | Encrypted evals + decryption key, project location, credentials | Builder prompts, builder reasoning, eval-generator prompts, quality results | Execution | Domain (1 call per iteration) |
| `product-strategist` | Scenario Writer | Generate manual test scenarios | Epic success scenarios, LLD exit gate, deployed URL | Evals, builder prompts, judge reports | Finalize | Domain (1 call) |
| `repo-orchestrator` | Git | Commit, push, evidence self-commit | Committed files | — | Finalize, Evidence | Utility (exempt) |

**Context isolation invariant (C4/C5/C6):** The orchestrator is the ONLY entity that touches multiple agent outputs. When passing information between agents, the orchestrator MUST filter. Specifically:
- Judge failures → Builder: orchestrator extracts remediation instructions only (C10). Strips eval IDs, eval text, pass criteria, raw scores.
- No other cross-agent information flow is permitted.

**Path resolution:**
All `.meridian/` paths are relative to the project root. STM artifacts go to `.meridian/project/issues/{issue}/`. Eval files go OUTSIDE the repo tree (C8).

## Pre-flight

Execute these checks before any domain work. Orchestrator owns — do not delegate.

| Check | Constraint | Action on Failure |

| LLD exists and target phase has Status: LOCKED | C1 | Hard halt — cannot implement DRAFT phase (F8) |

| All prerequisite phases complete | C3 | Hard halt — dependencies not met |

| Resolve issue number from `--issue` arg or branch name | — | Hard halt if unresolvable |

```bash
# Resolve paths
stm_base=$(grep '^\s*base-path:' core/config.yaml | awk '{print $2}')
epic_id="{--epic value}"    # e.g., E1
phase_num="{--phase value}"  # e.g., 1

# Artifact paths

epics_path="{artifact_base}/epics.yaml"
spec_path="{artifact_base}/product-spec.md"
scenarios_path="{artifact_base}/scenarios.md"
tech_approach_path="{artifact_base}/technical-approach.md"

# C1: Read lld.md — verify Status: LOCKED
# C1: Read Phase {phase_num} section — verify it exists (F8)
# C2: Read epics.yaml — find epic {epic_id}, verify success_scenarios.length >= 1, failure_conditions.length >= 1 (F9)
# C3: If phase_num > 1, verify Phase {phase_num - 1} exit gate was satisfied
# C12: Run build command — verify exit code 0

# STM for this implementation

quality_gates_path="{stm_dir}/quality-vision-gates.yaml"
```

**Resume check:** If `{stm_dir}/status/implement-epic.json` exists, resume — skip completed steps, reset any `in_progress` to pending, continue from first incomplete. Restore `eval_key` from status file for judge decryption.

## Workflow

### Phase: Preparation

**Step 1 — Build Context**
Owner: `tech-designer` (as Context Builder)
Depends on: pre-flight

The context builder reads the LLD phase, technical approach, and CLAUDE.md, then distills them into a focused CONTEXT.md scoped to THIS epic only. The output is <100 lines containing only what a builder needs — no full specs, no cross-epic information.

```json
{
  "intent_path": "core/components/recipes/implement-epic/reference/intent.yaml",
  "stm_base": "{stm_base}",
  "stm": {
    "input": {
      "lld_path": "{lld_path}",
      "tech_approach_path": "{tech_approach_path}",
      "claude_md_path": "CLAUDE.md",
      "epic_id": "{epic_id}",
      "phase_num": "{phase_num}"
    },
    "output": {
      "context_path": "{context_path}"
    }
  },
  "task_id": "build-context",
  "config": {
    "max_lines": 100,
    "required_sections": ["Scope", "Tech Decisions", "Domain Rules", "Do NOT", "Files You Own", "Exit Gate"],
    "scoping_rule": "Include ONLY information relevant to this epic's phase. Backend-only epic excludes frontend details. Frontend-only epic excludes backend internals."
  }
}
```

**Expected output:** CONTEXT.md at `{context_path}` with all 6 required sections, <100 lines.

**Step 1 Evals:**
- **SE-1 (C13):** CONTEXT.md exists at `{context_path}`, has all 6 required sections, is under 100 lines. Does NOT contain content from other epic phases.

**Step 2 Eval:**

**Step 3 Eval:**
- **SE-3 (C15):** Quality vision gates file exists at `{quality_gates_path}` with at least `build`, `typecheck`, `lint`, and `unit_tests` entries.

**Step 4 — Generate Encrypted Evals**
Owner: `eval-generator` — **CONTEXT-ISOLATED**
Depends on: pre-flight

**Critical isolation (C4):** The eval generator receives ONLY the following files. It does NOT receive the LLD (contains implementation file paths), technical approach (contains technology choices), CONTEXT.md, CLAUDE.md, or any implementation code.

```json
{
  "intent_path": "core/components/recipes/implement-epic/reference/intent.yaml",
  "stm_base": "{stm_base}",
  "stm": {
    "input": {
      "product_spec_path": "{spec_path}",
      "scenarios_path": "{scenarios_path}",
      "epics_path": "{epics_path}",
      "epic_id": "{epic_id}",
      "phase_num": "{phase_num}",
      "lld_exit_gate": "{extracted exit gate text — NOT the full LLD}"
    },
    "output": {
      "encrypted_eval_path": "{eval_dir}/phase-{phase_num}.enc",
      "manifest_path": "{eval_dir}/manifest.json"
    }
  },
  "task_id": "generate-evals",
  "config": {
    "encryption_key": "{eval_key}",
    "storage_dir": "{eval_dir}",
    "instructions": [
      "Read ONLY the input files listed above",
      "Do NOT read any files in app/, lib/, components/, or any source code",
      "Do NOT read CLAUDE.md, CONTEXT.md, technical-approach.md, or lld.md",
      "Generate YAML evals covering all relevant behaviors, scenarios, and exit gate criteria",
      "Encrypt with AES-256-CBC + PBKDF2",
      "Delete plaintext after encryption",
      "Write manifest.json with eval count and metadata"
    ]
  }
}
```

**Expected output:** Encrypted eval file at `{eval_dir}/phase-{phase_num}.enc`, manifest at `{eval_dir}/manifest.json`, no plaintext on disk.

**Step 4 Evals:**
- **SE-4 (C7, C8, F4):** Encrypted file exists at output path. No `.yaml` plaintext files exist in `{eval_dir}/`. No eval files (encrypted or plain) exist inside the project repo tree. Manifest exists with eval count > 0.
- **SE-13 (F7):** Eval generator's prompt (as passed by orchestrator) contains zero implementation code paths, zero builder outputs, zero prior eval results. Verify by inspecting the contract's `stm.input` — only spec_path, scenarios_path, epics_path, epic_id, phase_num, and lld_exit_gate are present.

### Phase: Execution

**Step 6 — Build Epic (TDD)**
Owner: `code-builder` (as Builder) — **CONTEXT-ISOLATED**
Depends on: Step 5

**Critical isolation (C5):** The builder receives ONLY CONTEXT.md. It does NOT receive the product spec, verification scenarios, evals, eval IDs, judge reports, or pass criteria. CONTEXT.md contains the distilled scope, tech decisions, domain rules, contamination warnings, file list, and exit gate — everything the builder needs without spec pollution.

**TDD requirement (C16):** The builder implements each scope item as a vertical story: write test first (red), then implement (green), then verify, before moving to the next scope item. Scope items are NOT batched.

```json
{
  "intent_path": "core/components/recipes/implement-epic/reference/intent.yaml",
  "stm_base": "{stm_base}",
  "stm": {
    "input": {
      "context_path": "{context_path}"
    },
    "output": {
      "build_report": "{stm_dir}/evidence/implement-epic/build-report.yaml"
    }
  },
  "task_id": "build-epic",
  "config": {
    "approach": "tdd-vertical-story",
    "instructions": [
      "Read CONTEXT.md as your ONLY source of truth for what to build",
      "For each scope item in 'Files You Own':",
      "  1. Write a test that verifies the scope item's expected behavior",
      "  2. Run the test — verify it FAILS (red phase)",
      "  3. Write the implementation code",
      "  4. Run the test — verify it PASSES (green phase)",
      "  5. Move to the next scope item",
      "After all scope items: run full build to verify zero errors",
      "Do NOT read any files in .meridian/ other than CONTEXT.md",
      "Do NOT read scenarios.md, product-spec.md, or any eval files"
    ]
  }
}
```

**Expected output:** All scope items implemented with corresponding tests. Build passes.

**Step 6 Evals:**
- **SE-5 (C12, F1):** Project build (`npm run build`) exits with code 0. Type check (`npx tsc --noEmit`) exits with code 0.
- **SE-6 (C16, F12):** Every scope item from CONTEXT.md "Files You Own" has at least one corresponding test file. Test files exist and pass.
- **SE-11 (F5):** Builder's prompt (as passed by orchestrator) contains zero eval IDs, zero eval text, zero pass criteria, zero judge report content. Verify by inspecting the contract — only `context_path` is in `stm.input`.

Depends on: Step 6

**Isolation (C14):** The quality agent receives only implemented code and quality vision gates. It does NOT receive evals, builder prompts, judge reports, or eval-generator output.

```json
  "intent_path": "core/components/recipes/implement-epic/reference/intent.yaml",
  "stm_base": "{stm_base}",

      "Do NOT read files outside the project root or in /tmp/"
    ]

}
```

**Expected output:** Quality report with per-gate PASS/FAIL.

**Step 7 Evals:**
- **SE-7 (C12, C14, F1, F13, F14):** Build passes. Lint passes with zero violations. Type check passes. Unit tests pass (if present). All required quality gates from vision gates file report PASS.

**Step 8 — Judge Evals**
Owner: `judge` — **CONTEXT-ISOLATED**
Depends on: Step 6

**Critical isolation (C6):** The judge receives ONLY the encrypted evals + decryption key + project location + credentials for verification queries. It does NOT receive builder prompts, builder reasoning, implementation rationale, eval-generator prompts, or quality results.

```json
{
  "intent_path": "core/components/recipes/implement-epic/reference/intent.yaml",
  "stm_base": "{stm_base}",
  "stm": {
    "input": {
      "encrypted_eval_path": "{eval_dir}/phase-{phase_num}.enc",
      "manifest_path": "{eval_dir}/manifest.json",
      "project_root": "."
    },
    "output": {
      "judge_report": "{stm_dir}/evidence/implement-epic/judge-report.yaml"
    }
  },
  "task_id": "judge-evals",
  "config": {
    "decryption_key": "{eval_key}",
    "instructions": [
      "Decrypt the eval file using the provided key",
      "Run every eval — execute the verification method described in each",
      "Record PASS/FAIL with evidence per eval",
      "Delete the decrypted plaintext file after evaluation",
      "Write the judge report with per-eval results, category breakdown, and summary",
      "Do NOT read any builder prompts, CONTEXT.md, or implementation reasoning",
      "Do NOT read any files in .meridian/ other than what evals require"
    ],
    "credentials": "{project-specific credentials for verification queries — e.g., Supabase URL/keys}"
  }
}
```

**Expected output:** Judge report with per-eval PASS/FAIL, category breakdown, overall pass rate.

**Step 8 Evals:**
- **SE-8 (F2):** Judge report exists. First-run pass rate is > 50% (F2 threshold). If <= 50%, recipe halts with structured failure — epic scope may need restructuring.
- **SE-12 (F6):** Judge's prompt (as passed by orchestrator) contains zero builder prompts, zero builder reasoning, zero implementation rationale. Verify by inspecting the contract — only `encrypted_eval_path`, `manifest_path`, `project_root`, and `decryption_key` are present.

### Phase: Fix Loop (conditional — max 3 iterations per C11)

**Gate:** If judge report shows 100% pass → skip to Finalize phase. Otherwise enter fix loop.

**Step 9 — Derive Remediation**
Owner: orchestrator (direct — information filtering, not domain work)
Depends on: Step 8

Read the judge report. Extract ONLY:
- Category of failure (e.g., "auth", "project", "dashboard")
- Description of what is wrong (e.g., "Project does not appear on dashboard after creation")
- Expected behavior (e.g., "Project should appear immediately without navigation")

**Strip (C10):** eval IDs, eval text, pass criteria, raw scores, eval methodology.

Write remediation instructions to `{stm_dir}/evidence/implement-epic/remediation-{iteration}.md`.

---

**Step 10 — Builder Fix**
Owner: `code-builder` (as Builder) — **CONTEXT-ISOLATED**
Depends on: Step 9

**Isolation (C5, C10):** The builder receives CONTEXT.md + remediation instructions. Remediation contains ONLY what is wrong and what the expected behavior is — not how it was tested, not eval IDs, not pass criteria.

```json
{
  "intent_path": "core/components/recipes/implement-epic/reference/intent.yaml",
  "stm_base": "{stm_base}",
  "stm": {
    "input": {
      "context_path": "{context_path}",
      "remediation_path": "{stm_dir}/evidence/implement-epic/remediation-{iteration}.md"

    "output": {
      "fix_report": "{stm_dir}/evidence/implement-epic/fix-report-{iteration}.yaml"
    }
  },
  "task_id": "fix-epic-{iteration}",
  "config": {
    "approach": "tdd-vertical-story",
    "instructions": [
      "Read CONTEXT.md and remediation instructions",
      "Fix each issue described in remediation",
      "Write or update tests for fixed behavior",
      "Verify all tests pass after fixes",
      "Run full build to verify zero errors"
    ]
  }
}
```


Same contract as Step 4, but with new `eval_key` and fresh agent instance.

### Phase: Scenario Validation

**Step 18 — Scenario Evals**
Owner: orchestrator
Depends on: Step 17

Run these scenario evals against the final artifacts:

- **SCE-1 (S1 — Engineer):** All evals passed on first judge run. Build report shows all scope items implemented with tests. Quality gate passed. Manual test scenarios produced. Code committed.

- **SCE-2 (S2 — Engineer):** Fix loop executed at least once. Each iteration used freshly generated evals (verify different manifest timestamps). Final judge run shows 100% pass.

- **SCE-3 (S3 — QA Engineer):** Test scenarios document exists. Each scenario has numbered steps and expected outcomes. A QA engineer can execute them against the deployed environment without reading code.

- **SCE-4 (S4 — Security Auditor):** Builder contract's `stm.input` contains only `context_path` (and `remediation_path` during fix loop). Judge contract's `stm.input` contains only `encrypted_eval_path`, `manifest_path`, `project_root`. Eval generator's `stm.input` contains only spec/scenario/epic paths. No cross-contamination.

- **SCE-5 (S5 — Engineering Lead):** If F3 triggered: failure report exists with persistent failure list, remediation history per iteration, and recommendation. An engineering lead can decide next steps without re-running the process.

- **SCE-6 (S6 — Engineer):** Test scenarios document contains at least one scenario marked as requiring a deployed environment for integration validation.

- **SCE-7 (S7 — Performance Auditor):** Quality report includes performance-related gates (if defined in vision gates). Results are traceable to specific metrics.

- **SCE-8 (S8 — Evals Engineer):** Judge report contains per-eval PASS/FAIL with evidence. Every eval was executed (count matches manifest). A coverage summary maps evals to behavior categories.

Only scenarios matching the actual execution path are evaluated. S2/S5 are skipped if no fix loop ran. S7 is skipped if no performance gates are defined.

## Pause and Resume

Steps execute in compiled order — top to bottom. The fix loop (Steps 9-13) iterates within the execution phase.

**Issue detection:** From `--issue` argument or branch name during pre-flight.

**Status file:** `{stm_dir}/status/implement-epic.json`

```json
{
  "recipe": "implement-epic",
  "issue": "{issue_number}",
  "epic_id": "{epic_id}",
  "phase_num": "{phase_num}",
  "started_at": "{timestamp}",
  "eval_key": "{current eval encryption key}",
  "fix_iteration": 0,
  "tasks": {
    "build-context":       { "status": "completed", "completed_at": "..." },
    "update-claude-md":    { "status": "completed", "completed_at": "..." },
    "capture-quality-gates": { "status": "completed", "completed_at": "..." },
    "generate-evals":      { "status": "completed", "completed_at": "..." },
    "build-epic":          { "status": "in_progress", "started_at": "..." },
    "quality-gate":        { "status": "pending" },
    "judge-evals":         { "status": "pending" },
    "fix-loop":            { "status": "pending", "iterations_completed": 0 },
    "test-scenarios":      { "status": "pending" },
    "commit-push":         { "status": "pending" },
    "cleanup":             { "status": "pending" },
    "report":              { "status": "pending" }
  }
}
```

**Executor loop:**
```
parse --epic, --phase, --issue arguments
resolve issue number
check for status file at {stm_dir}/status/implement-epic.json

if file exists (resume):
  restore eval_key and fix_iteration from status
  for each step in compiled order:
    if status == "completed" → skip
    if status == "in_progress" → reset to pending
    mark step "in_progress"
    execute step
    mark step "completed"
  for fix loop:
    restore iterations_completed
    continue from current iteration

if file absent (fresh start):
  execute all steps
  create status file on Step 1
```

**Fresh start:** No status file → execute all steps, create on first step.
**Resume:** Status file exists → skip completed, continue from first incomplete.
**Fix loop resume:** `fix_iteration` field tracks current iteration count. On resume, restore count and continue.

## Recovery

When an agent returns `status: "failed"`:

1. Read the error from the returned contract.
2. For build failures (F1): check if the error is in the builder's scope or a pre-existing issue.
3. For eval generation failures: retry once with the same specs. After 2 failures, halt.
4. For judge failures: verify encrypted file is readable. If corrupted, regenerate evals and retry.
5. For quality gate failures: these feed into the fix loop — they are expected during iteration.

**Pre-flight failures (C1, C2, C3, C12) are not recoverable** — hard halt with clear error message.

**F2 (>50% failure rate on first run):** This suggests the epic scope is too ambitious or the builder received insufficient context. Halt and present to user — do not enter the fix loop. The orchestrator presents: "First-run eval pass rate is below 50%. The epic may need restructuring before re-attempting implementation."
