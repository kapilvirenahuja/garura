# plan-roadmap

Plan a time-phased product roadmap from a locked product definition. Scopes IDD epics, assesses technical feasibility, runs a context-isolated reverse-coverage + confidence check, presents epics.yaml + feasibility.yaml as the human review surface, then composes roadmap.yaml directly from the locked upstream YAML artifacts. Briefs are NOT generated by this recipe — users may run `/briefs` separately on demand.

## Compiled From

This recipe was compiled from `reference/intent.yaml` by `/create-recipe`.
Intent defines constraints (C1, C5-C8, C11-C17), failure conditions (F1-F12), and scenarios (S1-S10). All failure conditions and scenarios have eval coverage.
To modify this recipe, update `reference/intent.yaml` and re-run `/create-recipe --rebake plan-roadmap`.
Do NOT edit this file manually — it is a compiled artifact.

Owner: `judge`
Depends on: Step 2

**Context isolation (C15):** The judge receives ONLY the product.yaml path and epics.yaml path. It does NOT receive product-strategist drafting notes, LTM paths used, profile reasoning, or any intermediate outputs from Step 2.

```json

  "intent_path": "core/components/recipes/plan-roadmap/reference/intent.yaml",

}
```

The judge extracts every discrete element from product.yaml (strategic goals, success_metrics, target_users, assumptions, out_of_scope, scope.in_scope items, profiles), then checks each against epics.yaml. Status values: `covered`, `partial`, `dropped`, `drifted`, `orphaned_scope` (C16), `deferred_violation` (C17).

**On dropped strategic goal:** halt — scoping failure.
**On orphaned_scope (C16, F11):** halt — scoping agent must create/amend epics to claim ownership.
**On deferred_violation (C17, F12):** halt — epic must be corrected.

Update status file: `check-coverage → completed`.

- **SE-22 (C16, F11) — coverage pass:** Read `stm.coverage_check_path` — verify every in_scope item has status `covered`. PASS if no `orphaned_scope`. FAIL if any in_scope item lacks an owning epic. Halt on FAIL.

PASS if a Product Manager can clearly describe each epic without additional research. FAIL otherwise.

- **SCE-2 (S2 — Product Owner):** Read `stm.roadmap_yaml_path` and `stm.epics_path`. Verify every epic has at least 2 binary testable success_scenarios and 2-4 observable failure_conditions suitable for deriving backlog acceptance criteria. PASS if all epics satisfy. FAIL otherwise.

- **SCE-3 (S3 — Engineering Lead):** Read `stm.roadmap_yaml_path`. For every entry in `feasibility[]`, verify risk_level, technical_risks with affected_systems, blockers, sequencing_constraints, and architecture_impact are populated; sequencing_constraints reference concrete upstream epics where dependencies exist. PASS if engineering lead can sequence work from the feasibility section alone. FAIL otherwise.

- **SCE-4 (S4 — Stakeholder):** Verify the checkpoint review surface was `stm.epics_path` + `stm.feasibility_path` (NOT an HTML brief), and that feasibility.yaml surfaces critical_blockers and open_questions for stakeholder review. PASS if both conditions met. FAIL if checkpoint surface is not the YAML pair, or feasibility.yaml does not surface blockers.

- **SCE-5a (S5 — Technical Architect):** Read `stm.roadmap_yaml_path` `risk_summary` section. Verify: (a) `foundation_epics` is non-empty when any epic has `foundation_investment: true`, (b) `high_risk_count` matches count of epics with `risk_level: high`, (c) at least one feasibility entry has `architecture_impact` identifying shared infrastructure. PASS if architect can identify cross-cutting risks. FAIL otherwise.

- **SCE-5b (S7 — Technical Architect):** Read `stm.product_yaml_path` — extract profiles. If profiles present, check PP-6: PP-6 <= 2 → MVP-depth epics; PP-6 >= 4 → epics include production hardening. PASS if epic depth correlates with profile values or product.yaml has no profiles section. FAIL if depth contradicts profiles.

- **SCE-6 (S6 — Product Manager):** When feasibility.yaml has non-empty critical_blockers or open_questions, pre-lock-resolutions.yaml exists listing each item with its user resolution and a timestamp, and roadmap.yaml was created only after that artifact. PASS if interview lists each item individually and resolutions are recorded. FAIL otherwise.

- **SCE-7 (S8 — Product Manager):** Read `stm.epics_path` and the evidence report. For each epic, verify `ltm_citations` is a non-empty list of file paths AND the evidence report surfaces the citations per epic. PASS if both conditions met. FAIL otherwise.

- **SCE-8 (S9 — Stakeholder):** Read `stm.confidence_report_path`. Verify: (a) `assessed_by` is "judge", (b) every epic has `confidence` (high/medium/low), non-empty `rationale`, `scenario_metric_coverage`, `falsifiability`, and `coverage_gaps`, (c) summary counts are consistent with per-epic scores. PASS if all conditions met. FAIL otherwise.

- **SCE-9 (S10 — Product Manager):** Read `stm.coverage_check_path`. Verify: (a) every product.yaml `scope.in_scope` item has status `covered`, (b) no entry has status `deferred_violation`, (c) any cross-cutting capabilities have `cross_cutting_justification` documented in the corresponding epic. PASS if full traceability exists. FAIL otherwise.

Record PASS/FAIL for each SCE. If any SCE fails, report to user with details but continue to Evidence — scenario failures are observations, not halts.

## Pause and Resume

Steps execute in compiled order — run top to bottom.

**Issue detection:** Slug is derived from product.yaml during pre-flight.

**Status file:** `.meridian/product/status/plan-roadmap.json`

```json
{
  "recipe": "plan-roadmap",
  "slug": "{slug}",
  "started_at": "{timestamp}",
  "product_yaml_path": "{--product value}",
  "tasks": {
    "scope-epics":         { "status": "completed", "completed_at": "..." },
    "check-coverage":      { "status": "completed", "completed_at": "..." },
    "score-confidence":    { "status": "completed", "completed_at": "..." },
    "assess-feasibility":  { "status": "completed", "completed_at": "..." },
    "human-review":        { "status": "in_progress", "started_at": "..." },
    "pre-lock-resolution": { "status": "pending" },
    "produce-roadmap":     { "status": "pending" },
    "validate-roadmap":    { "status": "pending" },
    "report":              { "status": "pending" }
  },
  "contract_snapshot": "{last enriched contract JSON}"
}
```

**Executor loop:**
```
check for status file at .meridian/product/status/plan-roadmap.json

if file exists (resume):
  read contract_snapshot — reconstruct enriched JSON contract
  for each step in compiled order:
    if status == "completed" → skip
    if status == "in_progress" → reset to pending
    mark step "in_progress" in status file
    execute step
    mark step "completed" in status file

if file absent (fresh start):
  execute all steps
  create status file on Step 1
```

**`/plan-roadmap --resume`** — No `--product` argument needed.

1. Find the status file at `.meridian/product/status/plan-roadmap.json`.
2. Read `contract_snapshot` — reconstruct the enriched JSON contract.
3. Route based on task statuses:
   - `human-review` status `in_progress` or `pending` → re-present epics.yaml + feasibility.yaml checkpoint.
   - `human-review` status `completed` AND `pre-lock-resolution` status `pending` → jump to Step 5b.
   - `pre-lock-resolution` status `completed` AND `produce-roadmap` status `pending` → jump to Step 6.
4. Report: "Resuming plan-roadmap for `{slug}` — {description of what it is doing}."
5. If no status file found → halt: "No plan-roadmap checkpoint found. Start with `/plan-roadmap --product <path>`."

## Recovery

When an agent returns `step_failure` (non-null):

1. Read `step_failure.domain_assessment.responsible_domain` — identify which agent can fix it.
2. Invoke the responsible agent with the current contract + retry context appended.
3. Max 2 retries per step. After 2 failures, halt with full failure context presented to user.

**Pre-flight failures are not recoverable** — hard halt with clear error message.
