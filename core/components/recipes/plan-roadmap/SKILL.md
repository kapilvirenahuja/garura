# plan-roadmap

Plan a time-phased product roadmap from a locked product definition. Scopes IDD epics, assesses technical feasibility, generates a human-reviewable brief, runs a feedback loop, then produces a full roadmap artifact post-approval. Human review of the brief is always required — it is the gate before any roadmap artifacts are produced.

## Compiled From

This recipe was compiled from `reference/intent.yaml` by `/create-recipe`.
Intent defines constraints (C1-C15), failure conditions (F1-F10), and scenarios (S1-S9). All 10 FCs (F1-F10) and 9 scenarios (S1-S9) have eval coverage.
To modify this recipe, update `reference/intent.yaml` and re-run `/create-recipe --rebake plan-roadmap`.
Do NOT edit this file manually — it is a compiled artifact.

### Step 2b — Score Epic Confidence (Context-Isolated)

Owner: `judge`
Depends on: Step 2a

**Context isolation (C15, F10):** The judge receives ONLY the epics.yaml path and product.yaml path. It does NOT receive product-strategist drafting notes, market context, profile derivation reasoning, or any intermediate outputs from Step 2.

```json
{
  "intent_path": "core/components/recipes/plan-roadmap/reference/intent.yaml",
  "stm_base": ".meridian/project/product/",
  "mode": "score-epic-confidence",
  "artifact_paths": {
    "epics_yaml_path": "{stm.epics_path}",
    "product_yaml_path": "{stm.product_yaml_path}"
  },
  "stm": {
    "output": {
      "confidence_report_path": ".meridian/project/product/{slug}/confidence-report.yaml"
    }
  },
  "task_id": "score-confidence"
}
```

The judge reads product.yaml (`strategic_goals` with IDs and `success_metrics` with targets) and epics.yaml, then performs a **structural coverage assessment** per epic:

1. **Scenario-to-metric coverage:** For the strategic goal referenced by `strategic_goal_ref`, read all `success_metrics` entries with matching `strategic_goal_ref`. For each metric, check if any of the epic's `success_scenarios` would produce evidence toward that metric's target. Score: full (every metric has a corresponding scenario), partial (some metrics covered), weak (no scenarios trace to any metric).

2. **Falsifiability of failure conditions:** For each entry in `failure_conditions`, assess whether it is observable and binary-testable — can a reviewer point to concrete evidence and say "this happened" or "this did not happen"? Vague conditions like "the epic didn't deliver value" are unfalsifiable. Score: strong (all conditions are observable), moderate (some are vague), weak (most are untestable).

3. **Gap detection:** Check if the epic's `strategic_goal_ref` points to a strategic goal whose success_metrics have NO corresponding success_scenario in this epic. Each unmatched metric is a coverage gap.

4. **Confidence scoring:**
   - `high`: scenario-to-metric coverage is full AND falsifiability is strong AND zero gaps
   - `medium`: coverage is partial OR falsifiability is moderate OR 1-2 gaps exist
   - `low`: coverage is weak OR falsifiability is weak OR 3+ gaps exist

**Expected output:** `confidence-report.yaml` at the specified path:

```yaml
confidence_report:
  assessed_at: "{ISO-8601}"
  assessed_by: "judge"
  epics:
    - epic_ref: "E1"
      confidence: "high|medium|low"
      rationale: "{paragraph citing specific scenarios, metrics, and gaps}"
      scenario_metric_coverage: "full|partial|weak"
      falsifiability: "strong|moderate|weak"
      coverage_gaps:
        - metric: "{success_metric.metric that has no scenario}"
          strategic_goal_ref: "SG1"
    - epic_ref: "E2"
      confidence: "medium"
      rationale: "..."
      scenario_metric_coverage: "partial"
      falsifiability: "moderate"
      coverage_gaps: []
  summary:
    high_confidence_count: {integer}
    medium_confidence_count: {integer}
    low_confidence_count: {integer}
    total_coverage_gaps: {integer}
```

Update status file: `score-confidence → completed`.
Set `stm.confidence_report_path` in enriched contract.

- **SE-17 (C15, F10):** The confidence scoring contract contains ONLY epics_yaml_path and product_yaml_path — no product-strategist notes, market context, or drafting reasoning. ## Pause and Resume

Steps execute in compiled order — run top to bottom.

**Issue detection:** Slug is derived from product.yaml during pre-flight.

**Status file:** `.meridian/project/product/{slug}/status/plan-roadmap.json`

```json
{
  "recipe": "plan-roadmap",
  "slug": "{slug}",
  "started_at": "2026-03-06T10:00:00+0530",
  "product_yaml_path": "{--product value}",
  "tasks": {
    "scope-epics":        { "status": "completed", "completed_at": "..." },
    "check-coverage":     { "status": "completed", "completed_at": "..." },
    "score-confidence":   { "status": "completed", "completed_at": "..." },
    "assess-feasibility": { "status": "completed", "completed_at": "..." },
    "produce-brief":      { "status": "completed", "completed_at": "..." },
    "human-review":       { "status": "in_progress", "started_at": "..." },
    "pre-lock-resolution": { "status": "pending" },
    "produce-roadmap":    { "status": "pending" },
    "validate-roadmap":   { "status": "pending" },
    "report":             { "status": "pending" }
  },
  "contract_snapshot": "{last enriched contract JSON}"
}
```

**Executor loop:**
```
check for status file at .meridian/project/product/{slug}/status/plan-roadmap.json

if file exists (resume):
  read contract_snapshot — reconstruct enriched JSON contract
  for each step in compiled order:
    if status == "completed" → skip
    if status == "in_progress" → reset to pending (may not have finished)
    mark step "in_progress" in status file
    execute step
    mark step "completed" in status file

if file absent (fresh start):
  execute all steps
  create status file on Step 1
```

**`/plan-roadmap --resume`** — No `--product` argument needed.

1. Find the status file at `.meridian/project/product/*/status/plan-roadmap.json` (most recent by `started_at` if multiple exist).
2. Read `contract_snapshot` — reconstruct the enriched JSON contract.
3. Route based on task statuses:
   - `human-review` status `in_progress` or `pending` → re-present brief from `stm.roadmap_brief_path`, continue from Step 5 feedback loop.
   - `human-review` status `completed` AND `pre-lock-resolution` status `pending` → jump to Step 5b.
   - `pre-lock-resolution` status `completed` AND `produce-roadmap` status `pending` → jump to Step 6.
4. Report: "Resuming plan-roadmap for `{slug}` — {description of what it is doing}."
5. If no status file found → halt: "No plan-roadmap checkpoint found. Start with `/plan-roadmap --product <path>`."

## Recovery

When an agent returns `step_failure` (non-null):

1. Read `step_failure.domain_assessment.responsible_domain` — identify which agent can fix it.
2. Invoke the responsible agent with the current contract + retry context appended:
   ```json
   "retry": {
     "previous_failure": "{step_failure.error}",
     "fix_applied": "{what was adjusted}",
     "attempt": 2
   }
   ```
3. Max 2 retries per step. After 2 failures, halt with full failure context presented to user.

**Pre-flight failures are not recoverable** — hard halt with clear error message.
