# prepare-implementation

Produce a complete set of implementation-ready design artifacts for a single epic, starting from a locked product.yaml and locked roadmap.yaml. The recipe requires an `--epic` flag referencing an epic ID from the roadmap. Before generating artifacts, the recipe resolves two-dimensional dependencies (epic ordering from roadmap + technology dependencies from codebase), scans the codebase for existing implementations, consults LTM for organizational standards and patterns, and produces a dependency resolution report for user approval. The output set — scoped to the target epic only — includes: features.yaml defining product behaviors and invariants, architecture.yaml naming concrete technologies (reading prior-epic artifacts as context), scenarios.yaml with verification scenarios, tech.yaml with project structure and components, and plan.yaml with execution order. Existing artifacts from prior epics are read-only inputs, not targets for regeneration. The artifacts must contain everything the implementation agent needs — port numbers, service endpoints, dev patterns, quality gates, ADR references, LTM standards — so that no codebase scanning or LTM reading is required at implementation time. Verification scenarios are compartmentalized from the implementer.

## Workflow

### Phase: Context Resolution (Steps 0-5)

**Step 0 — Read Locked Upstream Artifacts**
Owner: recipe
Depends on: pre-flight

Read `{product_path}/product.yaml` (LOCKED) and `{product_path}/roadmap.yaml` (LOCKED). Locate the target epic. Extract: epic_id, name, description, strategic_goal_ref, depends_on, priority. Collect prior-epic IDs for read-only context.

---

**Step 1 — Resolve Epic Dependency Chain**
Owner: recipe
Depends on: Step 0

For each dependency in the target epic's `depends_on`:
- Check for prepared artifacts: `{product_path}/epics/{dep_id}/plan.yaml` exists and `status: "LOCKED"`
- Check for built implementations: codebase contains the dependency epic's implementation
- If neither: mark as missing (halt condition per C15)

If any dependency is missing, halt and present the full dependency chain with recommended execution order.

### Phase: Scenario Validation

**SCE-1 (Product Owner — S1):** Given the locked features.yaml and scenarios.yaml, the product owner can confirm that every product behavior in features.yaml has at least one testable verification scenario in scenarios.yaml with clear pass/fail criteria, achieving full traceability between features and acceptance via feature_ref and behavior_ref back-links.

**SCE-2 (Technical Architect — S2):** Given the locked features.yaml and architecture.yaml, the architect can evaluate whether the chosen technology stack and architectural decisions faithfully implement the product behaviors and invariants, and can identify any architectural gaps or misalignments.

**SCE-3 (Engineering Lead — S3):** Given a single execution_order entry in plan.yaml and architecture.yaml, the engineering lead can assign the feature slice to an implementer with confidence that the scope_items, key_files, technology stack, and exit_gate provide sufficient context to implement without requiring access to other feature slices, the full features.yaml, or scenarios.yaml content.

**SCE-4 (Quality Lead — S4):** Given scenarios.yaml feature_gates and plan.yaml scenario_gate entries, the quality lead can construct a feature-gated validation plan where each feature slice has a defined set of scenarios (by ID) that must pass before the next feature begins, without exposing scenario content to implementers. The plan references scenario IDs only; content lives in scenarios.yaml which is validator-facing.

**SCE-5 (Product Manager — S5):** Given the locked features.yaml, the product manager can describe the product's identity, core behaviors, invariants, and scope boundaries to stakeholders without referencing any technology, implementation detail, or architectural decision. The identity section provides a technology-free product narrative.

**SCE-6 (Technical Architect — S6):** Given the locked plan.yaml summary section, the architect can determine the implementation sequence, understand what each feature slice delivers as a working capability, and verify that the cumulative scenario count in the summary accounts for all verification scenarios by the final feature. The per_feature summary with cumulative_scenarios supports this review.

**SCE-7 (Engineering Lead — S7):** Given the dependency resolution report for an epic, the engineering lead can verify that all epic dependencies are resolved (prepared or built), all technology dependencies from the codebase are identified (services, ports, frameworks, quality gates), all relevant ADRs are referenced, and LTM findings and gaps are documented — sufficient to approve proceeding to artifact generation with confidence that the context is complete.

**SCE-8 (Implementation Agent — S8):** Given the locked epic-scoped artifacts (features, architecture, tech, scenarios, plan), the implementation agent can implement any feature slice using only the provided artifacts without scanning the codebase, reading LTM, or resolving dependencies. Port numbers, service endpoints, development patterns, quality gates, and file paths (new vs modified) are all explicit in the artifacts.
