# prepare-implementation

Produce five implementation-ready design artifacts for a product: a features specification (product identity, invariants, scope, feature IDD fields, behaviors, blast_radius), an architecture design (principles, NFRs, stack, platforms, integrations, agentic PCAM), a tech design (project structure, libs, data models, components, feature mapping), verification scenarios (feature back-links, feature_gates, coverage), and an execution plan (ordered feature vertical slices, scope items with file paths, exit gates, scenario gates). Follows DRAFT → VALIDATE → LOCKED lifecycle invoked as separate phases.

## Compiled From

This recipe was compiled from `reference/intent.yaml` by `/create-recipe`.
Intent defines constraints (C1-C13), failure conditions (F1-F12), and scenarios (S1-S6).
To modify this recipe, update intent.yaml and re-run `/create-recipe --rebake prepare-implementation`.
Do NOT edit this file manually — it is a compiled artifact.

**Forbidden:** Direct artifact authoring, direct validation, direct file creation for design documents. All domain work goes through agents.

### Phase: Scenario Validation

**SCE-1 (Product Owner — S1):** Given the locked features.yaml and scenarios.yaml, the product owner can confirm that every product behavior in features.yaml has at least one testable scenario with clear pass/fail criteria. Full traceability between features and acceptance via scenario back-links (feature_ref, behavior_ref).

**SCE-2 (Technical Architect — S2):** Given the locked features.yaml and architecture.yaml, the architect can evaluate whether the technology stack and architecture faithfully implement the product behaviors and invariants, and can identify gaps or misalignments. architecture.yaml principles and NFRs are checkable against features.yaml invariants.

**SCE-3 (Engineering Lead — S3):** Given a single execution_order entry in plan.yaml and tech.yaml, the engineering lead can assign the feature slice to an implementer with confidence that the scope items, file paths, technology context, and exit gate provide sufficient context — without access to other slices, the full features.yaml, or scenarios.yaml content.

**SCE-4 (Quality Lead — S4):** Given scenarios.yaml feature_gates and plan.yaml scenario_gate entries, the quality lead can construct a feature-gated validation plan where each feature has a defined set of scenarios that must pass before the next begins — without exposing scenario content to implementers. The plan references scenario IDs only; content lives in scenarios.yaml which is validator-facing.

**SCE-5 (Product Manager — S5):** Given the locked features.yaml, the product manager can describe the product's identity, behaviors, invariants, and scope boundaries to stakeholders without referencing any technology or implementation detail. The identity section provides a technology-free product narrative.

**SCE-6 (Technical Architect — S6):** Given the locked plan.yaml summary table, the architect can determine the implementation sequence, understand what each feature slice delivers, and verify the cumulative scenario count accounts for all scenarios by the final feature. The per_feature summary with cumulative_scenarios supports this review.

**Executor loop:**
```
parse --phase argument
check for status file at {artifact_base}/status/prepare-implementation.json

for each step in compiled order for this phase:
  if status file shows step "completed" → skip
  if status file shows step "in_progress" → reset to pending (may not have finished)
  mark step "in_progress" in status file
  execute step
  mark step "completed" in status file
```

**Fresh start:** No status file → execute all steps for the requested phase, create status file on first step.
**Resume:** Status file exists → skip completed, continue from first incomplete within the active phase.
**Pause:** User terminates session. Status file reflects last completed step. Next invocation resumes.
