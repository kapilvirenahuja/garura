# generate-flow-tests

Owned by the `test-engineer` agent. Invoked once per flow-spec in the /decode target set.

## Purpose

Flow tests verify the user journey end-to-end — from the trigger action through every participating feature to the success criteria. They are the closest thing to a real user click-through and the most honest signal that a migration preserved behavior: if the E2E test passes against both the old stack and the new stack, the journey is equivalent.

Per /decode C24 Tier B, /decode generates one flow test per flow-spec, covering:
- The success path (one test per flow).
- Every documented failure_paths entry (one test per failure path).

Tests target the detected E2E framework (Playwright, Cypress, Selenium, Puppeteer, or equivalent). They use the framework's native idioms.

## Input

Receive via JSON contract from test-engineer.

- `flow_spec_path` (path, required) — the flow's spec from `extract-flow-spec`.
- `test_harness_path` (path, required) — `{stm_base}/{issue}/evidence/decode/test-harness.yaml`.
- `codebase_root` (path, required).
- `output_dir` (path, required) — `{stm_base}/{issue}/evidence/decode/proposals/generated-tests/flow/`.
- `app_base_url` (string, optional) — base URL for the running application (default resolved from TEST_HARNESS env). Used for framework config.
- `ltm_context` (object, required).

## Process

### 1. Validate inputs

- Confirm `flow_spec_path` parses as YAML and has the C4b shape.
- Confirm `test_harness_path` has a harness with role `e2e` (playwright, cypress, selenium, etc.). If absent, return `status: failure, what_failed: e2e_harness_missing` — /decode halts the flow per F18.

### 2. Select E2E framework

From the test_harness.yaml, pick the harness with role `e2e`. If multiple, prefer one already configured in the codebase (has existing e2e tests in scan-index) over a library merely listed in manifests. Framework ↔ file extension mapping:
- Playwright → `.spec.ts` or `.spec.js`
- Cypress → `.cy.ts` or `.cy.js`
- Selenium + Java → `.java` (typically under `src/test/e2e/`)
- Puppeteer → `.test.js`

### 3. Generate success-path test

Produce one test file at:

```
{output_dir}/{flow-id}__success.spec.{ext}
```

Content — framework-specific syntax, consistent shape:
- Setup: start at the trigger location (URL or app state).
- For each step in the flow-spec (in order):
  - Perform the `actor: user` action using framework-native selectors (click, fill, navigate).
  - Assert the `system_effect` is observable on the page or via subsequent navigation.
- Final assertion: the success_criteria is observably true (DOM state, URL state, or subsequent-page content).

Every step's assertion derives from the flow-spec's `cited_locations.excerpt` where the source indicates an observable outcome (e.g., a returned JSON field, a DOM element rendered, a URL changed).

### 4. Generate failure-path tests

For each failure_paths entry in the flow-spec, produce:

```
{output_dir}/{flow-id}__failure-{ordinal}.spec.{ext}
```

Content:
- Setup matching the trigger.
- Walk through steps until the failure-path `trigger` conditions are met (e.g., input invalid field value at step 2).
- Assert the `outcome` is observably true (e.g., error message shown, user redirected, form blocked).

### 5. Annotate with cited_specs

Every generated test file header:

```
// Generated by /decode — DO NOT HAND-EDIT; rerun the play
// covers:
//   - flow_spec: {flow_spec_path}
//   - path: "success" | "failure-{ordinal}"
//   - participating_features: [MEM-F001, MEM-F002, CMS-F001]
//   - steps: [STEP-FLOW-first-post-001-1, STEP-FLOW-first-post-001-2, ...]
```

Sidecar at:

```
{output_dir}/{flow-id}__cited-specs.yaml
```

### 6. Wire selectors and test-data strategy

Flow tests rely on selectors. The skill emits selectors using the codebase's existing convention:
- If data-testid attributes are present in scan-index component files, use those (most robust).
- Else use semantic role selectors (ARIA roles, button text).
- Least preferred: CSS selectors tied to class names (fragile).

Test data strategy detected from existing e2e tests (fixture file, database seeding via API, stub service worker). The skill follows the existing convention and notes it in the file header. If no convention is discernible, the skill emits tests using API-seeded data via the contract-test level endpoints — tests self-seed, isolating from shared state.

### 7. Environment requirements

Emit at `{output_dir}/{flow-id}__env-requirements.yaml`:

```yaml
flow_id: "{flow-id}"
requires:
  app_running: true
  app_base_url: "{app_base_url}"
  database_seeded: <bool>
  services_running: ["stripe-mock", "smtp-catcher"]   # or empty
  env_vars: [...]
```

/decode's test-runner consumes this to ensure the environment is prepared before execution. Missing environment → test-run marked deferred, flow surfaced at Phase 1.

### 8. Return contract

```yaml
flow_id: "{flow_id}"
generated_files:
  - path: "{output_dir}/FLOW-first-post-001__success.spec.ts"
    path_kind: "success"
    covers: ["FLOW-first-post-001"]
  - path: "{output_dir}/FLOW-first-post-001__failure-1.spec.ts"
    path_kind: "failure"
    covers_failure_trigger: "email-domain-blocklisted"
  # ...
total_files: <int>
framework: "playwright"
env_requirements_path: "{output_dir}/FLOW-first-post-001__env-requirements.yaml"
status: "success"
```

## Output

Primary artifacts: test files at `{output_dir}/`.
Companions: `{flow-id}__cited-specs.yaml`, `{flow-id}__env-requirements.yaml`.

## Failure Modes

```yaml
status: failure
what_failed: "e2e_harness_missing | flow_spec_malformed | selector_strategy_unresolvable | trigger_not_reachable"
detail: "<specific>"
evidence: {...}
```

`trigger_not_reachable` means the skill could not construct a reliable selector or URL to start the flow (e.g., trigger is behind auth and no test-auth strategy is discoverable). /decode surfaces this at Phase 1 for the user to provide an auth-seed strategy.

## Notes

- Flow tests are mostly portable across E2E frameworks. Migration-time porting from Cypress to Playwright (or vice versa) is mostly syntax substitution — the test intent is preserved. That is the Tier B promise.
- One test per flow per path (success + one per failure) keeps test count bounded. Extensive edge-case permutations are Tier C pure-unit tests' job, not flow tests.
- Environment provisioning (starting the app, seeding data) is NOT this skill's job. /decode's test-runner reads the env-requirements.yaml and either provisions the environment (if configured) or marks the flow deferred.
- When the codebase already has E2E tests for the flow (detected via map-test-surface matching the trigger), the skill produces a complementary test and records "prior: existing e2e at {path}" in the header. Deduplication review happens at /garura:enrich time.
