# Spec: Deterministic Eval Generation in create-play

## Problem Statement

When `/create-play` compiles a play, step evals (SE-n) and scenario evals (SCE-n) are manually authored as prose during the SKILL.md compilation step (6b). This produces three classes of defects:

1. **Coverage gaps** — constraints and failure conditions with zero eval coverage (discover-product: C4, C5, C6, C8, C9 uncovered by any eval)
2. **Misaligned evals** — evals that claim a failure condition mapping but test something different (discover-product SE-3 claims F2 "domain unresolvable" but checks ">=2 target user personas" which is F1's territory)
3. **Threshold invention** — evals inject numeric thresholds (>=3 goals, >=2 personas) not present in the intent's constraint language
4. **Drift** — when intent.yaml changes, manually-authored evals stay stale until manual rebake + manual rewrite

## Desired End State

After implementation, every compiled play's evals are:
- **Generated** by evals-creator (not hand-authored during compilation)
- **Traceable** — every eval cites its source constraint/failure-condition/scenario ID
- **Complete** — every C-n, F-n, and S-n in intent.yaml has at least one covering eval or structural check
- **Faithful** — eval language mirrors the intent's own language, no invented thresholds
- **Verified** — a coverage matrix is produced and checked before compilation completes

## Architecture

### Constraint Classification

Not all constraints are eval-verifiable. The compiler must classify each constraint before routing:

| Category | Definition | Enforcement Mechanism | Example |
|----------|-----------|----------------------|---------|
| **Artifact-verifiable** | Observable in a skill's output artifact | Eval (SE-n) generated by evals-creator | C7: "Strategic Goals terminology" — check product.yaml |
| **Pre-flight** | Environmental precondition | Pre-flight check in compiled play | C1: "must not run on main" |
| **Structural** | Process/architectural rule about the play itself | Compiler verification (Step 6c) | C4: "delegate to agent", C5: "max 3 dispatches" |

Every constraint MUST be classified into exactly one category. The coverage matrix tracks all three.

### Component Changes

#### 1. evals-creator skill (modify)

**File:** `core/components/skills/evals-creator/SKILL.md`

Changes:
- **Add constraint extraction** to Process step 1: parse `constraints` in addition to `failure_conditions` and `scenarios`
- **Add constraint eval generation** as a new Process step (between current steps 2 and 3): for each artifact-verifiable constraint, generate a step eval that checks the constraint's `rule` against the relevant skill's output. The eval's `check` and `pass_criteria` fields MUST use the constraint's own language from the `rule` field — no invented thresholds or reformulations.
- **Extend output format** to include `constraint_id` field on step evals (alongside existing `failure_condition_id`). A single eval can reference both a constraint_id and a failure_condition_id if they overlap.
- **Add constraint_evals section** to output format for constraint-sourced evals, or merge into step_evals with a `source_type: constraint | failure_condition` discriminator.
- **Change model from `sonnet` to `opus`** in frontmatter — eval generation is load-bearing and needs maximum reasoning quality.
- **Add `constraints` input field** — list of constraint objects with classification. evals-creator only processes constraints classified as `artifact-verifiable`. It ignores `pre-flight` and `structural` constraints (those are the compiler's responsibility).

New input contract:
```yaml
Input:
  intent_path: "core/components/plays/{play-name}/reference/intent.yaml"
  skill_contracts:
    - skill_name: "{skill-1}"
      contract_path: "core/components/skills/{skill-1}/SKILL.md"
  constraint_classifications:
    - id: "C1"
      category: "pre-flight"           # evals-creator ignores
    - id: "C7"
      category: "artifact-verifiable"   # evals-creator generates eval
    - id: "C4"
      category: "structural"            # evals-creator ignores
  output_path: "{stm}/evals.yaml"
```

Extended output format:
```yaml
step_evals:
  - id: SE-1
    source_type: failure_condition   # or "constraint" or "both"
    failure_condition_id: F1         # present when source_type is failure_condition or both
    constraint_id: null              # present when source_type is constraint or both
    skill: "draft-product-vision"
    check: "..."
    artifact_pattern: "..."
    pass_criteria: "..."             # MUST use intent's own language
    fail_criteria: "..."

scenario_evals:
  - id: SCE-1
    scenario_id: S1
    persona: "..."
    artifact_pattern: "..."
    check: "..."
    pass_criteria: "..."
    fail_criteria: "..."
```

New coverage summary in output:
```yaml
evals:
  path: "{output_path}"
  step_eval_count: {integer}
  scenario_eval_count: {integer}
  coverage:
    failure_conditions_covered: [F1, F2, F3]
    failure_conditions_uncovered: []
    constraints_covered: [C7, C10]
    constraints_uncovered: []
    scenarios_covered: [S1, S2, S3]
    scenarios_uncovered: []
```

#### 2. create-play SKILL.md (modify)

**File:** `core/components/plays/create-play/SKILL.md`

Changes to Step 6:

**Step 6a — Classify Constraints (NEW sub-step)**

Before invoking evals-creator, the compiler must classify every constraint in intent.yaml:

1. Read all constraints from intent.yaml
2. For each constraint, determine category:
   - If the constraint describes an environmental precondition that can be checked before domain work starts → `pre-flight`
   - If the constraint describes an observable property of a skill's output artifact → `artifact-verifiable`
   - If the constraint describes a process/structural rule about the play's own behavior → `structural`
3. Produce `constraint_classifications` list for evals-creator input

Classification heuristics (guidance for the compiler, not rigid rules):
- Constraints mentioning "must not run when", "must be provided", "must exist" → likely `pre-flight`
- Constraints mentioning artifact content, terminology, format, specific fields → likely `artifact-verifiable`
- Constraints mentioning "delegated to agent", "maximum N dispatches", "never directly" → likely `structural`

**Step 6b — Generate Evals (replaces current hand-authoring)**

Invoke evals-creator with the extended contract including constraint_classifications:

```yaml
Input:
  intent_path: "core/components/plays/{play-name}/reference/intent.yaml"
  skill_contracts: [... from skill manifest ...]
  constraint_classifications: [... from Step 6a ...]
  output_path: "{stm_base}/{issue}/evidence/create-play/{play-name}/evals.yaml"
```

The compiler MUST NOT hand-author any evals. All evals come from evals-creator's output file.

**Step 6c — Compile SKILL.md (replaces current compilation)**

When writing the compiled SKILL.md:
1. Read the evals from `{stm}/evals.yaml` (written by evals-creator)
2. For each workflow step, find all step evals that reference the skill invoked in that step
3. Embed the evals immediately after the step they validate, using the format:
   ```
   **Step N Eval:**
   - **SE-X (F-n/C-n):** {check} — pass: {pass_criteria}, fail: {fail_criteria}
   ```
4. In the Scenario Validation section, embed all scenario evals:
   ```
   - **SCE-X (S-n — {persona}):** {check} — pass: {pass_criteria}
   ```
5. The compiler copies eval language verbatim from the evals.yaml — no reformulation, no added thresholds

**Step 6d — Verify Coverage (replaces current verification)**

Produce a coverage matrix and verify completeness:

```markdown
## Coverage Matrix

| Intent Item | Type | Category | Covered By | Location |
|-------------|------|----------|------------|----------|
| C1 | constraint | pre-flight | Pre-flight check | Pre-flight table |
| C4 | constraint | structural | Compiler structure | Agent boundary table |
| C7 | constraint | artifact-verifiable | SE-1 | Step 2 Eval |
| F1 | failure_condition | — | SE-1, SE-2 | Step 2 Eval, Validate Step 1 Eval |
| S1 | scenario | — | SCE-1 | Scenario Validation |
```

Verification rules:
- Every `artifact-verifiable` constraint has >= 1 SE-n
- Every failure condition has >= 1 SE-n
- Every scenario has >= 1 SCE-n
- Every `pre-flight` constraint has >= 1 pre-flight check
- Every `structural` constraint has a verifiable structural element (agent boundary table, compilation rule, etc.)

If ANY item has zero coverage → compilation fails. The compiler must either:
1. Re-invoke evals-creator with additional context about the uncovered item
2. Reclassify the constraint (maybe it was misclassified)
3. Halt and report the gap

Write coverage matrix to `{stm_base}/{issue}/evidence/create-play/{play-name}/coverage-matrix.md`.

#### 3. Review mode gap checks (update)

The existing G1-G3 checks in Review mode (Step R2) should be updated to use the same coverage matrix logic:

- **G1 (Constraint Coverage):** Now checks against three categories — pre-flight constraints appear in pre-flight table, artifact-verifiable constraints appear in step evals, structural constraints appear in play structure
- **G2 (FC Coverage):** Unchanged — every F-n must map to at least one SE-n
- **G3 (Scenario Coverage):** Unchanged — every S-n must map to at least one SCE-n

### Eval Language Fidelity Rule

This is the most important behavioral change. Currently, hand-authored evals invent thresholds:

**Before (hand-authored, broken):**
```
SE-1 (F1): product.yaml has >=3 strategic goals and >=1 target audience
```

F1 says: "fewer than 3 strategic goals, or no target audience is identifiable." The ">=3" threshold happens to match F1 in this case, but the eval text should derive from the intent, not invent its own framing.

**After (generated, faithful):**
```
SE-1 (F1): product.yaml strategic goals count is not fewer than 3, and at least one target audience is identifiable from artifact content
```

The rule: evals-creator MUST use the constraint/failure-condition's own `rule`/`condition` language as the basis for eval criteria. It may make the language more concrete (pointing to a specific artifact and field), but it MUST NOT introduce numeric thresholds, counts, or criteria not present in the source text.

Exception: If the intent itself specifies a threshold (F1 says "fewer than 3"), the eval may reference that threshold because it comes from the intent.

### Model Upgrade

The evals-creator skill frontmatter changes from `model: sonnet` to `model: opus`. Eval generation requires understanding the semantic relationship between constraints, failure conditions, skill output shapes, and artifact schemas. This is high-stakes reasoning work — the quality difference between sonnet and opus matters here.

## Validation Gate

Before this change is considered complete, run the following validation:

### Gate 1: Rebake discover-product

1. Run `/create-play --build discover-product` with the updated pipeline
2. Compare generated evals vs current hand-authored evals:
   - **Coverage:** All 11 constraints, 7 failure conditions, 3 scenarios must be covered (currently 5+ constraints are uncovered)
   - **Alignment:** SE-3 should correctly map to F2 (domain unresolvable), not test persona count
   - **Language fidelity:** No eval should contain a threshold not present in intent.yaml
3. Diff the compiled SKILL.md — new version should have more evals, better traceability, no invented thresholds

### Gate 2: Rebake commit-code (cross-validation)

1. Run `/create-play --build commit-code` with the updated pipeline
2. Compare generated evals vs the hand-authored evals in the current compiled play
3. The generated output should be structurally equivalent but with better traceability and no threshold invention

### Gate 3: Coverage matrix completeness

For both rebaked plays, the coverage matrix must show:
- Zero uncovered items across all three categories
- Every constraint classified into exactly one category
- No eval referencing a constraint/failure-condition ID that does not exist in intent.yaml

### Gate Execution

Gates 1 and 2 should produce diff evidence written to:
- `{stm_base}/{issue}/evidence/create-play/discover-product/validation-diff.md`
- `{stm_base}/{issue}/evidence/create-play/commit-code/validation-diff.md`

Each diff file should contain:
1. Old evals (hand-authored) — listed with their IDs and text
2. New evals (generated) — listed with their IDs, source mapping, and text
3. Coverage comparison: old coverage gaps vs new coverage matrix
4. Assessment: any regressions? Any generated evals that are worse than the hand-authored version?

## Risks and Mitigations

| Risk | Mitigation |
|------|-----------|
| evals-creator generates vague evals for complex constraints | Opus model + constraint's own language rule limits vagueness. Validation gate catches it. |
| Constraint classification is subjective — compiler may misclassify | Provide classification heuristics in create-play. Review mode G1 check will catch misclassifications over time. |
| Generated evals are longer/more verbose than hand-authored ones | Acceptable tradeoff. Coverage and accuracy matter more than brevity in eval definitions. |
| Existing plays break if rebaked with new pipeline | Not a regression risk — rebake is an explicit action. Existing compiled SKILL.md files are untouched until explicitly rebaked. |

## Sequencing

1. **Modify evals-creator skill** — add constraint support, extend output format, change model
2. **Modify create-play** — add constraint classification, restructure Step 6 sub-steps, add coverage matrix
3. **Run Gate 1** — rebake discover-product, produce diff evidence
4. **Run Gate 2** — rebake commit-code, produce diff evidence
5. **Run Gate 3** — verify coverage matrices for both

Steps 1 and 2 are sequential (2 depends on 1's output format). Gates 1 and 2 can run in parallel after both modifications are complete.
