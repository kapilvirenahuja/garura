# Discovery — #388 Update Intent doctrine: 5 elements → 3, move Success Scenarios to Expectations

## Source

- **Tracking issue:** [#388](https://github.com/kapilvirenahuja/garura/issues/388) — "Update Intent doctrine: 5 elements → 3, move Success Scenarios to Expectations"
- **Originating article:** `~/cto/StormCaller/Content/articles/standalone/an-intent-is-three-things-the-goal.md` (commit `b21b586`) — "The Anatomy of Intent from ICE (in IDSD). Built from Where Spec-Driven Breaks."

This is **doctrine work**, not a code feature. The framework's documentation, agent
prompts, schema validators, knowledge-base extensions, and the abstraction-layer linter
currently describe Intent as **five** elements. The just-shipped article re-anchors
Intent as **three** elements. The framework must match.

## The new contract (three elements only)

Intent is what only the human writes:

1. **Goal** — one outcome, one sentence, no "and." Passes the two-implementations test:
   two completely different implementations could both satisfy it.
2. **Constraints** — qualities the outcome must carry. Directional, unconditional, in
   business language. NFR-shaped (performance, scale, reliability, accessibility,
   security, compliance). Never implementation patterns.
3. **Failure conditions** — binary checks the validator runs as evals. Observable.
   Post-output. A script or eval decides true/false without a human.

What changes structurally:

- **Success Scenarios leave Intent and live in Expectations**, generated by the system
  from intent + context, vetted at a checkpoint. Non-negotiable: builders reward-hack
  any scenarios they can see. Compartmentation is the only structural defense.
- **Connections** stays *conceptually* part of Intent but **not** as a fourth inline
  slot. It lives on a different surface. Design is a separate follow-on issue and is
  out of scope for this work.

## The compartmented evaluation contract

- **Builder agent** receives: goal + constraints + Context bundle. Nothing else.
- **Validator agent** receives: failure conditions (compiled into encrypted evals) +
  Expectations.success_scenarios (vetted at the craft-ice checkpoint).
- The builder cannot teach to a test it cannot see.

## Decision rule for borderline items

*"Would knowing this change how the builder writes code?"*

- Yes → constraint. Builder needs it to make the design call.
- No → failure condition. Validator catches it after.

Worked examples from the article:
- "Must not introduce a new runtime dependency" → constraint (affects library picks).
- "Unit test coverage must stay above 90%" → failure condition (coverage is post-output;
  giving it to the builder invites coverage gaming).

## Why this matters (deeper read from the article)

When intent grows beyond three slots, the method slides back into being a spec. The
five-element version was already drifting toward spec territory. The three-element
version is the boundary that keeps IDSD distinct from SDD.

## Scope decisions (Q&A with user, 2026-05-26)

1. **Reach:** Doctrine only — philosophy docs, intent-crafter / intent-resolver agents,
   `author-intent-yaml` schema, `validate-intent-epics`, `generate-intent-epics`, KB
   references under `core/components/memory/`, and the abstraction-layer linter.
   `reference/intent.yaml` fixtures and any unmigrated plays are deferred to a
   follow-on.
2. **Connections:** Stay silent in this round — a one-line pointer that the surface
   where Connections lives is defined by a future issue. No doctrine text about
   Connections beyond that.
3. **Expectation vetting:** Mirror the existing per-play HITL config. When HITL is on,
   a human approves the generated Expectation. When HITL is off, the system
   auto-approves and records that the approval was automatic. Doctrine codifies the
   pattern; it does NOT force a global behavior change.
4. **Builder isolation enforcement:** Doctrine + soft check. Agent contracts must state
   the rule, and affected plays add step evals that scan the builder contract for
   scenario leakage. No new validator agent. The compiler is trusted to write contracts
   correctly; the SE evals catch drift.

## Actors

- **The doctrine reader** (engineer, author, agent designer) — reads philosophy docs,
  agent prompts, schema validators, and the KB to learn what Intent is and how it
  splits from Expectation.
- **The agent at runtime** (intent-crafter, intent-resolver, builder, validator) —
  reads the doctrine indirectly through agent prompts, schemas, and the
  compartmentation rules baked into play contracts.
- **The reviewer at the craft-ice / create-play checkpoint** — receives a generated
  Expectation and is asked to flip `vetted.status` to approved.

## Out of scope (explicit non-goals)

- Rebuilding any compiled play's `SKILL.md` (no `/create-play --rebuild` runs in this
  work).
- Migrating remaining unmigrated `reference/intent.yaml` fixtures.
- Designing the Connections surface (separate issue).
- Adding a runtime validator agent that scans builder contracts before dispatch.
