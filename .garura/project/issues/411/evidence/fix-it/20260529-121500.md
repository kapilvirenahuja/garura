# Fix-It Delivery — #411

## Issue
**#411** — /implement halts on phase-internal .GATE depends_on — task-chain vs milestone-list mismatch

## Root Cause
`/implement` was milestone-first. Its pre-flight resolved every `depends_on` entry as a prior milestone ID, had no DAG walker, no self-location from branch state, and no two-boundary (gate vs milestone) halt model. When `/prepare` emitted a prerequisites phase (T-000, no milestone_id) and the first feature milestone depended on `T-000.GATE`, the pre-flight looked up a milestone named `T-000.GATE`, found none, and halted (F24) — leaving the first milestone permanently blocked. The reported gate-lookup halt was one symptom of the milestone-first model.

## Fix Strategy
Redesigned the `/implement` play to an agentic, self-locating, continuous DAG executor: it reads the plan DAG plus the current branch state, self-locates the next runnable node, and runs nodes continuously — halting ONLY at a gate (readiness stop, auto-resumes when the exit condition clears) or a milestone (human-acceptance stop via validate ACCEPTED, then resumes). `depends_on` edges resolve by node type (milestone=ACCEPTED, gate=exit_gate met, task=done in branch state). No milestone_id is required at invocation. Delivered as an INTENT change (intent.yaml) recompiled through the play compiler — SKILL.md never hand-edited for the intent portion.
