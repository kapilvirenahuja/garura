RCA + Design Checkpoint — Issue #261

.garura/ vs .meridian/ Migration Half-Done

Four load-bearing defects block Garura bootstrapping in consumer projects. Source components, installer, and sync scripts were never updated during the engine+STM migration to .garura/.

Fix Confidence
The codebase started migrating from .meridian/ to .garura/ (engine reads .garura/core/config.yaml; STM root is .garura/project/issues/; deployed memory is ~/.garura/core/memory/), but source components (66 files), the installer script, sync-claude, and two agents' hardcoded KB paths were never updated. When a consumer project runs the installer it writes .meridian/core/config.yaml, and every play halts reading config. This is a single coherent defect — incomplete migration — with one forward fix direction locked by the runtime.
Root Cause Summary Defect · Medium
  • RC1 Installer writes .meridian/core/config.yaml; all plays expect .garura/core/config.yaml
  • RC2 66 source files hardcode .meridian/core/config.yaml in grep/yq readiness commands
  • RC3 Two agents hardcode KB at core/components/memory/knowledge/ — source-repo path, fails in consumer projects

Fix Strategy Forward Only
Complete the .meridian/ → .garura/ migration. Engine runtime, STM root, deployed memory, repo name, and project identity are already at .garura/. Reverting is not viable without reversing engine changes. The fix is a bulk find-replace across 75 files with targeted surgical changes to the installer, sync scripts, and two agents' KB path documentation. All changes are load-bearing (not decorative); post-fix decorative renames are flagged for /enhance.
Chapter 1

Root Causes

Five sub-causes comprising a single incomplete migration defect.

RC1 Installer writes .meridian/ — consumer bootstrapping fails
Symptom
Installer writes .meridian/core/config.yaml; all plays expect .garura/core/config.yaml
Cause
installer/install.sh was never updated during the engine+STM migration. It still references .meridian/core/config.yaml (lines 56, 98, 209, 217, 280, 287) and downloads from the kapilvirenahuja/meridian repository (line 9). When a consumer runs the installer, it creates a .meridian/ structure that no play can find.
Files Affected
installer/install.sh
Why Wrong
core/engine/src/lib/constants.ts line 28 reads ".garura/core/config.yaml at runtime" per its docblock. STM root (.garura/project/issues/) is hard-coded in the engine. Installer is the entry point for consumer projects — it must write .garura/, not .meridian/.
RC2 66 source files hardcode .meridian/ in bash readiness commands
Symptom
66 source files read .meridian/core/config.yaml; breaks in consumer projects
Cause
Every agent markdown file (13), every play SKILL.md (22), and many skill files (15+) contain hardcoded grep/yq commands like:
grep -A5 '^stm:' .meridian/core/config.yaml | grep 'base-path' These execute during play initialization. When a consumer project has only .garura/, the grep commands fail with exit code 1 and every play halts.
Examples
  • core/components/plays/fix-it/SKILL.md:51 — grep on .meridian/core/config.yaml
  • core/components/plays/review-pr/SKILL.md:62-67 — five separate yq/grep calls
  • core/components/agents/intent-resolver.md:51 — documents config_path as .meridian/
Why Wrong
These are not decorative prose — they are hardcoded file paths in shell execution context. When the file doesn't exist, the entire play fails before any domain work begins.
RC3 Two agents hardcode source-repo KB path — unavailable in consumer projects
Symptom
Agents reference core/components/memory/knowledge/ — a path only available inside the Garura source repo
Cause
product-keeper.md (line 67) and market-analyst.md (lines 61-62) document their KB discovery path as core/components/memory/knowledge/domain/ and core/components/memory/knowledge/arch/. These paths do not exist in consumer projects where core/components/ is never present.
Files Affected
  • core/components/agents/product-keeper.md:67
  • core/components/agents/market-analyst.md:61-62
Fix Direction
Replace with deployed path guidance: ~/.garura/core/memory/knowledge/ (global) or .garura/core/memory/knowledge/ (project mode), resolved via ltm_context.core_base at invocation time.
RC4 Sync scripts deploy memory to ~/.meridian/ — plays find nothing at ~/.garura/
Symptom
sync-claude and sync-droids deploy memory to ~/.meridian/core/memory/, not ~/.garura/core/memory/
Cause
sync-claude/scripts/sync.sh (lines 24, 27) and sync-droids/scripts/sync.sh (lines 22, 25) set MEMORY_TARGET to .meridian/core/memory (project mode) or ~/.meridian/core/memory (global mode). Plays that read from ~/.garura/core/memory/ find nothing after sync.
Files Affected
  • core/components/skills/sync-claude/scripts/sync.sh:24,27
  • core/components/skills/sync-droids/scripts/sync.sh:22,25
RC5 Garura's own config still named "Meridian" and targets .meridian/ paths
Symptom
Project config (.meridian/core/config.yaml) still defines project.name as "Meridian" and all paths as .meridian/
Cause
The checked-in config file defines project.name: Meridian, github.repo: kapilvirenahuja/meridian.git, and all ltm/stm/product paths as .meridian/. When garura's own plays run, they read from .meridian/, perpetuating the mixed state and making it impossible to test the framework against its own config.
File Affected
.meridian/core/config.yaml (~60 lines, full rewrite needed)
Fields to Update
  • project.name: "Meridian" → "Garura"
  • github.repo: meridian.git → garura.git
  • ltm.project-target, ltm.global-target: .meridian/ → .garura/
  • stm.base-path, stm.pending-path, stm.archive-path: .meridian/ → .garura/
  • product.base-path: .meridian/ → .garura/

Chapter 4

Alternatives Considered

Two alternatives were evaluated and rejected. Forward migration is the only viable path.

Rejected Alternative 1 — Revert all .garura/ references back to .meridian/
Rationale
Roll back the engine+STM migration: update constants.ts, all STM paths, and deployed memory to .meridian/. This would make source components consistent with each other again.
Why Rejected
Reverting requires changing the engine's constants.ts and all STM paths — out of scope for fix-it. It contradicts the repo naming (garura, not meridian) and all infrastructure decisions already made. The runtime already depends on .garura/; reverting is a larger scope change than completing the migration.
Rejected Alternative 2 — Update only installer and sync scripts; leave 66 source files as .meridian/
Rationale
Minimal change set: installer writes .garura/, sync-claude/sync-droids deploy to ~/.garura/. Consumer projects would have .garura/ but source components would still reference .meridian/, relying on a symlink or alias from .meridian/ → .garura/.
Why Rejected
Requires either symlink hacks (fragile, breaks on Windows) or dual-path handling (complexity). More critically, the garura repo itself uses these scripts — keeping .meridian/ in source would mean garura's own workflows read from .meridian/ while consumer projects expect .garura/. This perpetuates the inconsistency and makes testing the framework against its own config impossible.
Chapter 5

Risks

5 risks identified. 1 high, 2 medium, 2 low. All have mitigations.

High
Breaking change to garura's own config location
After this fix, garura's own plays will read from .garura/core/config.yaml, not .meridian/core/config.yaml. Any local development or CI that expects .meridian/ will break immediately. This is a one-way door within the repo.
This is intentional — the fix corrects the inconsistency. The change is documented in the PR and this design. Testing must be done against .garura/ paths from this point forward. The break is expected and bounded to garura's own development workflows.
Medium
Existing local project deployments change memory target
Project-mode deployments (sync-claude --project, sync-droids --project) will change their memory target from .meridian/core/memory/ to .garura/core/memory/. Existing local projects using .meridian/core/memory/ will not find their deployed memory after this fix.
Project mode is ephemeral (gitignored). Re-running /sync-claude --project after the fix will automatically copy memory to the correct location. The sync is idempotent; no data migration needed.
Medium
Bulk find-replace may miss differently-formatted references
Bulk replace across 75 files carries a risk of missing references wrapped across lines, in comments, or formatted differently than expected (e.g., single quotes, env-var interpolation).
Step 10 (mandatory validation grep) catches any missed references before commit. The grep pattern covers the three distinct .meridian path forms used in the codebase.
Low
Decorative .meridian/ references remain in prose and templates
Reference docs, template fixtures, and example paths in comments may still reference .meridian/. These do not break functionality but create documentation inconsistency.
Explicitly flagged for /enhance as a post-fix decorative rename pass. Not blocking. All load-bearing references are covered by steps 1–9.
Low
Config file location remains at .meridian/core/config.yaml
The checked-in config file is still physically at .meridian/core/config.yaml after this fix (its contents are corrected, but its location is not moved). This is semantically odd — a .meridian/ file pointing to .garura/ structure.
Out of scope for fix-it (would require git mv and repo history impact). Consider a future /enhance task to move .meridian/core/config.yaml → .garura/core/config.yaml. Functionally harmless during the transition — contents now match .garura/ paths.
