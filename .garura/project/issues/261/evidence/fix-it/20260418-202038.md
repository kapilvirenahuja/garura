# fix-it Delivery — Issue #261

**Title:** .garura/ vs .meridian/ migration half-done — skills, agents, installer disagree on config path; KB path hardcoded workspace-relative in agents

## Root Cause

Forward migration from `.meridian/` to `.garura/` was partial. Engine/STM/memory targets had moved to `.garura/`, but 68 source files still hardcoded `.meridian/core/config.yaml` reads, the installer wrote `.meridian/`, and two agents hardcoded KB paths to source-only `core/components/memory/` locations. Result: consumer projects could not bootstrap; every play halted at its readiness check.

## Fix Strategy

Complete the forward migration in one coherent PR:

1. Installer writes `.garura/` and downloads from `kapilvirenahuja/garura`.
2. Sync scripts deploy memory to `~/.garura/core/memory/`.
3. Rename `.meridian/core/config.yaml` → `.garura/core/config.yaml` (via `git mv`) and update its internal paths + project name.
4. Agents (product-keeper, market-analyst) read KB via `ltm_context.core_base` — no hardcoded paths.
5. Bulk rename `.meridian/core/config.yaml` → `.garura/core/config.yaml` across 13 agents, 22 plays, 15+ skills, and memory reference docs.

Blast radius: 68 files, 8 commits grouped by concern.

## Notes

Blast radius (68 files) was 5× the fix-it threshold (~10). Because the 5 root causes were all facets of one coherent forward migration and splitting would have left the consumer-project bootstrap broken between PRs, the checkpoint explicitly surfaced the scope and the user approved (Tether) the single-PR execution.

The initial design proposed leaving `.meridian/core/config.yaml` in place as "functionally harmless during transition." That assumption proved wrong in practice — once 68 files were updated to read `.garura/core/config.yaml`, subsequent play invocations failed with config-not-found. The fix was added mid-execution via `git mv`. Captured as a risk-assessment calibration signal for future transitions but not a reusable domain learning per distill.
