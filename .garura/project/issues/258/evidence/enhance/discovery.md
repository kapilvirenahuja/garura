# Discovery — Issue #258

## Issue Body (current)

Split `/enhance` so that implementation and testing run as two separate agents that don't know about each other. Same pattern `/implement` uses, just smaller and faster.

Today `code-builder` writes the code and self-eval grades it at the end. One agent, one context, end-of-run batch verdict. Problems: the builder can teach to the test, and when things fail you don't know which task broke.

## Expectations (from issue)

**Implementer agent**
- Gets: the task spec, relevant code context.
- Does NOT get: test files, eval criteria, pass/fail conditions.
- Reports: "implementation attempted" + what it changed. No self-verdict.

**Tester agent**
- Gets: the task spec.
- Does NOT get: the implementer's output, narration, or rationale.
- Runs tests against the code on disk.
- Reports: pass or fail, with evidence.

**Play (coordinator)**
- Holds both contexts. Neither agent sees the other's.
- Uses the tester's verdict as the gate. Never the implementer's self-report.

## Q&A with User

**Q: What validator runs the tests — quality-auditor or play-direct execution?**
A: Whatever `/implement` already does. Mirror that pattern at smaller scope. Don't invent new infrastructure.

**Q: How detailed should the redesign be — per-task chains, TDD red-first, prompt capture, re-planning checkpoints?**
A: None of that. Explicit out-of-scope. Simple and crisp: two agents, no shared context, play gates on tester verdict.

**Q: What existing code does this connect to?**
A: `core/components/plays/enhance/reference/intent.yaml` (add constraints reflecting the split), `core/components/plays/enhance/SKILL.md` (rebuild via `/create-play --rebuild enhance`), likely `core/components/plays/implement/` as the pattern source.

**Q: Success criteria?**
A: A fresh `/enhance` run dispatches implementer and tester as two separate agents. Neither contract leaks the other's artifacts. Tester verdict gates advancement. Builder self-report is never the basis for pass/fail.

## Constraints

- Mirror `/implement`'s implementer/tester separation — do not design a new pattern.
- Keep it small. Don't re-architect `/enhance`.
- Out-of-scope items from the issue body are explicitly rejected — no per-task chains, no prompt capture, no TDD red-first authoring, no immutable task specs, no re-planning checkpoints.

## Scope confirmation

Expected files touched: intent.yaml (modify), SKILL.md (regenerated by `/create-play --rebuild`), possibly a new or updated eval spec. Likely 3–6 files, 1 domain (enhance play), effort under half a day. Right-sized for `/enhance`.
