## Solution Summary

Restructure the intent-epic schema to a single-file, dual-block layout: an intent triple at root level (intents[], constraints, failure_conditions) plus a nested expectation block (success_scenarios, recovery, vetted) generated by a new `draft-epic-expectation` skill and dispatched by a new `epic-expectation-crafter` agent. `generate-intent-epics` writes the intent triple plus a stub expectation block; the crafter agent fills the stub; a single Tether covers both. `validate-intent-epics` is updated to read the new shape. All 9 existing epics on disk are rewritten to the new shape in the same PR. The singular `intent` field is removed entirely and replaced by `intents[]`. Every downstream consumer that reads `failure_scenarios` or the singular `intent` field is updated in the same change. Quality bars (performance/security/accessibility/compliance) remain in `constraints` — renaming only, no new derivation mechanism.

## Connections

**Flow:** `/specify` Stage 5 → `product-keeper` dispatches `generate-intent-epics` → skill writes intent triple (intents[], constraints, failure_conditions) plus empty expectation stub to each epic file → `product-keeper` dispatches `epic-expectation-crafter` agent → agent calls `draft-epic-expectation` skill → skill reads intent block from the file, generates expectation block, writes back to same file (atomic read-merge-write), sets `vetted.status: pending` → agent surfaces single Tether (intent + expectation together) → human approves → agent sets `vetted.status: approved` → `product-keeper` dispatches `validate-intent-epics` → validator reads both intent block and expectation block, enforces new shape → `/specify` continues.

**Decision-manifest flow:** `generate-intent-epics` emits `decision-manifest-generate-intent-epics.yaml` (D-gie-001 to D-gie-005; D-gie-006 removed, D-gie-004 moves to expectation skill). `draft-epic-expectation` emits `decision-manifest-draft-epic-expectation.yaml` (new `D-dee-` prefix). `/specify` Step 9b surfaces both manifests at the decision-surfacing checkpoint.

**Validator coupling:** `validate-intent-epics` loads `intent-epic.yaml` schema as the mandatory-field source of truth. Because the schema's validation-rules comment block is updated in T1, the validator reads the new field list automatically when schema path is passed. The validator's per-epic checks also update explicitly (in T6) to iterate `intents[]` and read `expectation.success_scenarios`.

**Single Tether:** `epic-expectation-crafter` presents intent block + expectation block together at one checkpoint. No separate vetting of intent vs expectation. Single human response gates both to `approved`.
