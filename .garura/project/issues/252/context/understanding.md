# Context: Understanding — Issue #252 Component Linting

## Issue Summary

Issue #252 requests a deterministic Node.js linter for Garura components (plays, agents, skills under
`core/components/`). Three lint tiers: structural (per-file), semantic (cross-file), and cross-reference
integrity. Exposed as a `lint-components` skill. Severity mapping (error/warn/block) is agentic — the
skill reads the project quality profile to derive what's a blocker vs. warning.

---

## 1. Garura Component Structure

### Directory Layout — `core/components/` (2 levels)

```
core/components/
├── agents/                  19 files (one .md per agent)
├── memory/
│   ├── knowledge/           Domain knowledge files
│   └── standards/           Coding standards, templates, rules
├── plays/                   22 subdirectories (one per play)
│   └── {play-name}/
│       ├── SKILL.md         Compiled play (DO NOT edit manually)
│       └── reference/
│           ├── intent.yaml  Source of truth for the play
│           └── evals/       Eval YAML files
└── skills/                  92+ subdirectories (one per skill)
    └── {skill-name}/
        ├── SKILL.md         The skill definition
        └── [reference/]     Optional reference files, templates
```

### Component File Samples

#### Skill — `core/components/skills/manage-issue/SKILL.md`

**Frontmatter:**
```yaml
---
name: manage-issue
description: Read, create, close, comment, or resolve GitHub issues with optional sub-issue attachment
user-invocable: false
model: sonnet
allowed-tools: Bash, Read
---
```

**Required sections observed:**
- `## Purpose` — what the skill does
- `## Input` — inputs received from agent
- `## Process` — step-by-step instructions
- `## Output` — what to produce
- `## Constraints` — NEVER/ALWAYS rules
- `## Version` — version table (Version, Category)

Not all skills have all sections; some substitute `## Boundaries` for `## Constraints`, and some omit
`## Reference` or `## Version`. See "Key Patterns" below for the full picture.

#### Agent — `core/components/agents/tech-designer.md`

**Frontmatter:**
```yaml
---
name: tech-designer
domain: design
role: designer
description: Technical analysis, RCA, and solution design for features and bugs
model: sonnet
tools:
  - Bash
  - Read
  - Grep
  - Glob
  - Write
  - Skill
  - WebSearch
  - WebFetch
---
```

All agent frontmatter uses `tools:` as a YAML list (not inline `allowed-tools:`). Agents have:
`name`, `domain`, `role`, `description`, `model`, `tools`.

#### Play — `core/components/plays/enhance/SKILL.md` (first 60 lines)

**Frontmatter:**
```yaml
---
name: enhance
description: "Autonomous enhancement workflow..."
user-invocable: true
model: opus
---
```

Note: plays use `user-invocable` (not `user-invokable`) in 9/23 plays; 14 plays use `user-invokable`.
Both spellings appear in the codebase — this is itself a structural inconsistency the linter should flag.

The SKILL.md body for a play always begins with:
```
## Compiled From
This play was compiled from `reference/intent.yaml` by `/create-play`.
Do NOT edit this file manually — it is a compiled artifact.
```

Required sections in plays: `## Role`, `## Arguments` (for user-invocable plays), `## Pre-flight`,
`## Workflow` (phases and steps), `## Compilation Metadata`.

#### Intent.yaml — `core/components/plays/enhance/reference/intent.yaml`

Top-level keys:
```yaml
intent: |        # Plain text — what the play does
constraints:     # Array of {id: "C{N}", rule: "..."}
failure_conditions: # Array of {id: "F{N}", condition: "..."}
scenarios:       # Array of {id: "S{N}", persona, given, then}
```

YAML-only (no frontmatter delimiter). IDs are sequential within each section (C1, C2, ...; F1, F2, ...; S1, S2, ...).

## 2. Frontmatter Format Details

### Delimiter

All skill and play SKILL.md files use triple-dash YAML frontmatter:
```
---
key: value
---
```

Agent files use the same triple-dash format.

### Required Fields Per Component Type

| Component | Field | Notes |
|-----------|-------|-------|
| Skill | `name` | String, matches directory name |
| Skill | `description` | String |
| Skill | `user-invocable` | Boolean — ALL 63 skills use this spelling |
| Skill | `model` | `sonnet` or `opus` |
| Skill | `allowed-tools` | Comma-separated string (e.g., `Bash, Read`) |
| Play | `name` | String, matches directory name |
| Play | `description` | String |
| Play | `user-invocable` / `user-invokable` | Mixed — 9 use `user-invocable`, 14 use `user-invokable` |
| Play | `model` | `sonnet` or `opus` |
| Agent | `name` | String, matches filename (without .md) |
| Agent | `domain` | String |
| Agent | `role` | String |
| Agent | `description` | String |
| Agent | `model` | `sonnet` or `opus` |
| Agent | `tools` | YAML list (not comma-string) |

Note: Skills use `allowed-tools:` (comma string); agents use `tools:` (YAML list). This is intentional
and component-type-specific — the linter must enforce the right format per type.

## 6. Key Patterns for the Linter to Check

### 6a. Frontmatter Format

- **Delimiter:** Triple-dash (`---`) YAML block at start of file.
- **Skills:** Required fields = `name`, `description`, `user-invocable`, `model`, `allowed-tools`.
  `allowed-tools` is a comma-separated string.
- **Plays:** Required fields = `name`, `description`, `user-invocable` OR `user-invokable`, `model`.
  The linter should normalize and accept both spellings but flag them as a warning for consistency.
- **Agents:** Required fields = `name`, `domain`, `role`, `description`, `model`, `tools`.
  `tools` is a YAML list (not comma-string).

### 6b. Required Sections Per File Type

**Skills — minimum required sections:**
- `## Input`
- `## Process`
- `## Output`

**Plays — minimum required sections:**
- `## Compiled From`
- `## Role`
- `## Pre-flight`

**Intent.yaml — required keys:**
- `intent`
- `constraints` (array with `id` + `rule` per item)
- `failure_conditions` (array with `id` + `condition` per item)
- `scenarios` (array with `id` + `persona` + `given` + `then` per item)

### 6c. Constraint and Scenario ID Uniqueness

Within a single intent.yaml:
- All `constraints[*].id` values must be unique (C1, C2, C3…)
- All `failure_conditions[*].id` values must be unique (F1, F2…)
- All `scenarios[*].id` values must be unique (S1, S2…)

No cross-file constraint ID uniqueness requirement — IDs are scoped per intent.yaml.

### 6d. Cross-Reference Integrity

**Skill path references in plays:**
- Plays reference skills by name in their compiled SKILL.md bodies (e.g., `sync-claude`, `quality-check`).
- The linter should verify that any skill name referenced in a play's SKILL.md corresponds to a directory
  under `core/components/skills/{skill-name}/SKILL.md`.

**Agent references in plays:**
- Plays declare agent boundaries in a table (Agent | Type | Domain | Phases). Agents listed must exist
  at `core/components/agents/{agent-name}.md`.
- Pattern to match: the Agent Boundaries table in play SKILL.md:
  ```
  | `{agent-name}` | domain | ... |
  ```

**Template references in skills:**
- Skills reference templates with paths like:
  `~/.garura/core/memory/standards/templates/{template-name}.md`
  The linter checks these at the deployed path (global mode).

**Intent.yaml ↔ SKILL.md scenario/constraint coverage:**
- Scenario IDs in intent.yaml (`S1`, `S2`, ...) should correspond to `SCE-{N}` eval entries in SKILL.md.
- Constraint IDs in intent.yaml (`C1`, `C2`, ...) should be referenced in SKILL.md step descriptions.

### 6e. YAML Validity

- All `intent.yaml` files must be parseable YAML (js-yaml or similar).
- Frontmatter blocks in SKILL.md files must be parseable as YAML after stripping the `---` delimiters.

### 6f. How Plays Reference Agents (Task Tool Contract Pattern)

Plays dispatch agents via JSON contract embedded in the SKILL.md. Pattern observed in enhance:

```json
{
  "intent_path": "core/components/plays/{play}/reference/intent.yaml",
  "stm_base": "{stm_base}",
  "stm": {
    "input": { ... },
    "output": { ... }
  },
  "task_id": "{task-id}"
}
```

The `subagent_type` field is not present in the embedded JSON — it is set by the orchestrator at
runtime. Cross-reference: the agent name in the Boundaries table must match a `core/components/agents/`
file, not the JSON contract structure.

## 7. Scope Classification

Per discovery Q&A:
- **Target:** `core/components/**` only (skills, plays, agents, intent.yaml files)
- **Output form:** Deterministic Node.js script + `lint-components` skill wrapper
- **No changes to `create-play` pipeline** — linter is standalone
- **Severity mapping:** Agentic — `lint-components` skill reads quality profile to classify violations

Files the implementation will create:
1. `tools/lint-components/` — Node.js tool (new directory + package.json)
2. `core/components/skills/lint-components/SKILL.md` — skill wrapper

Files likely affected:
- `.github/workflows/release.yml` — optional wiring point for CI (not required per Q&A)
- `core/components/plays/enhance/SKILL.md` — may reference `lint-components` in quality-check step once wired

## 8. Observed Inconsistencies (Linter-Relevant)

| Inconsistency | Details |
|---------------|---------|
| `user-invocable` vs `user-invokable` in plays | 9 plays use one spelling, 14 use the other — linter should normalize + warn |
| Section naming variability in skills | `## Constraints` vs `## Boundaries` — both appear; linter should accept both as equivalent |
| `## Version` presence | Present in manage-issue, absent in most others — should be a warning, not an error |
| `allowed-tools` format | Skills use comma string; agents use YAML list — type-specific validation required |
