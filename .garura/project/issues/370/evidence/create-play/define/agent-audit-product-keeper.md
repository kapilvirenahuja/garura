# Agent Audit: product-keeper

**Context:** create-play rebake Step 4 — regression audit for issue #370.
**#370 change:** define's `intent.yaml` adds C17 (Decision Surfacing Discipline at the Phase 5 and Phase 9 human-review gates) and F11 (carrying a `surfaced_for_review` manifest entry past a gate without `user_response` is a structural violation). Both Phase 5 and Phase 9 are **play-owned** (orchestrator-owned) surfaces. No `product-keeper` definition change is part of #370. This audit asks: does the unchanged `product-keeper` still pass P1–P11 as `define` uses it?

**How define uses product-keeper:** Phases 2 (catalog match), 3 (constraint fit), 4 (KB/LTM grounding, conditional), 6 (update product spec via `manage-features`), 7 (epic generate + validate). It produces the three decision manifests (`decision-manifest-catalog-match.yaml` in Phase 2, `decision-manifest-manage-features.yaml` in Phase 6, `decision-manifest-generate-intent-epics.yaml` in Phase 7) that C17 consumes — but the surfacing of those manifests at the gates is orchestrator work, not agent work.

| Principle | Status | Finding |
|-----------|--------|---------|
| P1 JSON Contract | PASS | "JSON Contract Mode" section; "Intent Recognition" parses the play's JSON contract into an Intent object; key inputs/outputs and structured-failure JSON documented. define dispatches it with full JSON contracts (Phases 2,3,4,6,7). |
| P2 STM Path Handoff | PASS | Reads `stm.input` named paths, writes `stm.output` named paths; "NEVER … Write evidence/checkpoint/status files directly"; KB read protocol resolves paths from `ltm_context`, never inline data. define passes file paths only. |
| P3 Intent Awareness | PASS | "ALWAYS — Read intent.yaml from the contract first; let its constraints and failure conditions guide skill invocation"; Core Principle treats constraints as first-class inputs; agent self-selects relevant constraints. define passes `intent_path` in every Phase-2..7 contract. |
| P4 Structured Failure | PASS | Intent Recognition returns a structured `{status:failed, error:{...}}` JSON; "Escalation (Outside Domain) — return the JSON contract with status:failed and a structured error per ADR 016"; "Do NOT return raw errors." |
| P5 No Direct User Interaction | PASS | No `AskUserQuestion` anywhere. Escalation table routes user interaction to "Calling play interactive Q&A" / "Calling play cycle-back". define owns all user-facing phases (1,5,8,9). |
| P6 Output Contract Discipline | PASS | "ALWAYS — Return the enriched JSON contract — never raw skill output"; "Silent validation — put a 1-sentence summary in `notes`, not prose." |
| P7 Skill Delegation | FAIL (pre-existing-unrelated) | Agent delegates artifact production to skills ("Use the Skill tool for every skill invocation; never inline their logic") and has a rich skill table. BUT: define Phase 4 prose instructs the agent to "invoke `research-domain-context` skill" when KB coverage is thin (SKILL.md Phase 4 step 2; intent.yaml lines ~112, ~441), and `research-domain-context` is NOT listed in product-keeper's "Available Skills" table nor its Intent→Skill Mapping (grep confirmed: zero matches in product-keeper.md). The agent is asked to invoke a skill it does not declare. This is a real declared-inventory gap. **Classification: pre-existing-unrelated.** Phase 4 and its `research-domain-context` reference predate #370 (intent lines ~112/~441 are in the long-standing Phase-4 grounding block, not in C17/F11). C17/F11 add no new skill requirement to product-keeper — they govern orchestrator gate behaviour, not the agent's skill table. The #370 change neither introduces nor worsens this gap. |
| P8 Recovery and Escalation | PASS | "Recovery — Self-Recovery (Within Domain)" table with "Attempt one fix … then escalate if still failing"; "Escalation (Outside Domain)" returns failed contract with responsible_domain/suggested_agent. Bounded retry then escalate. |
| P9 Domain Boundaries | PASS | Domain is product-capability config + epic gen + quality profile. "NEVER — Touch design or arch artifacts directly. Those pipelines own their own outputs." define keeps issue ops with project-orchestrator and commits with repo-orchestrator — no boundary crossing. |
| P10 Task Graph Participation | FAIL (pre-existing-unrelated) | product-keeper has NO Task Graph / TaskUpdate section. It only echoes `task_id` in its failure JSON ("task_id": "<echoed>") and lists `task_id` as a key input. There is no documented "TaskUpdate task_id → in_progress / completed / failed" behaviour and no TaskCreate-on-discovery clause — unlike project-orchestrator and repo-orchestrator, which both have explicit "Task Graph" sections. define's Task DAG assigns T3,T4,T5,T7,T8 to product-keeper and the play itself drives the TaskUpdate transitions ("TaskUpdate T3 → in_progress → completed" etc. are written at play level in each phase). So task status moves, but the agent definition does not document its own participation. **Classification: pre-existing-unrelated.** This absence predates #370 and is independent of C17/F11; the surfacing change adds no task-graph obligation to the agent. Note: define compensates by driving TaskUpdate at the play layer, so runtime behaviour is not broken — this is a definition-completeness gap, not a runtime regression. |
| P11 Context Sufficiency | PASS | Has `WebSearch`/`WebFetch` tools, a "KB Reading Protocol" with selective LTM load rules resolving paths from `ltm_context.core_base`, and a research fallback (`research-domain-context`, referenced in define Phase 4). Domain requires KB/domain knowledge discovery; agent has the protocol and tools. Meets P11's documented-context-loading + research-fallback bar. (The P7 gap is that `research-domain-context` is not in the skill table — but the research-fallback capability for P11 purposes is present in tooling and protocol.) |

## Overall: FAIL

Two non-PASS principles (P7, P10), both classified **pre-existing-unrelated**. Neither is introduced or worsened by the #370 C17/F11 surfacing change. The #370 change is orchestrator-only (Phase 5 / Phase 9 are play-owned); it places no new field, tool, or section requirement on product-keeper.

### Regression verdict for #370

As used by define after the #370 change, product-keeper's contract behaviour relevant to C17/F11 is unchanged and adequate: it still produces the three decision manifests in Phases 2/6/7 to STM (P2 PASS), returns enriched contracts (P6 PASS), and the C17 surfacing of those manifests is performed by the orchestrator at the gates — outside this agent's boundary. No #370-introduced or #370-worsened defect. The two FAILs are standing definition-completeness gaps that the rebake does not need to fix and the #370 fix does not aggravate.
