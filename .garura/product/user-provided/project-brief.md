# Garura — Project Brief

**Vision:** Codifying Software Engineering

## What Garura Is

Garura is a DX platform that codifies software engineering. Rather than leaving engineering to ad-hoc AI use, it captures methodology, observability, governance, and intake as first-class, composable capabilities that climb an autonomy ladder over time.

## Core Capability Pillars

### 1. Agentic Methodology
Codifies *how* work is done. Progresses along a maturity ladder:
**Spec-Driven (L2) → Intent-Driven (L3) → Signal-Driven (L4) → Goal-Driven (L5)**.
Each level raises AI autonomy while preserving human checkpoints.

### 2. Control Tower
Cross-project observability. Makes visible and actionable:
- Quality (coverage, lint, build, security metrics + trends)
- Tech / domain / spec debt
- Knowledge base health (cross-project LTM)
- Project health (org-wide portfolio status)
- DxOps / security scanning
- AI-adoption maturity tracking

### 3. Governance
Enterprise controls over AI-assisted work:
- Token usage tracking (AI consumption)
- Memory / LTM curation and administration
- Usage tracking (who used what, when, how)
- CI/CD governance and pipeline enforcement
- Audits — AI vs human traceability, compliance reporting
- Learning pathways for team transformation

### 4. Work Intelligence
Knowledge-driven intake and prioritization:
- Intelligent triage (AI-driven issue routing)
- Agentic PI planning + sprint planning
- Backlog lifecycle (epics → features → stories → bugs → grooming)
- Test-case generation from acceptance criteria
- Tech-debt management, risk management, data governance
- Tooling integrations: repos, PM, CI, quality, hosting, testing, monitoring

### 5. Engineering Experience
The role-aware surfaces every adopter role touches — designers, product managers, engineers, engineering leads, platform engineers, compliance officers, and solo founders. The same platform, progressively revealed: a solo founder sees a lean CLI and a three-question wizard; a squad lead sees triage and sprint views; an enterprise admin sees the Control Tower and governance console. Surfaces include:
- IDE integrations (Claude Code, Copilot, Cursor, Codex CLI, Factory.ai)
- Engineering Portal (web UI for project creation, chat, dashboards, role-aware views)
- AI chat interface (per-project, role-aware response framing)
- Project creation wizard (greenfield init + brownfield onboarding)
- CLI / slash commands
- Design surface (Figma → design-to-code)
- Installers (npm, Claude Code, Copilot) and onboarding (bootcamp, demos)

## Cross-Cutting Dimensions

### Maturity Model (L1–L5)
Every capability climbs this ladder independently:
- L1 — Assisted / Vibe Coding (commoditized)
- L2 — Spec-Driven (LIVE today)
- L3 — Intent-Driven (building)
- L4 — Signal-Driven / Human-on-the-Loop
- L5 — Goal-Driven / Dark Factory

### Adoption Methodology
Organizational journey: **Prototype (2–3 wk) → Pilot (2–3 mo) → Scale (9–12 mo)**.

## Audience

Touches all software engineering roles — designers, engineers, product managers, platform engineers, engineering leads, solo founders, compliance officers. Feature set expands with the adopter segment: solo → small squad → enterprise.

## Delivery Horizon (2026)

- **Apr 2026** — MVP for solo-founders (greenfield projects)
- **May 2026** — Small-squads ready; brownfield project support added
- **Jul 2026** — Enterprise-ready (governance pillar hardened, Control Tower cross-project surfaces live)

## Vision Outcome

An organization adopting Garura stops treating AI as an assistant and starts treating engineering itself as a codified, observable, governed, continuously-improving system. Every capability is auditable, every decision traceable, every team climbs the autonomy ladder at its own pace.
